# 20장 무료 티어 운영과 한도 관리

이 책은 서문에서부터 월 0원 운영을 약속했다.
16장에서 Vercel을, 17장과 18장에서 Supabase를 고른 이유에도 무료 티어가 빠짐없이 등장했다.
그런데 무료 티어에 앱을 올린다는 것은 공짜로 얻는다는 뜻이 아니라, "이 한도 안에서 살겠다"는 계약을 맺는다는 뜻이다.

계약서를 읽지 않으면 두 방향으로 손해를 본다.
한도를 모르고 쓰다가 어느 날 서비스가 멈추는 것이 한쪽이고, 여유가 충분한데 미리 겁먹고 유료로 넘어가는 것이 다른 쪽이다.

이 장은 그 계약서를 읽는다.
Vercel과 Supabase의 한도를 한자리에 놓고, Flashcard의 규모를 대입해 무엇이 **가장 먼저** 터지는지 계산한다.
결론부터 말하면 용량도 트래픽도 아니다.
며칠 앱을 열지 않았을 때 찾아오는 Supabase의 일시정지다.

::: info [용어 풀이] 한도(Quota)와 스로틀링(Throttling)
한도는 정해진 기간에 쓸 수 있는 사용량의 상한이다.
스로틀링은 그 상한에 닿았을 때 서비스가 요청을 느리게 하거나 거절하는 조치를 말한다.
무료 플랜은 대개 초과분을 요금으로 받지 않고 기능을 잠그는 쪽을 택하므로, 한도는 "돈"이 아니라 "정지"로 청구된다.
:::

한 가지 미리 못 박아 둘 것이 있다.
이 장의 수치는 2026년 7월 기준이고, 무료 티어의 한도는 서비스 정책에 따라 언제든 바뀐다.
Vercel은 함수 과금 지표를 여러 번 개편했고 Supabase의 일시정지 기준도 조정된 적이 있다.
수치 자체보다 "어떤 종류의 한도가 있고 무엇을 먼저 확인해야 하는가"라는 구조를 가져가고, 실제 숫자는 그때그때 요금 페이지에서 확인하기 바란다.

## 두 서비스의 한도를 한자리에

### Vercel Hobby가 주는 것

16장에서 봤듯 Hobby 플랜은 Go 서버리스 함수와 CDN, 자동 HTTPS, GitHub 연동 배포를 결제 수단 없이 제공한다.
포함 범위보다 중요한 것은 한도다.

### Vercel Hobby의 한도

2026년 7월 시점의 Vercel 공식 문서와 요금 페이지 기준으로, Flashcard 같은 구성에 유의미한 한도는 다음과 같다.

| 항목 | Hobby 한도 |
|---|---|
| 대역폭(Fast Data Transfer) | 월 100GB |
| 엣지 요청(Edge Requests) | 월 100만 회 |
| 함수 호출(Function Invocations) | 월 100만 회 |
| 함수 활성 CPU(Active CPU) | 월 4시간 |
| 함수 프로비저닝 메모리(Provisioned Memory) | 월 360GB-시간 |
| 함수 실행 시간 | 요청당 최대 300초(5분) |
| 배포 횟수 | 하루 100회 |
| 빌드 | 1회당 최대 45분(2 vCPU, 8GB 메모리 컨테이너) |

::: info [용어 풀이] 대역폭(Bandwidth)과 전송량(egress)
서버에서 사용자 쪽으로 나간 데이터의 총량이다.
Vercel은 대역폭, Supabase는 전송량(egress)이라 부르지만 세는 것은 같다.
수도 요금이 물탱크 크기가 아니라 흘려보낸 물의 양으로 매겨지듯, 이 한도는 쌓아 둔 용량이 아니라 한 달 동안 흘러 나간 양을 잰다.
그래서 뒤에 나올 데이터베이스 500MB 같은 저장 용량 한도와는 성격이 다른 숫자다.
:::

엣지 요청과 함수 호출은 한도 수치가 같지만 세는 대상이 다르다.
엣지 요청은 Vercel 앞단의 서버에 도착한 요청을 전부 세고, 함수 호출은 그중 Go 함수까지 내려가 실제로 실행된 것만 센다.
캐시나 정적 파일로 앞단에서 끝나는 요청은 엣지 요청에만 잡히는데, 이 앱은 모든 경로를 함수로 보내므로(16장) 두 숫자가 거의 같이 움직인다.

활성 CPU라는 지표가 낯설 수 있는데, 함수가 켜져 있던 시간이 아니라 실제로 계산을 수행한 시간만 세는 값이다.
Flashcard의 API처럼 실행 시간 대부분이 데이터베이스 응답 대기인 워크로드는 그 대기 시간이 이 지표에 잡히지 않으므로, 요청에 걸린 전체 시간의 일부만 소모한다.

프로비저닝 메모리의 GB-시간은 함수 인스턴스에 할당된 메모리 크기에 인스턴스가 살아 있던 시간을 곱한 값이다.
1GB 메모리를 받은 인스턴스가 한 시간 떠 있으면 1GB-시간을 쓰므로, 360GB-시간은 1GB 인스턴스 기준 360시간(보름) 분량이다.
한 달(744시간)에 못 미치는 수치지만, 요청 처리가 모두 끝난 인스턴스는 멈추고 그동안은 소모도 멈추므로 실제로는 요청이 걸려 있던 시간만 쌓인다.
활성 CPU와 달리 이 지표는 데이터베이스 응답을 기다리는 동안에도 쌓인다.
계산은 멈춰도 메모리는 계속 붙들고 있기 때문이다.

주의할 점은 이 수치들이 고정 불변이 아니라는 것이다.
Vercel은 함수 과금 지표를 개편하는 등 요금 체계를 여러 번 바꿔 왔다.
월간 빌드 시간 총량처럼 이 표에 없는 항목이나 최신 수치는 요금 페이지(vercel.com/pricing)와 공식 문서에서 직접 확인하기 바란다.

### Vercel: 한도를 넘으면 차단이지 과금이 아니다

Hobby 플랜에는 청구서가 없다.
결제 수단을 등록하지 않으므로 초과 사용이 요금으로 전환될 경로 자체가 없고, 한도를 넘으면 해당 기능이 일시 중지되어 대개 30일이 지나야 다시 쓸 수 있다.

"자고 일어났더니 요금 폭탄"이 구조적으로 불가능하다는 점은 개인 개발자에게 결정적인 안전장치다.
반대로 서비스가 정말 성장했다면 어느 날 앱이 멈춘다는 뜻이기도 하므로, 사용량 대시보드가 한도의 절반을 넘기 시작하면 그때가 유료 전환을 검토할 시점이다.

### Vercel: 약관이 수치보다 먼저 걸린다

수치보다 먼저 걸리는 제약은 약관이다.
Vercel의 공정 사용 정책(fair use guidelines)은 Hobby 플랜을 개인적·비상업적 용도로 제한한다.
광고를 붙이거나, 유료 기능을 팔거나, 회사 서비스의 일부로 쓰는 순간 이 조건을 벗어난다.

판단 기준은 단순하게 잡는 것이 좋다.
이 앱이 돈을 벌기 시작하는가.
수익화하는 시점이 곧 Pro 플랜(사용자당 월 20달러) 전환 시점이고, 그 시점이라면 월 20달러를 감당할 근거도 이미 생긴 것이다.
학습용·개인용인 Flashcard는 이 제약과 무관하지만, 이 책의 구성을 상업 프로젝트에 가져다 쓸 독자라면 처음부터 Pro 기준으로 계산해야 한다.

### Supabase Free의 한도

2026년 7월 기준, Supabase 요금 페이지(supabase.com/pricing)가 밝히는 무료 플랜의 주요 한도는 다음과 같다.

| 항목 | 무료 플랜 한도 (2026년 7월 기준) |
|---|---|
| 관리형 PostgreSQL | 데이터베이스 500MB (공유 CPU, 메모리 500MB) |
| Auth | 월간 활성 사용자(Monthly Active Users, MAU) 5만 명 |
| 파일 스토리지 | 1GB |
| 전송량(egress) | 5GB (CDN 캐시 전송량 5GB 별도) |
| 활성 프로젝트 수 | 2개 |

Flashcard가 쓰는 것은 이 중 PostgreSQL과 Auth뿐이지만, 스토리지와 전송량 한도도 알아 두면 나중에 파일 업로드 기능을 붙일지 판단할 때 도움이 된다.
Edge Functions 호출 횟수처럼 이 표에 없는 항목의 한도나 최신 수치는 요금 페이지에서 직접 확인하자.
무료 티어의 한도는 서비스 정책에 따라 언제든 조정될 수 있는 값이다.

활성 프로젝트 2개라는 한도는 뜻밖의 곳에서 걸린다.
18장에서 구성한 대로 운영용과 개발용 프로젝트를 분리하면 Flashcard 하나만으로 이 한도가 찬다.
다만 세는 것은 활성 프로젝트뿐이라, 일시정지된 프로젝트는 몇 개든 수에 들지 않는다.
다음 앱을 시작할 때는 안 쓰는 프로젝트를 일시정지해 자리를 비우거나, 개발 환경을 Supabase CLI의 로컬 스택으로 대신하면 된다.

## 무엇이 가장 먼저 터지는가

두 서비스의 한도를 나란히 놓고 Flashcard의 실사용량을 대입해 보자.

### Flashcard 규모를 대입해 보면

혼자 쓰는 암기 앱에서 하루 학습 세션 몇 번, 화면과 API를 합친 함수 호출을 넉넉히 하루 100회로 잡아도 월 3,000회다.
함수 하나가 페이지도 정적 자산도 전부 서빙하는 구조라 모든 요청이 함수 호출로 잡히지만, 그래도 한도 100만 회의 0.3%에 불과하고, 요청당 활성 CPU를 후하게 10ms로 쳐도 월 30초로 4시간 한도의 0.2% 수준이다.
대역폭도 첫 방문에 CSS와 htmx 스크립트 수십 KB, 이후는 캐시와 수 KB짜리 HTML 응답뿐이라 월 100GB에 흔적도 내기 어렵다.
매일 쓰는 사용자가 100명으로 늘어도 자릿수가 두 개 커질 뿐 여전히 한도 안이며, 그 전에 비상업 조건이나 Supabase 쪽 한도가 먼저 신호를 보낸다.

정리하면 순서가 분명하다.
용량과 호출 수는 자릿수가 두세 개 커져도 여유가 있고, 수익화를 시작하면 Vercel의 비상업 약관이 수치보다 먼저 걸린다.
그런데 그 어느 것보다 먼저, 그리고 훨씬 자주 마주치는 것이 Supabase의 일시정지다.
사용자가 늘어서 생기는 문제가 아니라 **앱을 며칠 안 써서** 생기는 문제이므로, 성장과 무관하게 개인 앱이라면 누구나 만난다.

그래서 무료 티어 운영의 공은 한도 감시가 아니라 방치 방지에 들여야 한다.
한도 대시보드를 매달 들여다봐야 앱이 살아 있는 것이 아니라, 일시정지를 막는 장치 하나를 걸어 두어야 앱이 살아 있다.

### 가장 먼저 터지는 것: 미사용 프로젝트 일시정지

무료 운영에서 용량 한도보다 먼저 부딪히는 것은 일시정지(pause) 정책이다.
2026년 7월 기준, 무료 프로젝트는 7일 동안 활동이 적으면 자동으로 일시정지된다.
공식 문서 기준으로 지난 한 주간 하루 몇 건 수준의 데이터베이스 요청이 있으면 활성으로 간주되며, 일시정지 약 일주일 전에 경고 이메일이 온다.

일시정지되면 데이터와 설정은 보존되지만 프로젝트 전체가 응답을 멈춘다.
DB 연결도, Auth 로그인도, 그 위에 올라탄 API 호출도 전부 실패하므로 사용자 눈에는 앱이 죽은 것과 다름없다.
재개하려면 Supabase 대시보드에서 해당 프로젝트를 열고 복원(restore) 버튼을 눌러야 하는데, 자동으로 되살아나지 않고 복원에도 몇 분이 걸린다.
더 중요한 것은 기한이다.
일시정지 후 90일 안에만 대시보드에서 복원할 수 있고, 그 기한이 지나면 백업 파일을 내려받아 새 프로젝트로 옮기는 길만 남는다.

피하는 방법은 단순하다.
개발자 자신이 매일 쓰는 앱이라면 그 사용 자체가 활동이므로 아무것도 할 필요가 없다.
문제는 며칠씩 손을 놓는 시기인데, 이때는 주기적인 핑(ping)으로 활동을 만들어 주면 된다.
15장 끝에 예고해 둔 GitHub Actions의 스케줄 워크플로나 UptimeRobot 같은 무료 모니터링 서비스로, 로그인 없이 DB를 조회하는 공개 엔드포인트(Flashcard라면 `/api/shared-decks`)를 하루 몇 번 호출하는 식이다.
이때 핑 대상이 실제로 DB 쿼리를 일으켜야 한다는 점이 핵심이다.
DB를 건드리지 않는 로그인 화면이나 헬스 체크(health check)만 두드리면 Vercel 함수는 깨어나도 Supabase 기준으로는 여전히 무활동이다.
활동 여부를 세는 쪽은 Vercel이 아니라 Supabase의 데이터베이스이기 때문이다.
근본적인 해결책은 유료 전환뿐으로, 유료 플랜의 프로젝트는 일시정지 대상이 아니다.

::: info [용어 풀이] 핑(Ping)과 헬스 체크(Health Check)
핑은 무언가가 살아 있는지 확인하려고 밖에서 일정한 간격으로 보내는 짧은 요청이다.
헬스 체크는 그 물음에 서버가 "살아 있다"라고만 답하도록 만들어 둔 전용 주소인데, 빨리 답하려고 데이터베이스에는 아예 닿지 않게 만드는 것이 보통이다.
문을 두드려 인기척을 확인하는 일이 핑이고, 문 앞에 내건 "영업 중" 팻말이 헬스 체크다.
그래서 핑을 걸 때는 살아 있게 하려는 대상이 무엇인지에 따라 두드릴 주소를 골라야 한다.
:::

이 일시정지를 가장 먼저 겪는 것은 18장에서 만든 개발 프로젝트 쪽이다.
일상 개발이 로컬 SQLite에서 이루어지는 구성이라 개발 DB는 원래 뜸하게 쓰이기 때문이다.
개발 프로젝트가 잠드는 것은 사고가 아니므로 핑까지 걸어 줄 필요는 없고, 다시 필요해질 때 대시보드에서 깨우면 그만이며 운영 프로젝트에는 아무 영향이 없다.
90일 기한을 넘겨 복원 길이 막히더라도 개발 DB는 마이그레이션을 다시 실행하면 그만인 데이터라 잃을 것이 없다.
지켜야 할 것은 사용자가 바라보는 운영 프로젝트 하나다.

### 백업은 무료가 아니다

두 번째 함정은 백업이다.
2026년 7월 기준 무료 플랜에는 자동 백업이 포함되지 않고, Pro 플랜(월 25달러)부터 7일 보관 일일 백업이 제공된다.
실수로 `DELETE`를 잘못 날리거나 마이그레이션이 데이터를 망가뜨렸을 때, 무료 플랜에는 돌아갈 시점 자체가 없다는 뜻이다.

::: info [용어 풀이] 백업과 덤프(pg_dump)
백업은 데이터가 사라지거나 망가졌을 때 돌아갈 수 있도록 미리 떠 두는 사본이다.
덤프(dump)는 데이터베이스의 스키마와 데이터를 통째로 쏟아 내 하나의 파일로 만든 것이고, `pg_dump`는 PostgreSQL이 그 파일을 만들어 주는 명령이다.
덤프 파일 안에 든 것은 데이터베이스의 사진이 아니라 같은 상태를 다시 지어 내는 SQL 문장의 나열이다.
그래서 다른 서버에 그대로 부어 넣어 복원할 수 있고, 데이터베이스를 옮길 때 쓰는 이사 도구이기도 하다.
:::

1인 개발자의 현실적인 대안은 `pg_dump`다.
서버를 설치할 필요는 없고 클라이언트 도구만 따로 받으면 된다(macOS는 `brew install libpq`, 우분투는 `sudo apt install postgresql-client`).
17장에서 언급했듯 Supabase의 DB는 표준 PostgreSQL이므로, 18장에서 본 직접 연결 문자열을 넣어 `pg_dump '<직접 연결 문자열>' > backup.sql`을 실행하면 `backup.sql` 파일 하나가 떨어진다.
큰 변경 전에 수동으로 한 번 뜨는 습관만으로도 최악의 사고는 막을 수 있고, 핑과 같은 방식의 스케줄 워크플로로 자동화할 수도 있다.
다만 사용자 데이터가 담긴 덤프 파일을 공개 저장소에 올리는 실수는 백업 부재보다 큰 사고이니, 저장 위치는 내 컴퓨터나 비공개 저장소처럼 남이 볼 수 없는 곳으로 정하자.
이 백업 습관은 앞서 본 90일 복원 기한과도 이어진다.
외부에 내 손으로 뜬 백업이 있는 한, 플랫폼의 어떤 정책 변경도 데이터의 생사를 좌우하지 못한다.

### 무료의 대가: 감수해야 하는 것들

물론 공짜에는 대가가 있다.

첫째, 콜드스타트를 그대로 맞는다.
유료 플랜의 워밍업 옵션 없이 감수해야 하는데, 이를 Go로 최소화하는 이야기는 16장의 콜드스타트 절에서 다뤘다.

둘째, SLA(Service Level Agreement)가 없다.
가용성 보장도 장애 보상도 없고, 문제가 생겨도 이메일 지원 없이 문서와 커뮤니티로 해결해야 한다.

셋째, 런타임 로그가 1시간만 보존된다.
어젯밤에 난 에러를 아침에 들여다볼 수 없다는 뜻이다.

넷째, 무료 범위는 계약이 아니라 정책이다.
16장에서 봤듯 Fly.io는 무료 티어를 폐지했고, Koyeb는 신규 무료 가입을 막았으며, Render는 무료 데이터베이스의 만료 기한을 90일에서 30일로 줄였다.
Vercel과 Supabase의 무료 한도 역시 언젠가 바뀔 수 있다는 전제로 운영해야 한다.

이 대가 목록이 감수 가능한 동안 무료 티어는 정답이고, Flashcard는 그 전형적인 사례다.

### 언제 유료로 넘어가는가

유료 전환을 고민하게 되는 신호는 대체로 세 가지다.
데이터베이스가 500MB 한도에 다가갈 때, 자동 백업 없이는 불안할 만큼 데이터가 소중해졌을 때, 그리고 일시정지를 피하는 우회책이 번거로워질 만큼 실사용자가 생겼을 때다.
2026년 7월 기준 Pro 플랜은 월 25달러에 데이터베이스 8GB, MAU 10만 명, 7일 백업을 제공하고 일시정지가 없다.
세 신호 중 하나라도 켜졌다면 앱이 취미를 넘어섰다는 뜻이니, 이 시점의 월 25달러는 비용이 아니라 보험이다.

Supabase 쪽에서 Flashcard의 규모를 대입해 보면 여유가 크다.
카드 한 장은 앞뒷면 텍스트 몇백 바이트짜리 행이라, 카드 1만 장을 만들어도 학습 기록까지 합쳐 수십 MB 수준이다.
500MB 한도의 10%에도 미치지 못하는 양이다.
사용자 한 명의 MAU는 한도 5만 명의 0.002%이고, JSON 몇 KB짜리 API 응답으로는 월 5GB 전송량을 소진할 방법이 없다.
결국 이 앱이 무료 플랜에서 마주칠 현실적인 제약은 용량이 아니라, 며칠 앱을 열지 않았을 때 찾아오는 일시정지뿐이다.

## 운영자의 체크리스트

지금까지 본 것을 매달·매분기 반복할 수 있는 형태로 줄여 두자.

첫째, 일시정지 방지 장치를 하나 걸어 둔다.
DB를 실제로 건드리는 공개 엔드포인트를 하루 몇 번 호출하는 스케줄 워크플로면 충분하다.
15장에서 만든 CI 워크플로와 같은 저장소에 두고, `on: schedule`로 돌린다.

둘째, 큰 변경 전에는 `pg_dump`를 뜬다.
마이그레이션을 적용하기 직전이 가장 좋은 시점이고, 앞 절에서 본 `pg_dump '<직접 연결 문자열>' > backup.sql` 한 줄이면 된다.
덤프 파일을 어디에 두는지가 백업 자체보다 중요하다.

셋째, 두 대시보드의 사용량이 한도의 절반을 넘기 시작하면 유료 전환을 계산한다.
절반이라는 기준은 임의로 정한 것이 아니라, 남은 절반이 다음 달의 성장분을 흡수할 여유이기 때문이다.

넷째, 수익화하는 순간 Vercel Pro를 켠다.
수치와 무관하게 약관이 그렇게 정하고 있고, 그 시점이라면 비용을 감당할 근거도 이미 생겼다.

다섯째, 요금 페이지를 반년에 한 번 읽는다.
한도가 바뀌었을 때 그 사실을 알려 주는 곳은 이 장이 아니라 요금 페이지이기 때문이다.

## 에이전트 활용 아이디어

무료 티어 운영의 반복 작업은 대부분 에이전트에게 맡길 수 있는 형태다.

이 장의 체크리스트가 곧 지시문 재료다.
"Vercel과 Supabase 요금 페이지를 확인해 이 장의 한도 표와 달라진 항목을 보고해 달라"는 반년마다의 점검, "일시정지 방지 핑 워크플로가 최근 실행에서 성공했는지 확인해 달라"는 상태 점검이 그 예다.
이런 정기 점검 지시문을 스킬로 저장해 두면 운영이 명령 몇 개로 줄어든다.

핑 워크플로와 pg_dump 백업 같은 자동화 자체를 만드는 일도, 15장에서 본 대로 에이전트에게 맡기기 좋은 과제다.
다만 한도 초과 대응이나 유료 전환처럼 돈과 데이터에 닿는 판단은, 대시보드를 사람이 직접 확인하고 결정한다.

## 정리

첫째, 무료 티어는 공짜가 아니라 한도 안에서 살겠다는 계약이다.
Vercel Hobby는 초과분을 요금이 아니라 기능 정지로 청구하므로 요금 폭탄은 구조적으로 불가능하고, 대신 어느 날 앱이 멈춘다.

둘째, Flashcard 규모에서 용량과 호출 수는 한도의 1%에도 닿지 않는다.
수치를 대입해 보면 걱정할 것은 성장이 아니라 방치다.

셋째, 가장 먼저 터지는 것은 Supabase의 미사용 프로젝트 일시정지다.
DB 쿼리를 일으키는 핑을 스케줄 워크플로로 걸어 두는 것이 가장 값싼 대비책이고, DB를 건드리지 않는 헬스 체크는 아무 소용이 없다.

넷째, 무료 플랜에는 자동 백업이 없다.
직접 연결로 `pg_dump`를 떠 두는 습관이 90일 복원 기한과 정책 변경 양쪽으로부터 데이터를 지킨다.

다섯째, 수익화는 수치보다 먼저 오는 유료 전환 신호다.
Vercel의 공정 사용 정책이 Hobby를 개인·비상업 용도로 제한하기 때문이다.

다음 21장에서는 이 책이 일부러 비워 둔 자리를 하나씩 세어 보며 마무리한다.
