# 14장 Git: 개념과 브랜치 정책

1부가 끝난 자리를 돌아보자.
13장까지 따라왔다면 Echo Flip은 내 컴퓨터에서 완전히 돌아간다.
덱을 만들고, 카드를 뒤집고, 복습 큐가 차오르는 것까지 전부 로컬에서 확인했다.
2부는 이 앱을 세상에 공개하고, 공개한 뒤에도 오래 운영하는 이야기다.

공개를 향한 첫걸음은 배포 버튼이 아니다.
코드의 역사를 기록하고, 잘못된 변경을 되돌리고, 어느 시점의 코드가 사용자에게 나가 있는지 지목할 수 있게 만드는 체계, 곧 버전 관리(version control)다.

::: info [용어 풀이] 버전 관리(Version Control)
파일의 변경 이력을 시점마다 기록해 두고, 언제든 과거 상태를 열람하거나 그 시점으로 되돌릴 수 있게 하는 체계다.
문서의 복사본을 여러 개 만드는 대신, 문서 하나에 "언제 무엇을 왜 바꿨는지"의 목록을 붙여 두는 방식이라고 보면 된다.
소프트웨어 개발에서는 Git이 이 체계의 사실상 표준(de facto standard)이다.
:::

이번 장에서는 Git을 다루되, 명령어 매뉴얼과는 다른 각도로 접근하겠다.
명령어는 11장에서 본 에이전트가 대신 실행해 주므로 외울 필요가 없다.
반면 브랜치를 어떻게 운영할지는 에이전트에게 맡길 수 없는 정책의 문제이고, 이 장의 무게 중심은 거기에 있다.
버전 관리가 왜 필요한지와 Git의 핵심 개념을 짚은 뒤, 브랜치 정책의 후보들을 트레이드오프로 비교하고, Echo Flip의 선택인 main과 release 두 브랜치 구성을 살펴보겠다.

## 버전 관리는 왜 필요한가

버전 관리가 없는 곳에도 버전 관리의 욕구는 있다.
`보고서_최종.docx`, `보고서_최종_수정.docx`, `보고서_진짜최종.docx` 같은 파일명을 만들어 본 적이 있다면 이미 경험한 욕구다.
지금 상태를 잃고 싶지 않아서 복사본을 만들지만, 복사본이 쌓일수록 어느 것이 최신인지, 서로 무엇이 다른지 알 수 없게 된다.

Git은 이 복사본 만들기를 체계로 바꾼 도구다.
운영을 앞둔 앱에 이 체계가 필요한 이유는 셋이다.

첫째, 되돌릴 수 있는 역사가 생긴다.
운영은 실수를 전제로 설계해야 한다.
배포한 변경이 앱을 깨뜨리는 날은 반드시 오고, 그날의 대응은 "마지막으로 멀쩡했던 시점"으로 돌아가는 것에서 시작한다.
버전 관리는 특정 시점의 복사본 하나가 아니라 모든 시점을 보존하므로, 어느 과거로든 돌아갈 수 있다.

둘째, 변경의 이유가 남는다.
코드는 "지금 어떻게 되어 있는가"만 말해 주고, "왜 이렇게 되어 있는가"에는 답하지 못한다.
변경마다 남기는 한 줄의 기록이 그 질문에 답한다.
몇 달 뒤의 나에게, 그리고 코드를 읽는 에이전트에게 이 기록은 코드만큼 중요한 정보가 된다.

셋째, 배포의 단위가 된다.
16장에서 볼 Vercel은 기록된 변경 하나를 받아 배포 하나를 만든다.
"지금 사용자가 쓰는 앱이 어느 시점의 코드인가"라는 질문에 시점 하나로 답할 수 있어야, 문제가 생겼을 때 원인을 좁힐 수 있다.

혼자 만드는 앱이라면 이 셋이 과해 보일 수 있다.
그런데 에이전트와 함께 개발할수록 오히려 반대다.

11장에서 사람의 몫은 검토와 승인이라고 정리했다.
검토는 "무엇이 바뀌었는가"를 보여 주는 장치가 있어야 가능하고, 승인은 잘못 승인했을 때 되돌릴 길이 있어야 부담이 가벼워진다.
에이전트는 사람이 손으로 타이핑하던 시절과 비교할 수 없는 속도로 변경을 쏟아낸다.
그 속도를 받아 내는 안전망이 버전 관리다.
변경을 기록하지 않고 에이전트에게 코드를 맡기는 것은, 지우개 없는 펜을 쥐여 주는 것과 같다.

## Git의 핵심 개념

Git에는 개념이 많지만, 2부 내내 계속 쓸 것은 넷이다.
커밋, 브랜치, 병합, 그리고 원격 저장소다.

### 커밋: 프로젝트 전체의 스냅샷

커밋(commit)은 특정 순간의 프로젝트 전체를 통째로 기록한 스냅샷이다.
바뀐 부분의 목록이 아니라 그 순간의 장면 전체라는 점이 중요하다.
사진첩의 사진 한 장처럼, 커밋 하나를 열면 그 시점의 모든 파일이 그대로 있다.

::: info [용어 풀이] 커밋(Commit)
버전 관리에서 "이 시점을 기록해 둔다"고 확정하는 저장 단위다.
게임의 세이브 포인트처럼, 커밋을 만들어 둔 시점으로는 언제든 되돌아갈 수 있다.
커밋마다 무엇을 왜 바꿨는지 적는 메시지가 함께 저장된다.
:::

커밋에는 메시지가 붙는다.
"덱 공유 기능 추가"처럼, 이 스냅샷이 직전 스냅샷과 무엇이 다르고 왜 바꿨는지를 사람의 말로 남긴다.
앞 절에서 말한 "변경의 이유가 남는 이력"이 바로 이 메시지들의 목록이다.

### 브랜치: 이력에 붙인 움직이는 이름표

커밋은 직전 커밋을 가리키며 사슬처럼 이어진다.
브랜치(branch)는 이 사슬의 맨 끝 커밋에 붙인 이름표다.
새 커밋을 만들면 이름표가 새 커밋으로 따라 움직이므로, 고정된 표지판이 아니라 움직이는 이름표다.

::: info [용어 풀이] 브랜치(Branch)
커밋 이력의 한 갈래를 가리키는 이름표다.
새 커밋이 생기면 이름표가 최신 커밋을 따라 움직이고, 이름표를 여러 개 두면 이력이 여러 갈래로 나뉘어 서로 방해하지 않고 진행된다.
"가지"라는 이름 그대로, 한 줄기에서 갈라져 나와 따로 자라다가 다시 합쳐질 수 있다.
:::

브랜치를 새로 만드는 일은 프로젝트를 복사하는 것이 아니라, 같은 사슬에 이름표를 하나 더 붙이는 것이다.
그래서 만들고 지우는 비용이 0에 가깝다.
두 이름표가 각자 다른 커밋을 가리키기 시작하면 이력이 두 갈래로 갈라지고, 한쪽에서 실험하다 버려도 다른 쪽은 아무 영향을 받지 않는다.

### 병합과 풀 리퀘스트

갈라진 갈래는 언젠가 다시 만나야 한다.
병합(merge)은 한 브랜치에 쌓인 커밋들을 다른 브랜치로 합치는 작업이다.

풀 리퀘스트(pull request)는 "이 브랜치를 저 브랜치로 병합해 달라"는 요청을 하나의 화면으로 만든 GitHub의 기능이다.
바뀐 내용 전체를 훑어보고, 의견을 달고, 자동 검사 결과를 확인한 뒤에 병합 버튼을 누른다.
혼자 개발이라도 이 화면은 값어치가 있다.
병합하기 전에 변경 전체를 한 번은 눈으로 지나가게 만드는 자리이기 때문이다.

::: info [용어 풀이] 병합(Merge)과 풀 리퀘스트(Pull Request)
병합은 갈라진 두 브랜치의 이력을 하나로 합치는 작업이다.
풀 리퀘스트는 "이 브랜치를 병합해 달라"는 요청을 검토 화면으로 만든 GitHub의 기능으로, 병합 전에 변경 내용과 자동 검사 결과를 확인하는 관문이 된다.
GitLab에서는 같은 것을 머지 리퀘스트(merge request)라고 부른다.
:::

### 원격 저장소: GitHub와 push·pull

지금까지의 개념은 모두 내 컴퓨터 안의 이야기다.
Git은 내 컴퓨터의 로컬 저장소와, GitHub 같은 서비스에 있는 원격 저장소(remote repository)의 쌍으로 운영된다.
둘은 같은 이력의 사본이고, push로 로컬의 커밋을 원격에 올리고 pull로 원격의 커밋을 내려받아 맞춘다.

원격 저장소는 백업이면서 그 이상이다.
GitHub는 이력을 보관해 줄 뿐 아니라, 풀 리퀘스트 화면을 제공하고, 코드가 도착할 때마다 검사를 실행해 주고(15장), 배포 플랫폼이 코드를 가져가는 출발점이 된다(16장).
2부에서 만드는 공개와 운영의 체계는 전부 이 원격 저장소를 중심으로 돈다.

무엇을 이력에 넣지 않을지도 이력만큼 중요하다.
13장에서 내 학습 데이터가 담긴 `echo-flip.db`는 저장소에 커밋될 일이 없다고 했는데, 그것을 보장하는 장치가 저장소의 `.gitignore` 파일이다.
여기 적힌 파일은 Git이 없는 것으로 취급하므로, 데이터베이스 비밀번호 같은 비밀은 이력에 실려 원격으로 새어 나가지 않는다.

## 명령어는 에이전트에게, 판단은 사람에게

개념을 알았으니 명령어를 보자.
Git의 일상 명령은 여섯 갈래 정도로 압축된다.

- `git add`: 다음 스냅샷에 담을 변경을 고른다.
- `git commit`: 골라 둔 변경을 스냅샷으로 확정하고 메시지를 남긴다.
- `git branch`: 브랜치를 만들거나 나열한다.
- `git merge`: 다른 브랜치의 이력을 지금 브랜치로 합친다.
- `git push` / `git pull`: 로컬의 커밋을 원격에 올리거나, 원격의 커밋을 내려받는다.
- `git log` / `git diff`: 이력을 읽거나, 두 시점 사이의 차이를 본다.

명령마다 이 목록이 담지 못한 옵션이 수십 개씩 있다.
하지만 이 책의 관점에서는 외울 필요가 없다.
"이 변경을 커밋해 줘"라고 말하면 11장의 에이전트가 add와 commit을 알아서 실행하고 메시지 초안까지 써 준다.
실제로 이 저장소의 커밋 대부분은 에이전트가 명령을 실행하고 사람이 결과를 승인하는 방식으로 만들어졌다.
11장에서 본 권한 설계도 같은 구도다.
커밋처럼 로컬에서 되돌릴 수 있는 명령은 자동 허용해도, `git push`처럼 원격에 영향을 주는 명령은 사람의 승인을 유지한다.

그래서 사람에게 남는 몫은 타이핑이 아니라 두 가지 읽기다.
커밋 전에 diff를 읽고 승인하는 것, 그리고 쌓인 이력을 읽는 것이다.

이력 읽기가 어떤 가치가 있는지, 이 저장소의 실제 이력으로 보자.
`git log --oneline`은 커밋마다 식별자와 메시지를 한 줄로 보여 주는 명령인데, Echo Flip 초기 개발 시기의 출력에서 여섯 줄을 발췌하면 이렇다.

```
b02a70f 첫 API 요청 시 프로필 자동 생성: EnsureProfile 미들웨어 추가
579e09c Go 품질 검증 자동화: gofmt/vet hook + go-quality 서브에이전트(haiku) 추가
543bdb6 api/index.go에 internal 패키지 import 금지 제약 주석 추가
d50c98a Vercel Go 빌드 오류 수정: app 패키지를 internal에서 pkg/app으로 이동
8f992ea 덱 공유 기능 추가: 공유 링크·공유 덱 갤러리·내 덱으로 가져오기
82ad59a echo-flip 초기 구현: 영어 암기 카드 PWA (Next.js 정적 프론트 + Go/Gin Vercel 함수 + Supabase)
```

`git log`는 최신 커밋을 위에 보여 주므로, 시간 순서로 읽으려면 아래에서 위로 읽는다.
초기 구현이 올라갔고, 덱 공유 기능이 붙었고, Vercel 배포가 깨져서 고쳤다.
그 사고에서 배운 제약을 코드 주석으로 남겼고, 다음 커밋에서는 아예 훅과 서브에이전트로 검증을 자동화했다.
12장에서 해부한 그 설정 파일들이 태어난 순간이 이력에 그대로 남아 있다.

이력이 곧 프로젝트의 일지다.
어떤 문제가 언제 생겼고 어떻게 대응했는지를, 별도의 문서 없이 커밋 메시지의 나열만으로 복원할 수 있다.
이 일지의 품질은 메시지의 품질에 달려 있어서, "수정", "임시 저장" 같은 메시지가 쌓이기 시작하면 일지는 사라진다.
에이전트가 메시지 초안을 쓰는 개발이라면, 메시지 규칙을 11장의 프로젝트 지침에 적어 두는 것이 일지를 지키는 요령이다.

## 브랜치 정책은 사람이 정하는 정책이다

여기까지는 에이전트에게 맡겨도 되는 영역이었다.
지금부터는 아니다.

브랜치를 몇 개 둘지, 각 브랜치가 무엇을 의미할지, 어느 브랜치의 코드가 사용자에게 나갈지를 정하는 규칙을 브랜치 정책이라고 한다.
이 결정에는 기계적 정답이 없다.
gofmt 통과 여부는 예/아니오가 명확한 판정이라 12장에서 기계에게 넘겼지만, 브랜치 정책은 팀의 크기, 배포 주기, 실수를 되돌리는 비용을 어떻게 저울질하느냐에 따라 답이 달라지는 가치 판단이다.
에이전트는 어떤 정책이든 충실히 따라 주지만, 무엇을 따를지는 운영하는 사람이 정해서 알려 줘야 한다.

널리 쓰이는 정책 세 갈래를 트레이드오프와 함께 보자.

### 단일 브랜치: GitHub flow

main 브랜치 하나를 중심에 둔다.
변경은 짧게 살다 사라지는 작업 브랜치에서 만들어 풀 리퀘스트로 main에 병합하고, main에 들어간 코드는 곧바로 배포한다.
trunk 기반 개발(trunk-based development)이라고 불리는 흐름도 큰 틀에서 같은 계열이다.

장점은 단순함이다.
기억할 브랜치가 하나뿐이고, 병합 방향을 고민할 일이 없고, 코드가 사용자에게 닿는 길이 가장 짧다.
배포가 잦고 되돌리기 쉬운 웹 서비스라면 이 단순함이 그대로 힘이 되고, 1인 개발의 기본값으로도 가장 무난하다.

대가는 전제 조건에 있다.
main은 언제나 배포 가능해야 한다.
개발 환경과 운영 환경을 구분하지 않으므로, main에 들어간 미완성 변경이나 실험이 그대로 사용자에게 나간다.

### git flow: 층층이 쌓은 브랜치

반대편 끝에 git flow가 있다.
develop 브랜치에서 개발하고, 기능마다 feature 브랜치를 따고, 릴리스가 다가오면 release 브랜치에서 안정화하고, main에는 출시된 버전만 두고, 급한 수정은 hotfix 브랜치로 처리한다.

이 층층 구조가 더 나은 조건은 분명하다.
릴리스 주기가 길고, 여러 버전을 동시에 유지보수해야 하고(설치형 소프트웨어가 전형이다), 여러 사람의 작업을 릴리스 단위로 묶어야 할 때다.

혼자 만드는 웹 앱에는 과하다.
브랜치 사이의 병합 규칙을 관리하는 것 자체가 일이 되고, 웹 앱은 언제나 최신 버전 하나만 서비스하므로 "여러 버전 유지보수"라는 전제부터 해당하지 않는다.

### GitLab flow: 환경마다 브랜치 하나

둘 사이의 절충이 GitLab flow다.
개발은 GitHub flow처럼 main에서 단순하게 진행하되, 배포 환경마다 브랜치를 하나씩 둔다.
production 브랜치에 병합하는 것이 곧 운영 배포이고, 필요하면 그 앞에 staging 같은 중간 환경 브랜치를 끼운다.

이 정책이 더 나은 조건은, 개발 환경과 운영 환경이 나뉘어 있고 배포 시점을 명시적으로 통제하고 싶을 때다.
"main에 들어갔다"와 "사용자에게 나갔다"를 분리하고 싶은데 git flow만큼의 층은 필요 없을 때, 브랜치 하나를 더하는 비용으로 그 분리를 산다.

## Echo Flip의 선택: main과 release

Echo Flip은 GitLab flow를 가장 단순한 형태로 줄여서 쓴다.
브랜치는 둘이다.
main은 기본 브랜치이자 개발 환경이고, release는 운영 환경이다.

::: info [용어 풀이] 기본 브랜치(Default Branch)
저장소를 열었을 때 기준이 되는 브랜치다.
저장소를 복제하면 처음 받게 되는 갈래이고, 풀 리퀘스트를 만들 때 병합 대상의 기본값이 된다.
GitHub에서 만든 저장소의 기본 브랜치 이름은 관례상 main이다.
:::

1부까지는 main 하나로 충분했다.
사용자가 곧 개발자였고 앱은 내 컴퓨터에서만 돌았으므로, "개발 중인 코드"와 "쓰고 있는 코드"가 같아도 아무 문제가 없었다.
단일 브랜치 정책이 1인 개발의 기본값이라고 한 것과 같은 이유다.

운영을 시작하면 사정이 달라진다.
사용자가 쓰는 코드는 안정되어 있어야 하고, 개발 중인 코드는 마음껏 흔들 수 있어야 한다.
이 둘을 한 브랜치에 두면 개발의 자유와 운영의 안정 중 하나를 포기해야 한다.
그래서 둘을 브랜치로 가른다.
브랜치와 환경의 대응은 이렇게 정한다.

| | `main` | `release` |
|---|---|---|
| 역할 | 개발 | 운영 |
| 배포(16장) | Vercel 프리뷰 | Vercel 프로덕션 |
| 데이터베이스(18장) | 개발용 DB | 운영용 DB |
| 코드가 들어오는 길 | 일상 개발의 커밋 | main에서의 병합만 |

이력의 흐름으로 그리면 다음과 같다.

```
main    ──●──●──●──●──●──●──▶   개발 환경 (프리뷰 배포 + 개발용 DB)
               \           \
                \ 병합      \ 병합    ← 사람이 정하는 운영 배포의 순간
                 ▼           ▼
release ─────────●───────────●──▶   운영 환경 (프로덕션 배포 + 운영용 DB)
```

일하는 흐름은 이렇다.
개발은 전부 main에서 한다.
파일을 고칠 때마다 12장의 훅이 지키고, 푸시하면 15장에서 만들 원격 검사가 같은 규칙을 한 번 더 확인한다.
main에 쌓인 변경은 개발 환경, 즉 프리뷰 배포와 개발용 DB 위에서 실제로 굴려 본다.
사용자에게 내보낼 만큼 무르익었다고 판단하면 main을 release로 병합한다.
이 병합이 곧 운영 배포다.
release에 커밋이 도착하면 프로덕션이 새로 배포되고(16장), 그 코드는 운영용 DB를 바라본다(18장).

이 정책에서 "배포한다"는 별도의 행위가 아니라 병합이라는 Git 작업이 된다.
배포의 단위가 곧 이력의 단위라는 첫 절의 이야기가 여기서 완성된다.
그리고 언제 병합할지는 사람이 정한다.
에이전트가 기능을 완성했다고 그 순간 사용자에게 나가는 것이 아니라, 개발 환경에서 확인을 마친 사람이 내보낼 시점을 결정한다.

대가도 정직하게 적어 두자.
단일 브랜치였다면 없었을 병합 단계가 하나 늘었다.
release의 존재를 잊으면 "고쳤는데 왜 사용자 화면은 그대로지"라는 혼란이 생길 수 있고, main과 release가 오래 벌어지면 병합 한 번에 나가는 변경의 덩치가 커져서 문제가 생겼을 때 원인을 좁히기 어려워진다.
그래서 이 정책에는 "확인이 끝난 변경은 미루지 않고 내보낸다"는 운용 규율이 따라붙는다.
병합을 자주 해서 한 번에 나가는 덩치를 작게 유지하는 것이다.

마지막으로, 이렇게 정한 정책은 기록해 둬야 한다.
브랜치 정책은 저장소를 들여다봐도 보이지 않는다.
브랜치 이름은 보여도 "release에는 병합으로만 코드가 들어간다" 같은 규칙은 어디에도 적혀 있지 않기 때문이다.
11장에서 본 프로젝트 지침에 main이 개발, release가 운영, 운영 배포는 main에서 release로의 병합이라고 적어 두면, 에이전트도 다음 세션에서 같은 정책을 따른다.
정책을 정하는 것은 사람의 일이지만, 정해진 정책을 지키게 하는 일은 이렇게 다시 기계와 에이전트에게 넘길 수 있다.

## 정리

첫째, 버전 관리는 되돌릴 수 있는 역사, 변경의 이유가 남는 이력, 배포의 단위를 제공한다.
에이전트가 변경을 쏟아내는 개발일수록, 사람의 검토와 되돌림을 받쳐 주는 이 안전망이 더 필요하다.

둘째, 커밋은 프로젝트 전체의 스냅샷이고, 브랜치는 이력에 붙인 움직이는 이름표이고, 병합은 갈라진 이력을 합치는 작업이다.
로컬 저장소와 GitHub의 원격 저장소는 push와 pull로 맞추는 같은 이력의 쌍이다.

셋째, 명령어는 외울 필요가 없다.
에이전트가 대신 실행해 주기 때문이다.
사람의 몫은 diff와 이력을 읽고 승인하는 것이며, 커밋 메시지의 나열이 곧 프로젝트의 일지가 된다.

넷째, 브랜치 정책은 팀 규모와 배포 주기, 되돌림 비용을 저울질하는 가치 판단이라 에이전트에게 맡길 수 없다.
단일 브랜치는 가장 단순하지만 개발과 운영이 갈리지 않고, git flow는 층이 많은 대신 혼자에게 과하고, GitLab flow는 환경 브랜치로 그 사이를 절충한다.

다섯째, Echo Flip은 main(개발)과 release(운영) 두 브랜치를 둔다.
main에서 release로의 병합이 곧 운영 배포이고, 두 브랜치는 16장의 프리뷰·프로덕션 배포와 18장의 개발·운영 데이터베이스 분리로 이어진다.

이제 브랜치마다 의미가 생겼다.
main에 도착하는 코드는 개발 환경으로 가고, release에 도착하는 코드는 사용자에게 간다.
그렇다면 어느 쪽이든, 도착하는 코드가 정해진 검사를 통과했는지를 저장소 스스로 확인하게 만들어야 한다.
15장에서는 이 브랜치들에 거는 원격 품질 게이트, GitHub Actions를 다룬다.
