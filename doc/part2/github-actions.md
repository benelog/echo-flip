# 16장 GitHub Actions: 원격 품질 게이트

지금까지 Echo Flip의 품질 게이트는 두 곳에 있었다.
13장의 훅이 에이전트가 파일을 쓸 때마다 `gofmt`와 `go vet`을 돌리고, 14장의 `전체 검증` 태스크가 사람이 커밋하기 전에 다섯 검사를 묶어 돌린다.

둘 다 로컬의 게이트다.
그리고 로컬의 게이트에는 공통된 약점이 하나 있다.
통과하지 않고도 푸시할 수 있다는 것이다.

15장에서 정한 브랜치 정책을 떠올리면 이 약점은 더 무거워진다.
main으로 향하는 코드는 개발 환경에 닿고, release로 향하는 코드는 사용자가 쓰는 운영 환경에 닿는다.
어느 쪽이든 검사를 건너뛴 코드가 도착할 수 있다면, 정책은 있는데 지키는 장치가 없는 상태다.

이 장에서는 그 약점을 메우는 세 번째 게이트, 즉 저장소에 코드가 도착하는 순간 원격에서 자동으로 도는 검사를 그 브랜치들 위에 얹는다.
GitHub Actions로 `.github/workflows/ci.yml` 한 파일을 작성해 Go와 프런트엔드를 병렬로 검증하고, 이 게이트가 앞의 두 게이트와 어떻게 같은 명령을 공유하는지 짚는다.

## 로컬 게이트가 새는 곳

혼자 개발하고 커밋 전에 태스크를 꼬박꼬박 돌린다면, 원격 검사가 없다는 사실은 오래 드러나지 않는다.
드러나는 순간은 셋이다.

첫째, 사람이 깜빡할 때다.
급한 수정 하나를 바로 푸시하는 습관은 누구에게나 있고, 훅은 에이전트의 도구 호출에만 걸려 있지 사람의 `git push`에는 걸려 있지 않다.

둘째, 두 번째 사람이 합류할 때다.
풀 리퀘스트로 들어온 코드가 `go vet`을 통과하는지 확인하려면 그 코드를 내 기계로 내려받아 직접 돌려야 한다.
자동 검사가 없으면 리뷰어의 눈이 유일한 게이트가 된다.

셋째, 내 기계에만 있는 조건이 검사를 통과시킬 때다.
로컬에 설치된 오래된 의존성이나 캐시된 테스트 결과가 실패를 가려 주는 경우가 있다.
깨끗한 기계에서 처음부터 받아 돌리는 검사만이 이것을 잡아낸다.

세 번째 이유가 특히 중요하다.
CI는 "사람을 못 믿어서" 두는 장치라기보다, 아무것도 설치되지 않은 상태에서 저장소만으로 앱이 빌드되고 검증되는지를 확인하는 장치다.

::: info [용어 풀이] 지속적 통합(CI, Continuous Integration)
코드를 저장소에 합칠 때마다 빌드와 검사를 자동으로 돌려서, 깨진 코드가 오래 섞여 있지 않게 하는 관행이다.
합치는 주기를 짧게 가져갈수록 문제가 생긴 원인을 찾기 쉬워진다는 발상에서 나왔다.
검사를 대신 돌려 주는 서비스가 여럿 있는데, 저장소가 GitHub에 있다면 별도 가입 없이 쓸 수 있는 GitHub Actions가 가장 마찰이 적다.
:::

## GitHub Actions의 구조

GitHub Actions는 저장소에서 어떤 일(이벤트)이 벌어지면 정해진 절차를 실행해 주는 도구다.
용어가 네 겹으로 쌓여 있어 처음에는 헷갈리는데, 실제로는 단순하다.

워크플로(workflow)는 `.github/workflows/` 아래의 YAML 파일 하나다.
그 안에 잡(job)이 여럿 있고, 잡 하나가 러너(runner)라는 가상 머신 하나를 차지한다.
잡 안에는 스텝(step)이 순서대로 놓인다.

여기서 중요한 성질은 잡끼리는 기본적으로 병렬로 돈다는 것이다.
Echo Flip의 검증은 Go 쪽과 프런트엔드 쪽이 서로 의존하지 않으므로, 두 잡으로 나누면 전체 시간이 둘 중 느린 쪽으로 줄어든다.

## ci.yml 해부

저장소의 `.github/workflows/ci.yml` 전문이다.

```yaml
name: CI

on:
  push:
    branches: [main, release]
  pull_request:
  workflow_dispatch:

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  go:
    name: Go (build, vet, test, gofmt)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true

      - name: gofmt
        run: |
          unformatted=$(gofmt -l .)
          if [ -n "$unformatted" ]; then
            echo "다음 파일이 gofmt를 통과하지 못했습니다:"
            echo "$unformatted"
            exit 1
          fi

      - name: go build
        run: go build ./...

      - name: go vet
        run: go vet ./...

      - name: go test
        run: go test ./...

      - name: Vercel 호환 검사 (api/ 는 internal/ 을 import 할 수 없다)
        run: |
          if grep -rq 'benelog/echo-flip/internal' api/; then
            echo "api/ 가 internal/ 패키지를 import 합니다. pkg/ 로 옮기세요."
            exit 1
          fi

  web:
    name: Web (typecheck, test)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm

      - name: npm ci
        run: npm ci

      - name: tsc --noEmit
        run: npx tsc --noEmit

      - name: vitest
        run: npm test
```

위에서부터 읽어 보자.

### on: 언제 도는가

`push`는 15장에서 정한 두 브랜치, 곧 `main`이나 `release`에 커밋이 도착할 때 워크플로를 깨운다.
`main` 푸시가 개발 환경으로 가는 코드를 검사한다면, `release` 푸시는 운영 배포(17장)가 일어나기 직전의 마지막 관문이다.
개발 때 통과한 코드라도 운영으로 나가는 순간 같은 검사를 한 번 더 거치므로, 병합 과정에서 생긴 문제도 놓치지 않고 잡아낸다.
`pull_request`는 풀 리퀘스트가 열리거나 갱신될 때 워크플로를 깨운다.
`workflow_dispatch`는 GitHub 화면에서 버튼을 눌러 수동으로 돌릴 수 있게 해 준다.
디버깅할 때 이 한 줄이 있고 없고의 차이가 크다.

`push`에만 `branches` 필터를 건 것은 의도적이다.
필터가 없으면 브랜치를 밀어 올리고 곧바로 풀 리퀘스트를 열 때 같은 커밋에 워크플로가 두 번 돈다.

### concurrency: 낡은 실행을 취소한다

```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

같은 브랜치에 커밋을 연달아 밀면, 앞선 커밋의 검사 결과는 이미 쓸모가 없다.
`cancel-in-progress`가 진행 중인 실행을 취소해 러너 시간을 아낀다.
공개 저장소라면 러너 시간이 무료지만, 결과를 기다리는 사람의 시간은 무료가 아니다.

### setup-go: 버전을 저장소에서 읽는다

```yaml
- uses: actions/setup-go@v5
  with:
    go-version-file: go.mod
    cache: true
```

Go 버전을 워크플로에 적어 넣는 대신 `go.mod`에서 읽게 했다.
`go.mod`의 `go 1.26.4` 한 줄이 로컬과 CI의 단일 진실 공급원이 되므로, Go를 올릴 때 고칠 곳이 한 군데다.
`cache: true`는 모듈 다운로드 결과를 캐싱해 두 번째 실행부터 시간을 줄인다.

### gofmt: 고치지 않고 고발한다

이 스텝이 이 워크플로에서 가장 눈여겨볼 곳이다.

```yaml
- name: gofmt
  run: |
    unformatted=$(gofmt -l .)
    if [ -n "$unformatted" ]; then
      echo "다음 파일이 gofmt를 통과하지 못했습니다:"
      echo "$unformatted"
      exit 1
    fi
```

13장의 훅은 `gofmt -w`를 쓴다.
파일을 조용히 고쳐 놓고 넘어간다.
14장의 에디터도 저장하는 순간 같은 일을 한다.

그런데 CI에서 `gofmt -w`를 돌리면 아무 의미가 없다.
러너의 파일을 고쳐 봐야 그 러너는 곧 사라지고, 저장소에는 여전히 어긋난 파일이 남는다.
그래서 CI는 `-l`을 쓴다.
`gofmt -l`은 포맷이 어긋난 파일의 이름만 출력하고 고치지는 않으므로, 출력이 비어 있지 않으면 `exit 1`로 워크플로를 실패시킨다.

같은 도구, 같은 규칙인데 동사가 다르다.
로컬에서는 고치고, 원격에서는 고발한다.
이것이 세 게이트가 하나의 규칙을 공유하는 방식이다.

### Vercel 호환 검사: 서브에이전트가 하던 일을 기계로

```yaml
- name: Vercel 호환 검사 (api/ 는 internal/ 을 import 할 수 없다)
  run: |
    if grep -rq 'benelog/echo-flip/internal' api/; then
      echo "api/ 가 internal/ 패키지를 import 합니다. pkg/ 로 옮기세요."
      exit 1
    fi
```

17장에서 자세히 볼 제약을 미리 검사하는 스텝이다.
Vercel의 Go 런타임은 `api/index.go`를 모듈 바깥에서 컴파일하므로, 이 파일이 `internal/` 패키지를 import하면 로컬 `go build`는 통과하고 배포만 실패한다.
로컬에서는 절대 재현되지 않는 종류의 실패라, 사람의 주의력에 맡기기에 가장 나쁜 조건을 갖췄다.

13장의 `go-quality` 서브에이전트가 검사 항목으로 들고 있던 것이 바로 이 규칙이다.
에이전트에게 "확인해 달라"고 부탁하는 것과 `grep` 한 줄로 못 박는 것은 신뢰도가 다르다.
기계적으로 판정할 수 있는 규칙을 발견하면 곧바로 기계에게 넘기는 편이 낫다.

### web 잡: 프런트엔드

`npm ci`는 `npm install`과 다르다.
`package-lock.json`에 적힌 버전을 정확히 그대로 설치하고, 잠금 파일과 `package.json`이 어긋나면 실패한다.
CI에서 원하는 것이 정확히 이 동작이다.

그다음 `npx tsc --noEmit`이 9장에서 본 strict 모드 타입 검사를, `npm test`가 vitest를 돌린다.
두 명령 모두 14장의 `tasks.json`에 적힌 것과 글자 하나 다르지 않다.

## 같은 명령을 세 곳에서 부른다

세 게이트가 무엇을 어떻게 부르는지 한자리에 놓아 보자.

| 검사 | 13장 훅 (에이전트) | 14장 태스크 (사람) | 16장 CI (원격) |
|---|---|---|---|
| 포맷 | `gofmt -w` (자동 수정) | 저장 시 자동 수정 | `gofmt -l` (실패 처리) |
| 정적 분석 | `go vet` (수정 파일의 패키지) | `go vet ./...` | `go vet ./...` |
| 빌드 | 없음 | `go build ./...` | `go build ./...` |
| 테스트 | 없음 (서브에이전트가 담당) | `go test ./...` | `go test ./...` |
| 타입 검사 | 없음 | `npx tsc --noEmit` | `npx tsc --noEmit` |
| 프런트 테스트 | 없음 | `npm test` | `npm test` |
| Vercel 호환 | 서브에이전트가 확인 | 없음 | `grep` 한 줄 |

여기서 지켜야 할 원칙은 하나다.
**명령을 갈라지게 두지 않는다.**

CI에서만 `-race` 플래그를 붙이거나, 로컬에서만 특정 테스트를 건너뛰기 시작하면, 그 순간부터 "로컬에서는 통과하는데 CI에서만 깨진다"는 종류의 시간 낭비가 시작된다.
검사 항목을 늘리고 싶으면 세 곳 중 어디에 넣을지를 먼저 정하고, 넣기로 한 곳에서 같은 명령을 쓴다.

분담의 기준은 13장에서 세운 것과 같다.
빠르고 결정적인 검사는 앞쪽 게이트로, 느리고 포괄적인 검사는 뒤쪽 게이트로 보낸다.
`gofmt`는 밀리초 단위라 훅에 있고, 의존성 설치(`npm ci`)부터 다시 시작하는 검사는 수십 초가 걸리므로 CI에만 있다.

## Vercel의 자동 배포와는 무엇이 다른가

의문이 하나 남는다.
17장에서 볼 Vercel은 이미 `git push`마다 빌드하고 배포한다.
그렇다면 CI가 하는 일이 중복 아닌가.

아니다.
둘은 묻는 질문이 다르다.

Vercel의 빌드는 "이 코드로 앱을 만들 수 있는가"를 묻는다.
Next.js 빌드가 성공하고 Go 함수가 컴파일되면 통과다.
테스트는 돌지 않고, `go vet`도 돌지 않는다.
타입 오류가 있어도 Next.js는 기본 설정에서 빌드를 멈추지만, 테스트가 실패하는 코드는 아무 저항 없이 프로덕션으로 나간다.

CI는 "이 코드가 옳은가"를 묻는다.
그리고 이 질문에 답이 나오기 전에는 `main`에 코드가 들어가지 않게 막을 수 있다.
GitHub 저장소 설정에서 `main` 브랜치에 보호 규칙을 걸고 CI 통과를 필수 조건으로 지정하면, 검사가 실패한 풀 리퀘스트는 병합 버튼 자체가 잠긴다.
운영으로 나가는 코드의 통로인 `release` 브랜치라면 같은 규칙의 값어치가 더 크다.
혼자 개발하는 저장소에서도 이 설정은 값을 한다.
스스로 만든 규칙에 스스로 걸리는 장치이기 때문이다.

## 이 저장소의 다른 워크플로

`.github/workflows/`에는 파일이 하나 더 있다.
지금 읽고 있는 이 책을 GitHub Pages에 배포하는 `book.yml`이다.

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'doc/**'
      - '.github/workflows/book.yml'
  workflow_dispatch:
```

`paths` 필터가 눈여겨볼 대목이다.
원고를 고치지 않은 커밋에서는 책을 다시 빌드할 이유가 없으므로, `doc/` 아래가 바뀐 커밋에서만 이 워크플로가 깨어난다.
`ci.yml`에는 이 필터를 걸지 않았다.
원고만 고친 커밋에서도 Go 코드가 여전히 빌드되는지 확인하는 비용이 몇십 초에 불과하고, 필터를 잘못 걸어 검사를 건너뛰는 사고가 더 비싸기 때문이다.

필터는 캐시와 비슷하다.
아낀 시간보다 어긋났을 때의 대가가 크면 걸지 않는 편이 낫다.

## 무엇을 넣지 않았는가

이 워크플로에 없는 것도 의도의 결과다.

`staticcheck`는 넣지 않았다.
6장에서 봤듯 표준 도구가 아니라 `go install`로 설치해야 하고, 그만큼 러너 시간이 늘어난다.
검사 항목이 넓어지는 이득과 매 실행마다 붙는 설치 시간을 저울질한 결과였다.
프로젝트가 커지면 저울이 기울 것이다.

E2E 테스트도 없다.
브라우저를 띄우고 실제 데이터베이스를 붙여야 하므로, CI에서 돌리려면 테스트용 Supabase 프로젝트와 그 비밀을 저장소 시크릿에 넣는 일이 따라온다.
지금 이 앱의 규모에서는 그 복잡도가 얻는 것보다 크다.
이 판단이 언제 뒤집히는지는 22장에서 다시 이야기한다.

한편 이 워크플로에 나중에 더할 것이 하나 예고되어 있다.
21장에서 볼 Supabase 무료 프로젝트의 일시정지를 막는 스케줄 워크플로다.
`on: schedule`로 하루 몇 번 깨어나 데이터베이스를 건드리는 요청 하나를 보내는 것이 전부인데, CI와 같은 저장소, 같은 디렉터리에 나란히 놓인다.
GitHub Actions는 검사만 돌리는 도구가 아니라 저장소에 딸린 작은 크론(cron)이기도 하다.

## 정리

첫째, 훅과 에디터 태스크는 로컬 게이트라 통과하지 않고도 푸시할 수 있다.
CI는 깨끗한 러너에서 저장소만으로 앱이 빌드되고 검증되는지 확인하는 세 번째 게이트다.

둘째, `ci.yml`은 Go 잡과 web 잡을 병렬로 돌린다.
서로 의존하지 않는 검사를 잡으로 나누면 전체 시간이 느린 쪽으로 줄어든다.

셋째, CI의 `gofmt`는 `-w`가 아니라 `-l`이다.
러너의 파일을 고쳐도 저장소는 그대로이므로, 고치는 대신 어긋난 파일을 고발하고 실패한다.
같은 규칙을 게이트마다 다른 동사로 적용한다.

넷째, `api/`가 `internal/`을 import하지 않는지 확인하는 `grep` 한 줄은 로컬에서 절대 재현되지 않는 배포 실패를 막는다.
기계적으로 판정 가능한 규칙은 에이전트에게 부탁하지 말고 기계에게 맡긴다.

다섯째, Vercel의 빌드는 "만들 수 있는가"를, CI는 "옳은가"를 묻는다.
브랜치 보호 규칙으로 CI 통과를 병합 조건에 걸면, 혼자 개발하는 저장소에서도 스스로 만든 규칙에 스스로 걸리게 된다.

다음 17장에서는 그 Vercel을 다룬다.
`git push` 한 번으로 정적 프런트엔드와 Go 서버리스 함수가 한 플랫폼에 배포되는 구조, 그리고 방금 `grep`으로 막아 둔 그 제약이 왜 존재하는지를 코드 수준에서 확인한다.
