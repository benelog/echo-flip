# 7장 Claude Code: AI 에이전트와 개발하기

2부에서는 Echo Flip의 코드를 만들어 낸 AI 에이전트와, 앱을 배포하고 운영하는 인프라를 다룬다.
그 첫 순서로 이 장에서는 코딩 에이전트(Coding Agent)가 무엇이고 기존의 AI 챗과 어떻게 다른지, Anthropic의 Claude Code가 어떤 방식으로 동작하는지를 살펴보겠다.
이어서 프로젝트 컨텍스트를 전달하는 방법, 권한 모델, 그리고 Echo Flip 개발에서 에이전트를 실제로 활용한 흐름과 한계까지 짚어 보겠다.
구체적인 훅(Hook)과 서브에이전트(Subagent) 설정은 8장에서 파일 단위로 해부하므로, 이 장은 개념과 활용 방식에 집중한다.

## 코딩 에이전트란 무엇인가

### 질문-답변형 챗의 한계

ChatGPT나 Claude 같은 대화형 AI에게 코드를 요청해 본 경험은 이제 대부분의 개발자에게 익숙할 것이다.
"Go로 간격 반복 알고리즘을 구현해 줘"라고 물으면 그럴듯한 코드가 돌아온다.
그런데 그 코드를 실제 프로젝트에 넣으려면 사람이 해야 할 일이 많다.

::: info [용어 풀이] 대규모 언어 모델(LLM)
사람이 쓴 방대한 글을 학습해, 앞의 말에 이어질 다음 말을 확률적으로 골라 내는 AI다.
ChatGPT나 Claude처럼 문장을 주고받는 도구가 모두 이 모델 위에서 돌아간다.
정답을 어딘가에서 찾아오는 것이 아니라 그럴듯한 말을 이어 붙이는 방식이라, 문장은 매끄럽지만 내용이 틀릴 수 있다는 점이 이 장 전체를 관통하는 성질이다.
LLM은 Large Language Model의 약자다.
:::

먼저 브라우저에서 코드를 복사해 에디터에 붙여넣어야 한다.
프로젝트의 기존 코드와 이름이나 시그니처가 맞지 않으면 손으로 고쳐야 한다.
컴파일 에러가 나면 에러 메시지를 다시 복사해서 챗 창에 붙여넣고 수정본을 받아야 한다.
테스트가 실패하면 같은 과정을 반복한다.

이 왕복 과정에서 AI는 프로젝트의 전체 맥락을 보지 못한다.
사람이 골라서 보여 준 코드 조각만 보고 답하기 때문에, 프로젝트에 이미 있는 유틸리티 함수를 다시 만들거나 존재하지 않는 필드를 참조하는 일이 흔하다.
결국 챗 방식에서는 "맥락 전달"과 "결과 검증"이라는 두 가지 노동이 전부 사람 몫이다.

### 에이전트: 도구를 반복 호출하는 루프

코딩 에이전트는 이 두 가지 노동을 AI 쪽으로 옮긴 형태다.
에이전트는 한 번의 질문에 한 번의 답을 내놓고 끝나는 대신, 도구(Tool)를 반복 호출하면서 스스로 작업을 진행한다.
여기서 도구란 파일 읽기, 파일 쓰기, 셸 명령 실행, 코드 검색 같은 구체적인 행동을 말한다.

::: info [용어 풀이] AI 코딩 에이전트와 도구 호출 루프
사람의 지시를 받아 스스로 파일을 읽고 고치고 명령을 실행하며 일을 진행하는 AI다.
한 번 답하고 끝나는 챗과 달리, 도구를 부르고 그 결과를 본 뒤 다음 행동을 다시 정하는 일을 목표에 닿을 때까지 반복한다.
요리에 비유하면, 레시피를 한 번 읊어 주고 마는 사람이 아니라 직접 맛을 보며 간을 맞춰 가는 사람에 가깝다.
:::

동작 방식을 단순화하면 다음과 같은 루프다.

```text
사용자 요청
   ↓
[모델이 다음 행동을 결정]
   ↓
도구 호출 (파일 읽기 / 수정 / 셸 명령 / 검색)
   ↓
도구 실행 결과를 모델에 다시 입력
   ↓
[결과를 보고 다음 행동을 결정] ──→ 작업 완료 판단 시 종료
   ↑                                  │
   └──────────── 반복 ←───────────────┘
```

예를 들어 "SRS 알고리즘의 오답 처리 로직을 고쳐 줘"라는 요청을 받으면, 에이전트는 먼저 검색 도구로 관련 파일을 찾고, 파일 읽기 도구로 `internal/srs/` 아래의 코드를 읽는다.
그다음 수정안을 만들어 파일 쓰기 도구로 반영하고, 셸 도구로 `go test ./...`를 실행한다.
테스트가 실패하면 실패 메시지를 읽고 다시 수정한다.

핵심은 **"생성 → 실행 → 오류 관찰 → 수정"의 루프**다.
챗 방식에서는 사람이 이 루프의 전달자 역할을 했지만, 에이전트는 실행 결과를 스스로 관찰하고 다음 행동을 결정한다.
컴파일러의 에러 메시지, 테스트의 실패 출력, 린터의 경고가 모두 에이전트에게 피드백으로 들어간다.

이 구조가 중요한 이유는, AI가 만든 코드의 완성도가 한 번의 생성 품질이 아니라 **피드백 루프의 품질**에 좌우되기 때문이다.
같은 모델이라도 검증 신호가 풍부한 환경에서는 스스로 오류를 잡아내며 수렴하고, 검증 신호가 없는 환경에서는 그럴듯한 오류를 남긴 채 끝난다.
이 관점은 이 장 후반과 8장 전체를 관통하는 주제이므로 기억해 두자.

## Claude Code 소개

Claude Code는 Anthropic이 만든 터미널 기반 코딩 에이전트다.
IDE 플러그인이나 웹 서비스가 아니라, 명령어를 글자로 입력해 컴퓨터에 지시하는 화면인 터미널(셸)에서 실행하는 대화형 CLI(Command Line Interface)라는 점이 특징이다.
프로젝트 디렉터리 안에서 실행하면 그 디렉터리가 에이전트의 작업 공간이 된다.

에이전트가 쓰는 도구는 앞서 본 루프의 구성 요소 그대로다.
첫째, 파일 읽기·쓰기·수정 도구로 코드베이스를 탐색하고 변경한다.
둘째, 검색 도구로 파일 이름 패턴이나 코드 내용을 찾는다.
셋째, 셸 실행 도구로 빌드, 테스트, 버전 관리 명령을 실행한다.

터미널 기반이라는 선택에는 실용적인 이유가 있다.
개발 프로젝트의 검증 수단은 대부분 CLI 명령으로 존재한다.
Echo Flip만 해도 `go build`, `go vet`, `go test`, `npx tsc`, `npm test`, `npm run build`가 모두 셸 명령이다.
에이전트가 셸을 다룰 수 있으면 이 도구들을 그대로 피드백 루프에 편입할 수 있고, 언어나 프레임워크별로 별도의 통합을 만들 필요가 없다.

대화는 세션(Session) 단위로 이어진다.
한 세션 안에서 에이전트는 지금까지의 대화와 도구 실행 결과를 컨텍스트(Context)로 기억하면서 작업한다.
다만 컨텍스트에는 크기 제한이 있어서, 긴 작업에서는 오래된 내용이 요약되거나 잘려 나갈 수 있다.
이 제약이 실제 개발에 미치는 영향은 이 장 마지막의 한계 절에서 다시 짚는다.

::: info [용어 풀이] 세션(Session)
에이전트와 나누는 하나의 연속된 대화 단위다.
한 세션 안에서는 앞서 주고받은 말과 실행 결과를 기억하지만, 대화를 새로 시작하면 그 기억은 사라지고 백지에서 출발한다.
사람으로 치면 한 번 앉아서 이어 가는 회의와 같아서, 회의가 끝나면 다음 회의는 처음부터 다시 맥락을 잡아야 한다.
:::

::: info [용어 풀이] 컨텍스트와 토큰
컨텍스트(Context)는 모델이 지금 이 순간 참고하고 있는 대화와 코드 전체, 곧 '눈앞에 펼쳐 둔 기억'이다.
그 분량은 토큰(Token) 단위로 세는데, 토큰은 글자보다 크고 단어보다 작은 말 조각이라 긴 단어 하나가 여러 토큰으로 쪼개지기도 한다.
모델이 한 번에 담아 둘 수 있는 토큰 수에는 한계가 있어서, 대화가 길어지면 오래된 내용은 요약되거나 밀려난다.
:::

버전에 따라 세부 기능은 계속 바뀌므로 여기서 기능 목록을 나열하지는 않겠다.
이 책에서 중요한 것은 "코드베이스를 읽고, 수정하고, 검증 명령을 실행하는 루프를 터미널에서 돌린다"는 동작 원리와, 그 루프를 프로젝트에 맞게 조정하는 설정 방법이다.

## 프로젝트 컨텍스트 전달: CLAUDE.md

### 매 세션 시작 시 로드되는 프로젝트 지침

에이전트는 세션을 새로 시작할 때마다 백지 상태에서 출발한다.
어제의 대화에서 "이 프로젝트는 테스트를 `go test ./...`로 돌린다"라고 알려 줬더라도, 오늘 새 세션에는 그 기억이 없다.
매번 같은 설명을 반복하는 대신, Claude Code는 프로젝트 루트의 `CLAUDE.md` 파일을 세션 시작 시 자동으로 읽어 들인다.

::: info [용어 풀이] 프로젝트 지침(CLAUDE.md)
프로젝트 폴더에 두면 에이전트가 매 세션을 시작할 때마다 자동으로 읽어 들이는 안내 문서다.
자주 쓰는 명령어, 폴더 구조, 이 프로젝트만의 규칙과 조심할 점을 담아 두어, 세션이 바뀌어도 같은 설명을 되풀이하지 않게 해 준다.
파일 이름을 그대로 `CLAUDE.md`로 저장해야 에이전트가 알아보고 읽는다.
:::

`CLAUDE.md`는 사람으로 치면 새로 합류한 팀원에게 주는 온보딩 문서다.
프로젝트의 구조, 자주 쓰는 명령어, 코딩 규약, 조심해야 할 부분을 담아 두면 에이전트가 매 세션 그 지침 아래에서 작업한다.
`/init` 명령으로 코드베이스를 분석해 초안을 생성할 수도 있고, 처음부터 직접 작성해도 된다.

좋은 `CLAUDE.md`가 갖출 내용을 정리하면 다음과 같다.

첫째, **자주 실행하는 명령어**다.
빌드, 테스트, 린트, 로컬 서버 실행 명령을 적어 두면 에이전트가 검증 단계에서 올바른 명령을 바로 실행한다.

둘째, **프로젝트 구조와 아키텍처의 요약**이다.
어떤 디렉터리에 무엇이 있고 계층 간 의존 방향이 어떻게 되는지를 적어 두면, 에이전트가 코드를 넣을 위치를 잘못 고르는 일이 줄어든다.

셋째, **프로젝트 고유의 규약과 함정**이다.
일반적인 관례와 다른 부분, 겉보기와 다르게 동작하는 부분일수록 명시할 가치가 크다.
에이전트는 학습 데이터에서 배운 일반적인 패턴을 기본값으로 삼기 때문에, 프로젝트가 그 기본값에서 벗어난 곳을 알려 주지 않으면 그럴듯하지만 틀린 코드를 만든다.

넷째, **하지 말아야 할 것**이다.
건드리면 안 되는 파일, 실행하면 안 되는 명령, 커밋 규칙 같은 금지 사항은 지침으로 적어 두는 편이 매번 승인 단계에서 거르는 것보다 안전하다.

반대로 피해야 할 것은 분량 욕심이다.
`CLAUDE.md`는 매 세션의 컨텍스트에 포함되므로, 길어질수록 정작 중요한 지침이 묻히고 컨텍스트 예산을 잡아먹는다.
문서 전체를 복사해 넣기보다, 에이전트가 틀리기 쉬운 대목만 골라 짧게 유지하는 편이 효과적이다.

### Echo Flip의 경우

정직하게 밝히면, 이 책을 쓰는 시점의 Echo Flip 저장소에는 `CLAUDE.md`가 없다.
프로젝트 규모가 작아 `README.md`의 로컬 개발 절차와 테스트 명령만으로도 에이전트가 충분히 방향을 잡았고, 반복적으로 틀리는 것은 문서 대신 `.claude/` 디렉터리의 훅과 서브에이전트 설정으로 강제했기 때문이다.
지침 문서는 "지켜 달라고 부탁"하는 방식이고 훅은 "어기면 실패시키는" 방식인데, 이 프로젝트에서는 후자를 택한 부분이 많다.

그래도 만약 이 프로젝트에 `CLAUDE.md`를 만든다면 어떤 내용이 들어갈지 예시로 정리해 보자.
앞의 원칙을 Echo Flip에 적용하면 다음 정도가 된다.

```markdown
# CLAUDE.md (작성 예시 — 실제 저장소에는 없는 파일이다)

## 명령어
- Go 검증: go build ./... && go vet ./... && go test ./...
- 프런트 검증: npx tsc --noEmit && npm test
- 로컬 서버: go run ./cmd/server (8080), npm run dev (3000)

## 구조
- api/index.go: Vercel 서버리스 진입점. internal/ 패키지를 import하면
  Vercel 빌드가 실패하므로 pkg/app만 사용한다.
- internal/: 도메인 로직 (srs, smartrules, auth, db)
- src/: Next.js 프런트엔드 (App Router, 정적 export)

## 규약
- DB 접근은 Go API를 통해서만 한다. 프런트에서 Supabase에 직접 붙지 않는다.
- 마이그레이션 파일(internal/db/migrations/)은 수정하지 말고 새 파일을 추가한다.
```

이 예시에서 눈여겨볼 부분은 `api/index.go`에 관한 항목이다.
Vercel이 이 파일을 모듈 외부에서 단독 컴파일하기 때문에 Go의 `internal/` 패키지를 import할 수 없다는 제약은, 일반적인 Go 지식만으로는 예측할 수 없는 프로젝트 고유의 함정이다.
실제로 이 제약은 `.claude/agents/go-quality.md` 서브에이전트의 검사 항목으로 들어가 있는데, 그 내용은 8장에서 자세히 다룬다.

## 권한 모델과 안전장치

파일을 수정하고 셸 명령을 실행하는 에이전트는 그만큼 위험하기도 하다.
잘못된 명령 하나가 파일을 지우거나, 의도하지 않은 커밋을 원격 저장소에 밀어 넣을 수 있다.
그래서 Claude Code는 행동의 위험도에 따라 승인 흐름을 두는 권한 모델을 갖고 있다.

기본 원칙은 단순하다.
읽기처럼 되돌릴 필요가 없는 행동은 자유롭게 허용하고, 파일 수정이나 셸 명령처럼 상태를 바꾸는 행동은 실행 전에 사용자에게 보여 주고 승인을 받는다.
승인 요청 화면에는 어떤 파일을 어떻게 바꾸려는지, 어떤 명령을 실행하려는지가 표시되므로 사용자는 실행 전에 내용을 검토할 수 있다.

매번 승인하는 것이 번거롭다면 특정 행동을 자동 허용 목록에 등록할 수 있다.
예를 들어 `go test` 같은 검증 명령은 반복 실행해도 해가 없으므로 자동 허용해 두면 루프가 끊기지 않고, `git push`나 파일 삭제처럼 되돌리기 어려운 명령은 승인을 유지하는 식이다.
이런 정책은 프로젝트의 `.claude/settings.json` 같은 설정 파일에 저장해 팀 전체가 공유할 수도 있다.

자동 허용의 범위를 정할 때 기준으로 삼을 만한 질문은 두 가지다.
첫째, 이 행동은 되돌릴 수 있는가.
둘째, 잘못됐을 때 피해가 작업 디렉터리 안에 머무는가.
둘 다 "예"라면 자동 허용해서 에이전트의 루프 속도를 높이는 이득이 크고, 하나라도 "아니오"라면 사람의 확인을 거치는 비용이 정당화된다.

안전장치는 승인 흐름만이 아니다.
버전 관리 자체가 가장 강력한 안전망이다.
에이전트가 무엇을 바꾸든 `git diff`로 전체 변경을 확인할 수 있고, 마음에 들지 않으면 되돌릴 수 있다.
Echo Flip 개발에서도 에이전트의 작업 단위를 커밋 단위와 맞춰서, 검토를 통과한 변경만 커밋하는 방식으로 진행했다.

## 에이전트 시대의 개발 흐름

### 작성자에서 검토자로

에이전트가 코드 작성의 상당 부분을 맡으면서 개발자의 역할이 바뀌고 있다.
개발자는 AI에게 요구 사항과 제약 조건, 방향성을 제시하고, AI가 만든 코드를 검토하고 승인한다.
코드를 한 줄씩 타이핑하는 시간은 줄고, 무엇을 만들지 정의하고 만들어진 것을 판단하는 시간이 늘어난다.

이 전환에서 가장 어려운 질문은 "AI가 만든 코드를 어디까지 자세히 봐야 하는가"다.
모든 줄을 사람이 정독한다면 에이전트를 쓰는 이득이 크게 줄어든다.
반대로 검토 없이 승인만 한다면 그럴듯한 오류가 그대로 쌓인다.
이 사이 어딘가에서 각자 자신의 답을 찾아야 하는 상황이 되었다.

### 신뢰 수준을 판단하는 두 축

일괄적인 답 대신, 코드의 성격에 따라 검토 강도를 달리하는 기준을 세워 보자.
쓸 만한 기준은 두 가지 축이다.

첫째 축은 **오류를 발견하는 난이도**다.
틀리면 즉시 눈에 보이는 코드가 있고, 한참 뒤에야 드러나는 코드가 있다.
CSS가 어긋나면 화면을 보는 순간 알지만, SRS 복습 간격 계산이 미묘하게 틀리면 몇 주 뒤의 복습 큐가 이상해질 때까지 아무도 모른다.

둘째 축은 **오류의 비용**이다.
같은 오류라도 개발 중 화면의 버튼 위치가 틀린 것과, 사용자 데이터를 지우는 마이그레이션이 틀린 것은 비용이 다르다.

두 축을 조합하면 검토 강도를 다음과 같이 배분할 수 있다.

| | 오류 발견이 쉬움 | 오류 발견이 어려움 |
|---|---|---|
| **오류 비용 낮음** | 가볍게 훑고 승인 | 테스트로 커버하고 승인 |
| **오류 비용 높음** | 실행 결과를 직접 확인 | 사람이 정독 + 테스트 + 단계적 적용 |

Echo Flip에 대입하면, 화면 컴포넌트의 마크업 수정은 왼쪽 위에 가깝고, `internal/db/migrations/`의 스키마 변경이나 인증 로직(`internal/auth/`)은 오른쪽 아래에 가깝다.
실제 개발에서도 UI 변경은 화면 확인 위주로 빠르게 승인했고, SRS 알고리즘과 마이그레이션은 코드를 정독하고 테스트 케이스를 함께 검토했다.

### AI 코드의 품질은 검증 신호의 품질이다

여기서 한 걸음 더 나아가 보자.
"오류 발견이 어려운" 코드를 "발견이 쉬운" 코드로 옮길 수 있다면, 사람의 정독이 필요한 영역 자체를 줄일 수 있다.
그 수단이 바로 컴파일, 테스트, 정적 분석 같은 자동 검증 신호다.

SRS 간격 계산이 위험했던 이유는 오류가 늦게 드러나기 때문이었다.
그런데 `internal/srs/srs_test.go`처럼 경계 조건을 망라한 테이블 주도 테스트가 있으면, 계산 오류는 커밋 전에 테스트 실패로 드러난다.
검증 신호를 추가하는 순간 이 코드는 표의 오른쪽에서 왼쪽으로 이동하고, 에이전트에게 더 안심하고 위임할 수 있는 영역이 된다.

그래서 이렇게 말할 수 있다.
**AI가 만든 코드의 품질은 그 코드를 검증해 주는 신호의 품질에 달려 있다.**
1부에서 각 장마다 품질 검사 도구를 함께 다룬 이유가 이것이다.
Go의 `gofmt`·`go vet`·테스트(2장), TypeScript의 strict 모드와 vitest(4장), 데이터베이스의 제약 조건(6장)은 모두 사람을 위한 도구인 동시에 에이전트의 피드백 루프를 구성하는 신호다.

그리고 이 신호들을 에이전트의 루프에 자동으로 연결하는 장치, 곧 코드를 수정할 때마다 검사를 실행하는 훅과 커밋 전에 종합 검증을 수행하는 서브에이전트가 바로 8장의 주제다.

## Echo Flip 개발에서의 실제 활용 흐름

이제 Echo Flip을 만들면서 실제로 반복했던 작업 흐름을 소개하겠다.
큰 틀은 네 단계다.

첫째, **요구사항을 대화로 전달한다**.
"학습 화면에서 덱을 다 돌면 틀린 카드만 모아서 재도전 라운드를 시작하고 싶다. 라운드는 틀린 카드가 없어질 때까지 반복한다"처럼, 원하는 동작과 제약을 문장으로 설명한다.
이 단계에서 들이는 정성이 결과 품질을 가장 크게 좌우했다.
모호하게 시작하면 에이전트는 그럴듯한 해석을 하나 골라서 진행하는데, 그 해석이 의도와 다르면 뒤에서 고치는 비용이 더 크다.

::: info [용어 풀이] 프롬프트(Prompt)
에이전트에게 건네는 지시나 요청을 담은 말이다.
"이렇게 동작하게 해 줘"처럼 무엇을 원하는지 문장으로 적어 주는 것이 프롬프트이며, 여기에 담긴 정보가 구체적일수록 결과가 의도에 가까워진다.
막연한 한 줄보다 원하는 동작과 제약을 또렷이 적은 몇 줄이 훨씬 나은 결과를 부르는 것도 이 때문이다.
:::

둘째, **에이전트가 설계를 제안하고 코드를 작성한다**.
에이전트는 관련 파일을 읽고 어디를 어떻게 바꿀지 계획을 먼저 제시한다.
계획 단계에서 방향을 교정하는 것이 코드가 나온 뒤에 고치는 것보다 싸기 때문에, 규모가 있는 변경은 계획을 먼저 합의하고 진행했다.

셋째, **자동 검증을 돌린다**.
Go 코드는 `go build ./...`, `go vet ./...`, `go test ./...`로, 프런트엔드는 `npx tsc --noEmit`과 `npm test`(vitest), `npm run build`로 검증한다.
에이전트는 실패 출력을 읽고 스스로 수정한 뒤 다시 실행한다.
사람은 이 반복이 수렴한 결과만 받아 본다.

넷째, **사람이 리뷰하고 커밋한다**.
`git diff`로 변경 전체를 훑고, 앞 절의 두 축에 따라 정독할 부분을 골라 읽는다.
문제가 없으면 커밋하고, 의도와 다르면 피드백을 주고 둘째 단계로 돌아간다.

### 아키텍처 결정도 대화에서 나온다

에이전트의 쓰임이 코드 작성에만 머물지 않았다는 점도 기록해 둘 만하다.
Echo Flip의 배포 구조를 정할 때의 일이다.

초기 검토안은 흔한 구성이었다.
프런트엔드는 Vercel에 정적 배포하고, Go API 서버는 무료 티어를 제공하는 별도 호스팅에 상주 서버로 올리는 안이다.
그런데 배포 구성을 논의하던 대화에서 "호스팅을 두 군데로 나누지 말고 Go만 쓰면 안 되나"라는 질문을 던졌고, 에이전트가 Vercel이 Go를 서버리스 함수(Serverless Function)로 실행할 수 있다는 선택지를 정리해 줬다.
상주 서버가 없어지면 관리할 인프라가 Vercel 하나로 줄고 비용도 무료 티어 안에 들어온다는 트레이드오프까지 표로 비교한 뒤, 서버리스 안을 택했다.

그 결과가 지금의 구조다.
로컬에서는 `cmd/server`의 상주 서버로 개발하고, 배포 시에는 같은 Gin 앱을 `api/index.go`의 서버리스 함수로 감싼다.
이 전환의 기술적 세부 사항과 제약은 9장에서 다룬다.

여기서 얻은 교훈은, 에이전트가 유용한 국면이 "정답이 정해진 구현"보다 "선택지를 넓히는 탐색"일 때도 많다는 것이다.
Vercel의 Go 런타임 지원 같은 사실은 검색하면 나오는 정보지만, 내 프로젝트의 맥락에 대입해 트레이드오프를 정리해 주는 것은 대화형 도구의 강점이었다.
물론 이렇게 얻은 정보는 틀릴 수 있으므로, 실제로 최소 구성을 배포해 보고 나서 확정했다.

### 이 프로젝트의 에이전트 설정 미리 보기

Echo Flip 저장소에는 에이전트의 루프에 검증 신호를 연결하는 설정이 들어 있다.
구조만 미리 보면 다음과 같다.

```text
.claude/
├── settings.json          # 훅 등록: 파일 수정 직후 go-check.sh 실행
├── hooks/
│   └── go-check.sh        # gofmt 자동 적용 + go vet, 실패 시 에이전트에 피드백
└── agents/
    └── go-quality.md      # 커밋 전 종합 검증 서브에이전트 (build/vet/test + Vercel 호환)
```

`.claude/settings.json`은 에이전트가 파일을 쓰거나 수정할 때마다 실행되는 훅을 등록한다.
이 설정 덕분에 에이전트가 Go 파일을 수정하면 사람이 시키지 않아도 `gofmt`와 `go vet`이 즉시 실행되고, 문제가 있으면 그 결과가 에이전트에게 피드백으로 돌아간다.
`.claude/agents/go-quality.md`는 커밋 전에 빌드·정적 분석·테스트에 더해 "Vercel 서버리스 빌드 호환성"까지 검사하는 서브에이전트를 정의한다.
각 파일이 왜 이렇게 작성됐는지, 훅과 서브에이전트를 어떤 기준으로 나눠 썼는지는 8장에서 한 줄씩 해부한다.

## 한계와 주의점

에이전트의 강점을 다뤘으니 한계도 같은 비중으로 짚어 보겠다.
과장 없이 말하면, 아래 항목들은 Echo Flip 개발 중에 실제로 겪은 문제들이다.

### 그럴듯하지만 틀린 코드

가장 근본적인 한계다.
언어 모델은 통계적으로 그럴듯한 코드를 생성하므로, 결과물은 문법적으로 완벽하고 스타일도 자연스럽지만 논리가 틀릴 수 있다.
겉모습의 완성도가 높을수록 사람의 경계심은 낮아지기 때문에, 이런 오류는 어설픈 코드의 오류보다 오히려 위험하다.
컴파일과 테스트라는 자동 신호로 잡히지 않는 종류의 오류(요구사항 해석이 미묘하게 어긋난 경우)는 결국 사람의 리뷰로만 걸러진다.

### 학습 데이터에 없는 최신 API

모델의 지식에는 시간 기준점(Cutoff)이 있다.
그 이후에 나온 라이브러리 버전이나 바뀐 API는 모델이 알지 못하며, 더 나쁘게는 옛 버전의 API를 확신에 찬 어조로 생성한다.
빠르게 변하는 프런트엔드 생태계에서 특히 자주 마주치는 문제다.
대응책은 두 가지였다.
공식 문서의 해당 부분을 대화에 직접 붙여 넣어 주거나, 버전 확인이 필요한 부분임을 명시해서 에이전트가 문서를 조회하도록 유도하는 것이다.

### 컨텍스트 제한

에이전트가 한 번에 기억할 수 있는 대화와 코드의 양에는 한계가 있다.
세션이 길어지면 초반에 합의한 제약 조건을 잊은 듯한 행동이 나타나기도 한다.
Echo Flip 개발에서는 작업을 커밋 단위의 작은 덩어리로 나누고, 덩어리마다 세션을 새로 시작하는 방식으로 대응했다.
세션을 넘어 유지되어야 하는 지침이라면 대화가 아니라 `CLAUDE.md`나 훅 같은 설정에 담아야 한다는 원칙도 여기서 나온다.

### 사람이 기반 기술을 이해하고 있어야 한다

세 가지 한계의 공통 대응책은 결국 사람의 검증이다.
그리고 검증은 검증할 지식이 있어야 가능하다.
도입에서 밝혔듯, 이 책이 2부의 도구 이야기에 앞서 1부에서 언어와 프레임워크의 기초를 다진 것도 이 검증의 판단 근거를 쌓기 위해서였다.

## 정리

이 장에서는 코딩 에이전트의 동작 원리와 Claude Code를 활용한 개발 방식을 살펴봤다.

- 코딩 에이전트는 한 번의 질문-답변으로 끝나는 챗과 달리, 파일 읽기·쓰기, 셸 실행, 검색 같은 도구를 반복 호출하며 스스로 작업을 진행한다. 핵심은 "생성 → 실행 → 오류 관찰 → 수정"의 루프다.
- Claude Code는 터미널에서 실행되는 Anthropic의 코딩 에이전트로, 프로젝트 디렉터리를 작업 공간 삼아 코드베이스를 읽고 수정하며 빌드·테스트 명령을 실행한다.
- `CLAUDE.md`는 매 세션 시작 시 로드되는 프로젝트 지침 파일이다. 자주 쓰는 명령어, 구조 요약, 프로젝트 고유의 규약과 함정을 짧게 담는 것이 요령이다. Echo Flip은 이 파일 대신 훅과 서브에이전트로 규약을 강제하는 쪽을 택했다.
- 권한 모델의 기본은 상태를 바꾸는 행동에 대한 승인이다. 되돌릴 수 있고 피해 범위가 좁은 행동만 자동 허용하고, 버전 관리를 최종 안전망으로 삼는다.
- 개발자의 역할은 요구사항과 제약을 제시하고 결과를 검토하는 쪽으로 이동한다. 검토 강도는 "오류 발견의 난이도"와 "오류의 비용" 두 축으로 배분하고, 자동 검증 신호를 늘려 정독이 필요한 영역 자체를 줄인다.
- Echo Flip에서는 대화로 요구사항 전달 → 설계 합의 → 코드 작성 → `go test`·vitest·빌드 검증 → 사람 리뷰의 흐름을 반복했고, Vercel 서버리스 전환 같은 아키텍처 결정도 대화 속 질문에서 출발했다.
- 그럴듯하지만 틀린 코드, 학습 시점 이후의 API, 컨텍스트 제한이라는 한계가 있으며, 최종 검증자는 기반 기술을 이해한 사람이다.

AI 코드의 품질이 검증 신호의 품질에 달려 있다면, 다음 할 일은 그 신호를 에이전트의 루프에 자동으로 연결하는 것이다.
다음 장에서는 Echo Flip의 `.claude/` 디렉터리에 있는 훅과 서브에이전트 설정을 한 줄씩 해부하면서, 이 프로젝트의 품질 게이트가 어떻게 만들어졌는지 살펴보겠다.
