= 15장 GitHub Actions: 원격 품질 게이트

지금까지 Flashcard의 품질 게이트는 두 곳에 있었다.
8장의 훅이 에이전트가 파일을 쓸 때마다 `gofmt`와 `go vet`을 돌리고, 13장의 `전체 검증` 태스크가 사람이 커밋하기 전에 세 검사를 묶어 돌린다.

둘 다 로컬의 게이트다.
그리고 로컬의 게이트에는 공통된 약점이 하나 있다.
통과하지 않고도 푸시할 수 있다는 것이다.

14장에서 정한 브랜치 정책을 떠올리면 이 약점은 더 심각해진다.
main으로 향하는 코드는 개발 환경에 닿고, release로 향하는 코드는 사용자가 쓰는 운영 환경에 닿는다.
어느 쪽이든 검사를 건너뛴 코드가 도착할 수 있다면, 정책은 있는데 지키는 장치가 없는 상태다.

이 장에서는 그 약점을 메우는 세 번째 게이트, 즉 저장소에 코드가 도착하는 순간 원격에서 자동으로 도는 검사를 그 브랜치들 위에 얹는다.
GitHub Actions로 `.github/workflows/ci.yml` 한 파일을 작성해 빌드·정적 분석·테스트를 원격에서 돌리고, 이 게이트가 앞의 두 게이트와 어떻게 같은 명령을 공유하는지 짚는다.
장 뒤쪽에서는 같은 도구가 검사를 넘어 저장소 바깥을 바꾸는 일까지 맡는 모습을 본다.
브랜치에 따라 개발 DB와 운영 DB에 스키마를 적용하는 `migrate.yml`이 그것으로, 검사 워크플로와는 설정의 방향이 어떻게 달라지는지가 볼거리다.

== 로컬 게이트가 새는 곳

혼자 개발하고 커밋 전에 태스크를 꼬박꼬박 돌린다면, 원격 검사가 없다는 사실은 오래 드러나지 않는다.
드러나는 순간은 셋이다.

첫째, 사람이 깜빡할 때다.
급한 수정 하나를 바로 푸시하는 습관은 누구에게나 있고, 훅은 에이전트의 도구 호출에만 걸려 있지 사람의 `git push`에는 걸려 있지 않다.

둘째, 두 번째 사람이 합류할 때다.
풀 리퀘스트로 들어온 코드가 `go vet`을 통과하는지 확인하려면 그 코드를 내 기계로 내려받아 직접 돌려야 한다.
자동 검사가 없으면 리뷰어의 눈이 유일한 게이트가 된다.

셋째, 내 기계에만 있는 조건이 검사를 통과시킬 때다.
로컬에 설치된 오래된 의존성이나 캐시된 테스트 결과가 실패를 가려 주는 경우가 있다.
깨끗한 기계에서 처음부터 받아 돌리는 검사만이 이것을 잡아낸다.

세 번째 이유가 특히 중요하다.
CI는 "사람을 못 믿어서" 두는 장치라기보다, 아무것도 설치되지 않은 상태에서 저장소만으로 앱이 빌드되고 검증되는지를 확인하는 장치다.

.[용어 풀이] 지속적 통합(CI, Continuous Integration)
[NOTE]
====
코드를 저장소에 합칠 때마다 빌드와 검사를 자동으로 돌려서, 깨진 코드가 오래 섞여 있지 않게 하는 관행이다.
합치는 주기를 짧게 가져갈수록 문제가 생긴 원인을 찾기 쉬워진다는 발상에서 나왔다.
검사를 대신 돌려 주는 서비스가 여럿 있는데, 저장소가 GitHub에 있다면 별도 가입 없이 쓸 수 있는 GitHub Actions가 가장 마찰이 적다.
====

== GitHub Actions의 구조

GitHub Actions는 저장소에서 어떤 일(이벤트)이 벌어지면 정해진 절차를 실행해 주는 도구다.
워크플로, 잡, 러너, 스텝이라는 용어 네 개가 한꺼번에 등장해 처음에는 헷갈리는데, 실제로는 큰 것이 작은 것을 담는 포함 관계 하나로 이어진다.
워크플로(workflow) 하나가 잡(job)을 여럿 담고, 잡 안에는 스텝(step)이 순서대로 놓인다.

.[용어 풀이] 워크플로(Workflow)·잡(Job)·스텝(Step)
[NOTE]
====
워크플로는 `.github/workflows/` 아래에 두는 설정 파일 하나로, 저장소에서 정해 둔 일이 벌어지면 깨어나 그 안의 절차를 실행한다.
잡은 워크플로 안에서 컴퓨터 한 대를 통째로 차지하고 도는 작업 묶음이다.
스텝은 잡 안에서 위에서 아래로 하나씩 실행되는 단위로, 셸 명령 한 줄이거나 남이 만들어 공개해 둔 재사용 부품(액션, action) 하나다.
요리에 비유하면 워크플로는 조리법 한 장, 잡은 그 조리법을 통째로 맡은 요리사 한 명, 스텝은 요리사가 순서대로 밟는 손질과 가열 단계다.
====

워크플로를 적는 파일의 형식은 YAML이다.
프로그램을 짜는 것이 아니라 무엇을 언제 어떤 순서로 실행할지만 적어 두는 형식이라, 뒤에서 볼 `ci.yml`도 처음부터 끝까지 항목의 나열이다.

.[용어 풀이] YAML(YAML Ain't Markup Language)
[NOTE]
====
들여쓰기로 항목의 포함 관계를 나타내는 설정 파일 형식이다.
`이름: 값`으로 항목을 적고, 한 칸 더 들여쓴 줄은 바로 위 항목에 속한 것으로 읽힌다.
4장에서 본 JSON과 담을 수 있는 내용은 같지만 중괄호와 따옴표가 없어서 사람이 읽고 쓰기 편하고, 그래서 손으로 적는 설정 파일에 널리 쓰인다.
들여쓰기가 곧 구조이므로 공백 한 칸이 어긋나면 다른 뜻이 되고, 탭 문자는 아예 허용되지 않는다.
====

잡이 차지하는 그 컴퓨터가 러너(runner)다.

.[용어 풀이] 러너(Runner)
[NOTE]
====
GitHub가 검사를 돌릴 때마다 새로 빌려 주는, 아무것도 깔려 있지 않은 컴퓨터 한 대다.
Go도 없고 이 저장소의 코드도 없는 상태에서 시작해, 코드를 내려받고 필요한 도구를 설치한 뒤에야 검사를 돌릴 수 있다.
검사가 끝나면 이 컴퓨터는 통째로 버려지므로, 러너 안에서 파일을 고쳐 봐야 저장소에는 아무것도 남지 않는다.
====

여기서 알아 둘 성질은 잡끼리는 기본적으로 병렬로 돈다는 것이다.
백엔드와 프런트엔드가 서로 다른 도구 체인으로 검증되는 저장소라면 잡을 둘로 나눠 나란히 돌릴 수 있고, 그러면 전체 시간이 두 검사 시간의 합이 아니라 둘 중 더 느린 쪽만큼으로 줄어든다.
Flashcard는 화면까지 Go 서버가 렌더링하므로(11장) 검증이 전부 Go 도구 체인 하나 위에서 돌고, 잡도 하나면 충분하다.

== ci.yml 해부

저장소의 `.github/workflows/ci.yml` 전문이다.
두 브랜치에 코드가 도착하거나 풀 리퀘스트가 갱신될 때 깨어나, 포맷 검사부터 Vercel 호환 검사까지 다섯 스텝을 순서대로 돌리는 파일이다.

[source,yaml]
----
name: CI

on:
  push:
    branches: [main, release]
    paths-ignore:
      - 'doc/**'
  pull_request:
    paths-ignore:
      - 'doc/**'
  workflow_dispatch:

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  go:
    name: Go (build, vet, test, gofmt)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true

      - name: gofmt
        run: |
          unformatted=$(gofmt -l .)
          if [ -n "$unformatted" ]; then
            echo "다음 파일이 gofmt를 통과하지 못했습니다:"
            echo "$unformatted"
            exit 1
          fi

      - name: go build
        run: go build ./...

      - name: go vet
        run: go vet ./...

      - name: go test
        run: go test ./...

      - name: Vercel 호환 검사 (api/ 는 internal/ 을 import 할 수 없다)
        run: |
          if grep -rq 'benelog/flashcard/internal' api/; then
            echo "api/ 가 internal/ 패키지를 import 합니다. pkg/ 로 옮기세요."
            exit 1
          fi
----

위에서부터 읽어 보자.

=== on: 언제 도는가

`push`는 14장에서 정한 두 브랜치, 곧 `main`이나 `release`에 커밋이 도착할 때 워크플로를 깨운다.
`main` 푸시가 개발 환경으로 가는 코드를 검사한다면, `release` 푸시는 운영 배포(16장)가 일어나기 직전의 마지막 관문이다.
개발 때 통과한 코드라도 운영으로 나가는 순간 같은 검사를 한 번 더 거치므로, 병합 과정에서 생긴 문제도 놓치지 않고 잡아낸다.
`pull_request`는 풀 리퀘스트가 열리거나 갱신될 때 워크플로를 깨운다.
`workflow_dispatch`는 GitHub 화면에서 버튼을 눌러 수동으로 돌릴 수 있게 해 준다.
이 한 줄이 없으면 워크플로를 손으로 돌릴 방법이 없어서, 시험 삼아 한 번 돌려 보려 해도 커밋을 만들어 푸시해야 한다.

`push`에만 `branches` 필터를 건 것은 의도적이다.
필터가 없으면 브랜치를 밀어 올리고 곧바로 풀 리퀘스트를 열 때 같은 커밋에 워크플로가 두 번 돈다.

`paths-ignore`는 반대 방향의 필터다.
`doc/` 아래의 원고만 고친 커밋에서는 이 검사를 건너뛴다.
책 원고는 앱 바이너리에 들어가지 않으므로 그 파일들이 Go 빌드나 테스트 결과를 바꿀 방법이 없고, 결과가 달라질 수 없는 검사를 돌리는 것은 러너 시간과 기다리는 시간을 함께 쓰는 일이다.
필터를 걸어도 되는 조건은 "이 경로가 바뀌지 않으면 검사 결과가 절대 달라지지 않는다"가 참일 때뿐이라는 점을 기억해 두자.
자신이 없으면 걸지 않는 편이 낫다.
검사를 건너뛰어 깨진 코드를 통과시키는 대가가, 몇십 초를 아끼는 이득보다 언제나 크기 때문이다.

=== concurrency: 낡은 실행을 취소한다

같은 브랜치에서 아직 돌고 있는 검사가 있으면 그것을 취소하고 가장 최근 실행만 남기라는 설정이다.

[source,yaml]
----
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
----

같은 브랜치에 커밋을 연달아 밀면, 앞선 커밋의 검사 결과는 이미 쓸모가 없다.
`cancel-in-progress`가 진행 중인 실행을 취소해 러너 시간을 아낀다.
`${{ }}`는 워크플로가 돌 때 실제 값이 채워지는 자리이고, `github.ref`에는 이 실행을 깨운 브랜치의 이름이 들어간다.
그래서 `group`의 이름이 브랜치마다 달라지고, 취소는 같은 브랜치 안에서만 일어난다.
main에서 돌고 있던 검사가 release에 밀어 넣은 커밋 때문에 취소되는 일은 없다.
공개 저장소라면 러너 시간이 무료지만, 결과를 기다리는 사람의 시간은 무료가 아니다.

=== setup-go: 버전을 저장소에서 읽는다

잡의 첫 스텝인 `actions/checkout@v4`는 저장소의 코드를 빈 러너로 내려받는 액션이다.
그다음이 아직 Go가 깔려 있지 않은 러너에 Go를 설치하는 이 스텝이다.

[source,yaml]
----
- uses: actions/setup-go@v5
  with:
    go-version-file: go.mod
    cache: true
----

Go 버전을 워크플로에 적어 넣는 대신 `go.mod`에서 읽게 했다.
`go.mod`의 `go 1.26.4` 한 줄이 로컬과 CI가 함께 참조하는 단일 진실 공급원(single source of truth)이 되므로, Go를 올릴 때 고칠 곳이 한 군데다.
`cache: true`는 모듈 다운로드 결과를 캐싱해 두 번째 실행부터 시간을 줄인다.

=== gofmt: 고치지 않고 고발한다

이 스텝이 이 워크플로에서 가장 눈여겨볼 곳이다.
`gofmt -l .`로 포맷이 어긋난 파일의 목록을 받아 두고, 그 목록이 비어 있지 않으면 파일 이름을 찍은 뒤 실패로 끝내는 셸 스크립트다.

[source,yaml]
----
- name: gofmt
  run: |
    unformatted=$(gofmt -l .)
    if [ -n "$unformatted" ]; then
      echo "다음 파일이 gofmt를 통과하지 못했습니다:"
      echo "$unformatted"
      exit 1
    fi
----

8장의 훅은 `gofmt -w`를 쓴다.
파일을 조용히 고쳐 놓고 넘어간다.
13장의 에디터도 저장하는 순간 같은 일을 한다.

그런데 CI에서 `gofmt -w`를 돌리면 아무 의미가 없다.
러너의 파일을 고쳐 봐야 그 러너는 곧 사라지고, 저장소에는 여전히 어긋난 파일이 남는다.
그래서 CI는 `-l`을 쓴다.
`gofmt -l`은 포맷이 어긋난 파일의 이름만 출력하고 고치지는 않으므로, 출력이 비어 있지 않으면 `exit 1`로 워크플로를 실패시킨다.

같은 도구, 같은 규칙인데 동사가 다르다.
로컬에서는 고치고, 원격에서는 고발한다.
이것이 세 게이트가 하나의 규칙을 공유하는 방식이다.

=== Vercel 호환 검사: 서브에이전트가 하던 일을 기계로

`api/` 아래 파일에 `internal` 패키지를 가리키는 import가 있는지 `grep`(여러 파일을 훑어 특정 문자열이 든 줄을 찾아 주는 명령)으로 찾아, 하나라도 있으면 실패시키는 검사다.

[source,yaml]
----
- name: Vercel 호환 검사 (api/ 는 internal/ 을 import 할 수 없다)
  run: |
    if grep -rq 'benelog/flashcard/internal' api/; then
      echo "api/ 가 internal/ 패키지를 import 합니다. pkg/ 로 옮기세요."
      exit 1
    fi
----

16장에서 자세히 볼 제약을 미리 검사하는 스텝이다.
Vercel의 Go 런타임은 `api/index.go`를 모듈 바깥에서 컴파일하므로, 이 파일이 `internal/` 패키지를 import하면 로컬 `go build`는 통과하고 배포만 실패한다.
로컬에서는 절대 재현되지 않는 종류의 실패라, 사람의 주의력에 맡기기에 가장 나쁜 조건을 갖췄다.

8장의 `go-quality` 서브에이전트가 검사 항목으로 들고 있던 것이 바로 이 규칙이다.
에이전트에게 "확인해 달라"고 부탁하면 확인을 빠뜨릴 수도 있지만, `grep` 한 줄로 못 박으면 위반이 있는 한 예외 없이 실패한다.
기계적으로 판정할 수 있는 규칙을 발견하면 곧바로 기계에게 넘기는 편이 낫다.

세 게이트가 무엇을 어떻게 부르는지 한자리에 놓아 보자.

|===
| 검사 | 8장 훅 (에이전트) | 13장 태스크 (사람) | 15장 CI (원격)

| 포맷 | `gofmt -w` (자동 수정) | 저장 시 자동 수정 | `gofmt -l` (실패 처리)
| 정적 분석 | `go vet` (수정 파일의 패키지) | `go vet ./...` | `go vet ./...`
| 빌드 | 없음 | `go build ./...` | `go build ./...`
| 테스트 | 없음 (서브에이전트가 담당) | `go test ./...` | `go test ./...`
| Vercel 호환 | 서브에이전트가 확인 | 없음 | `grep` 한 줄
|===

여기서 지켜야 할 원칙은 하나다.
*명령을 갈라지게 두지 않는다.*

CI에서만 8장에서 본 `-race` 플래그를 붙이거나, 로컬에서만 특정 테스트를 건너뛰기 시작하면, 그 순간부터 "로컬에서는 통과하는데 CI에서만 깨진다"는 종류의 시간 낭비가 시작된다.
검사 항목을 늘리고 싶으면 세 곳 중 어디에 넣을지를 먼저 정하고, 넣기로 한 곳에서 같은 명령을 쓴다.

분담의 기준은 8장에서 세운 것과 같다.
빠르고 결정적인(언제 돌려도 같은 결과가 나오는) 검사는 앞쪽 게이트로, 느리고 포괄적인 검사는 뒤쪽 게이트로 보낸다.
`gofmt`는 밀리초 단위라 훅에 있고, 깨끗한 러너를 준비해 의존성 내려받기부터 다시 시작하는 검사는 수십 초가 걸리므로 CI에만 있다.

== Vercel의 자동 배포와는 무엇이 다른가

의문이 하나 남는다.
16장에서 볼 Vercel은 이미 `git push`마다 빌드하고 배포한다.
그렇다면 CI가 하는 일이 중복 아닌가.

아니다.
둘은 묻는 질문이 다르다.

Vercel의 빌드는 "이 코드로 앱을 만들 수 있는가"를 묻는다.
Go 함수가 컴파일되면 통과다.
테스트는 돌지 않고, `go vet`도 돌지 않는다.
컴파일만 되면 테스트가 실패하는 코드도 아무 저항 없이 프로덕션으로 나간다.

CI는 "이 코드가 옳은가"를 묻는다.
그리고 이 질문에 답이 나오기 전에는 `main`에 코드가 들어가지 않게 막을 수 있다.
GitHub 저장소 설정의 브랜치 보호 규칙(branch protection rule)으로 CI 통과를 병합의 필수 조건으로 지정하면 된다.

.[용어 풀이] 브랜치 보호 규칙(Branch Protection Rule)
[NOTE]
====
특정 브랜치에 코드가 들어오는 조건을 GitHub 저장소 설정에 못 박아 두는 장치다.
CI 통과를 필수 조건으로 걸면 검사가 실패한 풀 리퀘스트는 병합 버튼 자체가 잠기고, 직접 푸시를 막아 두면 풀 리퀘스트를 거치지 않은 코드는 그 브랜치에 들어오지 못한다.
14장의 브랜치 정책이 문서에 적어 둔 약속이라면, 브랜치 보호 규칙은 그 약속을 저장소가 강제하게 만드는 설정이다.
운영으로 나가는 코드의 통로인 `release` 브랜치라면 이 규칙의 값어치가 더 크다.
====

혼자 개발하는 저장소에서도 이 설정은 값을 한다.
스스로 만든 규칙에 스스로 걸리는 장치이기 때문이다.

== 이 저장소의 다른 워크플로

`.github/workflows/`에는 파일이 둘 더 있다.
하나는 지금 읽고 있는 이 책을 GitHub Pages에 배포하는 `book.yml`이다.

[source,yaml]
----
on:
  push:
    branches: [main]
    paths:
      - 'doc/**'
      - '.github/workflows/book.yml'
  workflow_dispatch:
----

`ci.yml`의 `paths-ignore`와 정확히 반대인 `paths` 필터가 눈여겨볼 대목이다.
원고를 고치지 않은 커밋에서는 책을 다시 빌드할 이유가 없으므로, `doc/` 아래가 바뀐 커밋에서만 이 워크플로가 깨어난다.
두 워크플로의 필터가 서로의 여집합이라 커밋 하나는 대개 둘 중 하나만 깨운다.

=== migrate.yml: 검사가 아니라 바꾸는 워크플로

다른 하나는 성격이 다르다.
`migrate.yml`은 데이터베이스의 스키마를 실제로 바꾼다.
지금까지 본 두 워크플로가 저장소 안에서 끝나는 일(검사, 정적 사이트 빌드)을 했다면, 이것은 러너 바깥의 살아 있는 데이터베이스에 접속해 되돌리기 어려운 변경을 남긴다.
그 차이가 파일의 모든 줄에 드러난다.

[source,yaml]
----
on:
  push:
    branches: [main, release]
    paths:
      - 'internal/db/migrations/**'
      - 'internal/db/migrate.go'
      - 'cmd/migrate/**'
      - '.github/workflows/migrate.yml'
  workflow_dispatch:

concurrency:
  group: migrate-${{ github.ref }}
  cancel-in-progress: false

jobs:
  migrate:
    name: ${{ github.ref_name == 'release' && '운영' || 'dev' }} DB 마이그레이션
    runs-on: ubuntu-latest
    steps:
      # ... (checkout, setup-go)
      - name: go run ./cmd/migrate
        env:
          MIGRATE_DATABASE_URL: ${{ github.ref_name == 'release' && secrets.PROD_MIGRATE_DATABASE_URL || secrets.DEV_MIGRATE_DATABASE_URL }}
        run: |
          if [ -z "$MIGRATE_DATABASE_URL" ]; then
            echo "${{ github.ref_name }} 브랜치에 해당하는 마이그레이션 시크릿이 비어 있습니다."
            exit 1
          fi
          go run ./cmd/migrate
----

네 곳을 차례로 보자.

첫째, `paths` 필터가 여기서는 시간을 아끼는 장치가 아니라 안전장치다.
필터가 없으면 스키마와 무관한 커밋마다 운영 데이터베이스에 접속을 열게 된다.
바꿀 것이 없으니 결과는 같지만, 아무 일도 하지 않기 위해 운영 DB의 문을 하루에도 몇 번씩 여는 구성은 사고의 여지만 늘린다.
목록에 워크플로 파일 자신(`.github/workflows/migrate.yml`)이 들어 있는 것도 눈여겨보자.
이 워크플로를 고친 커밋에서는 고친 결과가 실제로 도는지 확인할 수 있어야 한다.

둘째, `${{ github.ref_name == 'release' && ... || ... }}`가 브랜치를 보고 접속할 DB를 정한다.
`github.ref_name`에는 이 실행을 깨운 브랜치 이름이 들어 있으므로, `release`면 운영 DB의 연결 문자열이, 그 밖의 브랜치(곧 `main`)면 개발 DB의 연결 문자열이 `MIGRATE_DATABASE_URL`에 담긴다.
14장에서 정한 브랜치 정책이 여기서 데이터베이스 선택으로 이어진다.
잡의 `name`에도 같은 표현식을 써서, 실행 목록에 "운영 DB 마이그레이션"인지 "dev DB 마이그레이션"인지가 드러나게 했다.
사람이 나중에 로그를 볼 때 가장 먼저 확인하고 싶은 것이 그것이기 때문이다.

셋째, `cancel-in-progress`가 `false`다.
`ci.yml`에서는 낡은 검사를 취소하는 것이 이득이었지만, 마이그레이션을 중간에 끊으면 데이터베이스가 어중간한 상태로 남는다(18장에서 볼 dirty 플래그다).
검사는 언제든 다시 돌리면 그만이지만, 바꾸는 작업은 중단 자체가 사고다.
같은 이름의 설정이라도 워크플로의 성격에 따라 값이 뒤집힌다.

넷째, 시크릿이 비어 있으면 곧바로 실패시킨다.
등록되지 않은 시크릿을 참조하면 GitHub은 오류 대신 빈 문자열을 건네주므로, 이 검사가 없으면 워크플로는 "연결 문자열이 없다"는 뜻 모를 오류를 뱉거나 엉뚱한 곳에 접속을 시도한다.
빈 값을 이른 단계에서 명시적인 실패로 바꾸는 이 세 줄은, 18장에서 볼 서버의 기동 시점 설정 검사와 같은 발상이다.

접속 정보 두 벌은 저장소 시크릿으로 등록해 둔다.
`DEV_MIGRATE_DATABASE_URL`과 `PROD_MIGRATE_DATABASE_URL`이고, 값은 각각 개발·운영 Supabase 프로젝트의 연결 문자열이다.
이 문자열이 왜 평소 API가 쓰는 것과 다른 포트여야 하는지는 18장의 몫이다.

.[용어 풀이] 저장소 시크릿(Repository Secret)
[NOTE]
====
비밀번호나 접속 키처럼 코드에 적어 둘 수 없는 값을 GitHub에 따로 맡겨 두고, 워크플로가 돌 때만 꺼내 쓰게 하는 보관함이다.
저장소 설정 화면에서 이름과 값을 등록해 두면 워크플로 파일에서는 `secrets.이름`으로만 가리키므로, 값 자체는 저장소의 어느 파일에도 남지 않는다.
한 번 등록한 값은 다시 열어 볼 수 없고 덮어쓰기만 되며, 실행 로그에 찍히더라도 GitHub이 별표로 가려 준다.
값을 읽을 수 없다는 성질이 오히려 안전장치다. 유출이 의심되면 확인할 방법을 찾는 대신 새 값으로 덮어쓰면 된다.
====

DB 비밀번호가 든 문자열을 저장소가 아니라 GitHub의 보관함에 두는 이 구성 덕분에, 개발자의 노트북에는 운영 접속 정보가 없어도 된다.
18장에서 다시 이야기하겠지만, 값이 없는 곳에서는 그 값으로 저지르는 사고도 일어나지 않는다.

== 무엇을 넣지 않았는가

이 워크플로에 없는 것도 의도의 결과다.

`staticcheck`는 넣지 않았다.
8장에서 봤듯 표준 도구가 아니라 `go install`로 설치해야 하고, 그만큼 러너 시간이 늘어난다.
검사 항목이 넓어지는 이득과 매 실행마다 붙는 설치 시간을 저울질한 결과였다.
프로젝트가 커지면 저울이 기울 것이다.

브라우저를 띄워 앱을 통째로 돌려 보는 E2E 테스트도 없다(21장에서 다룬다).
CI에서 돌리려면 테스트용 Supabase 프로젝트를 따로 만들고 그 접속 정보를 저장소 시크릿에 넣는 일이 따라온다.
지금 이 앱의 규모에서는 그 복잡도가 얻는 것보다 크다.
이 판단이 언제 뒤집히는지는 21장에서 다시 이야기한다.

한편 이 워크플로에 나중에 더할 것이 하나 예고되어 있다.
20장에서 볼 Supabase 무료 프로젝트의 일시정지를 막는 스케줄 워크플로다.
`on: schedule`로 하루 몇 번 깨어나 데이터베이스를 건드리는 요청 하나를 보내는 것이 전부인데, CI와 같은 저장소, 같은 디렉터리에 나란히 놓인다.
GitHub Actions는 검사만 돌리는 도구가 아니라 저장소에 딸린 작은 크론(cron, 정해 둔 시각마다 명령을 대신 실행해 주는 예약 장치)이기도 하다.

== 에이전트 활용 아이디어

CI 영역은 에이전트와 궁합이 특히 좋다.

CLAUDE.md에는 "CI 워크플로는 로컬 검증과 같은 명령을 돌린다"는 이 장의 원칙과 워크플로 파일의 위치를 적어 둘 만하다.
같은 명령을 공유해야 한다는 원칙을 에이전트가 알아야, 검증 명령을 바꾸는 수정에서 CI 쪽을 빠뜨리지 않는다.

CI가 깨졌을 때가 에이전트의 활약 무대다.
실패 로그를 붙여 넣고 원인을 찾아 고쳐 달라고 하면, 로그 읽기라는 지루한 일을 대신해 준다.
로그가 길면 통째로 붙이는 대신 실패한 스텝의 출력만 골라 주는 것이 컨텍스트를 아끼는 요령이다.

워크플로 추가도 좋은 위임 대상이다.
20장에서 다룰 일시정지 방지 핑이나 정기 백업처럼 정해진 시각에 도는 스케줄 워크플로는, 요구사항만 또렷하면 에이전트가 한 번에 만들어 내는 종류의 일이다.
다만 시크릿을 다루는 부분만은 값이 로그에 찍히지 않는지 사람이 정독한다.

== 정리

첫째, 훅과 에디터 태스크는 로컬 게이트라 통과하지 않고도 푸시할 수 있다.
CI는 깨끗한 러너에서 저장소만으로 앱이 빌드되고 검증되는지 확인하는 세 번째 게이트다.

둘째, `ci.yml`은 잡 하나에 gofmt·빌드·정적 분석·테스트·Vercel 호환 검사를 순서대로 담는다.
이 앱의 검증은 전부 Go 도구 체인 하나 위에서 돌기 때문인데, 서로 의존하지 않는 검증 묶음이 여럿이라면 잡을 나눠 병렬로 돌리면 된다.

셋째, CI의 `gofmt`는 `-w`가 아니라 `-l`이다.
러너의 파일을 고쳐도 저장소는 그대로이므로, 고치는 대신 어긋난 파일을 고발하고 실패한다.
같은 규칙을 게이트마다 다른 동사로 적용한다.

넷째, `api/`가 `internal/`을 import하지 않는지 확인하는 `grep` 한 줄은 로컬에서 절대 재현되지 않는 배포 실패를 막는다.
기계적으로 판정 가능한 규칙은 에이전트에게 부탁하지 말고 기계에게 맡긴다.

다섯째, Vercel의 빌드는 "만들 수 있는가"를, CI는 "옳은가"를 묻는다.
브랜치 보호 규칙으로 CI 통과를 병합 조건에 걸면, 혼자 개발하는 저장소에서도 스스로 만든 규칙에 스스로 걸리게 된다.

여섯째, GitHub Actions는 검사만 돌리는 도구가 아니다.
`migrate.yml`은 마이그레이션 SQL이 바뀐 커밋에서만 깨어나 브랜치가 정한 데이터베이스에 스키마를 적용하며(`main`은 개발 DB, `release`는 운영 DB), 접속 정보는 저장소 시크릿에 둔다.
저장소 바깥을 바꾸는 워크플로는 검사 워크플로와 설정의 방향이 달라서, 진행 중인 실행을 취소하지 않고 시크릿이 비면 즉시 실패한다.

다음 16장에서는 그 Vercel을 다룬다.
`git push` 한 번으로 Go 서버리스 함수 하나가 페이지·정적 자산·API를 도맡아 서빙하는 구조로 배포되는 과정, 그리고 방금 `grep`으로 막아 둔 그 제약이 왜 존재하는지를 코드 수준에서 확인한다.
