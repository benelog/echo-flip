# 1장 기술 선택: 요구사항에서 아키텍처까지

기술 선택에는 정답이 없고 제약 조건에 따른 트레이드오프(trade-off)만 있다.
트레이드오프란 무언가를 얻는 대신 다른 무언가를 내주는 맞바꿈이다.
공짜 점심은 없다는 말처럼 속도를 높이면 비용이 늘고 안전을 챙기면 손이 더 가는 식으로, 모든 선택에는 반대급부가 따른다.
기술 선택에서도 "무엇이 최고인가"보다 "무엇을 얻고 무엇을 잃는가"를 따지는 편이 정확하다.
그래서 이 책은 각 기술의 사용법으로 들어가기 전에, 선택의 이유를 한 장에 모아 먼저 정리한다.
Echo Flip의 스택은 Go와 Gin, 화면까지 서버에서 그리는 html/template과 htmx, 그리고 1부의 로컬 개발에서는 SQLite로 시작해 2부의 운영에서 PostgreSQL로 옮겨 가는 데이터베이스인데, 이 조합은 유행을 따라 고른 것이 아니라 요구사항에서 도출된 결과다.
그래서 이 장은 도입의 기능 요구사항에 이어, 기술 선택을 실제로 끌고 간 비기능 요구사항부터 정리한다.
이어서 데이터베이스·백엔드 언어·웹 프레임워크·화면 렌더링이라는 네 계층의 선택을 "왜 이 기술인가"와 "언제 다른 선택이 더 나은가"의 짝으로 하나씩 짚고, 마지막에 그 선택들이 모인 전체 아키텍처를 조망한다.

## 비기능 요구사항: 무료로, 혼자서, 오래 운영하기

기능 요구사항보다 기술 선택을 더 강하게 규정한 것은 비기능 요구사항이었다.
네 가지를 짚어 보겠다.

첫째, 완전 무료 인프라로 운영한다.
개인 학습 앱은 수익 모델이 없으므로 고정비가 발생하는 순간 운영을 접게 된다.
월 몇 달러라도 서버비가 나가기 시작하면, 쓰는 사람이 몇 명뿐인 앱을 언제까지 유지할지 매달 고민하게 되기 때문이다.
호스팅·데이터베이스·인증까지 전부 무료 티어 안에서 해결해야 한다.
유료 전환을 전제로 한 "체험판 무료"가 아니라, 트래픽이 적은 개인 앱이라면 영구적으로 무료인 티어를 골랐다.

::: info [용어 풀이] 호스팅(Hosting)
만든 웹 앱을 인터넷에 늘 접속 가능한 상태로 올려 두는 일, 또는 그 일을 대신해 주는 서비스다.
가게를 열려면 손님이 찾아올 자리를 빌려야 하듯, 앱도 사용자가 주소로 찾아올 수 있는 자리가 필요하다.
Echo Flip은 Vercel이라는 호스팅 서비스에 앱을 올려 둔다.
:::

::: info [용어 풀이] 무료 티어(Free Tier)
클라우드 서비스가 일정 사용량까지는 요금을 받지 않는 무료 구간이다.
트래픽이 적은 개인 앱은 이 구간 안에 머무르면 월 이용료 없이 서비스를 운영할 수 있다.
여기서 말하는 무료 티어는 "체험판"과 달리 기간이 아니라 사용량으로 한도가 정해지는 영구 무료 구간을 가리킨다.
:::

둘째, 관리할 플랫폼을 최소화한다.
혼자 운영하는 앱에서 대시보드가 세 개, 네 개로 늘어나면 어느 순간 어딘가의 설정이 만료되거나 어긋난다.
Echo Flip은 배포 대상 플랫폼을 Vercel 하나로, 데이터 계층을 Supabase 하나로 묶어 관리할 곳을 둘로 줄였다.
Vercel은 웹 앱을 올려 두는 호스팅 서비스로, 코드 저장소를 연결해 두면 빌드와 배포를 자동으로 처리해 준다.
Supabase는 PostgreSQL 데이터베이스와 로그인 인증을 대신 운영해 주는 서비스다.
둘 다 트래픽이 적은 개인 앱이라면 무료 티어 안에서 운영할 수 있다.
한발 더 나아가 개발 단계에서는 외부 서비스를 아예 쓰지 않고 내 컴퓨터 안에서만 개발하므로, 두 플랫폼이 관리 대상이 되는 것은 2부에서 앱을 공개하는 시점부터다.
Vercel은 16장에서, Supabase는 인증과 데이터베이스 연결로 나누어 17장과 18장에서 다루고, 두 서비스의 무료 한도는 20장에서 한자리에 놓고 비교한다.
화면과 API를 서로 다른 곳에 배포하는 구성(예: 화면은 정적 파일 호스팅, API는 별도 컨테이너 호스팅)도 검토했다.
하지만 이렇게 나누면 화면과 API의 인터넷 주소가 서로 달라지고, 브라우저는 주소가 다른 곳으로 가는 요청을 일단 의심하고 막는다.
결국 오리진이 갈라지면 이를 허용해 주는 CORS 설정과 두 곳의 배포 시점을 맞추는 동기화라는 운영 부담이 따라온다.
상주 서버가 꼭 필요한 앱(웹소켓, 장시간 작업)이라면 그쪽이 낫지만, 요청-응답형 API만 있는 이 앱에는 한 플랫폼 올인원이 맞았다.

::: info [용어 풀이] PostgreSQL
무료로 쓸 수 있는 오픈 소스 데이터베이스다.
데이터를 스프레드시트 비슷한 표(테이블) 형태로 저장하고, 표끼리 연결해 조회할 수 있어 관계형 데이터베이스로 분류된다.
수십 년간 다듬어져 개인 프로젝트부터 대기업 서비스까지 두루 쓰이며, Echo Flip의 테이블 설계는 5장에서 다룬다.
:::

::: info [용어 풀이] 프런트엔드와 백엔드
프런트엔드는 사용자의 화면에서 눈에 보이는 부분, 백엔드는 그 뒤에서 데이터를 처리하고 저장하는 부분이다.
식당에 비유하면 손님을 맞는 홀이 프런트엔드, 주문을 받아 요리하는 주방이 백엔드다.
Echo Flip에서는 브라우저에 표시되는 화면이 프런트엔드이고, 그 화면(HTML)을 만들어 보내고 데이터베이스와 대화하는 Go 서버가 백엔드다.
:::

셋째, PWA(Progressive Web App)로 웹과 Android를 모두 커버한다.
네이티브 앱을 만들면 스토어 등록비와 심사, 별도 코드베이스라는 비용이 생긴다.
암기 카드 앱은 카메라나 푸시 같은 네이티브 기능이 필수가 아니므로, 웹 앱에 매니페스트를 더해 홈 화면에 설치하는 것으로 충분하다.
매니페스트의 `display: "standalone"` 선언 덕분에 Android Chrome에서 "홈 화면에 추가"를 누르면 주소창 없는 독립 실행형 앱처럼 동작하고, iOS Safari에서도 같은 방식으로 설치할 수 있다.
이 매니페스트 파일(`internal/web/static/manifest.webmanifest`)은 19장에서 다룬다.

::: info [용어 풀이] PWA(Progressive Web App)
설치해 쓰는 앱처럼 동작하는 웹 앱이다.
브라우저로 여는 웹사이트지만, 홈 화면에 추가하면 주소창 없이 독립 실행형 앱처럼 열려 앱 스토어를 거치지 않고도 설치한 앱 같은 경험을 준다.
덕분에 웹과 모바일을 하나의 코드로 함께 커버할 수 있다.
:::

넷째, 서버리스(Serverless) 콜드스타트(Cold Start)를 최소화한다.
무료 티어의 서버리스 함수는 요청이 없으면 잠들고, 첫 요청에서 깨어나는 시간이 사용자 체감 지연이 된다.
깨어나는 속도는 결국 프로그램의 크기가 얼마나 작고 실행 준비가 얼마나 간단한가에 달렸다.
즉 콜드스타트는 런타임의 기동 속도와 바이너리 크기에 좌우되므로, 백엔드 언어로 컴파일 언어인 Go를 선택했다.
실행에 필요한 것이 파일 하나에 담기는 단일 바이너리로 빌드되고 기동이 수십 밀리초 수준이라 서버리스와 궁합이 좋다.
Node.js나 Python 같은 인터프리터 언어로 백엔드를 만드는 선택지도 있었지만, 이 프로젝트는 콜드스타트와 견고한 핵심 알고리즘 구현을 우선해 Go를 택했다.
이 트레이드오프는 이 장의 "왜 Go인가" 절에서 본격적으로 다룬다.

::: info [용어 풀이] 서버리스(Serverless)
서버를 늘 켜 두지 않고, 요청이 올 때만 잠깐 실행됐다가 사라지는 방식이다.
"서버가 없다"는 뜻이 아니라, 서버를 직접 마련하고 지키는 일을 클라우드에 맡긴다는 뜻이다.
요청이 없을 때는 아무것도 돌지 않으므로, 트래픽이 적은 앱은 비용을 크게 아낄 수 있다.
:::

::: info [용어 풀이] 콜드스타트(Cold Start)
잠들어 있던 서버리스 함수가 첫 요청에 깨어나 준비를 마칠 때까지 걸리는 지연이다.
한동안 덮어 둔 노트북을 열 때 잠깐 뜸을 들이는 것과 비슷해서, 이 시간이 길면 사용자가 첫 화면을 그만큼 기다리게 된다.
그래서 기동이 빠른 언어를 고르면 이 지연을 줄일 수 있다.
:::

같은 이유로 지연 시간의 다른 축인 API↔DB 거리도 관리한다.
Vercel 함수 리전을 iad1(미국 동부)로 고정하고 Supabase 프로젝트도 같은 East US 리전에 만들어, 함수와 데이터베이스가 같은 지역 안에서 통신하게 했다.
사용자가 한국에 있어도 브라우저↔서버 왕복은 한 번이지만 API↔DB 왕복은 요청당 여러 번일 수 있으므로, DB를 함수 옆에 두는 쪽이 전체 지연에 유리하다.

## 네 계층의 선택에 공통으로 적용한 기준

이 장이 다루는 선택은 데이터베이스, 백엔드 언어, 웹 프레임워크, 화면 렌더링 방식이라는 네 계층이다.
개별 기술을 살펴보기 전에, 모든 선택에 반복해서 적용한 잣대를 먼저 세워 두자.
백엔드 언어를 고를 때도, 화면 렌더링 방식을 고를 때도, 데이터베이스를 고를 때도 결국 같은 네 가지 질문을 던졌다.

첫째, 무료 티어(free tier)로 운영할 수 있는가.
앞 절에서 본 대로 수익 모델이 없는 개인 앱은 고정비가 생기는 순간 운영이 흔들린다.
이 기준은 인프라만이 아니라 언어와 프레임워크 선택까지 거슬러 올라간다.
예를 들어 메모리를 많이 쓰는 런타임은 무료 티어의 함수 메모리 한도와 충돌하고, 상주 서버를 전제하는 프레임워크는 무료 서버리스 모델에 얹기 어렵다.

::: info [용어 풀이] 런타임(Runtime)
프로그램이 돌아가는 동안 그 코드를 실제로 실행해 주는 토대를 말한다.
언어로 쓴 코드는 그 자체로 실행되지 않고, 자바스크립트라면 브라우저나 Node.js, 자바라면 JVM처럼 코드를 받아 돌려 주는 실행 환경이 필요한데 이것이 런타임이다.
이 토대가 크면 메모리를 많이 차지하고 켜지는 데도 시간이 더 걸린다.
:::

둘째, 관리할 것이 적은가.
혼자 운영하는 앱에서 대시보드와 설정할 곳이 늘어나면 어느 순간 어딘가가 만료되거나 어긋난다.
의존성이 적은 언어, 배포 산출물이 단순한 빌드, 스키마와 제약을 한곳에서 지켜 주는 데이터베이스처럼, 신경 쓸 표면적이 작은 쪽을 우선했다.

::: info [용어 풀이] 의존성(Dependency)
내 코드가 제대로 돌아가려고 끌어다 쓰는, 남이 만든 코드 꾸러미를 말한다.
필요한 부품을 직접 다 짜는 대신 검증된 것을 가져다 쓰는 셈이라 편하지만, 부품이 많아질수록 각각의 업데이트와 보안 문제를 따라가야 해서 신경 쓸 일도 늘어난다.
그래서 의존성이 적을수록 혼자 오래 관리하기 쉽다.
:::

셋째, AI 에이전트가 잘 다루는 성숙한 생태계인가.
Echo Flip의 코드 상당 부분은 2장에서 다룰 Claude Code가 작성했다.
대규모 언어 모델(LLM, Large Language Model)은 학습 데이터에 많이 등장한 기술일수록 정확한 코드를 생성하므로, 사용자 층이 넓고 관용구가 안정된 기술이 곧 에이전트 협업의 품질로 이어진다.
여기에 더해, 에이전트가 스스로 검증할 수 있도록 컴파일·타입 검사·테스트 같은 기계적 검증 신호가 빠르고 결정적으로 도는 기술이 유리하다.
여기서 결정적이라는 말은 같은 코드에 같은 검사를 돌리면 언제나 같은 결과가 나온다는 뜻으로, 판정이 흔들리지 않으니 에이전트가 몇 번이고 돌려 보며 믿을 수 있는 신호가 된다.

넷째, 서버리스에 맞는가.
Echo Flip의 백엔드는 운영 환경에서 요청이 올 때만 깨어나는 서버리스 함수로 동작한다.
기동이 빠르고 메모리 사용이 적은 런타임, 함수가 아무것도 기억하지 않아도 되는 무상태 설계, 함수 인스턴스가 여럿 떠도 문제없는 외부 데이터베이스가 이 모델의 전제 조건이다.

이 네 기준은 서로 독립적이지 않다.
서버리스에 맞아야 무료 티어가 성립하고, 관리할 것이 적어야 혼자서 오래 운영할 수 있으며, 성숙한 생태계여야 에이전트가 만든 코드를 믿고 쌓아 갈 수 있다.
한편 이 기준의 최종 수혜자인 인프라 선택, 즉 Vercel과 Supabase를 고른 이유는 이 장에서 상세히 다루지 않고 16장과 17·18장에서 각각 다룬다.
이제 스택의 가장 아래층인 데이터베이스부터 시작해, 데이터에서 화면 쪽으로 한 층씩 올라가자.

## 왜 관계형 데이터베이스인가: SQLite로 시작해 PostgreSQL로 옮긴다

스택의 가장 아래층은 데이터가 머무는 곳이다.
여기서 정한 데이터의 모양이 그 위에 얹힐 백엔드와 화면의 생김새를 상당 부분 결정하므로 이 선택을 먼저 다룬다.
Echo Flip의 도메인을 한 문장으로 요약하면 "사용자가 덱을 만들고, 덱에 카드를 담고, 카드를 학습한 기록이 쌓인다"이다.
사용자-덱-카드-학습 기록이라는 관계가 도메인의 뼈대이고, 이 관계 자체가 데이터의 핵심 정보다.
이런 도메인에는 관계를 일급 개념으로 다루는 관계형 데이터베이스(RDB, Relational Database)가 자연스럽다.

::: info [용어 풀이] 관계형 데이터베이스
데이터를 표(테이블) 형태로 저장하고, 표와 표 사이의 연결 관계를 핵심으로 다루는 데이터베이스다.
엑셀 시트 여러 장이 서로를 가리키며 이어져 있는 모습에 가깝고, "이 카드는 저 덱에 속한다" 같은 이어짐을 데이터 자체가 붙들고 있다.
사용자·덱·카드·학습 기록처럼 서로 얽힌 정보를 다룰 때 자연스럽게 들어맞는다.
:::

관계형 데이터베이스가 주는 가치는 크게 두 가지다.

첫째, 제약(constraint)으로 데이터 정합성을 데이터베이스가 지켜준다.
"카드는 반드시 어떤 덱에 속한다", "SRS 상태는 카드당 정확히 하나다" 같은 규칙을 애플리케이션 코드가 아니라 외래 키(foreign key)와 기본 키(primary key)로 선언하면, 버그가 있는 코드가 끼어들어도 규칙이 깨진 데이터는 저장되지 않는다.
코드는 고치면 되지만 한번 오염된 데이터는 복구가 어렵다는 점에서, 마지막 방어선을 DB에 두는 가치는 크다.

둘째, 트랜잭션(transaction)이다.
복습 한 번을 기록할 때는 로그 삽입과 SRS 상태 갱신이 함께 일어나야 하는데, 트랜잭션은 이런 다중 쓰기를 "모두 성공하거나 모두 없던 일이 되거나"로 보장한다.
5장에서 보겠지만 Echo Flip의 store 계층은 이 성질에 전적으로 기대고 있다.

관계형이라는 방향은 이렇게 정해졌다.
그런데 Echo Flip은 관계형 데이터베이스를 하나가 아니라 둘 쓴다.
1부의 로컬 개발에서는 SQLite를, 2부의 운영에서는 PostgreSQL을 쓴다.
왜 하나로 통일하지 않았는지를 이어서 살펴보자.

### 1부의 선택: 파일 하나로 시작하는 SQLite

SQLite는 파일 하나로 동작하는 관계형 데이터베이스다.
별도의 데이터베이스 서버를 설치해 띄우는 대신 앱이 라이브러리처럼 품고 실행하며, 데이터는 내 컴퓨터의 파일 하나에 쌓인다.
설치할 것도, 가입할 계정도, 연결할 네트워크도 없으니 운영 부담이 0에 가깝다.
로컬 도구나 단일 서버 앱이라면 가장 먼저 검토할 만한 선택지이고, 1부에서 내 컴퓨터 안에 앱을 완성하는 동안의 데이터베이스로도 정확히 들어맞는다.
덕분에 독자는 데이터베이스 계정 하나 만들지 않고 4장의 SQL 실습부터 13장의 로컬 완성까지를 내 컴퓨터만으로 따라갈 수 있다.

### 2부에서 PostgreSQL로 옮겨 가는 이유

그러나 앱을 세상에 공개하면 사정이 달라진다.
Echo Flip의 운영 백엔드는 Vercel 서버리스 함수로 동작한다(16장에서 자세히 다룬다).
서버리스에서는 요청이 몰리면 같은 함수가 여러 개 복제되어 뜨고, 각 함수가 가진 저장 공간은 함수가 꺼질 때 내용까지 함께 사라진다.
그래서 파일 하나에 데이터를 쌓는 SQLite는 모두가 함께 보는 저장소가 될 수 없고, 네트워크 너머의 독립된 데이터베이스 서버가 필수가 된다.
운영 데이터베이스가 PostgreSQL인 이유다.

다행히 이 이사의 비용은 작다.
SQLite와 PostgreSQL은 같은 관계형 모델을 쓰고 SQL 문법도 거의 같아서, 1부에서 익힌 테이블·키·쿼리 지식이 그대로 이월된다.
코드 쪽도 마찬가지다.
데이터베이스와 대화하는 코드를 하나의 창구(인터페이스) 뒤로 모아 두고 SQLite용 구현과 PostgreSQL용 구현이 같은 창구를 함께 만족하게 해서, 두 데이터베이스의 교체는 코드 수정이 아니라 배포 대상의 선택이 된다.
이 구조의 코드 상세는 18장에서 다룬다.

### 대안 검토: 문서형 NoSQL

관계형이라는 방향 자체의 대안으로는 문서형 NoSQL(MongoDB, Firestore 등)이 있다.
문서 DB는 스키마를 미리 고정하기 어렵거나, 하나의 문서(애그리거트) 단위로만 읽고 쓰는 패턴이 지배적일 때 강점이 있다.
카드 앱이라도 "덱 문서 안에 카드 배열을 통째로 저장하고 통째로 읽는" 구조라면 문서 DB가 더 단순할 수 있다.
그러나 Echo Flip은 카드를 덱 단위가 아니라 개별 단위로 다룬다.
"오늘 복습할 카드"는 덱 경계를 넘어 due 시각으로 모으고, 스마트 덱은 오답률·태그 같은 조건으로 카드를 횡단 조회한다.
이런 조회는 조인과 인덱스, 집계가 자연스러운 관계형 모델의 홈그라운드이고, 문서 DB에서 구현하려면 비정규화와 수동 동기화가 늘어난다.

흥미로운 점은 PostgreSQL 안에서도 문서적 유연함이 필요한 자리에는 jsonb를 썼다는 것이다.
사용자 설정(`profiles.settings`)과 스마트 덱 규칙(`smart_decks.rule`)처럼 구조가 자주 바뀔 수 있는 데이터는 jsonb 컬럼에 담아, 관계가 중요한 곳은 테이블로, 유동적인 곳은 문서로 나누는 절충이 가능하다.

### 운영 데이터베이스로 PostgreSQL을 고른 이유

네트워크 너머에 둘 관계형 데이터베이스 서버는 PostgreSQL 말고도 여럿이지만, 운영용으로 PostgreSQL을 고른 데는 기능과 인프라 양쪽의 이유가 있다.
기능 면에서는 배열 타입과 GIN 인덱스(태그 검색), jsonb, 부분 인덱스(partial index), enum 타입, `generated always as identity`처럼 이 앱이 실제로 쓰는 기능들이 판단 기준이었다.
이름이 낯설어도 지금 다 알 필요는 없고, 태그 검색을 빠르게 하는 색인이나 유연한 JSON 저장처럼 이 앱의 기능 하나하나와 맞닿은 도구라는 것만 알아 두면 된다.
인프라 면에서는 Supabase 무료 티어와 맞물린다.
Echo Flip은 "무료 인프라로 운영한다"는 비기능 요구사항이 있고, Supabase는 관리형 PostgreSQL과 인증을 무료 티어로 함께 제공하므로 데이터베이스 선택이 곧 호스팅 선택이기도 했다.
Supabase 연결 방식과 풀러(pooler) 상세는 18장에서 다루고, 이 기능들이 5장의 SQLite 설계와 어떻게 대응하는지도 같은 장에서 확인한다.

## 왜 Go인가

데이터가 머물 곳을 정했으니 한 층 올라가, 그 데이터를 읽고 쓰는 백엔드 언어를 고를 차례다.
Echo Flip의 제약 조건은 무료 인프라, 최소한의 운영 부담, 그리고 개발의 상당 부분을 AI 에이전트에게 맡기는 개발 방식이다.
이 제약들을 하나씩 Go의 특성과 대조해 보겠다.

### 서버리스 함수에 맞는 언어의 조건

서버리스 함수는 요청이 없으면 잠들어 있다가, 요청이 오면 런타임을 띄우고 코드를 로드한 뒤에야 응답하는 모델이다.
이 준비 시간이 앞서 말한 콜드스타트다.
상주 서버라면 부팅이 하루에 한 번이므로 몇 초가 걸려도 문제가 아니지만, 서버리스에서는 콜드스타트가 사용자의 첫 화면 로딩에 그대로 얹힌다.

Go는 이 모델에 잘 맞는 특성을 여럿 갖고 있다.

첫째, 컴파일 결과가 단일 정적 바이너리(Static Binary)다.
가상 머신도, 인터프리터도, `node_modules` 같은 의존성 디렉터리도 함께 배포할 필요가 없다.
바이너리 하나를 메모리에 올리는 것으로 준비가 끝나므로 콜드스타트가 수십 밀리초 수준으로 짧다.

::: info [용어 풀이] 컴파일 언어와 인터프리터 언어
컴파일 언어는 코드를 미리 기계가 바로 읽는 형태로 통째 번역해 두고, 그 번역 결과물만 실행하는 언어다.
인터프리터 언어는 번역기를 옆에 두고 실행하는 순간 한 줄씩 통역해 가며 돌리는 언어라, 실행할 때 통역기(인터프리터)가 늘 함께 있어야 한다.
미리 번역해 둔 쪽은 실행이 빠르고 준비물이 적은 대신 고칠 때마다 다시 번역해야 하고, 그때그때 통역하는 쪽은 바로 실행해 보기 편한 대신 실행에 필요한 준비물이 더 많다.
Go는 앞쪽, 자바스크립트나 파이썬은 뒤쪽에 가깝다.
:::

둘째, 메모리 사용량이 적다.
서버리스 플랫폼은 대체로 함수의 메모리 상한으로 과금하거나 무료 한도를 정한다.
기본 수백 MB를 점유하는 JVM 같은 런타임과 달리 Go 프로세스는 수십 MB 안에서 동작하는 경우가 많아, 무료 티어의 한도 안에 여유 있게 들어간다.
수치로 감을 잡아 두자.
Echo Flip 규모의 Go 웹 서버는 유휴 상태 기준 10~20MB 수준의 메모리로 돈다.
같은 일을 하는 서버를 Node.js나 Python으로 만들면 인터프리터와 프레임워크를 띄우는 것만으로 대략 50~100MB에서 시작하는 경우가 많아, 앱 코드가 한 줄 돌기 전에 Go가 다 쓰는 양의 몇 배를 런타임이 먼저 점유한다.
정확한 값은 앱과 측정 방식에 따라 다르니 절대값보다 "몇 배"라는 배수를 기억해 두면 되는데, 이 배수가 무료 티어에서는 그대로 여유분의 차이가 된다.
무료 호스팅이 내주는 인스턴스는 메모리 256~512MB짜리가 흔해서, 기본 점유가 큰 런타임은 앱을 얹기 전에 그 절반을 쓰고 시작한다.

셋째, 컴파일이 빠르다.
Echo Flip의 백엔드 전체를 `go build ./...`로 빌드해도 몇 초면 끝난다.
빌드가 빠르면 배포가 빨라질 뿐 아니라, 8장에서 다룰 "수정 → 검증" 루프의 회전 속도가 올라간다.

넷째, HTTP 서버(`net/http`), JSON 직렬화(`encoding/json`), 시간 처리(`time`)가 모두 표준 라이브러리에 들어 있어 의존성을 최소로 유지할 수 있다.

### 단순한 문법이 AI 협업에 주는 이점

Go는 의도적으로 작은 언어다.
키워드가 25개뿐이고, 같은 일을 하는 코드는 누가 짜도 대체로 같은 모양으로 수렴한다.
상속이 없고, 예외(Exception) 대신 에러 값을 반환하며, 8장에서 볼 `gofmt`가 코드의 겉모양을 하나로 고정해 포맷 스타일을 논쟁거리에서 빼낸다.
설정 옵션이 없어 고를 것이 없기 때문인데, 대신 어떤 대가가 따르는지도 8장에서 함께 짚는다.

이 단순함은 사람에게도 이점이지만, AI 에이전트와 협업할 때 더 크게 작동한다.

첫째, 생성된 코드를 검토하기 쉽다.
에이전트가 작성한 코드라도 결국 사람이 읽고 책임져야 하는데, 숨은 제어 흐름이 없는 Go 코드는 diff만 보고도 동작을 따라갈 수 있다.
에러가 어디서 발생해 어디로 전파되는지가 `if err != nil` 분기로 코드에 전부 드러나기 때문이다.

둘째, 검증 루프가 빠르고 결정적이다.
`go build`, `go vet`, `go test`가 몇 초 안에 끝나고 결과가 명확한 성공/실패로 떨어지므로, 에이전트가 수정할 때마다 돌려서 스스로 피드백을 얻기 좋다.

셋째, 학습 데이터가 균질하다.
Go 커뮤니티는 "관용적인(idiomatic) Go"라는 단일 스타일을 강하게 공유하므로, 에이전트가 생성한 코드도 프로젝트의 기존 코드와 이질감 없이 섞인다.

물론 단순함의 대가도 있다.
에러 처리가 장황하고, 고차 함수 중심의 표현력 높은 추상화는 다른 언어보다 불편하며, Rails나 Spring 같은 풀스택 프레임워크의 생산성을 기대하기 어렵다.
Echo Flip처럼 화면이 십여 장이고 도메인이 단순한 앱에서는 이 대가가 작지만, 규모가 커지면 계산이 달라진다.
예를 들어 회원 관리와 게시판, 관리자 화면 수십 장을 갖춘 사내 시스템을 빠르게 세워야 한다면, 화면과 데이터 처리의 골격을 통째로 제공하는 Rails 쪽이 몇 배 빠르다.

### 정적 타입이 잡아 주는 실수

앞서 둘째로 말한 검증 신호의 바탕에는 Go가 정적 타입(static typing) 언어라는 사실이 있다.
Echo Flip에서 카드의 뜻을 담는 필드 이름은 `Meaning`인데, 코드 어딘가에서 `Meanig`처럼 오타를 내면 실행해 보기도 전에 `go build`가 그 줄을 짚어 준다.
동적 타입 언어였다면 잘못된 이름으로 꺼낸 값이 조용히 빈 값이 되고, 사용자가 "카드 뜻이 안 보인다"고 알려 주기 전까지 버그의 존재조차 모를 수 있다.

::: info [용어 풀이] 정적 타입과 동적 타입
값에 "숫자다", "문자다" 같은 종류(타입)를 언제 확인하느냐의 차이다.
정적 타입은 코드를 실행하기 전 미리 검사하는 단계에서 타입이 맞는지 따져, 어긋나면 그 자리에서 알려 준다.
동적 타입은 그런 사전 검사 없이 일단 실행하고, 잘못된 값이 실제로 문제를 일으키는 순간에야 오류가 드러난다.
빨리 쓰기엔 동적 타입이 편하지만, 오래 키울 앱에서는 실수를 일찍 잡아 주는 정적 타입이 유리하다.
:::

이 앱은 실제로 필드 이름을 한 차례 바꾼 이력이 있다.
카드의 앞뒤 면을 `front`/`back`으로 부르다가 학습 방향(단어를 보고 뜻 맞히기, 뜻을 보고 단어 맞히기)을 도입하면서 `text`/`meaning`으로 개명했다.
동적 타입 언어에서 이런 전면 개명은 문자열 검색으로 사용처를 찾고 놓친 곳은 실행 중 버그로 발견하는, 공포스러운 작업이다.
정적 타입 언어에서는 정의 한 곳을 바꾸면 옛 이름을 참조하는 모든 코드가 그 자리에서 컴파일 오류로 표시되고, 오류 목록이 곧 수정해야 할 작업 목록이 된다.
에이전트에게 개명을 맡겨도 컴파일러가 빠뜨린 곳을 전부 잡아 주니, 사람은 결과 diff만 검토하면 된다.

### Vercel이 공식 지원하는 런타임

Vercel은 Node.js 외에 Go를 공식 런타임으로 지원한다.
`api/` 디렉터리에 표준 `net/http` 시그니처를 따르는 핸들러 함수를 두면 Vercel이 알아서 컴파일해 서버리스 함수로 배포한다.
덕분에 화면 렌더링과 API를 모두 담은 Go 함수 하나를 무료 티어에 올리는 것으로 배포가 끝난다.
진입점 `api/index.go`의 코드는 9장에서, 배포 구조의 상세는 16장에서 다룬다.

### 대안 검토: Node.js, Python, JVM

Go가 유일한 답은 아니므로, 대안 세 가지를 놓고 언제 그쪽이 더 나은 선택인지 짚어 보자.

**Node.js/TypeScript**는 가장 강력한 대안이다.
Vercel의 1순위 런타임이고, 잠시 뒤 살펴볼 화면 렌더링에서 React 같은 브라우저 렌더링 프레임워크를 택한다면 브라우저 쪽 코드가 어차피 자바스크립트 계열이므로 백엔드까지 같은 언어로 통일할 수 있다.
이미 TypeScript에 능숙하거나 팀에 자바스크립트 경험이 쌓여 있다면 Node.js가 더 나은 선택이다.
반면 내주는 것도 있다.
TypeScript의 타입은 컴파일 시점에 지워져 코드가 실제로 도는 동안에는 값을 지켜 주지 못하고, 끌어 쓰는 의존성이 깊게 쌓이기 쉬우며, 같은 작업 기준 메모리 사용량이 Go보다 크다.

**Python**은 데이터 생태계가 강점이다.
영어 학습 앱이라는 도메인 특성상 형태소 분석, 발음 평가, 문장 난이도 채점 같은 자연어 처리(NLP) 기능을 붙일 계획이라면, 필요한 라이브러리가 대부분 Python 우선으로 나오므로 Python이 더 나은 선택이다.
반면 배포 아티팩트가 크고(인터프리터 + 의존성), 타입 힌트가 선택 사항이라 정적 검증 신호가 약하다.
Echo Flip은 사전 조회를 외부 API에 위임하고 SRS 계산만 직접 하므로 이 강점을 쓸 일이 없었다.

**JVM**(Java/Kotlin)은 성숙도가 강점이다.
Spring 생태계의 축적된 패턴과 대규모 팀을 위한 규율은 다른 스택이 따라오기 어렵다.
상주 서버를 운영할 수 있고, 도메인이 복잡하며, 조직의 표준 스택이 JVM이라면 굳이 벗어날 이유가 없다.
반면 서버리스에서는 초 단위 콜드스타트와 큰 메모리 기본값이 무료 티어와 정면으로 충돌한다.
GraalVM 네이티브 이미지로 완화할 수 있지만 빌드 복잡도라는 새 비용이 생긴다.

정리하면, "무료 서버리스 + 1인 유지보수 + AI 에이전트 주도 개발"이라는 제약 조합에서 Go가 가장 마찰이 적었고, 제약이 다르면 답도 달라진다.
Go 문법과 테스트, 품질 도구는 6장부터 8장까지 실제 코드로 익힌다.

## 왜 Gin인가

언어를 Go로 정했다면 다음 질문은 웹 프레임워크다.
흥미롭게도 Go에서는 "프레임워크가 정말 필요한가"부터가 진지한 질문이다.

::: info [용어 풀이] 프레임워크와 라이브러리
둘 다 남이 만들어 둔 코드 묶음이지만 주도권이 반대다.
라이브러리는 내가 필요할 때 불러다 쓰는 연장이고, 프레임워크는 정해진 골격 안에 내 코드를 끼워 넣는 틀이다.
연장은 내가 언제 쓸지 고르지만, 틀은 "여기에 네 코드를 넣어라"라고 전체 흐름을 미리 정해 두고 필요할 때 내 코드를 불러 준다.
그래서 프레임워크는 반복 작업을 대신 떠맡아 주는 대신 그 틀이 정한 방식을 따라야 한다.
:::

### net/http만으로도 서버는 만들 수 있다

Go의 큰 매력 중 하나는 표준 라이브러리 `net/http`만으로도 실용적인 HTTP 서버를 만들 수 있다는 점이다.
특히 Go 1.22부터는 표준 라우터인 `http.ServeMux`가 크게 개선되어, HTTP 메서드 매칭과 경로 파라미터(path parameter)를 지원한다.
다음 코드는 "/api/decks/ 뒤에 덱 이름이 붙은 주소로 GET 요청이 오면 이 함수를 실행하라"고 라우터에 등록하고, 주소에 담긴 덱 이름(slug)을 꺼내는 예다.

```go
mux := http.NewServeMux()
mux.HandleFunc("GET /api/decks/{slug}", func(w http.ResponseWriter, r *http.Request) {
	slug := r.PathValue("slug")
	// ...
})
```

라우팅만 놓고 보면 이 정도로 충분한 경우가 많다.
그렇다면 프레임워크는 무엇을 더해 주는가.
Echo Flip의 코드를 기준으로 보면 크게 세 가지다.

첫째, 라우트 그룹(route group)이다.
"이 경로 아래의 모든 API는 인증이 필요하다" 같은 규칙을 그룹 단위로 한 번만 선언할 수 있다.
표준 라이브러리에서는 핸들러마다 인증 검사를 감싸거나 래퍼 함수를 직접 만들어야 한다.

둘째, JSON 바인딩(binding)이다.
브라우저가 보내는 요청에는 "이 덱의 이름을 이렇게 바꿔 달라" 같은 데이터가 JSON이라는 문자 형식으로 담겨 오는데, 코드가 다루려면 이를 Go의 값 묶음(구조체)으로 옮겨 담아야 한다.
Gin에는 요청 본문의 JSON을 구조체로 역직렬화하고 실패를 일관되게 처리하는 `ShouldBindJSON` 같은 도우미가 있다.
표준 라이브러리로도 `json.NewDecoder(r.Body).Decode(&body)`를 쓰면 되지만, Content-Type 확인이나 검증 태그 연동 같은 잡무가 핸들러마다 반복된다.

셋째, 미들웨어 체인(middleware chain)이다.
패닉 복구, CORS, 인증처럼 모든 요청에 공통으로 적용할 처리를 등록 순서대로 실행하는 구조가 프레임워크에 내장되어 있다.
표준 라이브러리에서는 `http.Handler`를 겹겹이 감싸는 데코레이터 패턴을 손으로 조립해야 한다.

어느 것도 표준 라이브러리로 못 할 일은 아니지만, Echo Flip처럼 30개 가까운 엔드포인트에 인증·CORS·에러 응답 규약이 걸리는 API라면 이런 반복 코드를 프레임워크에 위임하는 편이 코드량과 실수 가능성을 함께 줄여 준다.

### 대안 비교: net/http, chi, Echo

Gin이 유일한 선택지는 아니다.
대안이 더 나은 경우를 함께 짚어 보자.

**net/http(1.22+ ServeMux)** 는 의존성이 전혀 없다는 것이 최대 장점이다.
엔드포인트가 몇 개 안 되고 미들웨어도 한두 개뿐인 소형 서비스, 혹은 외부 의존성을 최소화해야 하는 라이브러리성 도구라면 버전 업그레이드 부담도 학습 비용도 없는 표준 라이브러리만 쓰는 편이 낫다.

**chi**는 표준 `http.Handler` 인터페이스를 그대로 유지하는 얇은 라우터로, 표준 라이브러리 생태계의 미들웨어와 예제를 변환 없이 재사용할 수 있다.
"프레임워크에 종속되고 싶지 않다"는 요구가 강한 팀, 혹은 기존 net/http 코드에 라우팅만 보강하고 싶은 프로젝트라면 chi가 더 나은 선택이다.
대신 JSON 바인딩과 응답 도우미는 직접 갖춰야 한다.

**Echo**는 기능 범위가 Gin과 가장 비슷한 프레임워크다.
가장 큰 차이는 핸들러가 `func(c echo.Context) error`처럼 에러를 반환하고, 중앙의 에러 핸들러가 이를 HTTP 응답으로 변환한다는 점이다.
에러 처리를 한곳에 모으는 이 방식을 선호한다면 Echo가 더 맞을 수 있다.
Gin의 핸들러는 반환값이 없어서, 9장에서 보겠지만 Echo Flip은 `fail` 같은 도우미 함수로 같은 효과를 낸다.

성능은 이 규모의 앱에서 의미 있는 변별 요소가 아니며, 네 후보 모두 개인용 암기 카드 앱의 트래픽을 감당하고도 남는다.
Echo Flip이 Gin을 고른 결정적 이유는 다른 데 있다.

### 사용자 층이 넓다는 것의 새로운 의미

Gin은 Go 웹 프레임워크 중 가장 널리 쓰이고, GitHub 스타 수도 압도적이다.
사용자 층이 넓다는 것은 전통적으로 문서·튜토리얼·Stack Overflow 답변이 많다는 뜻이었다.
그런데 AI 에이전트와 함께 개발하는 시대에는 여기에 한 겹의 의미가 더해진다.

Echo Flip의 백엔드 코드 대부분은 2장에서 다룰 Claude Code로 작성했다.
LLM은 학습 데이터에 많이 등장한 라이브러리일수록 정확한 코드를 생성한다.
Gin처럼 공개 코드와 질의응답이 풍부한 프레임워크는 에이전트가 API 시그니처를 잘못 기억하거나 존재하지 않는 메서드를 지어내는 일이 드물고, 소수만 쓰는 프레임워크를 고르면 생성된 코드를 사람이 일일이 교정하는 비용이 커진다.

"많이 쓰이는 기술을 고르면 검색이 잘 된다"는 오래된 조언이 "많이 쓰이는 기술을 고르면 AI가 잘 짠다"로 확장됐다.
이 기준은 앞서 세운 세 번째 잣대 그대로이며, 잠시 뒤 화면 렌더링 방식을 고를 때도 같은 잣대가 다시 등장한다.
Gin의 라우팅과 미들웨어, 핸들러 작성법은 9장에서 실제 코드로 살펴본다.

## 왜 서버 렌더링인가: html/template과 htmx

백엔드 언어와 프레임워크를 정했으니 이제 마지막 층, 사용자가 직접 보는 화면 차례다.
화면에서는 "어느 프레임워크가 좋은가"보다 앞선 갈림길이 있다.
화면을 어디서 그릴 것인가, 즉 브라우저인가 서버인가라는 질문이다.

### 화면을 그리는 두 갈래

첫째는 브라우저에서 그리는 길이다.
서버는 데이터만 JSON으로 내려 주고, 브라우저 안에서 자바스크립트가 그 데이터를 받아 화면을 조립한다.
이 방식으로 만든 앱을 SPA(Single Page Application)라 부르고, React·Vue·Svelte가 이 갈래의 대표 프레임워크다.
첫 페이지를 받아온 뒤에는 화면 전환도 데이터 갱신도 브라우저 안에서 일어나므로, 설치형 앱처럼 매끄러운 사용감을 만들기 좋다.
대신 프런트엔드가 백엔드와 별개의 독립된 프로그램이 된다.
브라우저가 직접 실행하는 언어는 사실상 자바스크립트뿐이므로 이 프로그램은 자바스크립트로 짜야 하고, 규모가 커지면 타입 검사를 위한 TypeScript, 여러 파일을 하나로 묶는 번들러(bundler), 라이브러리를 내려받는 npm 생태계가 따라온다.

둘째는 서버에서 그리는 길이다.
서버가 데이터를 채워 넣은 완성된 HTML 문서를 보내고, 브라우저는 받은 문서를 그대로 표시한다.
화면 전환은 링크 클릭과 폼 제출이라는 웹의 기본 동작으로 일어나고, 즉각 반응해야 하는 곳만 부분 갱신 기법으로 다듬는다.
Rails와 Django가 오래 걸어온 길이고, Go에서는 표준 라이브러리에 든 html/template이 이 역할을 맡는다.

Echo Flip은 둘째 길을 택했다.
서버 렌더링이 항상 옳아서가 아니라, 이 책의 제약을 대입했을 때 저울이 그쪽으로 기울었기 때문이다.
공정한 비교를 위해 SPA가 더 나은 경우부터 짚는다.

### SPA가 더 나은 경우

첫째, 오프라인 우선(offline-first) 앱이다.
비행기 모드에서도 동작해야 하는 앱은 데이터와 화면을 조립하는 능력이 모두 브라우저 쪽에 있어야 하므로, 서버가 있어야 화면이 나오는 서버 렌더링으로는 성립하지 않는다.
둘째, 고도로 상호작용적인 편집기다.
문서 편집기나 드로잉 도구, 스프레드시트처럼 키 입력 하나하나에 화면이 반응해야 하는 제품은 상태를 브라우저에 두는 SPA 모델이 자연스럽다.
셋째, 팀에 프런트엔드 전문가가 있는 경우다.
백엔드는 JSON API에 집중하고 화면은 프런트엔드 담당이 전담하는 분업은 SPA가 만들어 주는 경계선 위에서 잘 돌아간다.
넷째, 구인 시장이다.
React 경험자는 어디서든 구할 수 있지만 htmx를 내건 채용 공고는 드물어서, 사람을 뽑아 팀을 키울 조직이라면 주류 기술을 따르는 편이 안전하다.

### 이 책의 기준을 대입하면

이제 이 책과 이 앱의 조건을 대입해 보자.
혼자 만들고, 무료로 운영하고, 배워야 할 양을 최소로 줄이고, AI 에이전트와 협업하며, 서버리스에 배포한다는 조건이다.

가장 크게 갈리는 것은 배우는 양이다.
SPA 길을 택하면 Go에 더해 TypeScript라는 두 번째 언어, React라는 프레임워크, 번들러와 npm이라는 도구 사슬, 그리고 두 프로그램을 잇는 API 규약까지 익혀야 한다.
서버 렌더링 길에서는 이 목록이 통째로 사라진다.
언어는 Go 하나이고, 화면은 표준 라이브러리에 든 html/template으로 그리며, npm도 번들러도 별도의 빌드 단계도 없다.
로컬에서는 `go run ./cmd/server` 한 줄로 화면까지 포함한 앱 전체가 뜬다.

관리할 표면적도 줄어든다.
프런트엔드의 의존성 목록과 잠금 파일, 프레임워크의 메이저 버전 업그레이드, 쉴 새 없이 올라오는 보안 권고를 따라가는 일이 없다.
화면을 위해 저장소에 들인 외부 코드는 잠시 뒤 소개할 htmx 파일 하나가 전부다.

AI 에이전트 협업이라는 세 번째 잣대에도 잘 맞는다.
HTML과 서버 렌더링은 수십 년 축적된 방식이라 학습 데이터가 풍부하고 관용구가 흔들리지 않으며, html/template은 표준 라이브러리라 버전에 따라 API가 널뛰는 일도 없다.
검증 루프도 하나로 합쳐진다.
화면 코드가 Go 프로젝트 안에 있으므로, 템플릿이 전부 문법에 맞게 파싱되는지 확인하는 테스트를 두면 `go build`와 `go test` 한 바퀴로 화면 쪽 오류까지 걸러진다(이 테스트는 11장에서 본다).

서버리스와의 궁합도 좋다.
함수가 HTML까지 만들어 돌려주므로 배포 단위가 Go 함수 하나이고, 프런트엔드 빌드를 언제 돌릴지, 정적 파일과 API를 어떻게 잇고 배포 시점을 어떻게 맞출지 같은 조율 거리가 없다.

그렇다면 SPA의 강점이던 매끄러운 상호작용은 어떻게 채우는가.
그 답이 htmx다.

### 부분 갱신을 맡는 htmx

서버 렌더링의 약한 부분은 갱신 단위가 페이지라는 것이다.
순수한 폼 제출만으로 앱을 만들면 버튼을 누를 때마다 브라우저가 화면 전체를 새로 로드한다.
설정을 저장하는 정도라면 참을 만하지만, Echo Flip의 학습 화면에서는 곤란하다.
카드 한 장을 채점할 때마다 페이지 전체가 다시 뜨면 화면이 번쩍이고 학습의 리듬이 끊긴다.

htmx는 이 틈을 메우는 라이브러리다.
HTML 속성으로 "이 폼을 제출하면 그 응답으로 저 영역만 갈아 끼워라"라고 선언하면, 요청을 보내고 응답을 받아 화면 일부를 교체하는 일을 htmx가 대신한다.
학습 화면의 채점 버튼이 실제로 이렇게 생겼다.
다음은 `internal/web/templates/partials/study.html`의 채점 폼으로, `/study/grade`에 제출한 결과로 받은 HTML 조각을 `#study-body` 영역에만 끼워 넣으라는 선언이다.

```html
<form class="grade-grid" hx-post="/study/grade" hx-target="#study-body">
  {{template "study_state_fields" $}}
  <button class="btn btn-red btn-big" name="correct" value="false">{{icon "x"}} 틀렸어요</button>
  <button class="btn btn-green btn-big" name="correct" value="true">{{icon "check"}} 맞았어요</button>
</form>
```

`hx-post`와 `hx-target`, 속성 두 개가 전부다.
서버는 이 요청을 받아 다음 카드가 담긴 HTML 조각을 렌더링해 돌려주고, htmx가 학습 영역만 그 조각으로 바꾼다.
사용자 눈에는 새로고침 없이 다음 카드로 넘어가는 화면이 보인다.

이 자리의 대안도 짚어 두자.
Alpine.js와 Stimulus는 HTML에 동작을 붙인다는 지향이 htmx와 같지만, 동작 자체는 자바스크립트로 직접 쓰는 도구라서 두 번째 언어가 도로 살아난다.
Echo Flip의 상호작용은 "서버가 새로 그린 조각으로 바꿔치기"만으로 전부 표현할 수 있었으므로, 속성 선언으로 끝나는 htmx가 배우는 양이 가장 적었다.
React와의 비교는 앞의 두 갈래 비교가 그대로 적용된다.
비용 면에서 htmx는 51KB짜리 `htmx.min.js` 파일 하나이고, npm으로 내려받는 대신 그 파일을 저장소에 직접 담아(vendoring) Go 서버가 함께 서빙한다.
덕분에 화면 쪽 의존성 관리는 가끔 새 버전 파일로 바꿔 두는 일이 전부다.

그래도 자바스크립트가 0줄은 아니다.
발음 읽어주기(음성 합성)나 클립보드 복사처럼 브라우저만 가진 기능은 자바스크립트로만 접근할 수 있다.
Echo Flip이 직접 쓴 자바스크립트는 이런 브라우저 전용 API를 감싸는 `static/app.js` 한 파일, 80여 줄이 전부다.
카드 뒤집기조차 자바스크립트가 아니라 숨긴 체크박스와 CSS의 3D 변환만으로 구현했는데, 이 재주는 10장에서 해부한다.

부수 효과로 보안 구조도 단순해진다.
로그인 토큰이 브라우저의 자바스크립트에 노출되지 않고 서버만 다루게 되는데, 이 이점은 17장에서 자세히 본다.

화면을 이루는 HTML과 CSS는 10장에서, html/template으로 화면을 그리는 방법은 11장에서, htmx로 부분 갱신을 붙이는 방법은 12장에서 전부 실제 코드로 다룬다.

## 전체 아키텍처 개요

이 장에서 정리한 요구사항과 네 계층의 선택이 수렴한 아키텍처를 그림 하나로 정리해 보자.
그 전에 하나 일러둔다.
지금부터 볼 그림은 앱을 세상에 공개한 뒤의 모습, 즉 2부가 끝났을 때 완성되는 운영 아키텍처다.
1부에서 다루는 앱은 같은 코드이지만 Vercel도 Supabase도 거치지 않고, 파일 하나로 동작하는 관계형 데이터베이스인 SQLite와 함께 내 컴퓨터 안에서만 돈다.
운영 설정이 주어지면 이 그림대로, 아무 설정이 없으면 로컬 모드로 동작하는 두 모드 구조라서, 계정 하나 만들지 않고도 1부의 앱을 끝까지 완성할 수 있다.

::: info [용어 풀이] 아키텍처(Architecture)
소프트웨어를 이루는 큰 구성 요소들과 그 사이의 연결 방식을 정한 전체 구조다.
건물 설계도가 방 배치와 배관을 정하듯, 화면·서버·데이터베이스가 각각 어디에 놓여 어떻게 통신하는지를 정한다.
벽지 바꾸기와 달리 집 구조 변경이 큰 공사이듯, 아키텍처도 나중에 바꾸기 어려우므로 초기에 신중하게 정한다.
:::

```text
[브라우저 / Android PWA]
   │
   │ ① 화면: Go 서버가 렌더링한 HTML 문서 받기
   │ ② 정적 자원: CSS·htmx·아이콘 내려받기
   │ ③ 조각 갱신: htmx가 보낸 요청에 HTML 조각으로 응답
   ▼
[Vercel — 무료 티어, 함수 리전 iad1]
   └─ Go 서버리스 함수 하나: api/index.go → Gin 라우터
          ├─ 화면 렌더링: html/template
          ├─ 정적 파일: 바이너리에 내장(embed)
          ├─ JSON API: /api/*
          └─ 로그인: Supabase Auth와 OAuth 왕복,
             세션은 HttpOnly 쿠키
          │
          │ ④ SQL — 트랜잭션 풀러(포트 6543)
          ▼
[Supabase — 무료 티어, East US 리전]
   ├─ PostgreSQL: profiles, decks, cards, card_srs,
   │              study_sessions, review_logs, smart_decks
   └─ Auth: Google/GitHub OAuth 처리, JWT 발급
```

구성 요소는 셋뿐이다.
화면을 표시하는 브라우저, 화면 렌더링부터 JSON API까지 도맡는 Vercel의 Go 서버리스 함수 하나, 그리고 데이터베이스와 인증을 겸하는 Supabase다.
각각을 조금 더 들여다보겠다.

### 화면: Go 서버가 렌더링하는 HTML

화면은 브라우저가 아니라 서버에서 만든다.
사용자가 주소를 열면 Go 서버가 Go 표준 라이브러리의 템플릿 엔진(html/template)으로 완성된 HTML 문서를 만들어 보내고, 브라우저는 받은 문서를 그리기만 한다.
카드 목록에서 한 장을 지우는 것 같은 부분 갱신은 htmx가 맡는다.
버튼에 적어 둔 속성대로 htmx가 서버에 요청을 보내고, 서버가 돌려준 HTML 조각으로 화면의 해당 부분만 갈아 끼우는 방식이라 페이지 전체를 새로 고치지 않아도 된다.
스타일은 프레임워크 없이 순수 CSS 파일 하나(`internal/web/static/app.css`)로, CSS 변수와 미디어 쿼리로 다크 모드까지 처리한다.
이 책이 다른 웹 개발 입문과 갈라지는 대목이 여기다.
직접 작성한 브라우저 자바스크립트는 음성 재생·클립보드 복사처럼 브라우저에서만 할 수 있는 일을 감싸는 파일 하나(`internal/web/static/app.js`, 약 80줄)뿐이고, 카드 뒤집기 애니메이션조차 자바스크립트 없이 CSS만으로 동작한다.
HTML과 CSS는 10장에서, 템플릿 구조는 11장에서, htmx는 12장에서 다룬다.

### 백엔드: Go/Gin을 Vercel 함수로

백엔드는 Go와 웹 프레임워크 Gin으로 작성한 서버 하나다.
화면 렌더링과 정적 파일 서빙, 그리고 외부 프로그램이 쓸 수 있는 JSON API(`/api/*`)까지 이 서버가 전부 처리한다.
로컬에서는 `cmd/server`로 띄우는 평범한 HTTP 서버지만, 배포 시에는 Vercel 서버리스 함수 하나로 감싼다.
진입점인 `api/index.go`는 들어온 요청을 Gin 엔진에 그대로 넘기는 함수 하나가 전부다(코드는 9장에서 본다).
Gin 엔진 조립은 `pkg/app`에 있고, 로컬 서버와 Vercel 함수가 같은 엔진을 공유한다.
덕분에 로컬에서 검증한 라우팅·미들웨어가 배포 환경에서도 동일하게 동작한다.
화면·경로마다 함수를 쪼개는 대신 함수 하나가 모든 요청을 받는 캐치올(catch-all) 구조라, 무료 티어의 함수 개수 제한과도 무관하고 콜드스타트도 한 곳에서만 발생한다.
이 연결을 완성하는 것이 `vercel.json`의 재작성(rewrite) 규칙으로, 모든 경로를 Go 함수로 보낸다(전문은 16장에서 본다).
CSS 같은 정적 파일도 별도 저장소에 올리는 대신 Go 바이너리 안에 내장(embed)해 같은 함수가 내보내므로, 배포 단위가 함수 하나로 끝난다.
화면과 API가 같은 곳에서 나오니 주소(오리진)가 갈라질 일이 없고, 운영 환경에서 CORS 문제가 아예 생기지 않는다.
Gin의 구조는 9장에서, Vercel 함수의 제약(예: `internal/` 패키지를 import할 수 없어 공유 코드를 `pkg/`에 두는 이유)은 16장에서 다룬다.

### 데이터와 인증: Supabase

Supabase는 관리형 PostgreSQL과 인증을 한 계정으로 제공한다.
Echo Flip은 이 중 두 가지만 쓴다.
PostgreSQL 데이터베이스, 그리고 OAuth를 이용한 Google/GitHub 로그인이다.

::: info [용어 풀이] OAuth
앱에 비밀번호를 직접 넘기지 않고, 이미 신뢰하는 구글이나 깃허브에 대신 로그인해 "이 사람이 맞다"는 확인만 받아 오는 방식이다.
앱은 당신의 구글 비밀번호를 끝내 보지 못하고, 구글이 발급한 확인 결과만 건네받는다.
그래서 비밀번호를 저장할 의무 자체가 앱에서 사라진다.
:::

인증 흐름은 이렇다.
사용자가 로그인 버튼을 누르면 Go 서버가 Supabase Auth와 OAuth를 왕복해 JWT를 받고, 이를 자바스크립트가 읽을 수 없는 HttpOnly 쿠키에 담아 브라우저에 맡긴다.
이후 브라우저는 요청마다 그 쿠키를 자동으로 실어 보내므로, 로그인 처리에 브라우저 자바스크립트가 한 줄도 필요 없다.
서버는 로그인한 사용자 명단을 따로 기억해 두지 않고, 쿠키에 실려 온 JWT가 진짜인지만 그때그때 확인한다.
이를 위해 Go 서버는 Supabase의 JWKS(JSON Web Key Set) 엔드포인트에서 위조 여부를 가리는 검증용 열쇠인 공개키를 받아 JWT 서명을 검증한다.
백엔드가 세션 저장소를 가질 필요가 없으니 서버리스와 잘 맞는 무상태(stateless) 구조다.

::: info [용어 풀이] JWT(JSON Web Token)
로그인에 성공했다는 사실을 서버가 서명해서 발급하는 긴 문자열이다.
놀이공원에서 입장 확인 후 손목에 채워 주는 밴드처럼, 한 번 받으면 매 요청마다 신분증을 다시 보이지 않아도 된다.
위조하면 서명이 깨지므로 서버는 밴드가 진짜인지 검사만 하면 된다.
:::

한 가지 특이한 결정이 있다.
Supabase는 브라우저에서 DB에 직접 접근하는 PostgREST API를 기본 제공하지만, Echo Flip은 이를 쓰지 않는다.
데이터베이스로 통하는 옆문을 잠그고 정문인 Go API 하나만 남긴 셈이다.
구체적으로는 모든 테이블에 행 수준 보안(Row Level Security, RLS)을 켜되 정책을 하나도 만들지 않아, 공개 키(anon key)로는 어떤 데이터에도 접근할 수 없게 막았다.
데이터 접근은 오직 Go API를 통해서만 이루어진다.
RLS 정책만으로 권한을 다 표현하는 구성도 가능하고 백엔드 코드가 줄어드는 매력이 있지만, SRS 채점이나 덱 복사처럼 여러 테이블을 묶는 트랜잭션 로직은 결국 서버 코드가 필요했다.
예를 들어 덱 복사가 카드 수십 장을 옮기다 중간에 끊겼을 때, 절반만 복사된 덱이 남지 않게 전체를 한 묶음으로 처리하는 일이 그렇다.
접근 경로를 하나로 통일하는 쪽이 보안 검토도 단순해진다.
이 설계와 트레이드오프는 17장과 18장에서 자세히 다룬다.

### 기술 스택 요약

의존성 목록으로 스택을 요약해 두겠다.
Go 쪽의 직접 의존성은 웹 프레임워크 Gin, JWT 검증 라이브러리(golang-jwt, keyfunc), PostgreSQL 드라이버 pgx, 로컬 모드가 쓰는 SQLite 드라이버, 마이그레이션 도구 golang-migrate, CORS 미들웨어, UUID 라이브러리까지 여덟 개가 전부다.
화면 쪽에는 의존성 목록이라 부를 것 자체가 없다.
템플릿(html/template)과 CSV 처리(encoding/csv)는 Go 표준 라이브러리이고, 브라우저에서 쓰는 외부 라이브러리는 htmx 하나뿐인데 이마저 파일 한 개를 저장소에 그대로 넣어 두었다.
패키지 매니저도, 빌드 도구도, 버전 잠금 파일도 화면 쪽에는 없다.
개인 프로젝트에서 의존성 하나는 곧 업그레이드 부담 하나다.
라이브러리 하나에 보안 결함이 공지되면 새 버전으로 올리고 그 여파로 깨진 곳을 고치는 일까지 혼자 감당해야 한다.
그래서 표준 라이브러리와 프레임워크 기본 기능으로 해결되는 일에는 라이브러리를 추가하지 않는다는 원칙을 지켰다.
`go.mod`의 의존성 목록은 6장에서 발췌해 다룬다.

## 정리

이 장에서는 비기능 요구사항에서 출발해 네 계층의 기술을 골랐고, 그 선택들이 방금 본 세 조각짜리 아키텍처로 수렴하는 것까지 확인했다.
무료 티어와 서버리스와 1인 운영이라는 제약은 각각 다른 계층을 조였지만, 결론은 화면까지 그려 주는 Go 함수 하나와 관리형 데이터베이스라는 단순한 그림 하나였다.

여기에 AI 에이전트 협업이라는 기준이 네 계층 모두를 관통한다.
Go의 균질한 스타일과 빠른 빌드, 컴파일 시점의 타입 검증, Gin의 풍부한 학습 데이터, 수십 년 동안 안정된 HTML과 표준 라이브러리 템플릿은 전부 "에이전트가 짜고, 기계가 검증하고, 사람이 검토한다"는 개발 방식을 뒷받침하는 특성이다.

선택을 표로 요약하면 다음과 같다.
가로줄 하나가 한 계층의 선택이고, '결정적 이유' 칸은 그 선택으로 얻은 것, '검토한 대안' 칸은 조건이 달랐다면 손이 갔을 다른 후보다.
이 두 칸을 나란히 읽으면 각 선택이 무엇을 얻고 무엇을 포기했는지, 즉 비교의 축이 드러난다.

| 계층 | 선택 | 결정적 이유 | 검토한 대안 | 사용법 |
| --- | --- | --- | --- | --- |
| 데이터베이스 | SQLite(1부 로컬)·PostgreSQL(2부 운영) | 관계·제약·트랜잭션, 파일 하나로 시작해 무료 관리형 티어로 | 문서형 NoSQL, 처음부터 PostgreSQL 하나만 쓰기 | 4·5장 |
| 백엔드 언어 | Go | 콜드스타트·단일 바이너리·균질한 스타일 | Node.js, Python, JVM | 6~8장 |
| 웹 프레임워크 | Gin | 반복 코드 위임, 가장 넓은 사용자 층 | net/http, chi, Echo | 9장 |
| 화면 렌더링 | 서버 렌더링(html/template) + htmx | 언어 하나·도구 사슬 없음, 필요한 곳만 부분 갱신 | SPA(React·Vue·Svelte), 순수 폼 제출만 | 10~12장 |

이 그림을 실제로 받쳐 주는 인프라, 즉 Vercel과 Supabase를 고른 이유와 설정 상세는 16장과 17·18장에서 다룬다.

마지막으로 한 번 더 강조하면, 이 조합은 "무료 서버리스 + 1인 개발 + 배우는 양 최소화 + AI 에이전트 주도 개발"이라는 Echo Flip의 제약에 최적화된 답이지 보편적인 정답이 아니다.
자바스크립트에 능숙한 팀이라면 Node.js가, NLP 기능이 핵심이라면 Python이, 오프라인 우선이거나 편집기처럼 상호작용이 극단적으로 많은 앱이라면 SPA가, 문서 단위 접근이 지배적이라면 NoSQL이 더 나은 선택일 수 있다.
반대로 상주 서버 한 대에 배포하는 앱이라면 굳이 옮기지 않고 SQLite로 운영까지 가는 길도 성립한다.
각 절에서 대안이 더 나은 조건을 함께 적어 둔 이유는, 독자가 자기 프로젝트의 제약을 대입해 다른 답을 낼 수 있게 하기 위해서다.

다음 2장부터 이렇게 고른 기술들로 내 컴퓨터 안에 앱을 완성하는 여정으로 들어간다.
첫 순서는 기술 각론이 아니라, 이 조합의 코드를 실제로 써 내려갈 도구다.
2장에서 AI 코딩 에이전트 Claude Code의 동작 원리와 기본기를 익히고, 3장에서 이 장의 결론을 에이전트에게 지시해 앱의 골격을 만든다.
데이터베이스·Go·화면이라는 각 층의 기초는 그 뒤 4장부터, 에이전트가 만든 코드를 읽고 검증하는 눈을 기른다는 목표로 한 층씩 다진다.
