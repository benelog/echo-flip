# 1장 기술 선택: 왜 이 조합인가

기술 선택에는 정답이 없고 제약 조건에 따른 트레이드오프(trade-off)만 있다.
트레이드오프란 무언가를 얻는 대신 다른 무언가를 내주는 맞바꿈이다.
공짜 점심은 없다는 말처럼 속도를 높이면 비용이 늘고 안전을 챙기면 손이 더 가는 식으로, 모든 선택에는 반대급부가 따른다.
기술 선택에서도 "무엇이 최고인가"보다 "무엇을 얻고 무엇을 잃는가"를 따지는 편이 정확하다.
그래서 이 책은 각 기술의 사용법으로 들어가기 전에, 선택의 이유를 한 장에 모아 먼저 정리한다.
Echo Flip의 스택은 Go와 Gin, TypeScript, React와 Next.js, PostgreSQL인데, 이 조합은 유행을 따라 고른 것이 아니라 도입 장에서 정리한 요구사항에서 도출된 결과다.

그 요구사항을 짧게 상기하면 이렇다.
완전 무료 인프라로 운영하고, 1인 개발자가 AI 코딩 에이전트와 협업해 만들며, 운영 부담을 최소화한다.
네이티브 앱 대신 PWA(Progressive Web App)로 웹과 Android를 함께 커버하고, 서버리스(Serverless) 함수의 콜드스타트(Cold Start)를 최소화한다.
이 장에서는 이 제약들을 각 기술의 특성과 하나씩 대조하면서, "왜 이 기술인가"와 함께 "언제 다른 선택이 더 나은가"를 짚는다.

## 다섯 가지 선택에 공통으로 적용한 기준

개별 기술을 살펴보기 전에, 모든 선택에 반복해서 적용한 잣대를 먼저 세워 두자.
백엔드 언어를 고를 때도, 프런트엔드 프레임워크를 고를 때도, 데이터베이스를 고를 때도 결국 같은 네 가지 질문을 던졌다.

첫째, 무료 티어(free tier)로 운영할 수 있는가.
도입에서 말한 대로 수익 모델이 없는 개인 앱은 고정비가 생기는 순간 운영이 흔들린다.
이 기준은 인프라만이 아니라 언어와 프레임워크 선택까지 거슬러 올라간다.
예를 들어 메모리를 많이 쓰는 런타임은 무료 티어의 함수 메모리 한도와 충돌하고, 상주 서버를 전제하는 프레임워크는 무료 서버리스 모델에 얹기 어렵다.

::: info [용어 풀이] 런타임(Runtime)
프로그램이 돌아가는 동안 그 코드를 실제로 실행해 주는 토대를 말한다.
언어로 쓴 코드는 그 자체로 실행되지 않고, 자바스크립트라면 브라우저나 Node.js, 자바라면 JVM처럼 코드를 받아 돌려 주는 실행 환경이 필요한데 이것이 런타임이다.
이 토대가 무거우면 메모리를 많이 차지하고 켜지는 데도 시간이 더 걸린다.
:::

둘째, 관리할 것이 적은가.
혼자 운영하는 앱에서 대시보드와 설정할 곳이 늘어나면 어느 순간 어딘가가 만료되거나 어긋난다.
의존성이 적은 언어, 배포 산출물이 단순한 빌드, 스키마와 제약을 한곳에서 지켜 주는 데이터베이스처럼, 신경 쓸 표면적이 작은 쪽을 우선했다.

::: info [용어 풀이] 의존성(Dependency)
내 코드가 제대로 돌아가려고 끌어다 쓰는, 남이 만든 코드 꾸러미를 말한다.
필요한 부품을 직접 다 짜는 대신 검증된 것을 가져다 쓰는 셈이라 편하지만, 부품이 많아질수록 각각의 업데이트와 보안 문제를 따라가야 해서 신경 쓸 일도 늘어난다.
그래서 의존성이 적을수록 혼자 오래 관리하기 쉽다.
:::

셋째, AI 에이전트가 잘 다루는 성숙한 생태계인가.
Echo Flip의 코드 상당 부분은 10장에서 다룰 Claude Code가 작성했다.
대규모 언어 모델(LLM, Large Language Model)은 학습 데이터에 많이 등장한 기술일수록 정확한 코드를 생성하므로, 사용자 층이 넓고 관용구가 안정된 기술이 곧 에이전트 협업의 품질로 이어진다.
여기에 더해, 에이전트가 스스로 검증할 수 있도록 컴파일·타입 검사·테스트 같은 기계적 검증 신호가 빠르고 결정적으로 도는 기술이 유리하다.
여기서 결정적이라는 말은 같은 코드에 같은 검사를 돌리면 언제나 같은 결과가 나온다는 뜻으로, 판정이 흔들리지 않으니 에이전트가 몇 번이고 돌려 보며 믿을 수 있는 신호가 된다.

넷째, 서버리스에 맞는가.
Echo Flip의 백엔드는 요청이 올 때만 깨어나는 서버리스 함수로 동작한다.
기동이 빠르고 메모리 사용이 적은 런타임, 서버 없이 배포되는 프런트엔드, 함수 인스턴스가 여럿 떠도 문제없는 외부 데이터베이스가 이 모델의 전제 조건이다.

이 네 기준은 서로 독립적이지 않다.
서버리스에 맞아야 무료 티어가 성립하고, 관리할 것이 적어야 혼자서 오래 운영할 수 있으며, 성숙한 생태계여야 에이전트가 만든 코드를 믿고 쌓아 갈 수 있다.
한편 이 기준의 최종 수혜자인 인프라 선택, 즉 Vercel과 Supabase를 고른 이유는 이 장에서 상세히 다루지 않고 12장과 13장에서 각각 다룬다.
이제 스택의 가장 아래층인 백엔드 언어부터 시작하자.

## 왜 Go인가

Echo Flip의 제약 조건은 무료 인프라, 최소한의 운영 부담, 그리고 개발의 상당 부분을 AI 에이전트에게 맡기는 개발 방식이다.
이 제약들을 하나씩 Go의 특성과 대조해 보겠다.

### 서버리스 함수에 맞는 언어의 조건

서버리스 함수는 요청이 없으면 잠들어 있다가, 요청이 오면 런타임을 띄우고 코드를 로드한 뒤에야 응답하는 모델이다.
이 준비 시간이 앞서 말한 콜드스타트다.
상주 서버라면 부팅이 하루에 한 번이므로 몇 초가 걸려도 문제가 아니지만, 서버리스에서는 콜드스타트가 사용자의 첫 화면 로딩에 그대로 얹힌다.

Go는 이 모델에 잘 맞는 특성을 여럿 갖고 있다.

첫째, 컴파일 결과가 단일 정적 바이너리(Static Binary)다.
가상 머신도, 인터프리터도, `node_modules` 같은 의존성 디렉터리도 함께 배포할 필요가 없다.
바이너리 하나를 메모리에 올리는 것으로 준비가 끝나므로 콜드스타트가 수십 밀리초 수준으로 짧다.

::: info [용어 풀이] 컴파일 언어와 인터프리터 언어
컴파일 언어는 코드를 미리 기계가 바로 읽는 형태로 통째 번역해 두고, 그 번역 결과물만 실행하는 언어다.
인터프리터 언어는 번역기를 옆에 두고 실행하는 순간 한 줄씩 통역해 가며 돌리는 언어라, 실행할 때 통역기(인터프리터)가 늘 함께 있어야 한다.
미리 번역해 둔 쪽은 실행이 가볍고 빠른 대신 고칠 때마다 다시 번역해야 하고, 그때그때 통역하는 쪽은 바로 실행해 보기 편한 대신 실행에 필요한 준비물이 더 많다.
Go는 앞쪽, 자바스크립트나 파이썬은 뒤쪽에 가깝다.
:::

둘째, 메모리 사용량이 적다.
서버리스 플랫폼은 대체로 함수의 메모리 상한으로 과금하거나 무료 한도를 정한다.
기본 수백 MB를 점유하는 JVM 같은 런타임과 달리 Go 프로세스는 수십 MB 안에서 동작하는 경우가 많아, 무료 티어의 한도 안에 여유 있게 들어간다.

셋째, 컴파일이 빠르다.
Echo Flip의 백엔드 전체를 `go build ./...`로 빌드해도 몇 초면 끝난다.
빌드가 빠르면 배포가 빨라질 뿐 아니라, 4장에서 다룰 "수정 → 검증" 루프의 회전 속도가 올라간다.

넷째, HTTP 서버(`net/http`), JSON 직렬화(`encoding/json`), 시간 처리(`time`)가 모두 표준 라이브러리에 들어 있어 의존성을 최소로 유지할 수 있다.

### 단순한 문법이 AI 협업에 주는 이점

Go는 의도적으로 작은 언어다.
키워드가 25개뿐이고, 같은 일을 하는 코드는 누가 짜도 대체로 같은 모양으로 수렴한다.
상속이 없고, 예외(Exception) 대신 에러 값을 반환하며, 4장에서 볼 `gofmt`가 포맷 논쟁을 원천 차단한다.

이 단순함은 사람에게도 이점이지만, AI 에이전트와 협업할 때 더 크게 작동한다.

첫째, 생성된 코드를 검토하기 쉽다.
에이전트가 작성한 코드라도 결국 사람이 읽고 책임져야 하는데, 숨은 제어 흐름이 없는 Go 코드는 diff만 보고도 동작을 따라갈 수 있다.
에러가 어디서 발생해 어디로 전파되는지가 `if err != nil` 분기로 코드에 전부 드러나기 때문이다.

둘째, 검증 루프가 빠르고 결정적이다.
`go build`, `go vet`, `go test`가 몇 초 안에 끝나고 결과가 명확한 성공/실패로 떨어지므로, 에이전트가 수정할 때마다 돌려서 스스로 피드백을 얻기 좋다.

셋째, 학습 데이터가 균질하다.
Go 커뮤니티는 "관용적인(idiomatic) Go"라는 단일 스타일을 강하게 공유하므로, 에이전트가 생성한 코드도 프로젝트의 기존 코드와 이질감 없이 섞인다.

물론 단순함의 대가도 있다.
에러 처리가 장황하고, 고차 함수 중심의 표현력 높은 추상화는 다른 언어보다 불편하며, Rails나 Spring 같은 풀스택 프레임워크의 생산성을 기대하기 어렵다.
Echo Flip처럼 백엔드가 얇은 API 계층인 앱에서는 이 대가가 작지만, 화면 렌더링까지 서버가 맡는 전통적 웹 앱이라면 계산이 달라진다.
예를 들어 회원 관리와 게시판, 관리자 화면 수십 장을 갖춘 사내 시스템을 빠르게 세워야 한다면, 화면과 데이터 처리의 골격을 통째로 제공하는 Rails 쪽이 몇 배 빠르다.

### Vercel이 공식 지원하는 런타임

Vercel은 Node.js 외에 Go를 공식 런타임으로 지원한다.
`api/` 디렉터리에 표준 `net/http` 시그니처를 따르는 핸들러 함수를 두면 Vercel이 알아서 컴파일해 서버리스 함수로 배포한다.
덕분에 프런트엔드 정적 파일과 백엔드 API를 한 플랫폼의 무료 티어에 함께 올릴 수 있다.
진입점 `api/index.go`의 코드는 5장에서, 배포 구조의 상세는 12장에서 다룬다.

### 대안 검토: Node.js, Python, JVM

Go가 유일한 답은 아니므로, 대안 세 가지를 놓고 언제 그쪽이 더 나은 선택인지 짚어 보자.

**Node.js/TypeScript**는 가장 강력한 대안이다.
프런트엔드가 이미 TypeScript이므로 백엔드까지 통일하면 언어가 하나로 줄고, 타입 정의와 검증 로직(예: zod 스키마)을 프런트와 공유할 수 있다.
Vercel의 1순위 런타임이기도 하므로, 팀이 TypeScript에 능숙하고 프런트·백엔드 간 공유 코드가 많다면 Node.js가 더 나은 선택이다.
반면 내주는 것도 있다.
TypeScript의 타입은 컴파일 시점에 지워져 코드가 실제로 도는 동안에는 값을 지켜 주지 못하고, 끌어 쓰는 의존성이 깊게 쌓이기 쉬우며, 같은 작업 기준 메모리 사용량이 Go보다 크다.

**Python**은 데이터 생태계가 강점이다.
영어 학습 앱이라는 도메인 특성상 형태소 분석, 발음 평가, 문장 난이도 채점 같은 자연어 처리(NLP) 기능을 붙일 계획이라면, 필요한 라이브러리가 대부분 Python 우선으로 나오므로 Python이 더 나은 선택이다.
반면 배포 아티팩트가 무겁고(인터프리터 + 의존성), 타입 힌트가 선택 사항이라 정적 검증 신호가 약하다.
Echo Flip은 사전 조회를 외부 API에 위임하고 SRS 계산만 직접 하므로 이 강점을 쓸 일이 없었다.

**JVM**(Java/Kotlin)은 성숙도가 강점이다.
Spring 생태계의 축적된 패턴과 대규모 팀을 위한 규율은 다른 스택이 따라오기 어렵다.
상주 서버를 운영할 수 있고, 도메인이 복잡하며, 조직의 표준 스택이 JVM이라면 굳이 벗어날 이유가 없다.
반면 서버리스에서는 초 단위 콜드스타트와 큰 메모리 기본값이 무료 티어와 정면으로 충돌한다.
GraalVM 네이티브 이미지로 완화할 수 있지만 빌드 복잡도라는 새 비용이 생긴다.

정리하면, "무료 서버리스 + 얇은 API + AI 에이전트 주도 개발"이라는 제약 조합에서 Go가 가장 마찰이 적었고, 제약이 다르면 답도 달라진다.
Go 문법과 테스트, 품질 도구는 2장부터 4장까지 실제 코드로 익힌다.

## 왜 Gin인가

언어를 Go로 정했다면 다음 질문은 웹 프레임워크다.
흥미롭게도 Go에서는 "프레임워크가 정말 필요한가"부터가 진지한 질문이다.

::: info [용어 풀이] 프레임워크와 라이브러리
둘 다 남이 만들어 둔 코드 묶음이지만 주도권이 반대다.
라이브러리는 내가 필요할 때 불러다 쓰는 연장이고, 프레임워크는 정해진 골격 안에 내 코드를 끼워 넣는 틀이다.
연장은 내가 언제 쓸지 고르지만, 틀은 "여기에 네 코드를 넣어라"라고 전체 흐름을 미리 정해 두고 필요할 때 내 코드를 불러 준다.
그래서 프레임워크는 반복 작업을 대신 떠맡아 주는 대신 그 틀이 정한 방식을 따라야 한다.
:::

### net/http만으로도 서버는 만들 수 있다

Go의 큰 매력 중 하나는 표준 라이브러리 `net/http`만으로도 실용적인 HTTP 서버를 만들 수 있다는 점이다.
특히 Go 1.22부터는 표준 라우터인 `http.ServeMux`가 크게 개선되어, HTTP 메서드 매칭과 경로 파라미터(path parameter)를 지원한다.
다음 코드는 "/api/decks/ 뒤에 덱 이름이 붙은 주소로 GET 요청이 오면 이 함수를 실행하라"고 라우터에 등록하고, 주소에 담긴 덱 이름(slug)을 꺼내는 예다.

```go
mux := http.NewServeMux()
mux.HandleFunc("GET /api/decks/{slug}", func(w http.ResponseWriter, r *http.Request) {
	slug := r.PathValue("slug")
	// ...
})
```

라우팅만 놓고 보면 이 정도로 충분한 경우가 많다.
그렇다면 프레임워크는 무엇을 더해 주는가.
Echo Flip의 코드를 기준으로 보면 크게 세 가지다.

첫째, 라우트 그룹(route group)이다.
"이 경로 아래의 모든 API는 인증이 필요하다" 같은 규칙을 그룹 단위로 한 번만 선언할 수 있다.
표준 라이브러리에서는 핸들러마다 인증 검사를 감싸거나 래퍼 함수를 직접 만들어야 한다.

둘째, JSON 바인딩(binding)이다.
브라우저가 보내는 요청에는 "이 덱의 이름을 이렇게 바꿔 달라" 같은 데이터가 JSON이라는 문자 형식으로 담겨 오는데, 코드가 다루려면 이를 Go의 값 묶음(구조체)으로 옮겨 담아야 한다.
Gin에는 요청 본문의 JSON을 구조체로 역직렬화하고 실패를 일관되게 처리하는 `ShouldBindJSON` 같은 도우미가 있다.
표준 라이브러리로도 `json.NewDecoder(r.Body).Decode(&body)`를 쓰면 되지만, Content-Type 확인이나 검증 태그 연동 같은 잡무가 핸들러마다 반복된다.

셋째, 미들웨어 체인(middleware chain)이다.
패닉 복구, CORS, 인증처럼 모든 요청에 공통으로 적용할 처리를 등록 순서대로 실행하는 구조가 프레임워크에 내장되어 있다.
표준 라이브러리에서는 `http.Handler`를 겹겹이 감싸는 데코레이터 패턴을 손으로 조립해야 한다.

어느 것도 표준 라이브러리로 못 할 일은 아니지만, Echo Flip처럼 30개 가까운 엔드포인트에 인증·CORS·에러 응답 규약이 걸리는 API라면 이런 반복 코드를 프레임워크에 위임하는 편이 코드량과 실수 가능성을 함께 줄여 준다.

### 대안 비교: net/http, chi, Echo

Gin이 유일한 선택지는 아니다.
대안이 더 나은 경우를 함께 짚어 보자.

**net/http(1.22+ ServeMux)** 는 의존성이 전혀 없다는 것이 최대 장점이다.
엔드포인트가 몇 개 안 되고 미들웨어도 한두 개뿐인 소형 서비스, 혹은 외부 의존성을 최소화해야 하는 라이브러리성 도구라면 버전 업그레이드 부담도 학습 비용도 없는 표준 라이브러리만 쓰는 편이 낫다.

**chi**는 표준 `http.Handler` 인터페이스를 그대로 유지하는 얇은 라우터로, 표준 라이브러리 생태계의 미들웨어와 예제를 변환 없이 재사용할 수 있다.
"프레임워크에 종속되고 싶지 않다"는 요구가 강한 팀, 혹은 기존 net/http 코드에 라우팅만 보강하고 싶은 프로젝트라면 chi가 더 나은 선택이다.
대신 JSON 바인딩과 응답 도우미는 직접 갖춰야 한다.

**Echo**는 기능 범위가 Gin과 가장 비슷한 프레임워크다.
가장 큰 차이는 핸들러가 `func(c echo.Context) error`처럼 에러를 반환하고, 중앙의 에러 핸들러가 이를 HTTP 응답으로 변환한다는 점이다.
에러 처리를 한곳에 모으는 이 방식을 선호한다면 Echo가 더 맞을 수 있다.
Gin의 핸들러는 반환값이 없어서, 5장에서 보겠지만 Echo Flip은 `fail` 같은 도우미 함수로 같은 효과를 낸다.

성능은 이 규모의 앱에서 의미 있는 변별 요소가 아니며, 네 후보 모두 개인용 암기 카드 앱의 트래픽을 감당하고도 남는다.
Echo Flip이 Gin을 고른 결정적 이유는 다른 데 있다.

### 사용자 층이 넓다는 것의 새로운 의미

Gin은 Go 웹 프레임워크 중 가장 널리 쓰이고, GitHub 스타 수도 압도적이다.
사용자 층이 넓다는 것은 전통적으로 문서·튜토리얼·Stack Overflow 답변이 많다는 뜻이었다.
그런데 AI 에이전트와 함께 개발하는 시대에는 여기에 한 겹의 의미가 더해진다.

Echo Flip의 백엔드 코드 대부분은 10장에서 다룰 Claude Code로 작성했다.
LLM은 학습 데이터에 많이 등장한 라이브러리일수록 정확한 코드를 생성한다.
Gin처럼 공개 코드와 질의응답이 풍부한 프레임워크는 에이전트가 API 시그니처를 잘못 기억하거나 존재하지 않는 메서드를 지어내는 일이 드물고, 소수만 쓰는 프레임워크를 고르면 생성된 코드를 사람이 일일이 교정하는 비용이 커진다.

"많이 쓰이는 기술을 고르면 검색이 잘 된다"는 오래된 조언이 "많이 쓰이는 기술을 고르면 AI가 잘 짠다"로 확장됐다.
이 기준은 앞서 세운 세 번째 잣대 그대로이며, 잠시 뒤 React를 고를 때도 같은 논리가 다시 등장한다.
Gin의 라우팅과 미들웨어, 핸들러 작성법은 5장에서 실제 코드로 살펴본다.

## 왜 TypeScript인가

백엔드 언어와 프레임워크를 정했으니 이제 화면 쪽으로 넘어가자.
프런트엔드에서는 언어 선택의 질문 자체가 백엔드와 다르게 생겼다.

### 브라우저의 언어는 하나뿐이다

백엔드 언어는 선택지가 넓다.
앞의 "왜 Go인가" 절에서 본 것처럼 Go, Node.js, JVM 계열 등 여러 후보를 놓고 서버 환경의 제약에 맞춰 고를 수 있었다.
그러나 프런트엔드는 사정이 다르다.
브라우저가 직접 실행하는 언어는 사실상 자바스크립트(JavaScript)뿐이다.
웹어셈블리(WebAssembly)라는 선택지가 있지만 DOM 조작(화면 요소를 읽고 바꾸는 일)과 UI 프레임워크 생태계를 생각하면 일반적인 웹 앱의 주 언어가 되기는 아직 어렵다.

그래서 프런트엔드 언어 선택의 질문은 "어떤 언어를 쓸 것인가"가 아니라 "자바스크립트를 어떻게 쓸 것인가"에 가깝다.
그대로 쓰는 길과 자바스크립트로 컴파일되는 다른 언어를 쓰는 길 사이에서, TypeScript는 그 중간에 있는 답이다.
자바스크립트 문법을 그대로 포함하는 상위 집합(superset)이면서, 컴파일 시점에만 존재하는 타입 계층을 얹는다.

### 동적 타입의 한계: 런타임에야 드러나는 오류

자바스크립트는 동적 타입(dynamic typing) 언어다.
변수와 함수 인자에 타입 제약이 없어 코드를 빨리 쓸 수 있지만, 잘못된 값이 흘러 들어가도 실행해 보기 전에는 알 수 없다.

::: info [용어 풀이] 정적 타입과 동적 타입
값에 "숫자다", "문자다" 같은 종류(타입)를 언제 확인하느냐의 차이다.
정적 타입은 코드를 실행하기 전 미리 검사하는 단계에서 타입이 맞는지 따져, 어긋나면 그 자리에서 알려 준다.
동적 타입은 그런 사전 검사 없이 일단 실행하고, 잘못된 값이 실제로 문제를 일으키는 순간에야 오류가 드러난다.
빨리 쓰기엔 동적 타입이 편하지만, 오래 키울 앱에서는 실수를 일찍 잡아 주는 정적 타입이 유리하다.
:::

Echo Flip의 도메인으로 예를 들어 보자.
카드 객체의 뜻 필드 이름은 `meaning`이다.
순수 자바스크립트에서 `card.meanig`처럼 오타를 내면 그 값은 조용히 `undefined`가 되고, 화면에는 빈 문자열이 렌더링된다.
컴파일러도 오류도 없고, 사용자가 "카드 뜻이 안 보인다"고 알려 주기 전까지는 버그의 존재조차 모른다.

이 앱은 실제로 필드 이름을 한 차례 바꾼 이력이 있다.
카드의 앞뒤 면을 `front`/`back`으로 부르다가 학습 방향(text→meaning, meaning→text)을 도입하면서 `text`/`meaning`으로 개명했다.
동적 타입 언어에서 이런 전면 개명은 문자열 검색으로 사용처를 찾고 놓친 곳은 런타임 버그로 발견하는, 공포스러운 작업이다.
TypeScript에서는 타입 정의 한 곳을 바꾸면 옛 이름을 참조하는 모든 코드가 그 자리에서 컴파일 오류로 표시된다.
오류 목록이 곧 수정해야 할 작업 목록이 된다.

### 타입은 컴파일 시점의 검증 신호다

정적 타입의 가치를 다른 각도에서 보면, 코드가 맞는지 실행 없이 판정해 주는 자동 검증 신호라는 점이다.
`tsc --noEmit` 한 번이면 프로젝트 전체의 타입 정합성을 수 초 안에 확인할 수 있다.

이 성질은 AI 에이전트와 함께 개발할 때 특히 중요해진다.
Echo Flip은 Claude Code로 개발한 앱이고, 에이전트가 생성한 코드는 사람이 전부 정독하기 전에 기계적인 검증을 먼저 통과해야 한다.
동적 타입 언어라면 "실행해 보고 눌러 봐야" 알 수 있는 오류의 상당수를, TypeScript에서는 컴파일러가 즉시 잡아 에이전트에게 되돌려 준다.
에이전트는 그 오류 메시지를 읽고 스스로 고친다.
타입 검사가 사람의 개입 없이 도는 피드백 루프의 첫 관문이 된다.
Go에서 `go build`와 `go vet`이 하던 역할을 프런트엔드에서는 `tsc`가 맡는다고 보면 되고, 이런 검증을 훅(hook)으로 자동화하는 방법은 11장에서 자세히 다룬다.

또 하나, 타입은 실행 가능한 문서다.
`src/lib/types.ts` 파일 하나를 읽으면 이 앱의 도메인 모델 전체가 파악되는데, 이는 사람만이 아니라 코드베이스를 처음 읽는 에이전트에게도 유용하다.

### React 생태계의 사실상 표준

React 생태계에서 TypeScript는 사실상 표준(de facto standard)이 됐다.
Echo Flip의 `package.json` 의존성 목록으로 확인해 보자.
`package.json`은 이 앱이 끌어다 쓰는 라이브러리들의 이름과 버전을 적어 둔 목록 파일이다.

```json
{
  "dependencies": {
    "@supabase/ssr": "^0.12.0",
    "@supabase/supabase-js": "^2.110.1",
    "@tanstack/react-query": "^5.101.2",
    "lucide-react": "^1.23.0",
    "next": "16.2.10",
    "papaparse": "^5.5.4",
    "react": "19.2.4",
    "react-dom": "19.2.4",
    "zod": "^4.4.3"
  }
}
```

Next.js는 프로젝트 생성 시 TypeScript가 기본값이고, TanStack Query는 타입 추론을 전제로 API를 설계했다.
zod처럼 타입 시스템과의 연동이 존재 이유인 라이브러리도 이 생태계의 산물이다.
papaparse처럼 자바스크립트로 작성된 라이브러리도 `@types/papaparse` 같은 타입 선언 패키지가 별도로 제공된다.
생태계가 타입을 전제로 움직이므로, TypeScript를 쓰지 않으면 라이브러리가 제공하는 안전망의 절반을 포기하는 셈이다.

### 대안과 트레이드오프

그래도 대안을 짚고 넘어가자.

첫째, 순수 자바스크립트에 JSDoc 주석을 더하는 방법이 있다.
`/** @param {string} name */` 같은 주석을 달면 TypeScript 컴파일러가 이를 읽어 `.js` 파일도 타입 검사할 수 있다.
빌드 단계가 전혀 없다는 것이 장점이라, 빌드 도구를 얹기 싫은 소규모 스크립트나 라이브러리에 어울린다.
다만 문법이 장황하고 제네릭·유니온 같은 복잡한 타입 표현이 불편해서, 앱 규모가 커지면 오히려 비용이 늘어난다.

둘째, 자바스크립트로 컴파일되는 다른 언어들이 있다.
Elm, ReScript, PureScript는 TypeScript보다 엄격한 타입 시스템과 함수형 패러다임을 제공하고, Dart는 Flutter를 통해 웹까지 커버한다.
언어 자체의 완성도는 뛰어나지만, React·Next.js 생태계와의 접점이 좁고 커뮤니티 규모가 작다.
Echo Flip처럼 Next.js 정적 export, TanStack Query, Supabase 클라이언트를 조합하는 앱이라면 생태계 이탈 비용이 언어의 이점을 넘어선다.
예를 들어 Supabase 로그인 연동 예제를 찾으면 TypeScript 코드는 공식 문서에 바로 있지만, 소수 언어에서는 그 코드를 직접 옮겨 적으며 어긋나는 부분을 스스로 메워야 한다.

셋째, TypeScript가 과할 수도 있다.
한 페이지짜리 데모, 수명이 짧은 프로토타입, 혼자 쓰는 브라우저 스크립트라면 타입 정의에 들이는 시간이 회수되지 않는다.
반대로 여러 사람이(혹은 사람과 에이전트가) 오래 유지보수할 앱, 도메인 모델이 있는 앱이라면 TypeScript의 초기 비용은 금방 회수된다.
Echo Flip은 후자이므로 TypeScript를 선택했다.
TypeScript의 타입 문법은 6장에서, 비동기 처리와 런타임 검증은 7장에서 실제 코드로 다룬다.

## 왜 React인가

언어를 TypeScript로 정했으니 다음은 그 위에 얹을 UI 프레임워크다.
프런트엔드 프레임워크 선택은 단순한 취향 문제가 아니라 앱의 수명 전체에 영향을 주는 결정이다.
Echo Flip이 React를 선택한 이유를 세 가지 관점에서 정리해 보자.

### 컴포넌트 기반 UI 모델

React의 첫 번째 축은 컴포넌트(component) 기반 UI 모델이다.
화면을 독립적인 조각으로 나누고, 각 조각을 입력(props)과 출력(렌더링 결과)이 명확한 함수로 만든다.

Echo Flip의 학습 화면을 예로 들면 구조가 이렇게 쌓인다.
`StudyView`가 진행률 바와 채점 버튼을 배치하고, 그 안에 카드 한 장을 그리는 `Flashcard`를 놓고, `Flashcard` 안에 발음 재생 버튼인 `TtsButton`을 놓는다.
각 컴포넌트는 자기 역할만 알면 되므로, 카드 디자인을 바꿀 때 세션 진행 로직을 건드릴 일이 없다.

### 선언적 렌더링

두 번째 축은 선언적(declarative) 렌더링이다.
jQuery 시절처럼 "정답이 공개되면 이 요소를 찾아서 클래스를 붙여라"라고 절차를 지시하는 대신, "상태가 이러하면 화면은 이렇게 생겼다"라고 결과만 선언한다.
`src/components/StudyView.tsx`의 실제 구조를 요약하면 이렇다.
정답이 공개된 상태(`revealed`)면 채점 버튼 두 개를, 아직이라면 "정답 보기" 버튼 하나를 보여 준다는 선언이다.

```tsx
{state.revealed ? (
  <div className="grid grid-cols-2 gap-3">{/* 틀렸어요 / 맞았어요 버튼 */}</div>
) : (
  <button onClick={reveal}>정답 보기</button>
)}
```

`revealed`라는 상태 하나가 바뀌면 React가 알아서 이전 화면과 다음 화면의 차이를 계산해 DOM을 갱신한다.
개발자는 상태 전이만 설계하면 되고, 화면 동기화 버그가 끼어들 틈이 크게 줄어든다.

### 생태계, 그리고 AI 에이전트

세 번째 축은 압도적인 생태계다.
TanStack Query 같은 서버 상태 라이브러리, lucide-react 같은 아이콘 세트, Tailwind CSS 통합까지, 필요한 조각이 대부분 검증된 형태로 존재한다.

여기에 이 책의 2부와 이어지는 관점이 하나 더 있다.
AI 코딩 에이전트로 개발할 때, 학습 데이터에 React 코드가 압도적으로 많다는 사실이 실질적인 이점이 된다.
경험상 에이전트가 생성하는 React 코드는 관용구(idiom)를 잘 따르고 한 번에 동작하는 비율이 높은 반면, 소수 프레임워크에서는 미묘하게 낡은 API를 섞어 쓰는 일이 잦았다.
Echo Flip은 10장에서 다룰 Claude Code로 대부분의 화면을 작성했으므로, 이 점이 선택에 실제로 작용했다.
Gin을 고를 때 확인한 "많이 쓰이는 기술을 고르면 AI가 잘 짠다"는 논리가 프런트엔드에서도 그대로 성립했다.

### 대안과 트레이드오프

물론 React가 항상 정답은 아니니 대안을 짚어 보자.

첫째, Vue는 템플릿 문법과 반응성(reactivity) 시스템 덕분에 진입 장벽이 낮고, 공식이 유지하는 라우터·상태 관리 도구가 잘 정돈되어 있다.
HTML에 가까운 템플릿을 선호하는 팀, 특히 백엔드 중심 팀이 화면을 얹는 경우라면 Vue가 더 매끄러울 수 있다.

둘째, Svelte는 컴파일 시점에 프레임워크 코드를 제거해 번들이 작고 런타임이 빠르다.
위젯 임베드처럼 전송 크기가 민감한 제품이라면 Svelte가 유리하다.
다만 생태계 규모와 AI 생성 코드 품질에서는 아직 React에 못 미친다.

셋째, SPA(Single Page Application) 프레임워크 없이 서버 렌더링 HTML로 가는 길도 있다.
Go 템플릿이나 Rails에 htmx 정도를 곁들이는 방식이다.
폼 제출과 목록 조회가 중심인 관리 도구라면 이쪽이 오히려 단순하고 빠르다.
그러나 Echo Flip의 핵심 화면은 카드 뒤집기, 라운드 반복, 진행률 갱신처럼 서버 왕복 없이 즉각 반응해야 하는 상호작용이므로, 클라이언트에 상태를 두는 SPA 모델이 자연스럽다.

정리하면, 상호작용이 많고 에이전트 협업을 전제하는 이 앱에서는 React의 생태계 이점이 학습 곡선 비용을 상회한다고 판단했다.

## 왜 Next.js이고 왜 정적 export인가

React는 뷰 라이브러리일 뿐, 라우팅과 빌드는 스스로 해결하지 않는다.
Echo Flip은 그 틀로 Next.js를 쓰되, 흔한 서버 렌더링 구성이 아니라 정적 export(static export)를 택했다.

### 서버 없는 Next.js를 만드는 output: 'export'

`next.config.ts`에 `output: "export"`를 지정하면 `next build`가 Node.js 서버용 산출물 대신 순수 HTML·CSS·JS 정적 파일을 만든다.
서버 컴포넌트의 서버 실행, 이미지 최적화 서버, API 라우트 같은 서버 기능은 포기하는 대신, 배포가 "파일 올리기"로 단순해진다.

이 선택의 근거는 두 가지다.
첫째, Echo Flip의 API는 전부 이 장 앞부분에서 본 Go/Gin 서버리스 함수가 담당하므로 Next.js 쪽에 Node 서버를 둘 이유가 없다.
둘째, 정적 파일만 서빙하면 Vercel 무료 요금제 안에서 CDN 배포가 끝나고, Node 서버 렌더링 함수를 함께 돌리는 것보다 비용·운영 부담이 확실히 가볍다.

트레이드오프도 명확하다.
SEO(Search Engine Optimization)는 검색 결과에 내 페이지가 잘 노출되게 만드는 일인데, 정적 export가 만든 첫 HTML은 알맹이가 비어 있어 검색에 불리할 수 있다.
블로그나 쇼핑몰처럼 검색 유입이 생명인 콘텐츠 사이트라면 내용이 채워진 HTML을 즉석에서 만들어 주는 요청 시점 서버 렌더링이나 ISR(Incremental Static Regeneration)이 필요하므로 정적 export가 부적합하다.
Echo Flip은 로그인 후 개인 데이터를 다루는 앱이라 검색 노출 요구가 거의 없고, 첫 화면이 빈 껍데기여도 문제가 없다.

대신 동적 라우트 세그먼트(`/decks/[slug]`)를 쓸 수 없다는 제약이 생기는데, Echo Flip은 고정 경로의 정적 페이지에 Vercel rewrite로 예쁜 URL을 입히고 나머지는 쿼리 파라미터 라우팅으로 우회한다.
이 우회의 구현은 8장에서, rewrite를 포함한 배포 구성 전체는 12장에서 자세히 다룬다.

## 왜 관계형 데이터베이스이고 왜 PostgreSQL인가

프런트엔드와 백엔드가 정해졌으니 마지막은 데이터가 머무는 곳이다.
Echo Flip의 도메인을 한 문장으로 요약하면 "사용자가 덱을 만들고, 덱에 카드를 담고, 카드를 학습한 기록이 쌓인다"이다.
사용자-덱-카드-학습 기록이라는 관계가 도메인의 뼈대이고, 이 관계 자체가 데이터의 핵심 정보다.
이런 도메인에는 관계를 일급 개념으로 다루는 관계형 데이터베이스(RDB, Relational Database)가 자연스럽다.

::: info [용어 풀이] 관계형 데이터베이스
데이터를 표(테이블) 형태로 저장하고, 표와 표 사이의 연결 관계를 핵심으로 다루는 데이터베이스다.
엑셀 시트 여러 장이 서로를 가리키며 이어져 있는 모습에 가깝고, "이 카드는 저 덱에 속한다" 같은 이어짐을 데이터 자체가 붙들고 있다.
사용자·덱·카드·학습 기록처럼 서로 얽힌 정보를 다룰 때 자연스럽게 들어맞는다.
:::

관계형 데이터베이스가 주는 가치는 크게 두 가지다.

첫째, 제약(constraint)으로 데이터 정합성을 데이터베이스가 지켜준다.
"카드는 반드시 어떤 덱에 속한다", "SRS 상태는 카드당 정확히 하나다" 같은 규칙을 애플리케이션 코드가 아니라 외래 키(foreign key)와 기본 키(primary key)로 선언하면, 버그가 있는 코드가 끼어들어도 규칙이 깨진 데이터는 저장되지 않는다.
코드는 고치면 되지만 한번 오염된 데이터는 복구가 어렵다는 점에서, 마지막 방어선을 DB에 두는 가치는 크다.

둘째, 트랜잭션(transaction)이다.
복습 한 번을 기록할 때는 로그 삽입과 SRS 상태 갱신이 함께 일어나야 하는데, 트랜잭션은 이런 다중 쓰기를 "모두 성공하거나 모두 없던 일이 되거나"로 보장한다.
9장에서 보겠지만 Echo Flip의 store 계층은 이 성질에 전적으로 기대고 있다.

### 대안 검토: SQLite와 문서형 NoSQL

관계형이라는 큰 방향 안에서도 대안은 있었다.

SQLite는 파일 하나로 동작하는 훌륭한 관계형 데이터베이스로, 로컬 도구나 단일 서버 앱이라면 운영 부담이 0에 가까워서 가장 먼저 검토할 만하다.
그러나 Echo Flip의 API는 Vercel 서버리스 함수로 동작한다(12장에서 자세히 다룬다).
서버리스에서는 요청이 몰리면 같은 함수가 여러 개 복제되어 뜨고, 각 함수가 가진 저장 공간은 함수가 꺼질 때 내용까지 함께 사라진다.
그래서 파일 하나에 데이터를 쌓는 SQLite는 모두가 함께 보는 저장소가 될 수 없고, 네트워크 너머의 독립된 데이터베이스 서버가 필수가 된다.

문서형 NoSQL(MongoDB, Firestore 등)도 생각해 볼 수 있다.
문서 DB는 스키마를 미리 고정하기 어렵거나, 하나의 문서(애그리거트) 단위로만 읽고 쓰는 패턴이 지배적일 때 강점이 있다.
카드 앱이라도 "덱 문서 안에 카드 배열을 통째로 저장하고 통째로 읽는" 구조라면 문서 DB가 더 단순할 수 있다.
그러나 Echo Flip은 카드를 덱 단위가 아니라 개별 단위로 다룬다.
"오늘 복습할 카드"는 덱 경계를 넘어 due 시각으로 모으고, 스마트 덱은 오답률·태그 같은 조건으로 카드를 횡단 조회한다.
이런 조회는 조인과 인덱스, 집계가 자연스러운 관계형 모델의 홈그라운드이고, 문서 DB에서 구현하려면 비정규화와 수동 동기화가 늘어난다.

흥미로운 점은 PostgreSQL 안에서도 문서적 유연함이 필요한 자리에는 jsonb를 썼다는 것이다.
사용자 설정(`profiles.settings`)과 스마트 덱 규칙(`smart_decks.rule`)처럼 구조가 자주 바뀔 수 있는 데이터는 jsonb 컬럼에 담아, 관계가 중요한 곳은 테이블로, 유동적인 곳은 문서로 나누는 절충이 가능하다.

### PostgreSQL을 고른 이유

관계형으로 방향을 정한 뒤 PostgreSQL을 고른 데는 기능과 인프라 양쪽의 이유가 있다.
기능 면에서는 배열 타입과 GIN 인덱스(태그 검색), jsonb, 부분 인덱스(partial index), enum 타입, `generated always as identity`처럼 이 앱이 실제로 쓰는 기능들이 판단 기준이었다.
이름이 낯설어도 지금 다 알 필요는 없고, 태그 검색을 빠르게 하는 색인이나 유연한 JSON 저장처럼 이 앱의 기능 하나하나와 맞닿은 도구라는 것만 알아 두면 된다.
인프라 면에서는 Supabase 무료 티어와 맞물린다.
Echo Flip은 "무료 인프라로 운영한다"는 비기능 요구사항이 있고, Supabase는 관리형 PostgreSQL과 인증을 무료 티어로 함께 제공하므로 데이터베이스 선택이 곧 호스팅 선택이기도 했다.
Supabase 연결 방식과 풀러(pooler) 상세는 13장에서 다루고, 이 기능들이 실제 스키마에서 어떻게 쓰이는지는 9장에서 확인한다.

## 정리

이 장의 다섯 가지 선택을 떼어 놓고 보면 각각 독립적인 결정 같지만, 한 걸음 물러서면 하나의 그림이다.

프런트엔드는 순수 정적 파일이라 서버 없이 CDN에서 서빙되고, 상호작용은 전부 브라우저 안의 React가 처리한다.
데이터가 필요한 순간에만 `/api/*` 요청이 Go 서버리스 함수를 깨우는데, 단일 바이너리인 Go는 수십 밀리초 만에 일어나 Gin 라우터로 요청을 처리한다.
함수는 상태를 들고 있지 않으므로 몇 개가 뜨든 상관없고, 정합성과 트랜잭션은 네트워크 건너편의 관리형 PostgreSQL이 책임진다.
정적 프런트, 찰나에 깨어나는 얇은 API, 관리형 데이터베이스라는 세 층이 맞물려야 "무료 티어 + 서버리스 + 1인 운영"이라는 제약이 비로소 성립한다.
어느 한 층이라도 상주 서버를 요구하는 순간 이 그림은 무너진다.

여기에 AI 에이전트 협업이라는 기준이 다섯 선택 모두를 관통한다.
Go의 균질한 스타일과 빠른 빌드, Gin과 React의 풍부한 학습 데이터, TypeScript의 컴파일 시점 검증은 전부 "에이전트가 짜고, 기계가 검증하고, 사람이 검토한다"는 개발 방식을 뒷받침하는 특성이다.

선택을 표로 요약하면 다음과 같다.
가로줄 하나가 한 계층의 선택이고, '결정적 이유' 칸은 그 선택으로 얻은 것, '검토한 대안' 칸은 조건이 달랐다면 손이 갔을 다른 후보다.
이 두 칸을 나란히 읽으면 각 선택이 무엇을 얻고 무엇을 포기했는지, 즉 비교의 축이 드러난다.

| 계층 | 선택 | 결정적 이유 | 검토한 대안 | 사용법 |
| --- | --- | --- | --- | --- |
| 백엔드 언어 | Go | 콜드스타트·단일 바이너리·균질한 스타일 | Node.js, Python, JVM | 2~4장 |
| 웹 프레임워크 | Gin | 반복 코드 위임, 가장 넓은 사용자 층 | net/http, chi, Echo | 5장 |
| 프런트엔드 언어 | TypeScript | 컴파일 시점 검증 신호, 생태계 표준 | JSDoc, Elm/ReScript 등 | 6·7장 |
| UI 프레임워크 | React + Next.js 정적 export | 생태계·AI 코드 품질, 서버 없는 배포 | Vue, Svelte, htmx/SSR | 8장 |
| 데이터베이스 | PostgreSQL | 관계·제약·트랜잭션, 무료 관리형 티어 | SQLite, 문서형 NoSQL | 9장 |

이 그림을 실제로 받쳐 주는 인프라, 즉 Vercel과 Supabase를 고른 이유와 설정 상세는 12장과 13장에서 다룬다.

마지막으로 한 번 더 강조하면, 이 조합은 "무료 서버리스 + 얇은 API + 상호작용 중심 화면 + AI 에이전트 주도 개발"이라는 Echo Flip의 제약에 최적화된 답이지 보편적인 정답이 아니다.
공유 코드가 많은 팀이라면 Node.js 통일이, NLP 기능이 핵심이라면 Python이, 콘텐츠 중심 사이트라면 서버 렌더링이, 문서 단위 접근이 지배적이라면 NoSQL이 더 나은 선택일 수 있다.
각 절에서 대안이 더 나은 조건을 함께 적어 둔 이유는, 독자가 자기 프로젝트의 제약을 대입해 다른 답을 낼 수 있게 하기 위해서다.

2장부터는 이렇게 고른 기술들의 사용법으로 들어간다.
먼저 2장에서 백엔드 언어 Go의 모듈 구조와 변수, 함수 같은 기초를 Echo Flip의 실제 코드로 익히는 것부터 시작하자.
