# 12장 서브에이전트와 훅으로 만드는 품질 게이트

11장에서 "AI가 만든 코드의 품질은 그 코드를 검증해 주는 신호의 품질에 달려 있다"는 관점을 제시했다.
이번 장에서는 그 검증 신호를 자동화한 Echo Flip의 실제 장치, 즉 훅(hook)과 서브에이전트(subagent) 설정을 해부한다.
`.claude/settings.json`, `.claude/hooks/go-check.sh`, `.claude/agents/go-quality.md` 세 파일을 전문 인용하며 한 줄씩 짚어 보겠다.
파일 세 개, 백 줄이 안 되는 설정이지만, 이 장이 이 책에서 가장 실전적인 Claude Code 활용법을 담고 있다.

::: info [용어 풀이] 품질 게이트(Quality Gate)
정해진 검사를 통과하지 못한 코드는 다음 단계로 넘어가지 못하게 막는 관문이다.
공항 검색대처럼, 조건을 충족한 것만 통과시키고 그렇지 못한 것은 붙잡아 돌려보낸다.
사람이 매번 신경 써서 지키는 규율이 아니라 시스템이 자동으로 세워 두는 장벽이라는 점이 핵심이다.
:::

## 왜 자동 품질 게이트인가

에이전트에게 Go 코드를 수정시켜 본 사람이라면 누구나 비슷한 프롬프트를 써 봤을 것이다.
"수정 후에는 반드시 gofmt과 go vet을 실행해."
CLAUDE.md에 적어 두든 매번 지시하든, 이 방식은 세 가지 이유로 한계가 있다.

첫째, 에이전트는 잊어버린다.
대화가 길어져 컨텍스트가 압축되거나, 여러 파일을 연달아 고치는 중에 검사 지시는 뒤로 밀린다.
열 번 중 아홉 번 실행해도 배포를 깨뜨리는 것은 빠뜨린 한 번이다.

둘째, 토큰이 낭비된다.
에이전트가 검사를 직접 실행하면 명령 호출과 출력이 모두 대화 컨텍스트에 쌓인다.
`go vet`이 아무 문제도 찾지 못한 성공 케이스조차 도구 호출 한 번 분량의 토큰을 소비한다.
검사는 수정할 때마다 반복되므로 이 비용은 세션 내내 누적된다.

셋째, 비결정적이다.
LLM은 확률적으로 동작하므로 같은 지시라도 어떤 날은 `go vet ./...`을, 어떤 날은 `go vet ./internal/srs`만 실행할 수 있다.
"검사했다"는 에이전트의 보고가 실제로 무엇을 검사했다는 뜻인지 사람이 매번 확인할 수 없다.

여기서 이 장 전체를 관통하는 원칙이 나온다.
**결정적(deterministic) 검사는 프롬프트가 아니라 시스템 설정으로 강제해야 한다.**
gofmt이 통과하는지, vet이 통과하는지는 예/아니오가 명확한 기계적 판정이다.
이런 판정을 확률적인 LLM의 성실함에 맡기는 것은 도구를 잘못 고른 것이다.
Claude Code는 이를 위한 장치로 훅을 제공한다.

## 훅: 라이프사이클에 셸 명령을 거는 장치

훅(hook)은 Claude Code의 라이프사이클 이벤트(lifecycle event)에 셸 명령을 연결하는 설정이다.
에이전트가 도구를 사용하기 직전(PreToolUse), 사용한 직후(PostToolUse), 사용자가 프롬프트를 제출할 때(UserPromptSubmit), 응답을 마칠 때(Stop) 등 여러 시점에 개입할 수 있다.
훅으로 등록한 명령은 LLM의 판단을 거치지 않고 Claude Code 자체가 실행하므로, 조건이 맞으면 100% 실행된다.
"검사를 잊지 마라"고 부탁하는 것과 "검사 없이는 진행할 수 없다"고 시스템이 강제하는 것의 차이다.

::: info [용어 풀이] 훅(Hook, Claude Code의 훅)
Claude Code가 특정 시점에 자동으로 실행하도록 미리 걸어 두는 스크립트다.
"파일을 저장하면 자동으로 검사가 돌아간다"는 식으로, 정해진 순간마다 사람 대신 기계가 알아서 개입하게 하는 장치다.
이름은 같지만 React 같은 프런트엔드 프레임워크에서 말하는 훅과는 다른 것이다.
:::

Echo Flip의 훅 설정 전체가 저장소의 `.claude/settings.json`에 들어 있다.
에이전트가 파일을 쓰거나 고칠 때마다 검사 스크립트 go-check.sh를 실행하라는 등록 하나가 내용의 전부다.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR/.claude/hooks/go-check.sh\"",
            "timeout": 60,
            "statusMessage": "gofmt + go vet 검사 중..."
          }
        ]
      }
    ]
  }
}
```

구조를 한 항목씩 해설해 보겠다.

**`hooks.PostToolUse`**: 훅을 걸 이벤트다.
PostToolUse는 도구 호출이 성공적으로 끝난 직후에 발생한다.
에이전트가 파일을 쓰거나 고친 바로 그 순간이 검사하기 가장 좋은 시점이므로 이 이벤트를 골랐다.
PreToolUse에 걸면 수정 전 내용을 검사하게 되어 의미가 없다.

**`matcher: "Write|Edit"`**: 어떤 도구 호출에 반응할지 거르는 조건이다.
세로 막대(|)가 "또는"을 뜻하는 정규식(regular expression) 표기라, 도구 이름이 Write이거나 Edit일 때만 매칭된다.
파일 내용을 바꾸는 도구는 Write(새 파일 작성·전체 덮어쓰기)와 Edit(부분 수정) 두 가지이므로 이 둘만 골라 놓았다.
Bash나 Read 같은 도구 호출에는 훅이 아예 실행되지 않아 불필요한 프로세스 기동을 피한다.

**`type: "command"`**: 훅의 종류다.
셸 명령을 실행하는 가장 기본적인 형태다.
실행되는 명령은 도구 호출 정보를 표준 입력(stdin)으로 건네받는다.
표준 입력은 프로그램이 시작할 때 데이터를 받아들이는 기본 통로로, 방금의 도구 호출 정보가 여기에 JSON으로 담겨 온다.

**`command`**: 실행할 명령이다.
`$CLAUDE_PROJECT_DIR`는 Claude Code가 훅 실행 시 넣어 주는 프로젝트 루트 경로 환경 변수다.
훅은 현재 디렉터리가 어디든 실행될 수 있으므로, 스크립트 경로를 이 변수 기준 절대 경로로 지정해야 안전하다.
경로에 공백이 있어도 깨지지 않도록 따옴표로 감쌌다.

**`timeout: 60`**: 훅의 최대 실행 시간(초)이다.
`go vet`은 첫 실행 시 빌드 캐시를 채우느라 수 초가 걸릴 수 있어 여유를 뒀다.
이 시간을 넘기면 훅이 강제 종료되므로, 훅에는 애초에 오래 걸리는 검사를 넣지 않는 것이 원칙이다.
이 원칙은 뒤에서 서브에이전트와의 역할 분담을 다룰 때 다시 나온다.

**`statusMessage`**: 훅이 도는 동안 터미널에 표시할 문구다.
훅은 도구 호출을 잠시 붙잡고 실행되므로, 왜 기다리는지 사용자에게 보여 주는 배려다.
기능에는 영향이 없지만 "훅이 돌고 있다"는 사실 자체를 사용자가 인지하게 해 디버깅에도 도움이 된다.

이 설정 파일은 저장소에 커밋되어 있다.
`.claude/settings.json`은 프로젝트 공유 설정이므로, 이 저장소를 클론해서 Claude Code를 여는 모든 사람(그리고 모든 에이전트 세션)에게 같은 품질 게이트가 적용된다.
개인 취향 설정은 `.claude/settings.local.json`에 두고 커밋하지 않는 관례와 구분된다.

## go-check.sh 해부: 무음 성공, 시끄러운 실패

훅이 실제로 실행하는 스크립트를 보자.
방금 수정된 파일이 Go 파일이면 포맷을 자동으로 맞추고 정적 검사를 돌린 뒤, 문제가 있을 때만 에이전트에게 알리는 내용이다.
`.claude/hooks/go-check.sh` 전문이다.

```bash
#!/usr/bin/env bash
# PostToolUse hook: auto-gofmt + go vet on the edited Go file.
# Silent on success (zero LLM tokens); exit 2 feeds vet errors back to Claude.
set -u

f=$(jq -r '.tool_input.file_path // .tool_response.filePath // empty')
case "$f" in
  *.go) ;;
  *) exit 0 ;;
esac
[ -f "$f" ] || exit 0

cd "${CLAUDE_PROJECT_DIR:-$(pwd)}" || exit 0

gofmt -w "$f" 2>/dev/null

dir=$(dirname "$f")
rel=$(realpath --relative-to="$PWD" "$dir" 2>/dev/null) || rel="$dir"
if ! out=$(go vet "./$rel" 2>&1); then
  {
    echo "go vet failed for ./$rel:"
    echo "$out"
  } >&2
  exit 2
fi
exit 0
```

30줄이 안 되는 스크립트지만 단계마다 설계 의도가 있다.
순서대로 짚어 보겠다.

### 1단계: stdin의 JSON에서 파일 경로 추출

PostToolUse 훅은 방금 실행된 도구 호출의 정보를 JSON으로 표준 입력에 받는다.
Write와 Edit 도구는 입력 파라미터(`tool_input.file_path`)에 대상 파일 경로를 담고, 도구 응답(`tool_response.filePath`)에도 같은 정보가 실린다.
jq는 JSON 문서에서 원하는 값만 골라 꺼내 주는 명령줄 도구다.
`jq -r '... // ... // empty'`는 둘 중 먼저 존재하는 값을 꺼내고, 둘 다 없으면 빈 문자열을 돌려주는 안전한 추출이다.
도구마다 JSON 스키마가 조금씩 다를 수 있으므로 대안 경로를 `//`로 이어 둔 것이다.

### 2단계: .go 파일만 통과시키는 case 문

파일 확장자를 보고 Go 파일이 아니면 그 자리에서 조용히 끝내는 부분이다.

```bash
case "$f" in
  *.go) ;;
  *) exit 0 ;;
esac
```

matcher가 Write|Edit에 매칭되므로 훅은 마크다운, HTML, JSON 등 모든 파일 수정에서 실행된다.
Go 파일이 아니면 즉시 `exit 0`으로 빠져나온다.
이 조기 종료가 중요한 이유는 비용이다.
이 책의 원고(`doc/*.md`)를 고칠 때도 훅은 실행되지만, jq 한 번과 case 문 하나로 끝나므로 체감 지연이 없다.
이어지는 `[ -f "$f" ] || exit 0`은 파일이 실제로 존재할 때만 진행하는 방어 코드다.

### 3단계: gofmt를 조용히 자동 적용

수정된 파일의 코드 포맷을 표준 형태로 바로잡는 한 줄이다.

```bash
gofmt -w "$f" 2>/dev/null
```

포맷 위반을 **보고하는 게 아니라 그냥 고쳐 버린다**는 점이 핵심이다.
gofmt의 결과는 언제나 하나로 결정되므로, "포맷이 틀렸다"고 에이전트에게 알려서 다시 고치게 하는 왕복은 토큰과 시간의 순수한 낭비다.
기계가 결정할 수 있는 일은 기계가 끝내고, 에이전트에게는 알리지도 않는다.
`2>/dev/null`은 파일에 문법 오류가 있어 gofmt이 실패하는 경우를 대비한 것인데, 그런 오류는 어차피 다음 단계의 go vet이 더 좋은 메시지로 잡아 준다.

자동 수정에는 트레이드오프가 하나 있다.
에이전트가 기억하는 파일 내용과 디스크의 실제 내용이 포맷만큼 어긋날 수 있다.
다만 어긋남은 공백·정렬 수준이고, 어긋난 상태에서 에이전트가 Edit을 시도하면 문자열 불일치로 실패해 파일을 다시 읽게 되므로 실질적인 문제로 번지지는 않았다.

### 4단계: 수정된 패키지에만 go vet

수정된 파일이 속한 디렉터리를 알아내 그 패키지만 go vet으로 검사하는 부분이다.

```bash
dir=$(dirname "$f")
rel=$(realpath --relative-to="$PWD" "$dir" 2>/dev/null) || rel="$dir"
if ! out=$(go vet "./$rel" 2>&1); then
```

`go vet ./...`으로 모듈 전체를 검사하지 않고, 방금 수정된 파일이 속한 패키지 디렉터리(`./$rel`)만 검사한다.
훅은 파일 수정마다 실행되므로 속도가 곧 개발 리듬이다.
전체 검사는 정확하지만 느리고, 패키지 검사는 빠르지만 다른 패키지로 번진 영향을 놓친다.
그 빈틈은 뒤에 나올 go-quality 서브에이전트의 `go vet ./...`이 커밋 전에 메운다.
빠른 검사와 완전한 검사를 한 장치에 욱여넣지 않고 두 층으로 나눈 것이다.

`realpath --relative-to`로 상대 경로를 만드는 이유는 go 명령이 모듈 루트 기준의 `./패키지` 형태 인자를 기대하기 때문이다.
그 직전의 `cd "${CLAUDE_PROJECT_DIR:-$(pwd)}"`가 모듈 루트로 이동해 이 전제를 만들어 준다.

### 5단계: 실패하면 stderr에 쓰고 exit 2

검사에 걸렸을 때 오류 내용을 내보내고 실패 신호와 함께 끝나는 마무리 부분이다.

```bash
  {
    echo "go vet failed for ./$rel:"
    echo "$out"
  } >&2
  exit 2
```

여기가 훅과 에이전트가 만나는 접점이다.
표준 에러(stderr)는 프로그램이 일반 출력과 섞이지 않게 오류 메시지만 따로 내보내는 통로다.
Claude Code의 훅 규약에서 종료 코드(exit code) 2는 특별한 의미를 갖는다.
**훅이 exit 2로 끝나면 표준 에러에 쓴 내용이 에이전트에게 피드백으로 전달된다.**
에이전트는 자기가 방금 저지른 vet 위반을 에러 메시지 그대로 보게 되고, 다음 턴에서 스스로 고친다.
사람이 개입할 필요가 없다.

::: info [용어 풀이] 종료 코드(Exit Code)
프로그램이 끝나면서 남기는 한 자리 성적표다.
0이면 통과, 그 외의 숫자는 실패를 뜻한다.
Claude Code의 훅에서는 이 숫자가 단순한 성패 표시를 넘어 신호로 쓰이는데, 특히 2로 끝내면 그때 남긴 오류 메시지가 에이전트에게 그대로 피드백된다.
:::

반대로 exit 0으로 끝나면 에이전트에게는 아무것도 전달되지 않는다.
이 스크립트의 성공 경로를 다시 보면, `gofmt -w`는 출력을 버리고 vet 성공 시에는 아무것도 찍지 않고 `exit 0`한다.
즉 **검사가 통과하는 한 이 훅은 LLM 토큰을 단 하나도 쓰지 않는다.**
에이전트가 직접 `go vet`을 실행하던 방식과 비교하면 차이가 분명하다.
직접 실행은 성공해도 도구 호출·출력만큼 컨텍스트를 소비하지만, 훅은 실패라는 예외 상황에만 컨텍스트에 등장한다.

정리하면 이 스크립트의 설계 원칙은 두 가지다.
첫째, 성공은 무음이다. 좋은 소식은 에이전트가 알 필요가 없다.
둘째, 실패는 정확히 한 곳(stderr + exit 2)으로 시끄럽게 알린다. 나쁜 소식은 고칠 수 있는 형태로 즉시 전달한다.
그리고 이 모든 것이 프롬프트가 아니라 설정이므로, 에이전트가 아무리 딴생각을 해도 검사는 빠지지 않는다.

## 서브에이전트: 별도 컨텍스트에서 일하는 보조 에이전트

훅이 "빠르고 결정적인 검사"를 맡는다면, 오래 걸리고 출력이 긴 검증은 어디에 두어야 할까.
`go build ./...`, `go test ./...` 같은 종합 검증을 메인 대화에서 직접 돌리면 빌드 로그와 테스트 출력이 컨텍스트를 잠식한다.
검증 결과에서 정작 필요한 정보는 "통과했는가, 실패했다면 어디인가" 한두 줄인데, 그걸 얻으려고 수백 줄의 로그를 컨텍스트에 싣는 것은 배보다 배꼽이 크다.

서브에이전트(subagent)는 이 문제를 푸는 Claude Code의 장치다.
메인 에이전트가 작업을 위임하면, 서브에이전트는 **별도의 컨텍스트 창(context window)** 에서 실행된다.
서브에이전트가 어떤 명령을 돌리고 어떤 로그를 읽었든 그 과정은 메인 대화에 남지 않고, **마지막 응답 메시지만** 반환값으로 돌아온다.
함수 호출과 비슷하다. 함수 내부의 지역 변수가 호출자에게 보이지 않듯, 서브에이전트의 중간 과정은 캡슐화된다.

::: info [용어 풀이] 서브에이전트(Subagent)
메인 에이전트가 특정 작업을 통째로 맡기는 별도의 보조 에이전트다.
심부름을 시키면 과정은 알아서 처리하고 결과만 보고하는 조수와 같아서, 자기만의 작업 공간에서 일하고 결과 요약만 돌려준다.
덕분에 장황한 중간 과정이 메인 대화를 어지럽히지 않는다.
:::

서브에이전트 정의는 `.claude/agents/` 아래 마크다운 파일 하나로 이루어지며, 세 가지를 제어할 수 있다.

첫째, 컨텍스트 격리다.
장황한 탐색·빌드·테스트 로그가 메인 대화를 오염시키지 않는다.

둘째, 도구 제한이다.
frontmatter의 `tools`에 나열한 도구만 쓸 수 있어, 검증 에이전트가 실수로 코드를 고치는 일을 구조적으로 막을 수 있다.

셋째, 모델 지정이다.
작업 성격에 맞는 모델을 `model`로 지정할 수 있다.
설계나 구현은 고성능 모델이 필요하지만, "명령을 순서대로 돌리고 결과를 요약하라"는 작업은 저렴하고 빠른 모델로 충분하다.

## go-quality 에이전트 해부

Echo Flip의 검증 서브에이전트 정의를 보자.
다섯 가지 검사를 순서대로 실행하고 결과를 정해진 형식으로만 보고하라는 지시문이다.
`.claude/agents/go-quality.md` 전문이다.

```md
---
name: go-quality
description: Go 코드 전체 품질 검증(build/vet/test + Vercel 호환 검사)을 별도 컨텍스트에서 실행하고 요약만 반환한다. Go 코드를 수정한 뒤 커밋 전 종합 검증이 필요할 때 사용. 장황한 빌드/테스트 로그를 메인 대화에 남기지 않기 위한 저비용 검증 에이전트.
tools: Bash, Read, Grep, Glob
model: haiku
---

너는 echo-flip 저장소(Go 1.26 모듈, `github.com/benelog/echo-flip`)의 품질 검증 에이전트다.
저장소 루트에서 아래 검사를 순서대로 실행하라:

1. `gofmt -l .` — 출력이 있으면 포맷 위반
2. `go build ./...`
3. `go vet ./...`
4. `go test ./...`
5. `go build api/index.go` — Vercel 서버리스 빌드 호환 검사. Vercel은 이 파일을
   모듈 외부에서 단독 컴파일하므로 `internal/` 패키지를 import하면 여기서 실패한다.
   이 검사가 실패하면 Vercel 배포도 실패한다는 점을 명시하라.

보고 규칙 — 마지막 메시지가 그대로 반환값이 된다:
- 모두 통과: `PASS — build/vet/test/gofmt/vercel 모두 통과 (테스트 N개)` 한 줄만.
- 실패 시: 실패한 검사 이름과 원인이 되는 `파일:라인` + 에러 메시지 핵심만 발췌.
  전체 로그를 붙여넣지 마라. 통과한 항목은 나열하지 마라.
- 코드를 수정하지 마라. 검증과 보고만 한다.
```

### frontmatter: 정체성과 권한과 비용

**`name: go-quality`**: 에이전트의 식별자다.
메인 에이전트가 위임할 때, 또는 사용자가 직접 호출할 때 이 이름을 쓴다.

**`description`**: 이 필드는 단순한 설명문이 아니라 **자동 위임의 트리거**다.
메인 에이전트는 작업을 마주쳤을 때 등록된 서브에이전트들의 description을 보고 위임 여부를 판단한다.
"Go 코드를 수정한 뒤 커밋 전 종합 검증이 필요할 때 사용"이라고 사용 시점을 명시해 둔 덕에, 커밋을 앞둔 메인 에이전트가 별도 지시 없이도 이 에이전트를 호출할 확률이 높아진다.
서브에이전트의 약점(호출을 잊을 수 있다)을 description 작성으로 보완한 것이다.

**`tools: Bash, Read, Grep, Glob`**: 읽기와 실행만 허용하고 Edit·Write를 뺐다.
본문 마지막 줄의 "코드를 수정하지 마라"는 프롬프트 수준의 당부이고, tools 제한은 시스템 수준의 강제다.
같은 규칙을 두 층에 이중으로 걸어 둔 것인데, 앞서 본 "강제할 수 있는 것은 설정으로 강제한다"는 원칙이 여기서도 반복된다.

**`model: haiku`**: 이 에이전트는 가장 저렴한 모델로 돈다.
검증 작업의 본질은 "정해진 명령을 순서대로 실행하고, 종료 코드를 보고, 실패 메시지를 발췌"하는 것이다.
창의성도 설계 판단도 필요 없으므로 고성능 모델은 과잉이다.
커밋 전마다 반복 호출되는 작업이라 모델 단가 차이가 그대로 누적 절감으로 이어진다.
반대로 실패 원인 분석까지 시키고 싶다면 상위 모델이 낫다.
이 에이전트는 분석 대신 "파일:라인 발췌"라는 기계적 보고만 요구해서 haiku로 충분하도록 작업을 잘라 냈다.

### 본문: 다섯 단계 검사의 순서와 이유

검사 순서에도 논리가 있다.
싸고 빨리 실패하는 검사를 앞에 두었다.

1단계 `gofmt -l .`은 포맷 위반 파일 목록을 출력한다.
훅이 gofmt을 자동 적용하므로 보통은 깨끗하지만, 훅을 우회한 수정(셸에서 직접 생성한 파일 등)을 잡는 안전망이다.

2단계 `go build ./...`는 모듈 전체가 컴파일되는지 확인한다.
훅의 go vet은 수정된 패키지만 봤으므로, 다른 패키지로 번진 시그니처 변경 등은 여기서 처음 걸린다.

3단계 `go vet ./...`는 컴파일은 되지만 의심스러운 코드(사용되지 않는 결과값, 잘못된 포맷 문자열 등)를 모듈 전체에서 찾는다.

4단계 `go test ./...`는 전체 테스트를 실행한다.
6장에서 다룬 `internal/srs`의 테이블 주도 테스트 같은 것들이 여기서 돌아간다.
테스트는 다섯 검사 중 가장 오래 걸리므로 훅이 아니라 이곳에 있다.

### 5단계 검사가 특히 중요한 이유

5단계 `go build api/index.go`는 이 저장소 고유의 검사이며, 다섯 검사 중 가장 중요하다.
Echo Flip의 API는 2부에서 Vercel 서버리스 함수로 배포되는데, Vercel의 Go 런타임은 진입점인 `api/index.go`를 **모듈 컨텍스트 밖에서 단독 컴파일**한다.
Go의 `internal/` 패키지는 같은 모듈 안에서만 import할 수 있으므로, 이 파일이 `internal/` 패키지를 import하는 순간 로컬에서는 멀쩡히 빌드되지만 Vercel 배포는 실패한다.
그래서 이 파일이 필요로 하는 공유 코드는 전부 `pkg/app`에 있다.

이 제약은 `api/index.go` 상단 주석에도 박혀 있다(4장에서 인용한 그 주석이다).

`go build ./...`도 `go test ./...`도 이 문제를 잡지 못한다는 점이 핵심이다.
일반 빌드는 모듈 안에서 이루어지므로 internal import가 합법이기 때문이다.
`go build api/index.go`처럼 파일을 단독 지정해야 Vercel의 컴파일 방식이 근사하게 재현된다.
즉 이 검사는 **로컬 검증으로는 잡히지 않는 배포 실패를 커밋 전에 잡는 검사**다.
푸시하고 배포가 깨진 뒤에 알게 되면 수정→푸시→배포 대기의 왕복이 몇 분씩 걸리지만, 이 검사는 몇 초 만에 같은 답을 준다.
Vercel의 Go 함수 빌드 방식과 `pkg/app` 구조는 16장에서 자세히 다룬다.

### 보고 규칙: 반환값을 설계한다

정의 파일의 마지막 블록은 검사 방법이 아니라 **보고 방법**을 지시한다.
"마지막 메시지가 그대로 반환값이 된다"는 문장은 서브에이전트의 동작 원리를 프롬프트 안에 명시해 둔 것이다.
서브에이전트는 마지막 응답만 메인 에이전트에게 돌려주므로, 그 마지막 응답의 형식을 통제하는 것이 곧 인터페이스 설계다.

통과 시에는 `PASS — build/vet/test/gofmt/vercel 모두 통과 (테스트 N개)` 한 줄만 반환한다.
메인 에이전트의 컨텍스트에는 이 한 줄만 추가된다.
빌드 로그 수백 줄이 haiku의 컨텍스트에서 소화되고 요약 한 줄로 압축되어 돌아오는 구조다.

실패 시에는 실패한 검사 이름과 `파일:라인` 발췌만 반환한다.
"전체 로그를 붙여넣지 마라. 통과한 항목은 나열하지 마라"는 금지 규칙이 붙어 있는데, 이는 LLM이 성실함을 과시하려고 장황해지는 경향을 억누르기 위한 것이다.
메인 에이전트가 고치는 데 필요한 정보는 위치와 에러 메시지뿐이다.

눈치챘겠지만 이 보고 규칙은 훅의 "무음 성공, 시끄러운 실패" 원칙의 서브에이전트 버전이다.
훅은 성공 시 출력 0바이트, 서브에이전트는 성공 시 한 줄이다.
검증 장치는 좋은 소식일수록 말을 아끼도록 설계한다.

## 훅과 서브에이전트의 역할 분담

이제 두 장치를 나란히 놓고 분담의 논리를 정리해 보자.

| | 훅 (go-check.sh) | 서브에이전트 (go-quality) |
|---|---|---|
| 실행 시점 | 파일 수정 직후, 자동 | 커밋 전, 위임 호출 |
| 실행 보장 | 100% (시스템 강제) | 에이전트 판단에 의존 |
| 검사 범위 | 수정된 파일·패키지 | 모듈 전체 + 배포 호환 |
| 소요 시간 | 1초 안팎 | 수십 초 |
| 성공 시 비용 | 토큰 0 | 요약 한 줄 |
| 담당 검사 | gofmt 자동 적용, 패키지 go vet | build/vet/test 전체 + `go build api/index.go` |

분담 기준은 단순하다.
**빠르고 결정적이며 매번 필요한 검사는 훅으로, 느리고 포괄적이며 마일스톤마다 필요한 검증은 서브에이전트로.**
달리 말하면, 딱 잘라 예/아니오로 판정할 수 있는 검사는 기계인 훅에게 맡기고, 여러 정황을 함께 살펴 종합적으로 판단해야 하는 검증은 서브에이전트에게 맡기는 것이다.

이 분담에는 양방향의 트레이드오프가 있다.

훅 쪽의 제약은 지연이다.
훅은 모든 매칭 도구 호출을 잠시 붙잡으므로, 훅이 느리면 **모든 파일 수정이 느려진다**.
에이전트가 파일 열 개를 연달아 고치는 세션이라면 훅 지연은 열 배로 체감된다.
그래서 go-check.sh는 전체 테스트는커녕 전체 vet도 하지 않고, 수정된 패키지의 vet까지만 한다.
`go test ./...`를 훅에 넣고 싶은 유혹이 들 수 있지만, 그러면 파일 하나를 고칠 때마다 전체 테스트가 끝나기를 기다리게 된다.

서브에이전트 쪽의 제약은 비결정성이다.
서브에이전트 호출은 결국 메인 에이전트의 판단이므로, **호출을 잊을 수 있다**.
훅처럼 시스템이 강제하지 않는다.
Echo Flip이 쓰는 완화책은 앞서 본 description에 사용 시점("커밋 전")을 명시해 두는 것이다.
그래도 훅만큼의 보장은 아니므로, 더 확실히 하려면 커밋 절차를 다루는 프로젝트 지침에 go-quality 실행을 관례로 못 박거나 같은 검증을 CI(지속적 통합)에서 한 번 더 돌릴 수 있다.
훅 → 서브에이전트 → CI로 갈수록 느리지만 확실해지는 그물을 층층이 치는 셈이다.

반대 방향의 실수도 경계할 만하다.
모든 검사를 서브에이전트로 미루면 에이전트는 vet 위반이 쌓인 채 수십 분을 작업하다가 커밋 직전에 한꺼번에 실패를 마주한다.
오류는 발생 지점에 가까울수록 고치기 싸다.
훅이 수정 직후에 잡아 주는 즉시 피드백은 서브에이전트가 대체할 수 없다.

## 내 프로젝트에 적용하기

이 구조는 Go에 묶여 있지 않다.
자기 스택으로 옮기는 절차를 일반화해 보겠다.

첫째, 스택의 검사 도구를 두 무리로 나눈다.
기준은 "결정적이고 1~2초 안에 끝나는가"다.
포맷터(prettier, gofmt, black, rustfmt)와 빠른 린터·타입체커는 훅 후보다.
전체 빌드, 전체 테스트, 통합 검증은 서브에이전트 후보다.

둘째, 훅 스크립트를 go-check.sh의 골격 그대로 작성한다.
stdin JSON에서 경로 추출 → 확장자 필터 → 자동 수정 가능한 것은 조용히 수정 → 판정이 필요한 것만 검사 → 실패 시 stderr + exit 2.
TypeScript 프로젝트라면 이런 모양이 된다.

```bash
#!/usr/bin/env bash
set -u

f=$(jq -r '.tool_input.file_path // .tool_response.filePath // empty')
case "$f" in
  *.ts|*.tsx) ;;
  *) exit 0 ;;
esac
[ -f "$f" ] || exit 0

cd "${CLAUDE_PROJECT_DIR:-$(pwd)}" || exit 0

npx prettier --write "$f" >/dev/null 2>&1

if ! out=$(npx eslint "$f" 2>&1); then
  echo "eslint failed for $f:" >&2
  echo "$out" >&2
  exit 2
fi
exit 0
```

`tsc --noEmit`을 훅에 넣을지는 프로젝트 크기에 달렸다.
증분 컴파일로 1~2초면 훅에, 그 이상이면 서브에이전트로 보내는 것이 낫다.
훅에 넣은 검사가 느려졌다고 느껴지면 주저 없이 서브에이전트 쪽으로 옮기자.
분담선은 고정이 아니라 프로젝트가 자라며 조정하는 다이얼이다.

셋째, 서브에이전트 정의를 작성할 때는 go-quality.md의 네 가지 요소를 챙긴다.
description에 "언제 쓰는가"를 명시해 자동 위임을 유도하고, tools에서 수정 권한을 빼고, model은 기계적 검증이면 저비용 모델로 지정하고, 보고 규칙으로 반환값 형식(통과 시 한 줄, 실패 시 위치 발췌)을 통제한다.
특히 마지막 항목을 빼먹으면 서브에이전트가 로그를 통째로 반환해 격리의 의미가 사라지므로 주의하자.

넷째, 자기 프로젝트만의 "로컬에서 안 잡히는 배포 실패"를 찾아 서브에이전트 검사에 넣는다.
Echo Flip의 `go build api/index.go`에 해당하는 검사는 스택마다 다르다.
Next.js 정적 export라면 `next build`가 서버 전용 API 사용을 잡아 주고, Docker 배포라면 이미지 빌드가 그 역할을 한다.
배포가 한 번이라도 깨졌다면, 그 원인을 커밋 전에 재현하는 명령을 찾아 검사 목록에 추가하는 습관이 이 목록을 자라게 한다.

## 정리

이 장에서는 Echo Flip의 자동 품질 게이트를 이루는 세 파일을 해부했다.

프롬프트로 검사를 부탁하는 방식은 잊어버림, 토큰 낭비, 비결정성이라는 한계가 있다.
결정적 검사는 프롬프트가 아니라 시스템 설정으로 강제해야 한다는 것이 이 장의 첫 번째 원칙이다.

훅은 Claude Code의 라이프사이클 이벤트에 셸 명령을 거는 장치다.
`.claude/settings.json`은 Write·Edit 직후(PostToolUse)에 go-check.sh를 실행하도록 등록하고, 스크립트는 .go 파일만 골라 gofmt를 조용히 자동 적용한 뒤 수정된 패키지에 go vet을 돌린다.
성공하면 무음이라 토큰을 전혀 쓰지 않고, 실패하면 stderr + exit 2로 에러가 에이전트에게 피드백되어 스스로 고친다.
훅은 프롬프트와 달리 100% 실행이 보장된다.

서브에이전트는 별도 컨텍스트 창에서 실행되어 마지막 메시지만 반환하는 보조 에이전트다.
go-quality는 haiku 모델로 gofmt·build·vet·test에 더해 `go build api/index.go`로 Vercel 배포 호환성까지 검증하고, 통과 시 한 줄만 보고해 메인 대화의 컨텍스트를 지킨다.
특히 마지막 검사는 로컬 빌드·테스트로는 잡히지 않는 배포 실패를 커밋 전에 잡는다.

분담의 기준은 속도와 보장이다.
빠르고 결정적인 검사는 매 수정마다 훅으로 강제하고, 느리고 포괄적인 검증은 커밋 전에 서브에이전트로 격리 실행한다.
훅이 무거워지면 모든 편집이 함께 느려지고, 서브에이전트는 호출을 잊을 수 있다는 각각의 약점을 이해하고 분담선을 조정하는 것이 운영의 요령이다.

11장과 이 장에서 에이전트 쪽 작업 환경을 갖췄다면, 다음 13장에서는 사람 쪽 작업 환경을 정리한다.
저장소를 클론해 환경 변수 하나 없이 명령 하나로 서버 하나를 띄우고, VS Code 설정으로 포맷과 검증 명령을 공유하는 데까지가 그 내용이다.
이 장의 훅이 에이전트 쪽 입구에 건 규칙이라면, 13장의 에디터 설정은 같은 규칙을 사람 쪽 입구에 건 것이다.
두 입구가 모두 로컬에 있다는 약점은 15장에서 원격 게이트를 세워 메우고, 그렇게 검증된 앱을 16장에서 Vercel에 올린다.
