= 8장 Go 테스트와 품질 게이트: 도구, 훅, 서브에이전트

7장에서 읽은 SRS 알고리즘이나 스마트 덱 규칙이 정말 설계대로 동작하는지, 눈으로 코드를 읽는 것만으로는 확신하기 어렵다.
확인 작업을 사람 손에 맡기는 대신 검사 코드로 적어 두고 기계가 반복 실행하게 하는 것이 테스트다.

이 장에서는 먼저 Go가 테스트를 찾아 실행하는 규칙과 명령 사용법을 익힌다.
이어서 7장에서 읽은 코드들의 실제 테스트를 읽으며 테이블 주도 테스트(Table-driven Test) 패턴과 전수 검사, 경계값 검사를 확인한다.
DB를 붙여야 하는 테스트에서는 픽스처 헬퍼와 서브테스트가 어떻게 쓰이는지 보고, 커버리지 수치를 어디까지 믿을지도 짚는다.
그다음 gofmt, go vet 같은 품질 검사 도구를 살펴보고, gofmt가 강제하는 세로 정렬과 그 대가를 따로 짚는다.
이 테스트와 도구들은 사람을 위한 안전망인 동시에, 2장에서 본 AI 에이전트가 자기 작업을 스스로 검증하는 신호이기도 하다.
그래서 장의 후반부에서는 이 신호들을 에이전트의 루프에 자동으로 연결하는 실제 장치, 즉 훅과 서브에이전트 설정까지 해부한다.

== go test의 세 가지 규칙

Go는 테스트 러너를 별도 프레임워크 없이 언어 도구에 내장했고, 무엇이 테스트인지를 세 가지 규칙으로 정한다.

첫째, 테스트 코드는 파일 이름이 `_test.go`로 끝나는 파일에 담는다.
internal/srs/srs.go의 테스트는 같은 디렉터리의 internal/srs/srs_test.go에 있다.
둘째, 테스트 함수의 이름은 `Test`로 시작한다.
셋째, 테스트 함수는 `*testing.T` 타입의 매개변수 하나를 받는다.
실패 보고는 모두 이 `t`를 통해 이뤄진다.

이 규칙에 맞는 함수를 `go test` 명령이 알아서 찾아 실행한다.
`_test.go` 파일은 `go build`가 만드는 실행 파일에서 제외되므로, 테스트를 아무리 많이 써도 배포되는 서버의 크기에는 영향이 없다.

테스트 파일을 검증 대상과 같은 패키지로 선언하면, 소문자로 시작하는 비공개 식별자까지 직접 부를 수 있다.
internal/store/deckslug_test.go의 앞부분이 그 실례다.

[source,go]
----
package store

import (
	"strings"
	"testing"
)

func TestDeckSlugRoundTrip(t *testing.T) {
	for _, seq := range []int64{1, 2, 35, 36, 1295, 1296, 46656, 1_000_000, 1_679_615} {
		slug := encodeDeckSlug(seq)
		// ...
	}
}
----

첫 줄이 `package store`이므로 이 테스트 파일은 store 패키지의 일원이다.
그래서 패키지 밖에는 공개되지 않은 `encodeDeckSlug`를 아무 절차 없이 직접 호출한다.
6장에서 본 공개 규칙(대문자만 공개)이 같은 패키지의 테스트에는 장벽이 되지 않는다.

같은 패키지에 두는 것이 유일한 선택지는 아니다.
첫 줄을 `package store_test`로 적으면 테스트가 별도 패키지가 되어, 바깥의 다른 코드와 똑같이 공개된 식별자만 쓸 수 있다.
비공개 함수에 손을 대지 못하는 대신, 내부 구현을 아무리 바꿔도 공개 API가 그대로면 테스트가 깨지지 않는다는 장점이 있다.
이 저장소는 검증하고 싶은 대상이 비공개인 경우가 많아 같은 패키지 방식을 골랐다.
deckslug_test.go의 `encodeDeckSlug`가 그렇고, 뒤에서 볼 litestore_test.go는 DB 핸들을 직접 열어 저장된 행 수를 세며, templates_test.go는 내부 템플릿 목록을 들여다본다.
반대로 라이브러리를 만들어 공개하는 처지라면, 사용자와 같은 입구로만 들어오도록 강제하는 `_test` 패키지가 더 낫다.

`Test` 말고도 `go test`가 알아보는 접두사가 더 있다.
`Benchmark`로 시작하는 함수는 코드가 얼마나 빠른지 반복 측정하고(`go test -bench .`), `Fuzz`로 시작하는 함수는 입력을 무작위로 변형해 가며 프로그램을 깨뜨리는 값을 스스로 찾아 나선다(`go test -fuzz`).
Flashcard에는 성능이 문제가 된 코드가 없어서 벤치마크를 쓰지 않았고, 퍼징(fuzzing)도 넣지 않았다.
다만 뒤에 나올 슬러그 디코딩처럼 "아무 문자열이나 들어올 수 있는 입구"는 퍼징이 잘 맞는 대상이라, 경계값을 손으로 나열하는 것을 넘어 자동 탐색까지 원하게 되면 그때 꺼내 쓰면 된다.

== 테스트 실행 명령

테스트 실행은 `go test` 명령 하나로 끝난다.
명령 뒤에 플래그(flag)를 덧붙여 동작을 바꾼다.

.[용어 풀이] 플래그(Flag, 명령행 옵션)
[NOTE]
====
명령의 동작을 바꾸려고 명령 이름 뒤에 덧붙이는 짧은 옵션이다.
`-`나 `--`로 시작하는 표기를 쓰고(`go test -v`, `gofmt -w`), 값이 필요한 플래그는 뒤에 값을 함께 적는다(`go test -run TestEaseFloor`).
같은 명령에 어떤 스위치를 켜고 끄느냐로 다른 일을 시키는 장치라고 보면 된다.
이 책의 명령 예제에서 `-`로 시작하는 부분은 모두 플래그다.
====

자주 쓰는 변형은 다음과 같다.

[source,bash]
----
go test ./...                              # 모듈 전체 테스트
go test ./internal/srs                     # 특정 패키지만
go test -run TestEaseFloor ./internal/srs  # 이름이 매칭되는 테스트만
go test -v ./internal/srs                  # 개별 테스트의 이름과 결과까지 상세 출력
go test -count=1 ./...                     # 캐시된 지난 결과를 무시하고 다시 실행
go test -race ./...                        # 동시 실행 오류까지 검사
----

`./...`은 현재 디렉터리 아래의 모든 패키지를 뜻하는 경로 패턴이다.
`-run` 뒤에는 테스트 함수 이름의 일부를 적어, 방금 고친 코드와 관련된 테스트만 골라 빠르게 돌릴 수 있다.
`-v`(verbose)는 통과한 테스트의 이름까지 하나하나 보여 준다.
평소에는 패키지별 `ok` 한 줄과 실패만 보고하다가, 무엇이 실행됐는지 확인하고 싶을 때 붙인다.
나머지 두 플래그는 설명할 것이 있어 아래에서 따로 다룬다.

=== 성공과 실패의 출력

저장소 루트에서 `go test ./...`를 실행하면 이런 출력이 나온다.

----
?   	github.com/benelog/flashcard/api	[no test files]
?   	github.com/benelog/flashcard/cmd/migrate	[no test files]
?   	github.com/benelog/flashcard/cmd/server	[no test files]
?   	github.com/benelog/flashcard/internal/auth	[no test files]
ok  	github.com/benelog/flashcard/internal/config	0.003s
?   	github.com/benelog/flashcard/internal/db	[no test files]
?   	github.com/benelog/flashcard/internal/handlers	[no test files]
ok  	github.com/benelog/flashcard/internal/litestore	0.229s
ok  	github.com/benelog/flashcard/internal/smartrules	0.004s
ok  	github.com/benelog/flashcard/internal/srs	0.006s
ok  	github.com/benelog/flashcard/internal/store	0.010s
ok  	github.com/benelog/flashcard/internal/web	0.012s
?   	github.com/benelog/flashcard/pkg/app	[no test files]
----

패키지마다 한 줄이다.
`ok`는 그 패키지의 테스트가 모두 통과했다는 뜻이고 뒤의 숫자는 걸린 시간이다.
`[no test files]`는 `_test.go` 파일이 하나도 없는 패키지라는 표시이며 실패가 아니다.

실패하면 출력이 달라진다.
다음은 srs_test.go의 기대값을 일부러 틀리게 고쳐 놓고 돌린 결과다.

----
--- FAIL: TestCorrectProgression (0.00s)
    srs_test.go:18: rep 3: interval = 16, want 15
FAIL
FAIL	github.com/benelog/flashcard/internal/srs	0.004s
FAIL
----

실패한 테스트 함수의 이름, 실패를 보고한 파일과 줄 번호, 그리고 코드에 적어 둔 메시지가 차례로 찍힌다.
`rep 3: interval = 16, want 15`는 뒤에서 볼 `t.Fatalf`의 서식 문자열이 실제 값으로 채워진 것이다.
어느 테스트가, 어디서, 무엇이 기대와 달랐는지가 한 화면에 모두 담기기 때문에 사람도 에이전트도 이 출력만 보고 고칠 곳을 찾아간다.
실패 메시지에 기대값과 실제값을 함께 넣는 습관이 중요한 이유가 여기에 있다.
`t.Fatal("실패")`라고만 적어 두면 출력은 나오지만 무엇이 잘못됐는지는 알려 주지 않는다.
이 명확한 성공/실패 신호가 뒤에서 이야기할 자동화의 재료가 된다.

=== 테스트 결과는 캐시된다

같은 명령을 한 번 더 돌리면 출력이 이렇게 바뀐다.

----
ok  	github.com/benelog/flashcard/internal/config	0.004s
ok  	github.com/benelog/flashcard/internal/litestore	(cached)
ok  	github.com/benelog/flashcard/internal/smartrules	(cached)
ok  	github.com/benelog/flashcard/internal/srs	(cached)
----

`(cached)`는 그 패키지의 코드도 테스트도 지난번과 똑같아서, Go가 테스트를 다시 돌리지 않고 저장해 둔 결과를 그대로 보여 줬다는 뜻이다.
결과가 달라질 수 없는 테스트를 건너뛰므로 반복 실행이 빨라진다.
처음 보면 "테스트가 안 돌았다"고 오해하기 쉽지만 정상 동작이다.

그래도 정말 다시 돌리고 싶을 때가 있다.
그때 붙이는 것이 `-count=1`이다.
`-count`는 원래 각 테스트를 몇 번 반복할지 정하는 플래그인데, 캐시는 반복 횟수가 1일 때만 쓰이므로 `-count=1`이 캐시를 무시하는 관례적인 표기가 됐다.
13장에서 에디터의 테스트 실행 설정에 이 플래그를 넣어 두는 이유가 이것이다.
버튼을 눌러 본 결과가 "방금 정말 실행된 결과"임을 보장하기 위해서다.

=== -race: 동시 실행 오류 검사

`-race`는 여러 작업이 같은 데이터를 동시에 건드려 생기는 오류를 잡아내는 플래그다.

.[용어 풀이] 경쟁 상태(Race Condition)
[NOTE]
====
두 가지 일이 동시에 같은 데이터를 건드릴 때, 누가 먼저 닿느냐에 따라 결과가 달라지는 상태다.
같은 장부에 두 사람이 동시에 숫자를 고쳐 적으면 한쪽의 기록이 소리 없이 사라지는 것과 같다.
서버는 여러 사용자의 요청을 동시에 처리하므로 이런 오류가 생길 여지가 늘 있는데, 어쩌다 한 번씩만 재현되어 눈으로 찾아내기가 가장 어려운 종류의 버그다.
====

Go 서버는 요청 여러 개를 동시에 처리한다.
`-race`를 붙이면 테스트가 도는 동안 메모리 접근을 감시해 두 작업이 같은 값을 동시에 건드리는 대목을 신고한다.
대신 실행이 몇 배 느려지므로 평소에 붙이는 플래그는 아니다.
Flashcard의 코드는 요청 사이에 공유하는 상태를 두지 않아서 상시 검사 목록에는 넣지 않았다.
카운터나 캐시처럼 요청들이 함께 쓰는 값을 서버에 들이는 순간, 검사 목록에 가장 먼저 추가할 후보가 된다.
그때는 로컬과 CI 어느 한쪽에만 붙이지 말고 15장에서 볼 세 게이트에 함께 붙여야 한다.

== SRS 알고리즘 테스트 읽기

이제 실제 테스트를 읽어 보자.
internal/srs/srs_test.go에서 정답이 이어질 때의 간격 진행을 검증하는 테스트다.

[source,go]
----
var now = time.Date(2026, 7, 7, 12, 0, 0, 0, time.UTC)

func TestCorrectProgression(t *testing.T) {
	s := NewState()
	// ease: 2.5→2.6→2.7→2.8; rep3: round(6×2.7)=16, rep4: round(16×2.8)=45
	wantIntervals := []float64{1, 6, 16, 45}
	for i, want := range wantIntervals {
		var due time.Time
		s, due = Grade(s, true, now)
		if s.IntervalDays != want {
			t.Fatalf("rep %d: interval = %v, want %v", i+1, s.IntervalDays, want)
		}
		wantDue := now.Add(time.Duration(want * float64(24*time.Hour)))
		if !due.Equal(wantDue) {
			t.Fatalf("rep %d: due = %v, want %v", i+1, due, wantDue)
		}
	}
	// ...
}
----

이것이 테이블 주도 테스트의 가장 단순한 형태다.
기대값 목록 `wantIntervals`를 테이블로 놓고, `for ... range` 루프가 각 행에 같은 검증을 적용한다.
케이스 추가가 테이블에 한 줄 더하는 일이 되므로, 분기마다 테스트 함수를 복제하는 것보다 관리할 코드가 적다.

.[용어 풀이] 테이블 주도 테스트(단위 테스트 포함)
[NOTE]
====
단위 테스트는 함수 하나가 기대대로 동작하는지 자동으로 확인하는 작은 검사 코드다.
테이블 주도 테스트는 그 검사를 "입력과 기대값의 목록(표)"으로 만들어, 한 번의 반복문으로 표의 모든 줄을 검사하는 방식이다.
검사할 경우를 늘리고 싶으면 표에 한 줄만 더하면 되므로, 비슷한 테스트 함수를 복사해 늘리는 것보다 관리가 쉽다.
====

파일 상단의 `var now = time.Date(...)`도 중요한 장치다.
7장에서 `Grade`가 현재 시각을 인자로 받도록 설계한 것을 봤는데, 그 덕분에 테스트는 고정 시각을 주입해 언제 실행해도 같은 결과를 얻는다.
순수 함수 설계가 결정적인(항상 같은 결과를 내는) 테스트로 보상받는 장면이다.

실패를 보고하는 방법이 두 가지라는 것도 이 테스트에서 배울 수 있다.

|===
| 메서드 | 하는 일 | 쓰는 상황

| `t.Fatalf` | 실패를 보고하고 그 테스트 함수를 즉시 중단 | 뒤 검증이 앞 결과에 의존할 때
| `t.Errorf` | 실패를 기록하고 다음 검증을 계속 진행 | 독립적인 검증의 실패를 한 번에 모두 보고할 때
|===

`TestCorrectProgression`은 두 번째 복습의 검증이 첫 번째 복습의 결과 위에서 이뤄지는 진행형 테스트라, 앞 단계가 틀리면 뒤 단계는 의미가 없다.
그래서 즉시 중단하는 `t.Fatalf`가 맞다.
서로 독립적인 검증이 이어질 때는 `t.Errorf`로 끝까지 진행해, 한 번의 실행에서 실패를 최대한 많이 수집하는 편이 낫다.

같은 파일의 다른 테스트도 짧게 보자.
오답이 거듭될 때 용이도가 어디까지 내려가는지 확인하는 테스트다.

[source,go]
----
func TestEaseFloor(t *testing.T) {
	s := NewState()
	for range 10 {
		s, _ = Grade(s, false, now)
	}
	if s.EaseFactor != MinEase {
		t.Fatalf("ease = %v, want floor %v", s.EaseFactor, MinEase)
	}
}
----

열 번 연속 틀려도 용이도가 하한 1.3 아래로 내려가지 않는지 확인한다.
`for range 10`은 Go 1.22부터 지원되는 정수 순회 문법으로, 인덱스가 필요 없는 n회 반복을 간결하게 쓴다.
반환값 중 필요 없는 것은 빈 식별자 `_`로 버린다.
이 파일에는 오답 시 상태 초기화를 확인하는 `TestLapseResets`도 있는데, 경계 조건(리셋, 하한)마다 테스트가 하나씩 배치된 구성을 눈여겨보자.

== 규칙 파싱 테스트와 테이블 패턴

internal/smartrules/rules_test.go에서는 입력 자체를 테이블로 놓는 패턴을 볼 수 있다.
스마트 덱 규칙 네 종류가 모두 SQL 조회문으로 제대로 바뀌는지 확인하는 테스트다.

[source,go]
----
func TestQueriesBuild(t *testing.T) {
	for _, raw := range []string{
		`{"type":"high_error"}`,
		`{"type":"stale"}`,
		`{"type":"tag","tags":["verb"]}`,
		`{"type":"recent"}`,
	} {
		r, err := Parse([]byte(raw))
		if err != nil {
			t.Fatal(err)
		}
		if q, _ := r.Query(); q == "" {
			t.Fatalf("empty query for %s", raw)
		}
		// ...
	}
}
----

네 가지 규칙 타입의 JSON을 슬라이스 리터럴(literal, 값을 코드에 그대로 적어 넣는 표기)로 나열하고, 모든 타입이 빈 SQL을 만들지 않는지 한 루프로 검증한다.
새 규칙 타입을 추가하면 테이블에 JSON 한 줄만 더하면 된다.
같은 파일에는 기본값 채움(`TestParseDefaults`), 미지의 타입 거부(`TestParseRejectsUnknownType`), 상한 초과 시 되돌림(`TestLimitClamped`) 등 `Validate`의 분기마다 대응하는 테스트가 있다.

== 전수 검사와 경계값: 덱 슬러그 테스트

테이블 패턴이 가장 진지하게 쓰인 곳은 덱 슬러그 테스트다.
6장 상수 절에서 본 internal/store/deckslug.go는 덱 번호(seq)를 네 글자 슬러그로 바꾸는 `encodeDeckSlug`와 그 역변환 `decodeDeckSlug`를 담고 있다.
연속된 번호가 뻔한 슬러그로 드러나지 않도록 곱셈으로 흩뿌리는 계산이 들어 있어서, 산수가 조금이라도 어긋나면 서로 다른 덱이 같은 슬러그를 받는 사고가 난다.

이런 성질은 몇 개의 예시로 확인하는 대신 아예 전수로 검사할 수 있다.
internal/store/deckslug_test.go의 핵심 테스트다.

[source,go]
----
// A broad round trip guards against an arithmetic slip in the permutation or
// its inverse, and confirms slugs are always 4 chars and never collide.
func TestDeckSlugBijective(t *testing.T) {
	seen := make(map[string]int64)
	for seq := int64(1); seq < 20000; seq++ {
		slug := encodeDeckSlug(seq)
		if len(slug) != slugLen {
			t.Fatalf("encodeDeckSlug(%d) = %q, want %d chars", seq, slug, slugLen)
		}
		if prev, dup := seen[slug]; dup {
			t.Fatalf("slug %q collides: seq %d and %d", slug, prev, seq)
		}
		seen[slug] = seq
		if got, err := decodeDeckSlug(slug); err != nil || got != seq {
			t.Fatalf("round trip %d -> %q -> %d (err %v)", seq, slug, got, err)
		}
	}
}
----

1번부터 2만 번 직전까지 모든 번호를 순회하며 세 가지를 확인한다.
슬러그가 항상 네 글자인지, 앞서 나온 슬러그와 겹치지 않는지, 다시 풀면 원래 번호로 돌아오는지다.
충돌 검사에는 `seen` 맵이 쓰였는데, 7장에서 본 comma-ok 관용구(`prev, dup := seen[slug]`)로 "이 슬러그가 전에 나왔는지"를 확인하고, 나왔다면 어느 번호와 부딪혔는지까지 메시지에 담는다.
2만 번의 반복이 부담스러워 보일 수 있지만, DB도 네트워크도 건드리지 않는 순수 계산이라 전수 검사조차 몇 밀리초면 끝난다.

정상 입력의 반대편, 거부해야 할 입력도 테이블로 고정한다.

[source,go]
----
func TestDecodeDeckSlugInvalid(t *testing.T) {
	// Wrong length, non-Base36 byte, multibyte input, and the zero slug (which
	// maps back to seq 0) must all be rejected.
	for _, s := range []string{"", "abc", "abcde", "abc!", "한글", "0000"} {
		if _, err := decodeDeckSlug(s); err == nil {
			t.Errorf("decodeDeckSlug(%q) expected error", s)
		}
	}
}
----

빈 문자열, 세 글자와 다섯 글자(잘못된 길이), 허용되지 않는 문자, 한글 같은 멀티바이트 입력, 그리고 존재하지 않는 0번 덱으로 되돌아가는 `"0000"`까지, 여섯 가지 경계 입력이 한 줄짜리 테이블에 나란히 놓였다.
URL의 슬러그는 사용자가 아무 문자열이나 넣을 수 있는 입구라, 이런 경계값이 버그가 가장 잘 숨는 곳이다.
경계 사례를 발견할 때마다 테이블에 한 줄로 고정해 두면 같은 실수가 다시는 조용히 재발하지 못한다.
각 입력의 검증이 서로 독립이므로, 여기서는 `t.Errorf`로 끝까지 진행하며 실패를 한 번에 모두 보고한다.

== DB를 붙인 테스트: 픽스처 헬퍼와 서브테스트

여기까지 읽은 테스트는 DB도 네트워크도 없는 순수 계산이었다.
그런데 이 저장소에서 가장 큰 테스트 파일은 정반대편에 있다.
internal/litestore/litestore_test.go는 488줄로 저장소 테스트 코드의 절반이 넘고, 진짜 SQLite 파일을 열어 덱과 카드를 만들고 복습을 기록하고 통계를 읽는다.
13장의 로컬 모드가 파일 하나로 도는 DB를 쓰기 때문에 가능한 일이다.
Postgres였다면 테스트마다 서버가 필요했겠지만, SQLite는 임시 파일 하나면 된다.

DB를 붙이는 순간 순수 함수 테스트에는 없던 문제가 셋 생긴다.
테스트마다 깨끗한 DB가 필요하고, 끝나면 치워야 하고, 덱과 카드를 미리 만들어 두는 준비 과정이 매번 반복된다.
Go의 `testing` 패키지에 이 셋을 각각 맡는 도구가 있다.
파일 첫머리의 헬퍼 함수가 셋을 모두 쓴다.
테스트마다 새 임시 폴더에 SQLite DB 파일을 열고, 끝나면 닫히도록 예약해 두고, 시험용 사용자까지 만들어 돌려주는 준비 코드다.

[source,go]
----
func testStore(t *testing.T) (*Store, uuid.UUID) {
	t.Helper()
	s, err := Open(filepath.Join(t.TempDir(), "test.db"))
	if err != nil {
		t.Fatal(err)
	}
	t.Cleanup(func() { s.Close() })
	userID := uuid.New()
	if _, err := s.GetOrCreateProfile(context.Background(), userID, ""); err != nil {
		t.Fatal(err)
	}
	return s, userID
}
----

`t.TempDir()`은 이 테스트만 쓰는 임시 디렉터리를 만들어 주고, 테스트가 끝나면 안의 내용까지 통째로 지운다.
테스트마다 새 디렉터리가 나오므로 DB 파일이 서로 섞이지 않고, 앞 테스트가 남긴 데이터가 뒤 테스트의 결과를 바꾸는 일도 없다.
치우는 코드를 사람이 쓰지 않아도 되고, 테스트가 도중에 실패해도 정리는 보장된다.

`t.Cleanup()`은 테스트가 끝날 때 실행할 코드를 등록한다.
여기서는 열어 둔 DB 연결을 닫는다.
`defer`와 비슷하지만 등록한 함수가 반환될 때가 아니라 *테스트가* 끝날 때 실행된다는 점이 다르다.
그 덕분에 이렇게 헬퍼 안에서 등록해 두어도, 정리는 헬퍼를 호출한 테스트 함수가 끝나는 시점에 일어난다.

`t.Helper()`는 이 함수가 헬퍼임을 알리는 표시다.
이 한 줄을 적어 두면 헬퍼 안의 `t.Fatal`이 실패했을 때 실패 위치가 헬퍼의 줄 번호가 아니라 *헬퍼를 호출한 테스트의 줄 번호*로 보고된다.
헬퍼를 열 곳에서 부르는데 실패 위치가 늘 헬퍼 내부로 찍히면 어느 테스트가 깨졌는지 알 수 없으므로, 한 줄로 그것을 막는다.

이 헬퍼 덕분에 각 테스트의 첫 줄은 `s, userID := testStore(t)` 하나로 끝난다.
같은 파일의 `mustDeck`과 `mustCard`도 같은 구조로, 덱과 카드를 만들다 에러가 나면 그 자리에서 테스트를 중단한다.
이렇게 테스트의 밑준비를 담당하는 코드를 픽스처라고 부른다.

.[용어 풀이] 픽스처(Fixture)
[NOTE]
====
테스트를 돌리기 전에 갖춰 두는 고정된 환경과 데이터다.
빈 DB, 미리 만들어 둔 사용자와 덱처럼 "이 상태에서 시작한다"고 정해 두는 밑준비를 가리킨다.
모든 테스트가 같은 밑준비에서 출발해야 결과가 재현되므로, 준비하는 코드와 치우는 코드를 헬퍼 함수로 묶어 두는 것이 관례다.
====

=== t.Run: 이름이 붙은 서브테스트

앞에서 본 테이블 주도 테스트는 루프가 테이블을 돌며 같은 검증을 반복했다.
케이스가 늘고 케이스마다 조건이 달라지면 각 행에 이름을 붙이고 싶어진다.
`t.Run`이 그 일을 한다.
다음은 스마트 덱 규칙 네 가지가 각각 어떤 카드를 어떤 순서로 골라내는지 검증하는 litestore_test.go의 테스트다.

[source,go]
----
tests := []struct {
	name string
	rule smartrules.Rule
	want []string // expected card texts in rule order
}{
	{"high_error",
		smartrules.Rule{Type: smartrules.HighError, MinAttempts: 3, MinErrorRate: 0.4, Limit: 20},
		[]string{"alpha"}},
	{"stale nulls first",
		smartrules.Rule{Type: smartrules.Stale, NotReviewedDays: 7, Limit: 20},
		[]string{"alpha", "beta"}},
	// ...
}
for _, tt := range tests {
	t.Run(tt.name, func(t *testing.T) {
		cards, err := s.CardsByRule(ctx, userID, tt.rule)
		// ...
	})
}
----

테이블의 각 행이 이름(`name`), 입력(`rule`), 기대값(`want`)을 가진 구조체가 됐다.
그리고 루프 본문이 `t.Run(tt.name, func(t *testing.T) { ... })`으로 감싸였다.
Go에서 테이블 주도 테스트라고 하면 가장 먼저 떠올리는 것이 이 형태다.

무엇이 좋아지는지는 `-v`를 붙인 출력에서 드러난다.

----
=== RUN   TestSmartRules
=== RUN   TestSmartRules/high_error
=== RUN   TestSmartRules/stale_nulls_first
=== RUN   TestSmartRules/tag_overlap
=== RUN   TestSmartRules/recent_newest_first
--- PASS: TestSmartRules (0.04s)
    --- PASS: TestSmartRules/high_error (0.00s)
    --- PASS: TestSmartRules/stale_nulls_first (0.00s)
    --- PASS: TestSmartRules/tag_overlap (0.00s)
    --- PASS: TestSmartRules/recent_newest_first (0.00s)
PASS
ok  	github.com/benelog/flashcard/internal/litestore	0.044s
----

각 케이스가 `부모테스트/케이스이름`이라는 독립된 이름을 갖는다(이름 안의 빈칸은 밑줄로 바뀐다).
실패하면 몇 번째 행인지가 아니라 어느 케이스인지가 이름으로 나온다.
그리고 그 이름을 `-run`에 그대로 넘겨 한 케이스만 돌릴 수 있다.

[source,bash]
----
go test -run 'TestSmartRules/tag_overlap' ./internal/litestore
----

앞 절의 `-run`이 테스트 함수 단위였다면, 서브테스트를 쓰면 케이스 단위까지 좁혀진다.
케이스가 스무 개인 테이블에서 열일곱 번째만 깨졌을 때 그 하나만 몇 밀리초에 돌려 가며 고칠 수 있다.
서브테스트는 각각 독립적으로 실패를 보고하기도 한다.
한 케이스 안에서 `t.Fatalf`로 중단해도 중단되는 것은 그 서브테스트뿐이고 나머지 케이스는 계속 실행된다.

케이스마다 이름을 지어 줄 가치가 있는지가 `t.Run`을 쓸지 정하는 기준이다.
앞서 본 `wantIntervals` 같은 단순한 기대값 목록은 루프만으로 충분하고, 케이스마다 조건과 기대 결과가 다른 이 규칙 테스트에는 `t.Run`이 맞다.

== 커버리지: 숫자가 말하는 것과 말하지 않는 것

"테스트를 충분히 썼는가"를 재는 도구도 `go test`에 들어 있다.
`-cover` 플래그는 테스트가 한 번이라도 실행한 코드가 전체의 몇 퍼센트인지 알려 준다.

[source,bash]
----
go test -cover ./internal/...
----

Flashcard의 실제 출력을 커버리지가 높은 순으로 다시 늘어놓으면 다음과 같다.

----
ok  	github.com/benelog/flashcard/internal/srs	0.006s	coverage: 100.0% of statements
ok  	github.com/benelog/flashcard/internal/smartrules	0.007s	coverage: 92.9% of statements
ok  	github.com/benelog/flashcard/internal/litestore	0.235s	coverage: 72.0% of statements
ok  	github.com/benelog/flashcard/internal/config	0.012s	coverage: 56.5% of statements
ok  	github.com/benelog/flashcard/internal/web	0.013s	coverage: 9.9% of statements
ok  	github.com/benelog/flashcard/internal/store	0.011s	coverage: 8.6% of statements
	github.com/benelog/flashcard/internal/auth		coverage: 0.0% of statements
	github.com/benelog/flashcard/internal/db		coverage: 0.0% of statements
	github.com/benelog/flashcard/internal/handlers		coverage: 0.0% of statements
----

테스트가 하나도 없는 패키지는 `ok`도 시간도 없이 0.0%로만 나온다.

이 숫자들은 이 프로젝트가 테스트를 어디에 놓았는지를 정직하게 보여 준다.
계산 규칙이 전부인 srs가 100%이고 규칙 파싱과 SQLite 스토어가 뒤를 잇는다.
반면 핸들러와 인증은 0%다.

낮은 숫자를 해석할 때는 조심할 것이 있다.
internal/store가 8.6%인 것은 이 패키지를 소홀히 해서가 아니다.
이 패키지는 대부분이 Postgres에 SQL을 보내는 코드이고(cards.go 288줄, sharing.go 218줄), 테스트가 도는 곳은 DB 없이 계산되는 슬러그 코드뿐이기 때문이다.
Postgres 코드를 검사하려면 테스트가 붙을 진짜 Postgres 서버가 필요한데, 그 준비 비용을 치르는 대신 같은 동작을 SQLite 구현으로 검증하는 쪽을 골랐다.
litestore의 72%가 그 결정의 결과다.
두 구현이 같은 인터페이스를 만족한다는 점(9장)이 이 선택의 근거이고, 그럼에도 Postgres 전용 SQL의 오류는 이 그물에 걸리지 않는다는 것이 대가다.

그래서 커버리지는 목표로 삼기에 나쁜 숫자다.
"90%를 채운다"는 규칙을 세우면 검증할 가치가 낮은 코드에도 형식적인 테스트가 붙고, 정작 숫자는 올라가는데 잡히는 버그는 늘지 않는다.
커버리지는 목표가 아니라 *테스트가 없는 곳을 확인하는 지도*로 읽을 때 쓸모가 있다.
핸들러가 0%라는 사실은 이 저장소의 핸들러가 얇게 유지되고 있다는 9장의 설계와 함께 읽어야 하며, 그 계층에 로직이 쌓이기 시작하면 가장 먼저 테스트를 붙일 곳이라는 신호이기도 하다.

== 테스트는 AI 에이전트의 검증 신호

이 테스트들은 사람을 위한 안전망인 동시에 AI 에이전트를 위한 명세다.
기대 동작이 테스트로 고정되어 있을수록 에이전트에게 맡길 수 있는 작업의 범위가 넓어진다.
그래서 이 프로젝트는 틀려도 화면에서 티가 나지 않는 곳부터 테스트로 덮었다.
사람이 암산으로 검산하기 어려운 계산(srs), 무엇이든 들어올 수 있는 입구(슬러그 디코딩, 규칙 파싱), 여러 테이블을 한꺼번에 고치는 DB 흐름(복습 기록, 공유 덱)이 그런 곳이다.
반대로 핸들러와 화면처럼 브라우저에서 바로 확인되는 계층은 사람이 눌러 보는 쪽에 남겨 두었고, 그 선택이 앞 절의 커버리지 표에 그대로 드러난다.
테스트가 AI 협업에서 갖는 의미는 2장에서 봤고, 테스트 실행을 강제하는 자동화는 이 장의 후반부에서 다룬다.

== 품질 검사 도구

테스트가 "동작이 맞는가"를 확인한다면, 그 앞 단계인 "코드가 형식과 상식에 맞는가"는 도구가 기계적으로 검사한다.
Go 툴체인에는 이런 품질 검사 도구가 함께 들어 있다.

.[용어 풀이] 포매터와 정적 분석(gofmt·go vet)
[NOTE]
====
포매터는 줄 간격·들여쓰기 같은 코드의 겉모양을 정해진 하나의 형태로 자동 정리해 주는 도구이고(gofmt), 정적 분석은 코드를 실행하지 않고 읽기만 해서 버그가 될 만한 대목을 찾아 주는 도구다(go vet).
글을 소리 내어 낭독하지 않고도 맞춤법·문법 오류를 짚어 주는 검사기와 같다.
사람의 판단 없이 늘 같은 결과를 내므로, 사람이든 AI든 코드를 고칠 때마다 자동으로 돌리기에 알맞다.
====

*gofmt*는 표준 포맷터다.

[source,bash]
----
gofmt -l .   # 포맷이 어긋난 파일 나열
gofmt -w .   # 제자리 수정
----

들여쓰기, 정렬, 괄호 위치를 단 하나의 표준형으로 고정하며 설정 옵션이 아예 없다.
"탭이냐 스페이스냐" 같은 논쟁이 Go 커뮤니티에 존재하지 않는 이유다.
모든 Go 코드가 같은 모양이므로 사람이든 AI든 처음 보는 코드를 읽는 비용이 줄어든다.
gofmt가 고정하는 것 중 처음 보면 낯선 세로 정렬은 이 장 마지막 절에서 따로 다룬다.

*go vet*은 정적 분석기다.

[source,bash]
----
go vet ./...
----

컴파일은 통과하지만 십중팔구 버그인 코드, 예컨대 `Printf` 형식 문자열과 인자의 불일치, 도달 불가능한 코드, 잘못된 구조체 태그를 잡아낸다.

*go build*는 빌드 명령이지만 그 자체로 가장 기본적인 검사다.
Go는 사용하지 않는 임포트와 지역 변수를 컴파일 에러로 처리하므로, `go build ./...` 통과는 타입이 어긋난 코드와 쓰지 않는 코드가 없다는 확인까지 겸한다.

*staticcheck*은 Go 팀이 아니라 외부에서 만든 서드파티(third-party) 정적 분석기로, `go vet`보다 넓은 150여 가지 검사를 제공한다.

[source,bash]
----
staticcheck ./...
----

사용되지 않는 코드, 비효율적인 관용구, 오래된 API 사용 등을 짚어 준다.
표준 도구가 아니므로 `go install`로 설치해야 하지만, Go 커뮤니티에서 사실상의 표준으로 통한다.

이 도구들은 빠르고, 결정적이고, 결과 해석에 사람의 판단이 필요 없어서 자동화의 재료로 이상적이다.
이 프로젝트에서는 AI 에이전트가 Go 파일을 수정할 때마다 훅(Hook)으로 gofmt와 go vet이 자동 실행되도록 구성했는데, 그 구조와 설계 의도는 이 장 후반부의 훅 절에서 자세히 다룬다.

== 세로 정렬은 취향이 아니라 규칙이다

gofmt가 고정하는 항목 중에 처음 보면 낯선 것이 세로 정렬(vertical alignment)이다.
구조체 필드를 한 칸씩만 띄워 써도 gofmt가 컬럼을 맞춰 준다.

[source,go]
----
// 입력: 한 칸씩만 띄웠다
type User struct {
	ID int64 `json:"id"`
	Name string `json:"name"`
	EmailAddress string `json:"email"`
	OK bool // 활성 여부
}

// gofmt 출력: 컬럼이 맞춰졌다
type User struct {
	ID           int64  `json:"id"`
	Name         string `json:"name"`
	EmailAddress string `json:"email"`
	OK           bool   // 활성 여부
}
----

구조체의 필드명과 타입, 태그, 줄 끝 주석이 이렇게 정렬되고 `const`·`var` 블록과 맵 리터럴의 값도 마찬가지다.
반대로 함수 본문의 문장에서는 손으로 맞춘 정렬을 gofmt가 도로 지운다.

[source,go]
----
// 입력: 손으로 맞춘 정렬
a      := 1
bbbbbb := 2
do(1,    2,   3)

// gofmt 출력: 정렬이 지워졌다
a := 1
bbbbbb := 2
do(1, 2, 3)
----

정렬할지 말지를 사람이 고를 자리가 없다.
선언부는 정렬되고 문장은 정렬되지 않으며, 그 경계는 도구가 정한다.
Go에 정렬 스타일 논쟁이 없는 것은 커뮤니티가 합의를 잘해서가 아니라 고를 것이 없기 때문이다.
Go 격언(Go Proverbs)의 한 문장이 이 태도를 요약한다.
"gofmt의 스타일은 누구의 취향도 아니지만, gofmt는 모두가 좋아한다."

대가가 없지는 않다.
컬럼을 맞춘다는 것은 한 줄이 길어지면 이웃한 줄의 공백도 함께 바뀐다는 뜻이다.
아래 구조체에 `MaxIdleConnections` 필드 하나를 추가하면, 손대지 않은 네 줄까지 함께 수정된 것으로 기록된다.

[source,diff]
----
 type Config struct {
-	Host  string
-	Port  int
-	Debug bool
-	TLS   bool
+	Host               string
+	Port               int
+	Debug              bool
+	TLS                bool
+	MaxIdleConnections int
 }
----

한 줄을 더했을 뿐인데, 줄마다 마지막으로 고친 커밋을 보여 주는 `git blame` 명령은 나머지 네 줄의 책임 커밋도 이 커밋으로 옮긴다.
gofmt를 두고 남아 있는 실질적인 불만은 대부분 여기서 나온다.

완화 장치는 gofmt 안에 이미 들어 있다.
첫째, 빈 줄이 정렬 그룹을 끊는다.
논리 단위마다 빈 줄을 넣으면 정렬 범위가 그 안으로 제한되어, 긴 이름 하나가 구조체 전체를 흔들지 않는다.

[source,go]
----
type T struct {
	A                 int
	VeryLongFieldName int

	B int
	C int
}
----

둘째, 한 항목이 유난히 길면 gofmt는 나머지를 늘리는 대신 정렬을 포기한다.
긴 맵 키나 긴 구조체 태그가 섞인 블록이 정렬되지 않은 채 남아 있다면, 빠뜨린 것이 아니라 gofmt의 판단이다.

[source,go]
----
var m = map[string]int{
	"a": 1,
	"b": 2,
	"this_is_an_absurdly_long_key_that_goes_on_and_on_forever": 3,
	"c": 4,
}
----

한글로 코드를 쓸 때 걸리는 함정도 하나 있다.
gofmt는 컬럼 너비를 화면에 보이는 폭이 아니라 룬(rune, 글자 하나를 세는 Go의 단위) 개수로 센다.
한글 한 글자는 터미널에서 두 칸을 차지하지만 gofmt는 한 칸으로 세므로, 앞 컬럼에 한글이 들어가면 gofmt를 방금 돌린 코드인데도 눈에는 어긋나 보인다.

[source,go]
----
var m = map[string]string{
	"한국어":     "a",
	"abcdefg": "b",
	"짧다":      "c",
}
----

위 코드는 gofmt 출력 그대로다.
바로잡을 방법은 없고, 어긋나 보여도 gofmt의 출력이 옳다고 받아들이면 된다.
한글 주석은 대개 줄 끝의 마지막 컬럼이라 이 현상이 드러나지 않고, 위처럼 한글 문자열이 맵의 키로 쓰일 때 눈에 띈다.

끝으로 용어를 하나 구분해 두자.
Go에서 "정렬(alignment)"은 구조체 필드를 메모리에 어떤 순서로 배치하는가를 뜻하기도 한다.
그쪽은 gofmt와 무관하고 `go vet`의 기본 검사에도 들어 있지 않으며, `fieldalignment`라는 별도 분석기가 다룬다.
이 절에서 말한 정렬은 눈에 보이는 코드의 모양이다.

== 프롬프트가 아니라 설정으로: 자동 품질 게이트

여기까지의 테스트와 도구는 사람이 터미널에서 명령을 입력해야 돌아간다.
2장에서 "AI가 만든 코드의 품질은 그 코드를 검증해 주는 신호의 품질에 달려 있다"는 관점을 제시했는데, 신호는 만들어 두는 것만으로는 부족하고 에이전트의 루프에 빠짐없이 연결되어야 제값을 한다.
지금부터는 그 연결을 자동화한 Flashcard의 실제 장치, 즉 훅(hook)과 서브에이전트(subagent) 설정을 해부한다.
`.claude/settings.json`, `.claude/hooks/go-check.sh`, `.claude/agents/go-quality.md` 세 파일을 전문 인용하며 한 줄씩 짚어 보겠다.
파일 세 개, 백 줄이 안 되는 설정이지만, 이 책에서 가장 실전적인 Claude Code 활용법이 여기에 담겨 있다.

.[용어 풀이] 품질 게이트(Quality Gate)
[NOTE]
====
정해진 검사를 통과하지 못한 코드는 다음 단계로 넘어가지 못하게 막는 관문이다.
공항 검색대처럼, 조건을 충족한 것만 통과시키고 그렇지 못한 것은 붙잡아 돌려보낸다.
사람이 매번 신경 써서 지키는 규율이 아니라 시스템이 자동으로 세워 두는 장벽이라는 점이 핵심이다.
====

=== 왜 프롬프트로는 안 되는가

에이전트에게 Go 코드를 수정시켜 본 사람이라면 누구나 비슷한 프롬프트를 써 봤을 것이다.
"수정 후에는 반드시 gofmt과 go vet을 실행해."
CLAUDE.md에 적어 두든 매번 지시하든, 이 방식은 세 가지 이유로 한계가 있다.

첫째, 에이전트는 잊어버린다.
대화가 길어져 컨텍스트가 압축되거나, 여러 파일을 연달아 고치는 중에 검사 지시는 뒤로 밀린다.
열 번 중 아홉 번 실행해도 배포를 깨뜨리는 것은 빠뜨린 한 번이다.

둘째, 토큰이 낭비된다.
에이전트가 검사를 직접 실행하면 명령 호출과 출력이 모두 대화 컨텍스트에 쌓인다.
`go vet`이 아무 문제도 찾지 못한 성공 케이스조차 도구 호출 한 번 분량의 토큰을 소비한다.
검사는 수정할 때마다 반복되므로 이 비용은 세션 내내 누적된다.

셋째, 비결정적이다.
LLM은 확률적으로 동작하므로 같은 지시라도 어떤 날은 `go vet ./...`을, 어떤 날은 `go vet ./internal/srs`만 실행할 수 있다.
"검사했다"는 에이전트의 보고가 실제로 무엇을 검사했다는 뜻인지 사람이 매번 확인할 수 없다.

여기서 이 장 전체를 관통하는 원칙이 나온다.
*결정적(deterministic) 검사는 프롬프트가 아니라 시스템 설정으로 강제해야 한다.*
gofmt이 통과하는지, vet이 통과하는지는 예/아니오가 명확한 기계적 판정이다.
이런 판정을 확률적인 LLM의 성실함에 맡기는 것은 도구를 잘못 고른 것이다.
Claude Code는 이를 위한 장치로 훅을 제공한다.

== 훅: 라이프사이클에 셸 명령을 거는 장치

훅(hook)은 Claude Code의 라이프사이클 이벤트(lifecycle event, 에이전트가 일하는 도중의 정해진 순간들)에 셸(shell) 명령을 연결하는 설정이다.
에이전트가 도구를 사용하기 직전(PreToolUse), 사용한 직후(PostToolUse), 사용자가 프롬프트를 제출할 때(UserPromptSubmit), 응답을 마칠 때(Stop) 등 여러 시점에 개입할 수 있다.

.[용어 풀이] 셸(Shell)과 셸 스크립트(Shell Script)
[NOTE]
====
셸은 터미널(부록 A)에 친 명령을 받아 프로그램을 대신 실행하고 그 결과를 돌려주는 프로그램이다.
터미널이 명령을 주고받는 창이라면, 셸은 그 창 안에 앉아 명령을 알아듣고 처리하는 담당자다.
리눅스와 macOS에서는 보통 bash나 zsh가 그 자리를 맡는다.
셸 스크립트는 그렇게 한 줄씩 치던 명령들을 파일에 적어 두고 한꺼번에 실행하게 만든 것으로, 변수와 조건 분기를 쓸 수 있어 작은 프로그램이 된다.
이 장에서 해부할 `go-check.sh`가 바로 그 셸 스크립트다.
====

훅으로 등록한 명령은 LLM의 판단을 거치지 않고 Claude Code 자체가 실행하므로, 조건이 맞으면 100% 실행된다.
"검사를 잊지 마라"고 부탁하는 것과 "검사 없이는 진행할 수 없다"고 시스템이 강제하는 것의 차이다.

.[용어 풀이] 훅(Hook, Claude Code의 훅)
[NOTE]
====
Claude Code가 특정 시점에 자동으로 실행하도록 미리 걸어 두는 스크립트다.
"파일을 저장하면 자동으로 검사가 돌아간다"는 식으로, 정해진 순간마다 사람 대신 기계가 알아서 개입하게 하는 장치다.
이름은 같지만 React 같은 프런트엔드 프레임워크에서 말하는 훅과는 다른 것이다.
====

Flashcard의 훅 설정 전체가 저장소의 `.claude/settings.json`에 들어 있다.
에이전트가 파일을 쓰거나 고칠 때마다 검사 스크립트 go-check.sh를 실행하라는 등록 하나가 내용의 전부다.

[source,json]
----
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR/.claude/hooks/go-check.sh\"",
            "timeout": 60,
            "statusMessage": "gofmt + go vet 검사 중..."
          }
        ]
      }
    ]
  }
}
----

구조를 한 항목씩 해설해 보겠다.

*`hooks.PostToolUse`*: 훅을 걸 이벤트다.
PostToolUse는 도구 호출이 성공적으로 끝난 직후에 발생한다.
에이전트가 파일을 쓰거나 고친 바로 그 순간이 검사하기 가장 좋은 시점이므로 이 이벤트를 골랐다.
PreToolUse에 걸면 수정 전 내용을 검사하게 되어 의미가 없다.

*`matcher: "Write|Edit"`*: 어떤 도구 호출에 반응할지 거르는 조건이다.
세로 막대(|)가 "또는"을 뜻하는 정규식(regular expression) 표기라, 도구 이름이 Write이거나 Edit일 때만 매칭된다.
파일 내용을 바꾸는 도구는 Write(새 파일 작성·전체 덮어쓰기)와 Edit(부분 수정) 두 가지이므로 이 둘만 골라 놓았다.
Bash나 Read 같은 도구 호출에는 훅이 아예 실행되지 않아 불필요한 프로세스 기동을 피한다.

*`type: "command"`*: 훅의 종류다.
셸 명령을 실행하는 가장 기본적인 형태다.
실행되는 명령은 방금의 도구 호출 정보를 JSON으로 담아 표준 입력(stdin)으로 건네받는다.

.[용어 풀이] 표준 입력·표준 출력·표준 에러(stdin·stdout·stderr)
[NOTE]
====
명령줄 프로그램이 바깥과 데이터를 주고받도록 운영체제가 열어 주는 세 개의 기본 통로다.
표준 입력은 프로그램이 데이터를 받아들이는 입구, 표준 출력은 정상적인 결과를 내보내는 출구, 표준 에러는 오류 메시지만 따로 내보내는 별도의 출구다.
결과와 오류를 두 통로로 나눠 둔 덕에, 정상 출력은 파일에 저장하면서 오류만 화면에 남기는 식으로 각각을 다른 곳으로 보낼 수 있다.
셸에서 `2>/dev/null`은 표준 에러를 버리라는 뜻이고 `>&2`는 표준 에러로 내보내라는 뜻인데, 뒤에 나올 go-check.sh에 둘 다 등장한다.
====

*`command`*: 실행할 명령이다.
`$CLAUDE_PROJECT_DIR`는 Claude Code가 훅 실행 시 넣어 주는 프로젝트 루트 경로 환경 변수다.
훅은 현재 디렉터리가 어디든 실행될 수 있으므로, 스크립트 경로를 이 변수 기준 절대 경로로 지정해야 안전하다.
경로에 공백이 있어도 깨지지 않도록 따옴표로 감쌌다.

*`timeout: 60`*: 훅의 최대 실행 시간(초)이다.
`go vet`은 첫 실행 시 빌드 캐시를 채우느라 수 초가 걸릴 수 있어 여유를 뒀다.
이 시간을 넘기면 훅이 강제 종료되므로, 훅에는 애초에 오래 걸리는 검사를 넣지 않는 것이 원칙이다.
이 원칙은 뒤에서 서브에이전트와의 역할 분담을 다룰 때 다시 나온다.

*`statusMessage`*: 훅이 도는 동안 터미널에 표시할 문구다.
훅은 도구 호출을 잠시 붙잡고 실행되므로, 왜 기다리는지 사용자에게 보여 주는 배려다.
기능에는 영향이 없지만 "훅이 돌고 있다"는 사실 자체를 사용자가 인지하게 해 디버깅에도 도움이 된다.

이 설정 파일은 저장소에 커밋되어 있다.
`.claude/settings.json`은 프로젝트 공유 설정이므로, 이 저장소를 클론해서 Claude Code를 여는 모든 사람(그리고 모든 에이전트 세션)에게 같은 품질 게이트가 적용된다.
개인 취향 설정은 `.claude/settings.local.json`에 두고 커밋하지 않는 관례와 구분된다.

== go-check.sh 해부: 무음 성공, 시끄러운 실패

훅이 실제로 실행하는 스크립트를 보자.
방금 수정된 파일이 Go 파일이면 포맷을 자동으로 맞추고 정적 검사를 돌린 뒤, 문제가 있을 때만 에이전트에게 알리는 내용이다.
`.claude/hooks/go-check.sh` 전문이다.

[source,bash]
----
#!/usr/bin/env bash
# PostToolUse hook: auto-gofmt + go vet on the edited Go file.
# Silent on success (zero LLM tokens); exit 2 feeds vet errors back to Claude.
set -u

f=$(jq -r '.tool_input.file_path // .tool_response.filePath // empty')
case "$f" in
  *.go) ;;
  *) exit 0 ;;
esac
[ -f "$f" ] || exit 0

cd "${CLAUDE_PROJECT_DIR:-$(pwd)}" || exit 0

gofmt -w "$f" 2>/dev/null

dir=$(dirname "$f")
rel=$(realpath --relative-to="$PWD" "$dir" 2>/dev/null) || rel="$dir"
if ! out=$(go vet "./$rel" 2>&1); then
  {
    echo "go vet failed for ./$rel:"
    echo "$out"
  } >&2
  exit 2
fi
exit 0
----

30줄이 안 되는 스크립트지만 단계마다 설계 의도가 있다.
순서대로 짚어 보겠다.

=== 1단계: stdin의 JSON에서 파일 경로 추출

PostToolUse 훅은 방금 실행된 도구 호출의 정보를 JSON으로 표준 입력에 받는다.
Write와 Edit 도구는 입력 파라미터(`tool_input.file_path`)에 대상 파일 경로를 담고, 도구 응답(`tool_response.filePath`)에도 같은 정보가 실린다.
jq는 JSON 문서에서 원하는 값만 골라 꺼내 주는 명령줄 도구다.
`jq -r '... // ... // empty'`는 둘 중 먼저 존재하는 값을 꺼내고, 둘 다 없으면 빈 문자열을 돌려주는 안전한 추출이다.
도구마다 JSON 스키마가 조금씩 다를 수 있으므로 대안 경로를 `//`로 이어 둔 것이다.

=== 2단계: .go 파일만 통과시키는 case 문

파일 확장자를 보고 Go 파일이 아니면 그 자리에서 조용히 끝내는 부분이다.

[source,bash]
----
case "$f" in
  *.go) ;;
  *) exit 0 ;;
esac
----

matcher가 Write|Edit에 매칭되므로 훅은 마크다운, HTML, JSON 등 모든 파일 수정에서 실행된다.
Go 파일이 아니면 즉시 `exit 0`으로 빠져나온다.
이 조기 종료가 중요한 이유는 비용이다.
이 책의 원고(`doc/*.md`)를 고칠 때도 훅은 실행되지만, jq 한 번과 case 문 하나로 끝나므로 체감 지연이 없다.
이어지는 `[ -f "$f" ] || exit 0`은 파일이 실제로 존재할 때만 진행하는 방어 코드다.

=== 3단계: gofmt를 조용히 자동 적용

수정된 파일의 코드 포맷을 표준 형태로 바로잡는 한 줄이다.

[source,bash]
----
gofmt -w "$f" 2>/dev/null
----

포맷 위반을 *보고하는 게 아니라 그냥 고쳐 버린다*는 점이 핵심이다.
gofmt의 결과는 언제나 하나로 결정되므로, "포맷이 틀렸다"고 에이전트에게 알려서 다시 고치게 하는 왕복은 토큰과 시간의 순수한 낭비다.
기계가 결정할 수 있는 일은 기계가 끝내고, 에이전트에게는 알리지도 않는다.
`2>/dev/null`은 파일에 문법 오류가 있어 gofmt이 실패하는 경우를 대비한 것인데, 그런 오류는 어차피 다음 단계의 go vet이 더 좋은 메시지로 잡아 준다.

자동 수정에는 트레이드오프가 하나 있다.
에이전트가 기억하는 파일 내용과 디스크의 실제 내용이 포맷만큼 어긋날 수 있다.
다만 어긋남은 공백·정렬 수준이고, 어긋난 상태에서 에이전트가 Edit을 시도하면 문자열 불일치로 실패해 파일을 다시 읽게 되므로 실질적인 문제로 번지지는 않았다.

=== 4단계: 수정된 패키지에만 go vet

수정된 파일이 속한 디렉터리를 알아내 그 패키지만 go vet으로 검사하는 부분이다.

[source,bash]
----
dir=$(dirname "$f")
rel=$(realpath --relative-to="$PWD" "$dir" 2>/dev/null) || rel="$dir"
if ! out=$(go vet "./$rel" 2>&1); then
----

`go vet ./...`으로 모듈 전체를 검사하지 않고, 방금 수정된 파일이 속한 패키지 디렉터리(`./$rel`)만 검사한다.
훅은 파일 수정마다 실행되므로 속도가 곧 개발 리듬이다.
전체 검사는 정확하지만 느리고, 패키지 검사는 빠르지만 다른 패키지로 번진 영향을 놓친다.
그 빈틈은 뒤에 나올 go-quality 서브에이전트의 `go vet ./...`이 커밋 전에 메운다.
빠른 검사와 완전한 검사를 한 장치에 욱여넣지 않고 두 층으로 나눈 것이다.

`realpath --relative-to`로 상대 경로를 만드는 이유는 go 명령이 모듈 루트 기준의 `./패키지` 형태 인자를 기대하기 때문이다.
그 직전의 `cd "${CLAUDE_PROJECT_DIR:-$(pwd)}"`가 모듈 루트로 이동해 이 전제를 만들어 준다.

=== 5단계: 실패하면 stderr에 쓰고 exit 2

검사에 걸렸을 때 오류 내용을 내보내고 실패 신호와 함께 끝나는 마무리 부분이다.

[source,bash]
----
  {
    echo "go vet failed for ./$rel:"
    echo "$out"
  } >&2
  exit 2
----

여기가 훅과 에이전트가 만나는 접점이다.
`>&2`로 오류 내용을 표준 에러에 실어 보내고, 종료 코드 2로 끝낸다.
Claude Code의 훅 규약에서 종료 코드(exit code) 2는 특별한 의미를 갖는다.
*훅이 exit 2로 끝나면 표준 에러에 쓴 내용이 에이전트에게 피드백으로 전달된다.*
에이전트는 자기가 방금 저지른 vet 위반을 에러 메시지 그대로 보게 되고, 다음 턴에서 스스로 고친다.
사람이 개입할 필요가 없다.

.[용어 풀이] 종료 코드(Exit Code)
[NOTE]
====
프로그램이 끝나면서 남기는 한 자리 성적표다.
0이면 통과, 그 외의 숫자는 실패를 뜻한다.
Claude Code의 훅에서는 이 숫자가 단순한 성패 표시를 넘어 신호로 쓰이는데, 특히 2로 끝내면 그때 남긴 오류 메시지가 에이전트에게 그대로 피드백된다.
====

반대로 exit 0으로 끝나면 에이전트에게는 아무것도 전달되지 않는다.
이 스크립트의 성공 경로를 다시 보면, `gofmt -w`는 출력을 버리고 vet 성공 시에는 아무것도 찍지 않고 `exit 0`한다.
즉 *검사가 통과하는 한 이 훅은 LLM 토큰을 단 하나도 쓰지 않는다.*
에이전트가 직접 `go vet`을 실행하던 방식과 비교하면 차이가 분명하다.
직접 실행은 성공해도 도구 호출·출력만큼 컨텍스트를 소비하지만, 훅은 실패라는 예외 상황에만 컨텍스트에 등장한다.

정리하면 이 스크립트의 설계 원칙은 두 가지다.
첫째, 성공은 무음이다. 좋은 소식은 에이전트가 알 필요가 없다.
둘째, 실패는 정확히 한 곳(stderr + exit 2)으로 시끄럽게 알린다. 나쁜 소식은 고칠 수 있는 형태로 즉시 전달한다.
그리고 이 모든 것이 프롬프트가 아니라 설정이므로, 에이전트가 아무리 딴생각을 해도 검사는 빠지지 않는다.

== 서브에이전트: 별도 컨텍스트에서 일하는 보조 에이전트

훅이 "빠르고 결정적인 검사"를 맡는다면, 오래 걸리고 출력이 긴 검증은 어디에 두어야 할까.
`go build ./...`, `go test ./...` 같은 종합 검증을 메인 대화에서 직접 돌리면 빌드 로그와 테스트 출력이 컨텍스트를 잠식한다.
검증 결과에서 정작 필요한 정보는 "통과했는가, 실패했다면 어디인가" 한두 줄인데, 그걸 얻으려고 수백 줄의 로그를 컨텍스트에 싣는 것은 배보다 배꼽이 크다.

서브에이전트(subagent)는 이 문제를 푸는 Claude Code의 장치다.
메인 에이전트가 작업을 위임하면, 서브에이전트는 *별도의 컨텍스트 창(context window)* 에서 실행된다.
서브에이전트가 어떤 명령을 돌리고 어떤 로그를 읽었든 그 과정은 메인 대화에 남지 않고, *마지막 응답 메시지만* 반환값으로 돌아온다.
자료 조사를 맡겼을 때 어느 책을 몇 권이나 들췄는지는 듣지 않고 정리된 답만 받아 보는 것과 같다.
개발자에게는 함수 호출과 같다고 말할 수 있다.
함수 안에서만 쓰는 임시 변수가 그 함수를 부른 쪽에는 보이지 않는 것과 같은 이치다.

.[용어 풀이] 서브에이전트(Subagent)
[NOTE]
====
메인 에이전트가 특정 작업을 통째로 맡기는 별도의 보조 에이전트다.
심부름을 시키면 과정은 알아서 처리하고 결과만 보고하는 조수와 같아서, 자기만의 작업 공간에서 일하고 결과 요약만 돌려준다.
덕분에 장황한 중간 과정이 메인 대화를 어지럽히지 않는다.
====

서브에이전트 정의는 `.claude/agents/` 아래 마크다운 파일 하나로 이루어지며, 세 가지를 제어할 수 있다.

첫째, 컨텍스트 격리다.
장황한 탐색·빌드·테스트 로그가 메인 대화를 오염시키지 않는다.

둘째, 도구 제한이다.
정의 파일 맨 위에 적는 `tools`에 나열한 도구만 쓸 수 있어, 검증 에이전트가 실수로 코드를 고치는 일을 구조적으로 막을 수 있다.

셋째, 모델 지정이다.
작업 성격에 맞는 모델을 `model`로 지정할 수 있다.
설계나 구현은 고성능 모델이 필요하지만, "명령을 순서대로 돌리고 결과를 요약하라"는 작업은 저렴하고 빠른 모델로 충분하다.

== go-quality 에이전트 해부

Flashcard의 검증 서브에이전트 정의를 보자.
다섯 가지 검사를 순서대로 실행하고 결과를 정해진 형식으로만 보고하라는 지시문이다.
`.claude/agents/go-quality.md` 전문이다.

[source,md]
----
---
name: go-quality

너는 flashcard 저장소(Go 1.26 모듈, `github.com/benelog/flashcard`)의 품질 검증 에이전트다.
저장소 루트에서 아래 검사를 순서대로 실행하라:

1. `gofmt -l .` — 출력이 있으면 포맷 위반
2. `go build ./...`
3. `go vet ./...`
4. `go test ./...`
5. `go build api/index.go` — Vercel 서버리스 빌드 호환 검사. Vercel은 이 파일을
   모듈 외부에서 단독 컴파일하므로 `internal/` 패키지를 import하면 여기서 실패한다.
   이 검사가 실패하면 Vercel 배포도 실패한다는 점을 명시하라.

보고 규칙 — 마지막 메시지가 그대로 반환값이 된다:
- 모두 통과: `PASS — build/vet/test/gofmt/vercel 모두 통과 (테스트 N개)` 한 줄만.
- 실패 시: 실패한 검사 이름과 원인이 되는 `파일:라인` + 에러 메시지 핵심만 발췌.
  전체 로그를 붙여넣지 마라. 통과한 항목은 나열하지 마라.
- 코드를 수정하지 마라. 검증과 보고만 한다.
----

=== frontmatter: 정체성과 권한과 비용

*`name: go-quality`*: 에이전트의 식별자다.
메인 에이전트가 위임할 때, 또는 사용자가 직접 호출할 때 이 이름을 쓴다.

*`description`*: 이 필드는 단순한 설명문이 아니라 *자동 위임의 트리거*다.
메인 에이전트는 작업을 마주쳤을 때 등록된 서브에이전트들의 description을 보고 위임 여부를 판단한다.
"Go 코드를 수정한 뒤 커밋 전 종합 검증이 필요할 때 사용"이라고 사용 시점을 명시해 둔 덕에, 커밋을 앞둔 메인 에이전트가 별도 지시 없이도 이 에이전트를 호출할 확률이 높아진다.
서브에이전트의 약점(호출을 잊을 수 있다)을 description 작성으로 보완한 것이다.

*`tools: Bash, Read, Grep, Glob`*: 읽기와 실행만 허용하고 Edit·Write를 뺐다.
본문 마지막 줄의 "코드를 수정하지 마라"는 프롬프트 수준의 당부이고, tools 제한은 시스템 수준의 강제다.
같은 규칙을 두 층에 이중으로 걸어 둔 것인데, 앞서 본 "강제할 수 있는 것은 설정으로 강제한다"는 원칙이 여기서도 반복된다.

*`model: haiku`*: 이 에이전트는 가장 저렴한 모델로 돈다.
검증 작업의 본질은 "정해진 명령을 순서대로 실행하고, 종료 코드를 보고, 실패 메시지를 발췌"하는 것이다.
창의성도 설계 판단도 필요 없으므로 고성능 모델은 과잉이다.
커밋 전마다 반복 호출되는 작업이라 모델 단가 차이가 그대로 누적 절감으로 이어진다.
반대로 실패 원인 분석까지 시키고 싶다면 상위 모델이 낫다.
이 에이전트는 분석 대신 "파일:라인 발췌"라는 기계적 보고만 요구해서 haiku로 충분하도록 작업을 잘라 냈다.

=== 본문: 다섯 단계 검사의 순서와 이유

검사 순서에도 논리가 있다.
싸고 빨리 실패하는 검사를 앞에 두었다.

1단계 `gofmt -l .`은 포맷 위반 파일 목록을 출력한다.
훅이 gofmt을 자동 적용하므로 보통은 깨끗하지만, 훅을 우회한 수정(셸에서 직접 생성한 파일 등)을 잡는 안전망이다.

2단계 `go build ./...`는 모듈 전체가 컴파일되는지 확인한다.
훅의 go vet은 수정된 패키지만 봤으므로, 다른 패키지로 번진 시그니처 변경 등은 여기서 처음 걸린다.

3단계 `go vet ./...`는 컴파일은 되지만 의심스러운 코드(사용되지 않는 결과값, 잘못된 포맷 문자열 등)를 모듈 전체에서 찾는다.

4단계 `go test ./...`는 전체 테스트를 실행한다.
앞에서 본 `internal/srs`의 테이블 주도 테스트 같은 것들이 여기서 돌아간다.
테스트는 다섯 검사 중 가장 오래 걸리므로 훅이 아니라 이곳에 있다.

=== 5단계 검사가 특히 중요한 이유

5단계 `go build api/index.go`는 이 저장소 고유의 검사이며, 다섯 검사 중 가장 중요하다.
Flashcard의 API는 2부에서 Vercel 서버리스 함수로 배포되는데, Vercel의 Go 런타임은 진입점인 `api/index.go`를 *모듈 컨텍스트 밖에서 단독 컴파일*한다.
Go의 `internal/` 패키지는 같은 모듈 안에서만 import할 수 있으므로, 이 파일이 `internal/` 패키지를 import하는 순간 로컬에서는 멀쩡히 빌드되지만 Vercel 배포는 실패한다.
그래서 이 파일이 필요로 하는 공유 코드는 전부 `pkg/app`에 있다.

이 제약은 `api/index.go` 상단 주석에도 박혀 있다(6장에서 인용한 그 주석이다).

`go build ./...`도 `go test ./...`도 이 문제를 잡지 못한다는 점이 핵심이다.
일반 빌드는 모듈 안에서 이루어지므로 internal import가 합법이기 때문이다.
`go build api/index.go`처럼 파일을 단독 지정해야 Vercel의 컴파일 방식이 근사하게 재현된다.
즉 이 검사는 *로컬 검증으로는 잡히지 않는 배포 실패를 커밋 전에 잡는 검사*다.
푸시하고 배포가 깨진 뒤에 알게 되면 수정→푸시→배포 대기의 왕복이 몇 분씩 걸리지만, 이 검사는 몇 초 만에 같은 답을 준다.
Vercel의 Go 함수 빌드 방식과 `pkg/app` 구조는 16장에서 자세히 다룬다.

=== 보고 규칙: 반환값을 설계한다

정의 파일의 마지막 블록은 검사 방법이 아니라 *보고 방법*을 지시한다.
"마지막 메시지가 그대로 반환값이 된다"는 문장은 서브에이전트의 동작 원리를 프롬프트 안에 명시해 둔 것이다.
서브에이전트는 마지막 응답만 메인 에이전트에게 돌려주므로, 그 마지막 응답의 형식을 통제하는 것이 곧 인터페이스 설계다.

통과 시에는 `PASS — build/vet/test/gofmt/vercel 모두 통과 (테스트 N개)` 한 줄만 반환한다.
메인 에이전트의 컨텍스트에는 이 한 줄만 추가된다.
빌드 로그 수백 줄이 haiku의 컨텍스트에서 소화되고 요약 한 줄로 압축되어 돌아오는 구조다.

실패 시에는 실패한 검사 이름과 `파일:라인` 발췌만 반환한다.
"전체 로그를 붙여넣지 마라. 통과한 항목은 나열하지 마라"는 금지 규칙이 붙어 있는데, 이는 LLM이 성실함을 과시하려고 장황해지는 경향을 억누르기 위한 것이다.
메인 에이전트가 고치는 데 필요한 정보는 위치와 에러 메시지뿐이다.

눈치챘겠지만 이 보고 규칙은 훅의 "무음 성공, 시끄러운 실패" 원칙의 서브에이전트 버전이다.
훅은 성공 시 출력 0바이트, 서브에이전트는 성공 시 한 줄이다.
검증 장치는 좋은 소식일수록 말을 아끼도록 설계한다.

== 훅과 서브에이전트의 역할 분담

이제 두 장치를 나란히 놓고 분담의 논리를 정리해 보자.

|===
|  | 훅 (go-check.sh) | 서브에이전트 (go-quality)

| 실행 시점 | 파일 수정 직후, 자동 | 커밋 전, 위임 호출
| 실행 보장 | 100% (시스템 강제) | 에이전트 판단에 의존
| 검사 범위 | 수정된 파일·패키지 | 모듈 전체 + 배포 호환
| 소요 시간 | 1초 안팎 | 수십 초
| 성공 시 비용 | 토큰 0 | 요약 한 줄
| 담당 검사 | gofmt 자동 적용, 패키지 go vet | build/vet/test 전체 + `go build api/index.go`
|===

분담 기준은 단순하다.
*빠르고 결정적이며 매번 필요한 검사는 훅으로, 느리고 포괄적이며 마일스톤마다 필요한 검증은 서브에이전트로.*
달리 말하면, 딱 잘라 예/아니오로 판정할 수 있는 검사는 기계인 훅에게 맡기고, 여러 정황을 함께 살펴 종합적으로 판단해야 하는 검증은 서브에이전트에게 맡기는 것이다.

이 분담에는 양방향의 트레이드오프가 있다.

훅 쪽의 제약은 지연이다.
훅은 모든 매칭 도구 호출을 잠시 붙잡으므로, 훅이 느리면 *모든 파일 수정이 느려진다*.
에이전트가 파일 열 개를 연달아 고치는 세션이라면 훅 지연은 열 배로 체감된다.
그래서 go-check.sh는 전체 테스트는커녕 전체 vet도 하지 않고, 수정된 패키지의 vet까지만 한다.
`go test ./...`를 훅에 넣고 싶은 유혹이 들 수 있지만, 그러면 파일 하나를 고칠 때마다 전체 테스트가 끝나기를 기다리게 된다.

서브에이전트 쪽의 제약은 비결정성이다.
서브에이전트 호출은 결국 메인 에이전트의 판단이므로, *호출을 잊을 수 있다*.
훅처럼 시스템이 강제하지 않는다.
Flashcard가 쓰는 완화책은 앞서 본 description에 사용 시점("커밋 전")을 명시해 두는 것이다.
그래도 훅만큼의 보장은 아니므로, 더 확실히 하려면 커밋 절차를 다루는 프로젝트 지침에 go-quality 실행을 관례로 못 박거나 같은 검증을 CI(지속적 통합)에서 한 번 더 돌릴 수 있다.
훅 → 서브에이전트 → CI로 갈수록 느리지만 확실해지는 그물을 층층이 치는 셈이다.

반대 방향의 실수도 경계할 만하다.
모든 검사를 서브에이전트로 미루면 에이전트는 vet 위반이 쌓인 채 수십 분을 작업하다가 커밋 직전에 한꺼번에 실패를 마주한다.
오류는 발생 지점에 가까울수록 고치기 싸다.
훅이 수정 직후에 잡아 주는 즉시 피드백은 서브에이전트가 대체할 수 없다.

== 내 프로젝트에 적용하기

훅과 서브에이전트로 검사를 나눠 거는 이 구조는 Go에 묶여 있지 않다.
자기 기술 스택으로 옮기는 절차를 일반화해 보겠다.
이 절은 Go가 아닌 다른 언어를 쓰는 독자를 위한 것이라, 다른 언어의 도구 이름이 여럿 나온다.
Go만 따라갈 독자는 건너뛰어도 좋다.

첫째, 기술 스택의 검사 도구를 두 무리로 나눈다.
기준은 "결정적이고 1~2초 안에 끝나는가"다.
포맷터(prettier, gofmt, black, rustfmt)와 빠른 린터·타입체커는 훅 후보다.
전체 빌드, 전체 테스트, 통합 검증은 서브에이전트 후보다.

둘째, 훅 스크립트를 go-check.sh의 골격 그대로 작성한다.
stdin JSON에서 경로 추출 → 확장자 필터 → 자동 수정 가능한 것은 조용히 수정 → 판정이 필요한 것만 검사 → 실패 시 stderr + exit 2.
TypeScript 프로젝트라면 이런 모양이 된다.
앞의 go-check.sh와 뼈대가 같고, gofmt 자리에 prettier(포맷 도구), go vet 자리에 eslint(검사 도구)가 들어갔을 뿐이다.
`npx`는 그 도구들을 내려받아 실행해 주는 명령이다.

[source,bash]
----
#!/usr/bin/env bash
set -u

f=$(jq -r '.tool_input.file_path // .tool_response.filePath // empty')
case "$f" in
  *.ts|*.tsx) ;;
  *) exit 0 ;;
esac
[ -f "$f" ] || exit 0

cd "${CLAUDE_PROJECT_DIR:-$(pwd)}" || exit 0

npx prettier --write "$f" >/dev/null 2>&1

if ! out=$(npx eslint "$f" 2>&1); then
  echo "eslint failed for $f:" >&2
  echo "$out" >&2
  exit 2
fi
exit 0
----

`tsc --noEmit`(타입 검사만 하고 결과 파일은 만들지 않는 명령)을 훅에 넣을지는 프로젝트 크기에 달렸다.
고친 부분만 다시 컴파일하는 증분 컴파일로 1~2초면 훅에, 그 이상이면 서브에이전트로 보내는 것이 낫다.
훅에 넣은 검사가 느려졌다고 느껴지면 주저 없이 서브에이전트 쪽으로 옮기자.
분담선은 고정이 아니라 프로젝트가 자라며 조정하는 다이얼이다.

셋째, 서브에이전트 정의를 작성할 때는 go-quality.md의 네 가지 요소를 챙긴다.
description에 "언제 쓰는가"를 명시해 자동 위임을 유도하고, tools에서 수정 권한을 빼고, model은 기계적 검증이면 저비용 모델로 지정하고, 보고 규칙으로 반환값 형식(통과 시 한 줄, 실패 시 위치 발췌)을 통제한다.
특히 마지막 항목을 빼먹으면 서브에이전트가 로그를 통째로 반환해 격리의 의미가 사라지므로 주의하자.

넷째, 자기 프로젝트만의 "로컬에서 안 잡히는 배포 실패"를 찾아 서브에이전트 검사에 넣는다.
Flashcard의 `go build api/index.go`에 해당하는 검사는 기술 스택마다 다르다.
Next.js 정적 export라면 `next build`가 서버 전용 API 사용을 잡아 주고, Docker 배포라면 이미지 빌드가 그 역할을 한다.
배포가 한 번이라도 깨졌다면, 그 원인을 커밋 전에 재현하는 명령을 찾아 검사 목록에 추가하는 습관이 이 목록을 자라게 한다.

== 정리

첫째, `go test`는 세 가지 규칙(`_test.go` 파일, `Test` 접두사, `*testing.T` 매개변수)만 지키면 별도 프레임워크 없이 테스트를 찾아 실행한다.
테스트 파일을 검증 대상과 같은 패키지에 두면 비공개 함수까지 직접 검증할 수 있다.
실패 출력에는 테스트 이름과 `파일:줄 번호`와 메시지가 함께 찍히므로 기대값과 실제값을 메시지에 넣어 두어야 하고, 통과한 결과는 캐시되어 `(cached)`로 표시되므로 정말 다시 돌리려면 `-count=1`을 붙인다.

둘째, 테이블 주도 테스트는 케이스 추가를 테이블 한 줄 추가로 만드는 Go의 표준 테스트 패턴이다.
시각을 인자로 받는 순수 함수 설계 덕분에 테스트가 결정적이 되고, 진행형 검증에는 `t.Fatalf`, 독립 검증에는 `t.Errorf`를 가려 쓴다.
순수 계산은 2만 건 전수 검사조차 값싸게 돌릴 수 있고, 거부해야 할 경계 입력은 테이블 한 줄로 고정해 재발을 막는다.

셋째, DB를 붙인 테스트는 `t.TempDir`으로 테스트마다 격리하고, `t.Cleanup`으로 뒷정리를 등록하고, `t.Helper`로 실패 위치가 헬퍼가 아니라 호출한 테스트로 찍히게 한다.
케이스마다 조건이 다른 테이블에는 `t.Run`으로 이름을 붙여, 실패한 케이스를 이름으로 확인하고 `-run '부모/케이스'`로 그 하나만 돌린다.
커버리지(`-cover`)는 채워야 할 목표가 아니라 테스트가 없는 곳을 확인하는 지도로 읽는다.

넷째, gofmt·go vet·go build·staticcheck는 빠르고 결정적인 품질 검사 도구다.
gofmt는 선언부의 세로 정렬까지 강제하며, 정렬 스타일을 고를 여지가 없어 논쟁이 사라지는 대신 diff가 넓어지는 대가가 따른다.

다섯째, 결정적 검사는 프롬프트가 아니라 시스템 설정으로 강제한다.
훅은 파일 수정 직후 100% 실행이 보장되는 장치로, 성공하면 무음이라 토큰을 전혀 쓰지 않고, 실패하면 stderr와 종료 코드 2로 에러가 에이전트에게 피드백되어 스스로 고치게 만든다.

여섯째, 느리고 포괄적인 검증은 서브에이전트로 격리 실행한다.
go-quality는 저비용 모델로 build·vet·test에 더해 `go build api/index.go`로 Vercel 배포 호환성까지 검증하고, 통과 시 한 줄만 보고해 메인 대화의 컨텍스트를 지킨다.

일곱째, 분담의 기준은 속도와 보장이다.
빠르고 결정적인 검사는 매 수정마다 훅으로, 느리고 포괄적인 검증은 커밋 전에 서브에이전트로 나누고, 프로젝트가 자라면 분담선을 조정한다.

이 장의 훅이 에이전트 쪽 입구에 건 규칙이라면, 13장의 에디터 설정은 같은 규칙을 사람 쪽 입구에 걸고, 15장의 CI는 두 입구가 모두 로컬이라는 약점을 원격에서 메운다.
다음 9장에서는 이 Go 코드 위에서 HTTP API를 조립하는 웹 프레임워크 Gin을 다룬다.
로컬 서버와 Vercel 함수가 같은 Gin 엔진을 공유하는 구조가 어떻게 만들어지는지 확인해 보자.
