# 2장 데이터베이스 기초: 테이블, SQL, 인덱스

도입에서 본 화면이 보여 주는 덱과 카드는 브라우저의 것이 아니다.
화면을 닫아도, 다른 기기에서 열어도 데이터가 그대로인 것은 서버 건너편의 데이터베이스가 원본을 지키고 있기 때문이다.
관계형 데이터베이스와 PostgreSQL을 선택한 이유는 1장에서 정리했다.
이 장에서는 관계형 데이터베이스가 데이터를 담는 방식(테이블과 키), 그 데이터를 다루는 언어인 SQL, 그리고 조회 속도를 좌우하는 인덱스를 Echo Flip의 실제 테이블과 쿼리로 익힌다.
다음 3장에서 실제 스키마 설계를 해부하는 데 필요한 기초를 다지는 장이다.

데이터의 모양을 먼저 익히는 데는 이유가 있다.
뒤에 나올 Go의 구조체도, API가 주고받는 JSON도, 화면이 그리는 목록도 결국 여기 있는 표와 열을 옮겨 담은 것이다.
바닥의 모양을 알고 나면 그 위층의 코드가 왜 그렇게 생겼는지가 훨씬 빨리 눈에 들어온다.

## 관계형 데이터베이스: 표에 담고 키로 잇는다

::: info [용어 풀이] 테이블·행·열(Table·Row·Column)
관계형 데이터베이스가 데이터를 담는 격자로, 스프레드시트의 시트 한 장과 거의 같다.
표 하나가 테이블이고, 가로 한 줄인 행(Row)이 카드 한 장·사용자 한 명 같은 개별 기록이며, 세로 칸인 열(Column)이 이름·생성 시각처럼 기록마다 채우는 항목이다.
잠시 뒤 볼 `create table` 문이 이 시트의 이름과 열 목록을 미리 정해 두는 일에 해당한다.
:::

Echo Flip의 덱을 담는 `decks` 테이블을 스프레드시트처럼 그려 보면 이런 모양이다.

| id | name | description | created_at |
| --- | --- | --- | --- |
| 3f2a… | 토익 필수 단어 | 시험 대비 | 2026-05-02 |
| 9c81… | 회화 표현 | (비어 있음) | 2026-06-11 |

행 하나가 덱 하나이고, 열은 모든 덱이 공통으로 갖는 항목이다.
스프레드시트와 다른 점은 이 표의 구조와 규칙을 데이터베이스가 강제한다는 것이다.
어떤 열이 있는지, 각 칸에 어떤 종류의 값만 들어갈 수 있는지를 미리 선언해 두면, 규칙에 어긋나는 데이터는 아예 저장되지 않는다.

### 왜 표를 나누는가

카드도 담아야 하니 표가 하나 더 필요한데, 여기서 관계형 모델의 핵심 질문이 나온다.
카드 표의 각 행에 소속 덱의 이름을 함께 적으면 어떨까.
당장은 편해 보이지만, 덱 이름을 바꾸는 순간 그 덱에 속한 카드 수백 행을 전부 고쳐야 하고, 한 행이라도 빠뜨리면 같은 덱이 두 이름을 갖게 된다.

관계형 모델의 답은 "같은 사실은 한 곳에만 적는다"이다.
덱의 이름은 `decks` 표에 한 번만 적고, 카드 표에는 어느 덱에 속하는지 가리키는 식별자만 둔다.
이렇게 중복을 걷어 내며 표를 주제별로 나누는 작업을 정규화(normalization)라 부른다.
그리고 나뉜 표를 잇는 수단이 키(key)다.

::: info [용어 풀이] 기본 키와 외래 키(Primary Key·Foreign Key)
기본 키는 한 행을 유일하게 가리키는 식별 값으로, 표 안에서 같은 값이 두 번 나오지 않게 막는 대표 열이다.
외래 키는 다른 표의 기본 키를 가리키는 열로, "이 카드는 저 덱에 속한다"처럼 표와 표를 잇는 끈이다.
스프레드시트로 치면 한 시트의 칸이 다른 시트의 특정 줄을 가리키도록 약속해 두고, DB가 그 약속이 깨지지 않게 지켜 주는 셈이다.
:::

카드를 담는 `cards` 표를 같은 방식으로 그려 보자.

| id | deck_id | text | meaning |
| --- | --- | --- | --- |
| a11f… | 3f2a… | trade-off | 맞바꿈, 절충 |
| b420… | 3f2a… | pull request | 코드 반영 요청 |

`id`가 이 표의 기본 키이고, `deck_id`가 `decks` 표의 `id`를 가리키는 외래 키다.
두 카드 모두 `deck_id`가 `3f2a…`이므로 "토익 필수 단어" 덱에 속한다는 사실이, 덱 이름을 한 글자도 중복해 적지 않고 표현된다.
Echo Flip은 이런 식으로 이어진 테이블 일곱 개로 이루어져 있는데, 전체 그림은 3장에서 요구사항으로부터 하나씩 도출한다.

## 데이터베이스와 대화하는 언어, SQL

표를 만들고, 행을 넣고, 원하는 행을 찾아오는 모든 일은 SQL 문장으로 이루어진다.

::: info [용어 풀이] SQL(Structured Query Language)
관계형 데이터베이스에 일을 시킬 때 쓰는 표준 언어다.
"덱 표에서 이 사용자의 행을 만든 날짜 순으로 달라"처럼 원하는 결과를 문장으로 서술하면, 그 결과를 어떤 순서로 어떻게 찾을지는 데이터베이스가 스스로 정한다.
PostgreSQL, MySQL, SQLite 등 어느 관계형 데이터베이스를 쓰든 대부분의 문법이 통하므로, 한번 익히면 오래 쓰는 지식이다.
:::

Go나 TypeScript가 "이 순서로 이렇게 하라"라고 절차를 지시하는 언어라면, SQL은 원하는 결과의 조건만 서술하는 선언형 언어다.
반복문을 돌며 행을 하나씩 확인하는 코드를 쓰는 대신 "이 조건에 맞는 행"이라고 말하면 끝이다.
7장에서 볼 store 계층이 데이터베이스에 보내는 문장들이 전부 SQL이고, 이 장에서 그 문장들을 하나씩 읽는다.

SQL 문장은 역할에 따라 크게 두 갈래로 나뉜다.

**DDL(Data Definition Language, 데이터 정의어)**: 테이블을 만들고 고치고 지우는, 데이터의 틀을 다루는 문장이다. `create table`, `alter table`, `drop table`이 대표다.

**DML(Data Manipulation Language, 데이터 조작어)**: 그 틀 안의 내용을 넣고 찾고 고치고 지우는 문장이다. `insert`, `select`, `update`, `delete`가 대표다.

집에 비유하면 DDL은 설계와 시공이고 DML은 입주 후의 생활인 셈이다.
차례로 살펴보자.

## DDL: 데이터의 틀을 만든다

앞에서 스프레드시트처럼 그려 본 `decks` 표의 실제 선언이다.
`internal/db/migrations/000001_init.up.sql`에서 발췌한, 표의 이름과 열 목록·규칙을 정하는 문장이다.

```sql
create table decks (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references profiles(id) on delete cascade,
  name text not null,
  description text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

한 줄이 열 하나이고, 각 줄은 "열 이름, 자료형, 규칙들" 순서로 읽는다.

자료형부터 보자.
`uuid`는 전역에서 겹치지 않는 식별자, `text`는 문자열, `timestamptz`는 시간대 정보를 품은 시각이다.
자료형은 그 칸에 담길 값의 종류를 못 박는 장치로, 8장에서 볼 TypeScript 타입이 컴파일 시점의 검증이라면 자료형은 저장 시점의 검증이다.
시각 열에 문자열을 넣으려는 문장은 실행되지 않는다.

`default`는 값을 주지 않았을 때 데이터베이스가 대신 채우는 기본값이다.
`gen_random_uuid()`는 새 UUID를 만들어 주는 함수이고 `now()`는 현재 시각이므로, 덱을 만들 때 id와 생성 시각은 호출자가 신경 쓸 필요가 없다.

나머지 규칙들이 제약이다.

::: info [용어 풀이] 제약(Constraint)
데이터가 지켜야 할 규칙을 DB에 직접 새겨, 어긴 값은 아예 저장되지 않게 막는 장치다.
"이 칸은 비어 있으면 안 된다"(not null), "이 값은 중복되면 안 된다"(unique)처럼 스프레드시트로는 강제하기 어려운 규칙을 DB가 대신 지켜 준다.
규칙을 애플리케이션 코드가 아니라 데이터 곁에 두면, 어떤 경로로 값이 들어와도 규칙이 똑같이 적용된다.
:::

`primary key`는 이 열이 행의 대표 식별자라는 선언이고, `not null`은 비워 둘 수 없다는 뜻이다(제약이 없는 `description`은 비워 둘 수 있다).
`references profiles(id)`는 외래 키 선언으로, `user_id`에는 `profiles` 표에 실제로 존재하는 사용자의 id만 들어갈 수 있다.
존재하지 않는 사용자의 덱, 이름 없는 덱 같은 데이터는 코드에 버그가 있어도 저장 단계에서 거부된다.
코드는 고치면 되지만 오염된 데이터는 복구가 어려우므로, 마지막 방어선을 DB에 두는 것이다.
뒤에 붙은 `on delete cascade`는 부모 행이 지워질 때의 동작을 정하는 옵션인데, 삭제 정책의 설계는 3장에서 다룬다.

### alter table: 운영 중에 틀을 고친다

테이블은 한 번 만들면 끝이 아니다.
Echo Flip에 덱 공유 기능이 추가되면서 `decks` 표에도 열이 필요해졌다.
`internal/db/migrations/000002_sharing.up.sql`에서 발췌한, 이미 운영 중인 표에 열 두 개를 더하는 문장이다.

```sql
alter table decks
  add column share_slug text,
  add column shared_at timestamptz;
```

`alter table`은 기존 데이터를 유지한 채 틀만 고친다.
이미 쌓여 있던 덱 행들은 새 열이 비어 있는 채로 남고, 이후 공유되는 덱만 값을 갖게 된다.

틀을 없애는 문장은 `drop`이다.
`drop table decks`는 표를 데이터까지 통째로 지우므로 DDL 중 가장 신중해야 할 문장이다.
이런 구조 변경 문장들을 순서 있는 파일로 쌓아 관리하는 방법(마이그레이션)은 3장에서 다룬다.

## DML: 넣고, 찾고, 고치고, 지운다

이제 틀 안의 데이터 차례다.
DML의 네 동사 `insert`, `select`, `update`, `delete`는 7장에서 볼 API의 생성·조회·수정·삭제와 일대일로 대응한다.
덱 API의 핸들러가 store 계층을 부르면, store 계층이 실행하는 것이 바로 이 문장들이다.

### insert: 행을 만든다

`internal/store/decks.go`의 `CreateDeck`이 실행하는 문장의 핵심이다.
`decks` 표에 새 행 하나를 만든다.

```sql
insert into decks (user_id, name, description) values ($1, $2, $3)
```

괄호 안의 열 목록과 `values`의 값 목록이 순서대로 짝을 이룬다.
`id`, `created_at`처럼 목록에 없는 열은 DDL에서 정한 `default`가 채워 주므로, 문장이 짧아진다.

`$1`, `$2`, `$3`은 자리표시자(placeholder)다.
실제 값은 문장과 별도로 전달되고, Go 코드에서는 `s.pool.QueryRow(ctx, 문장, userID, name, description)`처럼 인자로 넘긴다.
값을 문자열 이어붙이기로 문장 안에 직접 끼워 넣으면, 악의적인 입력에 SQL 조각을 심어 문장의 의미를 바꿔치기하는 SQL 인젝션(injection) 공격에 뚫린다.
자리표시자를 거친 값은 언제나 값으로만 취급되므로, "SQL은 반드시 자리표시자로"가 웹 개발의 철칙이다.

### select: 찾아 읽는다

가장 자주 쓰고 표현력도 가장 큰 문장이다.
덱 목록 화면이 부르는 `ListDecks`가 실행하는 문장으로, 한 사용자의 덱을 최신순으로 가져오면서 덱마다 카드 수까지 세어 붙인다.

```sql
select d.id, d.name, d.description,
       (select count(*) from cards c where c.deck_id = d.id) as card_count,
       d.created_at, d.updated_at, d.seq  -- ...일부 열 생략...
from decks d
where d.user_id = $1
order by d.created_at desc
```

절 단위로 끊어 읽으면 구조가 보인다.
`select`가 가져올 열, `from`이 대상 표(뒤의 `d`는 표에 붙인 별명), `where`가 행을 고르는 조건, `order by`가 정렬 순서다(`desc`는 내림차순).
여기에 행 수를 자르는 `limit`까지가 조회의 기본 재료다.

`where d.user_id = $1` 조건은 로그인한 사용자의 덱만 고른다.
Echo Flip의 store 계층 문장에는 이 조건이 빠짐없이 붙어 있는데, 사용자마다 자기 데이터만 보여야 하는 앱의 기본 규칙이 SQL 조건으로 구현된 것이다(이 설계는 3장의 RLS 논의로 이어진다).

`count(*)`는 행 수를 세는 집계 함수다.
괄호로 감싼 안쪽의 select(서브쿼리)가 "이 덱을 가리키는 카드 행 수"를 세어 `card_count`라는 이름으로 붙이고, 이 값이 10장에서 볼 덱 목록 화면의 `deck.cardCount`로 흘러가 "12장" 같은 표시가 된다.
집계에는 count 외에 합계(`sum`), 평균(`avg`) 등이 있고, `group by`와 묶으면 "날짜별 복습 수" 같은 통계도 SQL 한 문장으로 나온다.
실물 통계 쿼리는 3장에서 읽는다.

### 조인: 나눈 표를 다시 잇는다

표를 나누라고 해 놓고 조회는 어떻게 하나 싶을 텐데, 그 답이 조인이다.

::: info [용어 풀이] 조인(JOIN)
두 테이블을 공통 값으로 이어 붙여 하나의 표처럼 조회하는 연산이다.
카드 표와 복습 상태 표를 카드 id로 이으면, 카드 내용과 다음 복습 시각을 한 줄에 놓고 볼 수 있다.
데이터를 주제별 표로 나눠 담는 관계형 모델에서, 나눠 둔 것을 다시 합쳐 읽는 도구다.
:::

Echo Flip에서 카드의 복습 상태(다음 복습 시각 등)는 `cards`가 아니라 `card_srs`라는 별도 표에 있다(왜 나눴는지는 3장에서).
카드 내용과 복습 시각을 함께 보려면 두 표를 잇는다.

```sql
select c.text, c.meaning, s.due_at
from cards c
join card_srs s on s.card_id = c.id
where c.user_id = $1
```

`join ... on ...`의 `on`이 잇는 규칙이다.
"`card_srs`의 `card_id`가 `cards`의 `id`와 같은 행끼리 한 줄로 붙여라"라고 읽으면 된다.
외래 키로 이어 둔 관계를 조회 시점에 다시 합치는 것이므로, 표를 나눠도 조회가 불편해지지 않는다.
Echo Flip은 이 조인을 매번 쓰는 대신 뷰(view)라는 이름 붙은 조회로 저장해 두고 쓰는데, 그 실물인 `cards_with_stats`는 3장에서 해부한다.

### update와 delete: 고치고 지운다

`internal/store/cards.go`의 `UpdateCard`가 실행하는 문장이다.
카드 한 장의 내용을 통째로 새 값으로 바꾼다.

```sql
update cards set
  text = $3, meaning = $4, updated_at = now()  -- ...일부 열 생략...
where user_id = $1 and id = $2
```

`set`이 바꿀 열과 새 값, `where`가 바꿀 행을 정한다.
삭제도 구조는 같다.

```sql
delete from decks where user_id = $1 and id = $2
```

여기서 반드시 새겨 둘 것이 하나 있다.
`where`가 없는 update와 delete도 문법상 유효한 문장이고, 데이터베이스는 군말 없이 표의 모든 행을 고치거나 지운다.
"조건을 빠뜨린 delete"는 데이터베이스 사고의 고전이므로, 고치고 지우는 문장은 where부터 확인하는 습관이 필요하다.

두 문장 모두 `id`만으로 행이 정해지는데도 `user_id` 조건이 함께 붙어 있는 점도 눈여겨보자.
남의 카드 id를 알아내 지우려는 요청이 와도 자기 행이 아니면 조건에 걸리지 않는, 인가(authorization)를 겸한 이중 안전장치다.

고치거나 지운 뒤에는 몇 행이 영향을 받았는지가 돌아온다.
store 계층은 이 값으로 "그런 행이 없었다"를 판정한다.

```go
tag, err := s.pool.Exec(ctx, `delete from decks where user_id = $1 and id = $2`, userID, deckID)
// ...
if tag.RowsAffected() == 0 {
	return ErrNotFound
}
```

0행이면 5장에서 볼 센티널 에러 `ErrNotFound`를 돌려주고, 핸들러가 이를 404 응답으로 바꾼다.
SQL의 실행 결과가 Go의 에러 처리와 HTTP 상태 코드로 이어지는 연결 고리다.

한편 덱을 지우면 그 덱의 카드들은 어떻게 될까.
답은 앞서 DDL에서 본 `on delete cascade`에 있는데, 삭제 정책의 설계는 3장에서 정리한다.
그리고 여러 문장을 "전부 성공하거나 전부 취소"로 묶는 트랜잭션(transaction)도 복습 기록을 다루는 3장에서 실제 코드와 함께 만난다.

## 인덱스: 표가 커져도 빨리 찾는 법

마지막 기초는 조회 속도다.
`where user_id = $1` 같은 조건으로 행을 찾을 때, 데이터베이스가 표의 모든 행을 처음부터 끝까지 확인하는 것을 전체 스캔(full scan)이라 한다.
행이 백 개일 때는 아무 문제가 없지만, 복습 기록처럼 수십만 행씩 쌓이는 표에서는 조회마다 수십만 번의 확인이 일어난다.
훑는 양이 표 크기에 비례해서 늘어나는 것이 문제의 본질이다.

::: info [용어 풀이] 인덱스(Index)
특정 열의 값을 빠르게 찾도록 미리 정렬해 둔 보조 자료구조로, 책 뒤의 찾아보기(색인)와 같은 역할이다.
찾아보기가 있으면 책장을 처음부터 넘기지 않고 원하는 쪽으로 바로 가듯, 인덱스가 있으면 DB가 모든 행을 훑지 않고 원하는 행으로 곧장 간다.
대신 찾아보기를 최신으로 유지하는 품이 들듯, 인덱스도 데이터를 쓸 때마다 갱신 비용을 치른다.
:::

인덱스를 만드는 문장도 DDL이다.
`000001_init.up.sql`에서 발췌한, `decks` 표의 `user_id` 열에 인덱스를 거는 문장이다.

```sql
create index decks_user_idx on decks (user_id);
```

PostgreSQL 인덱스의 기본형은 B-tree로, 값들을 정렬된 상태로 유지하는 트리 구조다.
정렬되어 있으므로 이름순 전화번호부에서 원하는 이름을 찾듯 범위를 크게 좁혀 가며 내려갈 수 있고, 등호 조건(`=`)과 범위 조건(`<`, `>=`), 그리고 `order by` 정렬까지 인덱스의 정렬 순서로 해결된다.
앞서 본 `ListDecks`의 `where d.user_id = $1`이 이 인덱스 덕분에 덱이 아무리 쌓여도 해당 사용자의 행으로 곧장 간다.

### 복합 인덱스와 열 순서

인덱스는 여러 열을 묶어 하나로 만들 수도 있다.
Echo Flip의 심장인 "오늘 복습할 카드" 조회를 받치는 인덱스다.

```sql
create index card_srs_user_due_idx on card_srs (user_id, due_at);
```

복합 인덱스는 성(姓)으로 먼저 정렬하고 같은 성 안에서 이름으로 정렬한 명부와 같다.
성을 아는 순간 그 구역으로 바로 가고, 구역 안은 이름순이라 범위를 짚기도 쉽다.

그래서 열 순서가 중요하다.
"이 사용자의(user_id가 정확히 하나로 정해지는 등치 조건) 복습 시각이 지난(due_at이 일정 시점 이전이라는 범위 조건) 카드"라는 질문에는, 등치 조건의 열을 앞에 두고 범위 조건의 열을 뒤에 두는 (등치, 범위) 순서가 들어맞는다.
사용자 구역으로 바로 들어가 due_at 순서로 늘어선 행을 앞에서부터 끊어 읽으면 되기 때문이다.
이 인덱스가 실제 쿼리와 어떻게 맞물리는지는 3장에서 확인한다.

### 인덱스는 공짜가 아니다

인덱스가 이렇게 좋다면 모든 열에 걸면 되지 않을까.
그렇지 않다.
행을 넣거나 고칠 때마다 그 표의 모든 인덱스도 함께 갱신해야 하므로, 인덱스 하나하나가 쓰기 작업의 비용을 늘린다.
저장 공간도 따로 차지한다.
그래서 인덱스는 "일단 다 걸어 두는" 것이 아니라 실제 조회 패턴에서 역산해 필요한 곳에만 건다.

참고로 앞서 본 unique 제약은 내부적으로 유니크 인덱스로 구현된다.
중복 검사를 하려면 어차피 같은 값을 빨리 찾아야 하니, 제약과 인덱스는 한 몸이다.

Echo Flip이 어떤 조회를 위해 어떤 인덱스를 걸었고, 어떤 열에는 일부러 걸지 않았는지, 실제 인덱스 목록은 3장에서 해부한다.

## 정리

이 장에서는 다음 장의 스키마 해부에 필요한 기초 세 가지를 다졌다.

첫째, 관계형 데이터베이스는 데이터를 테이블(행과 열)에 담되, 같은 사실이 한 곳에만 적히도록 표를 주제별로 나누고 기본 키와 외래 키로 잇는다.

둘째, SQL은 원하는 결과만 서술하는 선언형 언어다.
DDL(create·alter·drop)로 틀을 만들고 제약으로 규칙을 새기며, DML(insert·select·update·delete)로 내용을 다룬다.
값은 자리표시자로 전달해 SQL 인젝션을 막고, 나눈 표는 조인으로 다시 합쳐 읽으며, update와 delete에서는 where 조건이 안전벨트다.

셋째, 인덱스는 조회 비용이 표 크기에 비례해 늘지 않게 하는 찾아보기다.
등치 조건 열을 앞에, 범위 조건 열을 뒤에 두는 복합 인덱스 순서 원칙을 봤고, 쓰기 비용 때문에 조회 패턴에서 역산해 걸어야 한다는 것도 확인했다.

다음 3장에서는 이 기초 위에서 Echo Flip의 실제 스키마를 해부한다.
도입의 요구사항이 일곱 개 테이블로 도출되고, 제약과 인덱스가 조회 패턴에서 역산되며, 스키마가 마이그레이션으로 진화하는 과정이다.
