= 3장 에이전트에게 지시하기: Plan 모드로 다듬는 요구사항과 아키텍처

2장에서 도구를 손에 익혔으니, 이제 실전에 투입할 차례다.
이 장의 목표는 도입에서 정리한 기능 요구사항과 1장에서 고른 기술 스택을 에이전트에게 지시해, 앱의 골격이 내 컴퓨터에서 처음 돌아가는 데까지 가는 것이다.
그 과정에서 이 책이 가장 힘주어 말하고 싶은 습관 하나를 익힌다.
코드를 만들기 전에 Plan 모드로 계획을 받아, 요구사항과 아키텍처를 그 계획 위에서 다듬는 습관이다.
이 장의 대화 예시는 Flashcard를 시작하던 과정을 재구성한 것으로, 프롬프트의 문구 자체보다 무엇을 지시문에 담고 무엇을 계획 단계에서 걸러 냈는지에 주목하며 읽으면 된다.

== 지시가 개발의 절반이다

2장에서 봤듯 에이전트는 모호한 지시를 받으면 그럴듯한 해석을 하나 골라 진행한다.
그 해석이 의도와 다르면 코드가 한참 만들어진 뒤에 방향을 되돌려야 하므로, 지시문에 들이는 정성이 곧 개발 속도다.

프로젝트를 처음 시작하는 지시에서는 특히 조심할 것이 하나 더 있다.
기술 스택을 못 박지 않으면, 에이전트는 학습 데이터에서 가장 많이 본 조합으로 기운다.
"암기 카드 웹 앱을 만들어 줘"라고만 하면 React와 Node.js 기반의 골격이 돌아올 가능성이 크다.
그 조합이 나빠서가 아니라 세상에 가장 흔한 조합이기 때문인데, 그대로 받아들이면 1장에서 따져 가며 내린 결론이 에이전트의 첫 응답에서 무의미해진다.
기술 선택은 사람이 제약을 따져 내리는 결정이고, 에이전트에게는 그 결정을 명시적으로 전달해야 한다.

== 첫 지시문: 시나리오와 기술 스택과 금지 사항

Flashcard의 시작 지시문을 재구성하면 다음과 같은 모양이다.

[source,text]
----
암기 카드 웹 앱을 만들려고 한다. 코드를 만들기 전에 계획부터 세워 달라.

[기능]
- 사용자는 덱을 만들고 카드(원문 text, 뜻 meaning)를 담는다.
- 학습: 방향(원문→뜻, 뜻→원문)을 고르고, 카드를 뒤집어 스스로
  맞음/틀림을 판정한다. 덱을 한 바퀴 돌면 틀린 카드만 모아
  재도전 라운드를 반복한다.
- 복습: SM-2를 맞음/틀림 두 값으로 단순화한 간격 반복으로
  "오늘 복습" 큐를 만든다. 재도전 라운드의 답은 채점에 반영하지 않는다.

[기술: 바꾸지 말 것]
- 백엔드: Go + Gin. 화면도 서버에서 html/template으로 렌더링하고,
  부분 갱신은 htmx를 쓴다. React 같은 브라우저 렌더링 프레임워크 금지.
- 데이터베이스: SQLite 파일 하나. 단, 나중에 PostgreSQL로 바꿀 수 있게
  DB 접근을 인터페이스 뒤로 모은다.
- CSS는 프레임워크 없이 파일 하나. 브라우저 자바스크립트는 최소로.

[지금 만들지 않는 것]
- 로그인, 배포 설정, 통계 화면. 골격이 돈 다음에 붙인다.
----

이 지시문에는 세 가지 층이 있고, 층마다 맡은 일이 다르다.

첫째, 기능은 목록이 아니라 시나리오로 적었다.
"학습 기능"이라고 쓰는 대신 방향 선택과 뒤집기, 자기 판정, 재도전 라운드라는 흐름을 문장으로 적었다.
동작의 순서와 규칙이 문장에 담겨 있어야 에이전트가 화면과 데이터의 모양을 스스로 도출할 수 있다.

둘째, 기술 스택에 "바꾸지 말 것"이라는 딱지를 붙였다.
앞서 말한 다수파로의 쏠림을 막는 장치이자, 1장의 결론을 대화의 전제로 승격시키는 문장이다.
왜 이 조합인지 이유까지 설명할 필요는 없지만, 협상 불가라는 사실은 분명히 해야 한다.

셋째, 지금 만들지 않는 것을 적었다.
에이전트는 성실함을 보이려고 범위를 부풀리는 경향이 있어서, 내버려 두면 첫 골격에 로그인과 설정 화면까지 딸려 온다.
범위를 좁혀 주는 문장은 낭비를 줄일 뿐 아니라, 뒤에서 볼 "커밋 단위로 나눠 만들기"의 출발점이 된다.

== Plan 모드로 계획을 받는다

지시문 첫 줄에 "계획부터 세워 달라"고 적었지만, 말로 부탁하는 대신 2장에서 본 Plan 모드를 켜는 쪽이 확실하다.
Plan 모드에서는 에이전트가 파일을 만들거나 고칠 수 없으므로, 지시가 어떻게 해석됐는지를 코드가 생기기 전에 계획서로 확인할 수 있다.

Flashcard 규모의 지시라면 계획에는 대체로 디렉터리 구조, 테이블 초안, 화면과 라우트(주소별 처리 경로) 목록, 구현 순서가 담겨 온다.
이 계획서를 읽을 때의 점검 목록은 세 줄이다.

첫째, 요구사항 해석이 맞는가.
재도전 라운드가 "틀린 카드만 모아 반복"으로 계획됐는지, 카드가 앞면/뒷면이 아니라 원문/뜻이라는 대칭 구조로 잡혔는지 같은 것들이다.

둘째, 기술 스택이 지시대로인가.
계획서의 의존성 목록에 지시에 없는 프런트엔드 빌드 도구나 ORM(SQL 대신 코드로 데이터베이스를 다루게 하는 라이브러리)이 슬쩍 들어와 있지 않은지 본다.
계획서에서 줄 하나 지우는 일이, 생성된 package.json과 설정 파일 무더기를 걷어 내는 일보다 훨씬 싸다.

셋째, 범위가 부풀지 않았는가.
"지금 만들지 않는 것"에 적어 둔 항목이 계획에 들어와 있으면 이번 단계에서 빼 달라고 요청한다.

== 계획 검토가 요구사항을 다듬는다

Plan 모드의 진짜 값어치는 계획서 검토가 아니라, 계획을 사이에 둔 대화에서 나온다.
계획을 세우던 에이전트가 이런 질문을 되돌려 준다.

"재도전 라운드에서 맞힌 답도 SRS 채점에 반영합니까?"

도입에서 이 규칙의 답(반영하지 않는다. 방금 본 카드를 다시 맞힌 것은 장기 기억의 성공이 아니므로)을 이미 봤지만, 원래 순서는 반대였다.
요구사항을 적을 때는 이 경우를 생각하지 못했고, 계획 단계의 질문이 스펙(요구사항 명세)의 빈틈을 드러냈으며, 답을 정하면서 요구사항이 한 줄 늘었다.
요구사항은 문서를 오래 들여다본다고 완성되지 않는다.
구현 계획이라는 구체적인 그림 앞에 세워 봐야 빈틈이 드러나고, Plan 모드는 코드를 만들기 전에 그 그림을 그려 주는 장치다.

아키텍처도 같은 방식으로 다듬어진다.
2장 끝에서 본 서버리스 전환이 그 사례다.
배포 구조를 논의하던 대화에서 "서버를 늘 켜 두지 않고 운영할 방법은 없나"라는 질문 하나가 상주 서버 방안을 뒤집었고, 트레이드오프 표를 받아 비교한 뒤 방향을 바꿨다.
계획과 코드 중 계획을 고치는 쪽이 압도적으로 싸다는 원칙은, 함수 하나의 설계에서든 앱 전체의 아키텍처에서든 똑같이 작동한다.

계획이 지시와 어긋나 있어도 실망할 일은 아니다.
어긋난 계획은 지시문의 어느 문장이 모호했는지를 알려 주는 피드백이고, 지시문을 고쳐 계획을 다시 받는 왕복은 몇 분이면 끝난다.

== 골격은 커밋 단위로 나눠 만든다

계획이 합의되면 승인하고 구현으로 넘어간다.
이때 계획 전체를 한 번에 만들게 하지 않고, 검증하고 커밋할 수 있는 덩어리로 나눠 진행한다.
Flashcard의 첫 골격은 대략 이런 순서였다.

첫째, 테이블 스키마(표의 구조 정의)와 SQLite 연결.
둘째, SRS 채점을 담당하는 순수 함수와 그 테스트.
셋째, 덱과 카드의 Gin 라우트.
넷째, 화면 템플릿과 htmx 학습 흐름.

한 덩어리가 끝날 때마다 `go build ./...`와 `go test ./...`가 통과하는지 확인하고 커밋한다.
이렇게 나누는 이유는 2장의 한계 절에서 본 컨텍스트 제한에 있다.
긴 세션에서는 초반의 합의가 흐려지므로, 작업을 커밋 단위로 끊고 덩어리마다 새 세션으로 시작하는 쪽이 결과가 안정적이다.
사람 입장에서도 검토 대상이 작아야 2장에서 세운 검토 기준(오류 발견 난이도와 오류 비용)을 적용할 여유가 생긴다.

마지막 덩어리까지 끝나면 첫 실행이다.
터미널에서 아래 명령 한 줄이면 내 컴퓨터에서 서버가 켜진다(Go 설치는 부록 A에 있다).

[source,bash]
----
go run ./cmd/server
----

서버가 떴다는 줄이 터미널에 출력되면, 브라우저 주소창에 `http://localhost:8080`을 입력한다.
localhost는 "내 컴퓨터 자신"을 가리키는 주소이고 뒤에 붙은 8080은 그 안에서 이 서버가 받는 창구 번호인데, 자세한 설명은 13장에 있다.
터미널 창은 닫지 말고 그대로 열어 두어야 하며, 서버를 끌 때는 그 창에서 Ctrl+C를 누른다.

브라우저에서 덱을 만들고 카드를 뒤집어 보는 순간, 도입의 요구사항이 처음으로 화면이 된다.

== 합의된 결정은 CLAUDE.md에 고정한다

계획을 다듬으며 내린 결정들은 이 세션의 대화에만 남아 있다.
2장에서 봤듯 다음 세션의 에이전트는 백지에서 시작하므로, 세션이 끝나기 전에 결정을 설정으로 옮겨 둔다.

기술 스택과 금지 사항(브라우저 렌더링 프레임워크 금지, DB 접근은 인터페이스 뒤로), 검증 명령, 디렉터리 구조 요약이 CLAUDE.md 초안의 재료다.
작성 요령은 2장에서 정리한 그대로, 에이전트가 틀리기 쉬운 대목만 골라 짧게 적는다.
이 시점에 CLAUDE.md를 만들어 두면, 이후의 모든 세션이 오늘 합의한 전제 위에서 시작된다.

== 정리

이 장에서는 요구사항과 기술 스택을 에이전트에게 지시해 앱의 골격을 만들었다.

첫째, 기술 스택을 못 박지 않으면 에이전트는 학습 데이터의 다수파 조합으로 기운다.
기능은 시나리오로, 기술 스택은 협상 불가로, 범위는 "지금 만들지 않는 것"까지 지시문에 담는다.

둘째, 구현 전에 Plan 모드로 계획을 받는다.
계획서에서는 요구사항 해석, 기술 스택 준수, 범위 세 가지를 점검하고, 어긋난 계획은 지시문을 고칠 피드백으로 쓴다.

셋째, 계획을 사이에 둔 대화가 요구사항과 아키텍처를 다듬는다.
에이전트의 질문이 스펙의 빈틈을 드러내고, 계획 단계의 방향 전환은 코드가 나온 뒤의 전환보다 압도적으로 싸다.

넷째, 구현은 검증하고 커밋할 수 있는 덩어리로 나눠 진행하고, 합의된 결정은 CLAUDE.md에 고정해 다음 세션의 전제로 만든다.

이제 골격이 도는 앱과, 에이전트가 작성한 코드 무더기가 손에 남았다.
남은 질문은 하나다.
이 코드가 옳은지 내가 판단할 수 있는가.
승인 버튼을 누르는 사람이 코드를 읽지 못하면 검토는 요식 행위가 되고, 2장에서 본 그럴듯하지만 틀린 코드가 그대로 쌓인다.
그래서 다음 4장부터는 이 앱을 이루는 기술을 바닥부터 한 층씩 익힌다.
첫 순서는 가장 아래층, 데이터가 저장되는 관계형 데이터베이스다.
에이전트가 만든 테이블이 왜 그런 모양인지 읽어 낼 수 있게 되는 것이 목표다.
