# 도입: 무엇을 만드는가

책의 본문에 들어가기 전에, 앞으로 계속 소재로 삼을 앱이 무엇인지부터 분명히 해 두려 한다.
이 장에서는 암기 카드 앱 Echo Flip의 기능 요구사항을 사용자 시나리오 중심으로 정리하고, 무료 인프라 운영이라는 비기능 요구사항이 기술 선택을 어떻게 끌고 갔는지 짚어 보겠다.
이어서 전체 아키텍처를 한눈에 조망하고, 이 책의 구성과 다루는 범위를 안내한다.
요구사항을 먼저 공유해 두면 이후 각 장에서 "왜 이렇게 만들었는가"라는 질문에 답하기가 훨씬 수월해진다.

::: info [용어 풀이] 아키텍처(Architecture)
소프트웨어를 이루는 큰 구성 요소들과 그 사이의 연결 방식을 정한 전체 구조다.
건물 설계도가 방 배치와 배관을 정하듯, 화면·서버·데이터베이스가 각각 어디에 놓여 어떻게 통신하는지를 정한다.
벽지 바꾸기와 달리 집 구조 변경이 큰 공사이듯, 아키텍처도 나중에 바꾸기 어려우므로 초기에 신중하게 정한다.
:::

## Echo Flip은 어떤 앱인가

Echo Flip은 암기 카드(flashcard) 앱이다.
영어 단어·숙어·문장은 물론, 전공 용어와 개념 설명처럼 짝을 이루는 지식이라면 무엇이든 카드로 만들어 암기할 수 있다.
카드 앞면에는 원문(text), 뒷면에는 뜻(meaning)이 적혀 있고, 사용자는 카드를 뒤집어 답을 확인한 뒤 스스로 맞았는지 틀렸는지를 판정한다.
종이 단어장을 넘기던 경험을 웹과 모바일로 옮기되, "오늘 무엇을 복습할지"를 앱이 대신 골라 주는 것이 핵심이다.
그 바탕이 되는 간격 반복(Spaced Repetition)은 복습 간격을 조금씩 늘려 가며 잊어버릴 때쯤 다시 보여 주는 학습 방법이다.
갓 외운 카드는 자주, 익숙해진 카드는 뜸하게 내보내니 사람이 복습 일정을 짜지 않아도 적은 노력으로 오래 기억할 수 있다.
이 책에 자주 나오는 SRS는 이 방식을 구현한 시스템(Spaced Repetition System)의 약자다.

이 앱에는 또 하나의 배경이 있다.
Echo Flip은 개인 개발자가 AI 코딩 에이전트인 Claude Code와 함께 만든 앱이다.
요구사항을 정리하고 기술을 선택하고 결과를 검증하는 일은 사람이 했고, 코드의 상당 부분은 에이전트가 작성했다.
그래서 이 저장소에는 앱 코드뿐 아니라 에이전트가 스스로 품질을 지키게 만드는 장치인 훅(Hook)과 서브에이전트(Subagent) 설정도 함께 들어 있다.
AI와 함께 개발하는 방식 자체도 이 책이 다루는 주제 중 하나이며, 11장과 12장에서 자세히 살펴본다.

혼자 만들고 혼자 운영하는 앱이라는 점은 이후 모든 기술 결정에 영향을 준다.
팀도 없고 예산도 없으므로, 기능은 욕심내되 인프라는 최대한 단순하고 저렴해야 한다.
이 제약이 어떤 아키텍처로 이어졌는지가 이 장의 후반부 주제다.

## 기능 요구사항: 사용자 시나리오로 살펴보기

기능 목록을 나열하는 대신, 사용자가 앱을 쓰는 흐름을 따라가며 요구사항을 정리해 보겠다.
학습 앱의 하루는 대체로 "재료를 만들고 → 학습하고 → 복습 큐를 소화하고 → 성취를 확인하는" 순서로 흘러간다.

### 학습 재료 만들기: 카드, 덱, 태그

사용자는 먼저 덱(Deck)을 만들고 그 안에 카드를 채운다.
카드는 양방향이다.
원문(text)에는 영어 단어·숙어·문장이나 전공 용어를, 뜻(meaning)에는 그 설명을 적는다.
"원문을 보고 뜻을 떠올리기"와 "뜻을 보고 원문을 떠올리기" 중 어느 방향으로도 학습할 수 있어야 하므로, 카드 모델은 앞면·뒷면(front/back)이 아니라 대칭적인 두 필드로 설계했다.
카드에는 태그를 붙일 수 있어 "동사", "TOEIC" 같은 분류로 여러 덱을 가로질러 묶을 수 있다.

카드를 한 장씩 입력하는 일은 지루하므로, 입력을 도와주는 부가 기능을 두 가지 마련했다.

첫째, CSV 가져오기/내보내기다.
`text,meaning,type,tags,phonetic,example` 헤더의 CSV 파일을 통째로 들여올 수 있고, 태그는 `|` 문자로 구분한다.
초기 버전의 `front,back` 헤더도 하위 호환으로 인식한다.
스프레드시트에 정리해 둔 단어장을 그대로 옮겨 오는 시나리오를 겨냥한 기능이다.

둘째, 무료 사전 API 자동 채우기다.
영어 단어를 입력하고 버튼을 누르면 Free Dictionary API(api.dictionaryapi.dev)에서 발음기호·뜻·예문을 받아 와 카드 입력란을 자동으로 채워 준다.
이 API는 키 발급이 필요 없어서 계정 하나 만들지 않고 쓸 수 있다.
조회는 Go 서버가 대신 수행하며(`internal/web/dictionary.go`), 받아 온 값으로 입력 폼의 빈칸만 채워 주는 화면 갱신 기법은 10장에서 htmx 예제로 다시 등장한다.

::: info [용어 풀이] API(Application Programming Interface)
한 프로그램이 다른 프로그램의 기능이나 데이터를 정해진 형식으로 요청해 받아 오는 창구다.
식당의 메뉴판처럼 무엇을 어떤 형식으로 주문하면 무엇이 돌아오는지 미리 약속돼 있어, 서로의 내부 사정을 몰라도 협력할 수 있다.
Echo Flip이 사전 뜻을 받아 오려고 부르는 Free Dictionary API가 그런 창구의 한 예다.
:::

덱 하나를 열면 이 재료들이 한 화면에 모인다.
카드 목록 위로 학습 시작 버튼이 있고, 그 아래 CSV 가져오기·내보내기와 덱 공유 버튼이 나란히 놓인다.

### 학습하기: 뒤집고, 판정하고, 다시 도전한다

학습 세션의 흐름은 다음과 같다.

첫째, 방향을 선택한다.
원문→뜻으로 볼지, 뜻→원문으로 볼지를 세션 시작 시점에 고른다.

둘째, 카드를 뒤집는다.
질문 면을 보고 답을 떠올린 뒤 카드를 탭하면 반대 면이 나타난다.
"Flip"이라는 이름이 여기서 나왔다.

셋째, 스스로 맞음/틀림을 판정한다.
객관식 문제를 자동 채점하는 방식이 아니라, 사용자가 정직하게 자기 답을 평가하는 방식이다.
채점 로직을 만들지 않아도 되니 구현이 단순해지고, 단답이 아닌 문장·개념 카드에도 똑같이 적용된다는 장점이 있다.
대신 판정의 신뢰도가 사용자에게 달려 있다는 한계는 감수한다.

넷째, 덱을 한 바퀴 소화하면 틀린 카드만 모아 재도전 라운드가 시작된다.
모든 카드를 맞힐 때까지 라운드가 반복되므로, 세션이 끝나는 시점에는 적어도 단기 기억에는 전부 들어가 있게 된다.
이 흐름을 관리하는 로직은 `internal/web/study.go`에 있고, 서버가 진행 상태를 저장하지 않고도 세션을 이어 가는 방식과 함께 10장에서 해부한다.

### 꾸준히 복습하기: 간격 반복과 "오늘 복습" 큐

암기 앱의 승부처는 학습 화면이 아니라 복습 스케줄링이다.
Echo Flip은 간격 반복 알고리즘의 고전인 SM-2의 이진(binary) 변형을 사용한다.
원래 SM-2는 0~5의 6단계 품질 점수를 입력받지만, "이번 답이 4점인지 5점인지"를 매번 고민하게 만드는 것은 카드 뒤집기의 리듬을 깨뜨린다.
그래서 UI는 맞음/틀림 두 버튼만 제공하고, 내부적으로 맞음을 품질 5, 틀림을 품질 2로 사상한다.

핵심 로직은 `internal/srs/srs.go`의 `Grade` 함수 하나로, Go로 작성한 순수 함수다.
맞히면 복습 간격이 1일 → 6일 → 이전 간격 × 용이도(ease factor) 순으로 늘어나고, 틀리면 간격이 1일로 초기화되면서 용이도가 깎인다.
잘 외운 카드는 점점 뜸하게, 헷갈리는 카드는 자주 나타나는 구조다.
재도전 라운드의 답은 채점에 반영하지 않는다는 규칙도 있는데, 방금 본 카드를 다시 맞힌 것을 "장기 기억 성공"으로 치면 간격이 부당하게 늘어나기 때문이다.

이 함수 덕분에 홈 화면의 "오늘 복습" 큐가 자동으로 구성된다.
사용자는 매일 앱을 열고 큐에 쌓인 카드만 소화하면 된다.

### 무엇을 공부할지 고르기: 규칙 기반 스마트 덱

복습 큐만으로는 부족한 순간이 있다.
"유난히 자주 틀리는 카드만 몰아서 보고 싶다"거나 "한동안 안 본 카드를 점검하고 싶다"는 요구다.
Echo Flip은 이를 스마트 덱으로 해결한다.
덱에 카드를 미리 담아 두는 대신 "이런 카드를 모아 달라"는 조건만 적어 두는 방식이다.
스마트 덱은 카드를 소유하지 않는 가상의 덱으로, 이 조건에 해당하는 규칙(rule)을 JSON으로 저장해 두었다가 학습 시점에 쿼리로 변환해 카드를 골라낸다.

규칙은 오답률이 높은 카드(high_error), 오래 안 본 카드(stale), 특정 태그의 카드(tag), 최근 추가한 카드(recent)의 네 가지를 지원한다.
규칙을 정의한 `internal/smartrules/rules.go`는 5장에서 해부한다.
예를 들어 high_error 규칙의 기본값은 "3회 이상 시도했고 오답률이 40% 이상인 카드"다.
머신러닝 같은 거창한 추천이 아니라 SQL 한 줄로 표현되는 규칙 기반 추천이지만, 개인 학습 앱에는 이 정도가 오히려 예측 가능하고 유지보수하기 쉽다.
학습 시점마다 쿼리를 다시 실행하므로 스마트 덱의 내용물이 낡을(stale) 일도 없다.

### 성취 확인과 나눔: 통계와 덱 공유

꾸준함을 유지하려면 성취가 보여야 한다.
Echo Flip은 일별 학습량, 정답률, 연속 학습일(streak), 덱별 성취도를 통계 화면에서 보여 준다.
이를 위해 모든 학습 세션과 카드별 답안을 `study_sessions`, `review_logs` 테이블에 기록한다.

마지막으로 덱 공유다.
잘 만든 단어장은 혼자 쓰기 아깝다.
사용자는 자기 덱의 공유 링크를 만들 수 있고, 공유된 덱은 공유 덱 갤러리에 노출된다.
다른 사용자는 로그인하지 않고도 공유 덱을 미리 볼 수 있으며, "내 덱으로 가져오기"를 누르면 카드가 복사되어 자기 계정으로 들어온다.
이때 학습 기록(SRS 상태)은 복사하지 않고 새로 시작한다.
원저자의 기억 상태가 내 기억 상태일 수는 없기 때문이다.

## 비기능 요구사항: 무료로, 혼자서, 오래 운영하기

기능 요구사항보다 기술 선택을 더 강하게 규정한 것은 비기능 요구사항이었다.
네 가지를 짚어 보겠다.

첫째, 완전 무료 인프라로 운영한다.
개인 학습 앱은 수익 모델이 없으므로 고정비가 발생하는 순간 운영을 접게 된다.
월 몇 달러라도 서버비가 나가기 시작하면, 쓰는 사람이 몇 명뿐인 앱을 언제까지 유지할지 매달 고민하게 되기 때문이다.
호스팅·데이터베이스·인증까지 전부 무료 티어 안에서 해결해야 한다.
유료 전환을 전제로 한 "체험판 무료"가 아니라, 트래픽이 적은 개인 앱이라면 영구적으로 무료인 티어를 골랐다.

::: info [용어 풀이] 호스팅(Hosting)
만든 웹 앱을 인터넷에 늘 접속 가능한 상태로 올려 두는 일, 또는 그 일을 대신해 주는 서비스다.
가게를 열려면 손님이 찾아올 자리를 빌려야 하듯, 앱도 사용자가 주소로 찾아올 수 있는 자리가 필요하다.
Echo Flip은 Vercel이라는 호스팅 서비스에 앱을 올려 둔다.
:::

::: info [용어 풀이] 무료 티어(Free Tier)
클라우드 서비스가 일정 사용량까지는 요금을 받지 않는 무료 구간이다.
트래픽이 적은 개인 앱은 이 구간 안에 머무르면 월 이용료 없이 서비스를 운영할 수 있다.
여기서 말하는 무료 티어는 "체험판"과 달리 기간이 아니라 사용량으로 한도가 정해지는 영구 무료 구간을 가리킨다.
:::

둘째, 관리할 플랫폼을 최소화한다.
혼자 운영하는 앱에서 대시보드가 세 개, 네 개로 늘어나면 어느 순간 어딘가의 설정이 만료되거나 어긋난다.
Echo Flip은 배포 대상 플랫폼을 Vercel 하나로, 데이터 계층을 Supabase 하나로 묶어 관리할 곳을 둘로 줄였다.
Vercel은 웹 앱을 올려 두는 호스팅 서비스로, 코드 저장소를 연결해 두면 빌드와 배포를 자동으로 처리해 준다.
Supabase는 PostgreSQL 데이터베이스와 로그인 인증을 대신 운영해 주는 서비스다.
둘 다 트래픽이 적은 개인 앱이라면 무료 티어 안에서 운영할 수 있다.
한발 더 나아가 개발 단계에서는 외부 서비스를 아예 쓰지 않고 내 컴퓨터 안에서만 개발하므로, 두 플랫폼이 관리 대상이 되는 것은 2부에서 앱을 공개하는 시점부터다.
Vercel은 16장에서, Supabase는 인증과 데이터베이스 연결로 나누어 17장과 18장에서 다루고, 두 서비스의 무료 한도는 20장에서 한자리에 놓고 비교한다.
화면과 API를 서로 다른 곳에 배포하는 구성(예: 화면은 정적 파일 호스팅, API는 별도 컨테이너 호스팅)도 검토했다.
하지만 이렇게 나누면 화면과 API의 인터넷 주소가 서로 달라지고, 브라우저는 주소가 다른 곳으로 가는 요청을 일단 의심하고 막는다.
결국 오리진이 갈라지면 이를 허용해 주는 CORS 설정과 두 곳의 배포 시점을 맞추는 동기화라는 운영 부담이 따라온다.
상주 서버가 꼭 필요한 앱(웹소켓, 장시간 작업)이라면 그쪽이 낫지만, 요청-응답형 API만 있는 이 앱에는 한 플랫폼 올인원이 맞았다.

::: info [용어 풀이] PostgreSQL
무료로 쓸 수 있는 오픈 소스 데이터베이스다.
데이터를 스프레드시트 비슷한 표(테이블) 형태로 저장하고, 표끼리 연결해 조회할 수 있어 관계형 데이터베이스로 분류된다.
수십 년간 다듬어져 개인 프로젝트부터 대기업 서비스까지 두루 쓰이며, Echo Flip의 테이블 설계는 3장에서 다룬다.
:::

::: info [용어 풀이] 프런트엔드와 백엔드
프런트엔드는 사용자의 화면에서 눈에 보이는 부분, 백엔드는 그 뒤에서 데이터를 처리하고 저장하는 부분이다.
식당에 비유하면 손님을 맞는 홀이 프런트엔드, 주문을 받아 요리하는 주방이 백엔드다.
Echo Flip에서는 브라우저에 표시되는 화면이 프런트엔드이고, 그 화면(HTML)을 만들어 보내고 데이터베이스와 대화하는 Go 서버가 백엔드다.
:::

셋째, PWA(Progressive Web App)로 웹과 Android를 모두 커버한다.
네이티브 앱을 만들면 스토어 등록비와 심사, 별도 코드베이스라는 비용이 생긴다.
암기 카드 앱은 카메라나 푸시 같은 네이티브 기능이 필수가 아니므로, 웹 앱에 매니페스트를 더해 홈 화면에 설치하는 것으로 충분하다.
매니페스트의 `display: "standalone"` 선언 덕분에 Android Chrome에서 "홈 화면에 추가"를 누르면 주소창 없는 독립 실행형 앱처럼 동작하고, iOS Safari에서도 같은 방식으로 설치할 수 있다.
이 매니페스트 파일(`internal/web/static/manifest.webmanifest`)은 19장에서 다룬다.

::: info [용어 풀이] PWA(Progressive Web App)
설치해 쓰는 앱처럼 동작하는 웹 앱이다.
브라우저로 여는 웹사이트지만, 홈 화면에 추가하면 주소창 없이 독립 실행형 앱처럼 열려 앱 스토어를 거치지 않고도 설치한 앱 같은 경험을 준다.
덕분에 웹과 모바일을 하나의 코드로 함께 커버할 수 있다.
:::

넷째, 서버리스(Serverless) 콜드스타트(Cold Start)를 최소화한다.
무료 티어의 서버리스 함수는 요청이 없으면 잠들고, 첫 요청에서 깨어나는 시간이 사용자 체감 지연이 된다.
깨어나는 속도는 결국 프로그램이 얼마나 가볍고 실행 준비가 얼마나 간단한가에 달렸다.
즉 콜드스타트는 런타임의 기동 속도와 바이너리 크기에 좌우되므로, 백엔드 언어로 컴파일 언어인 Go를 선택했다.
실행에 필요한 것이 파일 하나에 담기는 단일 바이너리로 빌드되고 기동이 수십 밀리초 수준이라 서버리스와 궁합이 좋다.
Node.js나 Python 같은 인터프리터 언어로 백엔드를 만드는 선택지도 있었지만, 이 프로젝트는 콜드스타트와 견고한 핵심 알고리즘 구현을 우선해 Go를 택했다.
이 트레이드오프는 1장에서 본격적으로 다룬다.

::: info [용어 풀이] 서버리스(Serverless)
서버를 늘 켜 두지 않고, 요청이 올 때만 잠깐 실행됐다가 사라지는 방식이다.
"서버가 없다"는 뜻이 아니라, 서버를 직접 마련하고 지키는 일을 클라우드에 맡긴다는 뜻이다.
요청이 없을 때는 아무것도 돌지 않으므로, 트래픽이 적은 앱은 비용을 크게 아낄 수 있다.
:::

::: info [용어 풀이] 콜드스타트(Cold Start)
잠들어 있던 서버리스 함수가 첫 요청에 깨어나 준비를 마칠 때까지 걸리는 지연이다.
한동안 덮어 둔 노트북을 열 때 잠깐 뜸을 들이는 것과 비슷해서, 이 시간이 길면 사용자가 첫 화면을 그만큼 기다리게 된다.
그래서 기동이 빠른 언어를 고르면 이 지연을 줄일 수 있다.
:::

같은 이유로 지연 시간의 다른 축인 API↔DB 거리도 관리한다.
Vercel 함수 리전을 iad1(미국 동부)로 고정하고 Supabase 프로젝트도 같은 East US 리전에 만들어, 함수와 데이터베이스가 같은 지역 안에서 통신하게 했다.
사용자가 한국에 있어도 브라우저↔서버 왕복은 한 번이지만 API↔DB 왕복은 요청당 여러 번일 수 있으므로, DB를 함수 옆에 두는 쪽이 전체 지연에 유리하다.

## 전체 아키텍처 개요

이상의 요구사항이 수렴한 아키텍처를 그림 하나로 정리해 보자.
그 전에 하나 일러둔다.
지금부터 볼 그림은 앱을 세상에 공개한 뒤의 모습, 즉 2부가 끝났을 때 완성되는 운영 아키텍처다.
1부에서 다루는 앱은 같은 코드이지만 Vercel도 Supabase도 거치지 않고, 파일 하나로 동작하는 관계형 데이터베이스인 SQLite와 함께 내 컴퓨터 안에서만 돈다.
운영 설정이 주어지면 이 그림대로, 아무 설정이 없으면 로컬 모드로 동작하는 두 모드 구조라서, 계정 하나 만들지 않고도 1부의 앱을 끝까지 완성할 수 있다.

```text
[브라우저 / Android PWA]
   │
   │ ① 화면: Go 서버가 렌더링한 HTML 문서 받기
   │ ② 정적 자원: CSS·htmx·아이콘 내려받기
   │ ③ 조각 갱신: htmx가 보낸 요청에 HTML 조각으로 응답
   ▼
[Vercel — 무료 티어, 함수 리전 iad1]
   └─ Go 서버리스 함수 하나: api/index.go → Gin 라우터
          ├─ 화면 렌더링: html/template
          ├─ 정적 파일: 바이너리에 내장(embed)
          ├─ JSON API: /api/*
          └─ 로그인: Supabase Auth와 OAuth 왕복,
             세션은 HttpOnly 쿠키
          │
          │ ④ SQL — 트랜잭션 풀러(포트 6543)
          ▼
[Supabase — 무료 티어, East US 리전]
   ├─ PostgreSQL: profiles, decks, cards, card_srs,
   │              study_sessions, review_logs, smart_decks
   └─ Auth: Google/GitHub OAuth 처리, JWT 발급
```

구성 요소는 셋뿐이다.
화면을 표시하는 브라우저, 화면 렌더링부터 JSON API까지 도맡는 Vercel의 Go 서버리스 함수 하나, 그리고 데이터베이스와 인증을 겸하는 Supabase다.
각각을 조금 더 들여다보겠다.

### 화면: Go 서버가 렌더링하는 HTML

화면은 브라우저가 아니라 서버에서 만든다.
사용자가 주소를 열면 Go 서버가 Go 표준 라이브러리의 템플릿 엔진(html/template)으로 완성된 HTML 문서를 만들어 보내고, 브라우저는 받은 문서를 그리기만 한다.
카드 목록에서 한 장을 지우는 것 같은 부분 갱신은 htmx가 맡는다.
버튼에 적어 둔 속성대로 htmx가 서버에 요청을 보내고, 서버가 돌려준 HTML 조각으로 화면의 해당 부분만 갈아 끼우는 방식이라 페이지 전체를 새로 고치지 않아도 된다.
스타일은 프레임워크 없이 순수 CSS 파일 하나(`internal/web/static/app.css`)로, CSS 변수와 미디어 쿼리로 다크 모드까지 처리한다.
이 책이 다른 웹 개발 입문과 갈라지는 대목이 여기다.
직접 작성한 브라우저 자바스크립트는 음성 재생·클립보드 복사처럼 브라우저에서만 할 수 있는 일을 감싸는 파일 하나(`internal/web/static/app.js`, 약 80줄)뿐이고, 카드 뒤집기 애니메이션조차 자바스크립트 없이 CSS만으로 동작한다.
HTML과 CSS는 8장에서, 템플릿 구조는 9장에서, htmx는 10장에서 다룬다.

### 백엔드: Go/Gin을 Vercel 함수로

백엔드는 Go와 웹 프레임워크 Gin으로 작성한 서버 하나다.
화면 렌더링과 정적 파일 서빙, 그리고 외부 프로그램이 쓸 수 있는 JSON API(`/api/*`)까지 이 서버가 전부 처리한다.
로컬에서는 `cmd/server`로 띄우는 평범한 HTTP 서버지만, 배포 시에는 Vercel 서버리스 함수 하나로 감싼다.
진입점인 `api/index.go`는 들어온 요청을 Gin 엔진에 그대로 넘기는 함수 하나가 전부다(코드는 7장에서 본다).
Gin 엔진 조립은 `pkg/app`에 있고, 로컬 서버와 Vercel 함수가 같은 엔진을 공유한다.
덕분에 로컬에서 검증한 라우팅·미들웨어가 배포 환경에서도 동일하게 동작한다.
화면·경로마다 함수를 쪼개는 대신 함수 하나가 모든 요청을 받는 캐치올(catch-all) 구조라, 무료 티어의 함수 개수 제한과도 무관하고 콜드스타트도 한 곳에서만 발생한다.
이 연결을 완성하는 것이 `vercel.json`의 재작성(rewrite) 규칙으로, 모든 경로를 Go 함수로 보낸다(전문은 16장에서 본다).
CSS 같은 정적 파일도 별도 저장소에 올리는 대신 Go 바이너리 안에 내장(embed)해 같은 함수가 내보내므로, 배포 단위가 함수 하나로 끝난다.
화면과 API가 같은 곳에서 나오니 주소(오리진)가 갈라질 일이 없고, 운영 환경에서 CORS 문제가 아예 생기지 않는다.
Gin의 구조는 7장에서, Vercel 함수의 제약(예: `internal/` 패키지를 import할 수 없어 공유 코드를 `pkg/`에 두는 이유)은 16장에서 다룬다.

### 데이터와 인증: Supabase

Supabase는 관리형 PostgreSQL과 인증을 한 계정으로 제공한다.
Echo Flip은 이 중 두 가지만 쓴다.
PostgreSQL 데이터베이스, 그리고 OAuth를 이용한 Google/GitHub 로그인이다.

::: info [용어 풀이] OAuth
앱에 비밀번호를 직접 넘기지 않고, 이미 신뢰하는 구글이나 깃허브에 대신 로그인해 "이 사람이 맞다"는 확인만 받아 오는 방식이다.
앱은 당신의 구글 비밀번호를 끝내 보지 못하고, 구글이 발급한 확인 결과만 건네받는다.
그래서 비밀번호를 저장할 의무 자체가 앱에서 사라진다.
:::

인증 흐름은 이렇다.
사용자가 로그인 버튼을 누르면 Go 서버가 Supabase Auth와 OAuth를 왕복해 JWT를 받고, 이를 자바스크립트가 읽을 수 없는 HttpOnly 쿠키에 담아 브라우저에 맡긴다.
이후 브라우저는 요청마다 그 쿠키를 자동으로 실어 보내므로, 로그인 처리에 브라우저 자바스크립트가 한 줄도 필요 없다.
서버는 로그인한 사용자 명단을 따로 기억해 두지 않고, 쿠키에 실려 온 JWT가 진짜인지만 그때그때 확인한다.
이를 위해 Go 서버는 Supabase의 JWKS(JSON Web Key Set) 엔드포인트에서 위조 여부를 가리는 검증용 열쇠인 공개키를 받아 JWT 서명을 검증한다.
백엔드가 세션 저장소를 가질 필요가 없으니 서버리스와 잘 맞는 무상태(stateless) 구조다.

::: info [용어 풀이] JWT(JSON Web Token)
로그인에 성공했다는 사실을 서버가 서명해서 발급하는 긴 문자열이다.
놀이공원에서 입장 확인 후 손목에 채워 주는 밴드처럼, 한 번 받으면 매 요청마다 신분증을 다시 보이지 않아도 된다.
위조하면 서명이 깨지므로 서버는 밴드가 진짜인지 검사만 하면 된다.
:::

한 가지 특이한 결정이 있다.
Supabase는 브라우저에서 DB에 직접 접근하는 PostgREST API를 기본 제공하지만, Echo Flip은 이를 쓰지 않는다.
데이터베이스로 통하는 옆문을 잠그고 정문인 Go API 하나만 남긴 셈이다.
구체적으로는 모든 테이블에 행 수준 보안(Row Level Security, RLS)을 켜되 정책을 하나도 만들지 않아, 공개 키(anon key)로는 어떤 데이터에도 접근할 수 없게 막았다.
데이터 접근은 오직 Go API를 통해서만 이루어진다.
RLS 정책만으로 권한을 다 표현하는 구성도 가능하고 백엔드 코드가 줄어드는 매력이 있지만, SRS 채점이나 덱 복사처럼 여러 테이블을 묶는 트랜잭션 로직은 결국 서버 코드가 필요했다.
예를 들어 덱 복사가 카드 수십 장을 옮기다 중간에 끊겼을 때, 절반만 복사된 덱이 남지 않게 전체를 한 묶음으로 처리하는 일이 그렇다.
접근 경로를 하나로 통일하는 쪽이 보안 검토도 단순해진다.
이 설계와 트레이드오프는 17장과 18장에서 자세히 다룬다.

### 기술 스택 요약

의존성 목록으로 스택을 요약해 두겠다.
Go 쪽의 직접 의존성은 웹 프레임워크 Gin, JWT 검증 라이브러리(golang-jwt, keyfunc), PostgreSQL 드라이버 pgx, 로컬 모드가 쓰는 SQLite 드라이버, 마이그레이션 도구 golang-migrate, CORS 미들웨어, UUID 라이브러리까지 여덟 개가 전부다.
화면 쪽에는 의존성 목록이라 부를 것 자체가 없다.
템플릿(html/template)과 CSV 처리(encoding/csv)는 Go 표준 라이브러리이고, 브라우저에서 쓰는 외부 라이브러리는 htmx 하나뿐인데 이마저 파일 한 개를 저장소에 그대로 넣어 두었다.
패키지 매니저도, 빌드 도구도, 버전 잠금 파일도 화면 쪽에는 없다.
개인 프로젝트에서 의존성 하나는 곧 업그레이드 부담 하나다.
라이브러리 하나에 보안 결함이 공지되면 새 버전으로 올리고 그 여파로 깨진 곳을 고치는 일까지 혼자 감당해야 한다.
그래서 표준 라이브러리와 프레임워크 기본 기능으로 해결되는 일에는 라이브러리를 추가하지 않는다는 원칙을 지켰다.
`go.mod`의 의존성 목록은 4장에서 발췌해 다룬다.

## 이 책의 구성

본격적인 안내에 앞서 한 가지 일러둔다.
이 책에는 곳곳에 "[용어 풀이]"라고 적힌 상자가 있는데, 개발이 처음인 독자를 위해 낯선 개발 용어를 두어 문장으로 풀어 둔 것이다.
이미 아는 용어라면 지나쳐도 좋고, 상자를 건너뛰어도 본문은 그대로 이어진다.

이 책은 두 부로 나뉜다.
1부에서는 내 컴퓨터 안에서 완전하게 동작하는 앱을 완성하고, 2부에서는 그 앱을 세상에 공개하고 오래 운영하는 체계를 만든다.
1부의 실습은 SQLite와 함께 내 컴퓨터에서 완결되므로, 설치할 프로그래밍 언어는 Go 하나뿐이다.
여기에 데이터베이스 실습에 쓰는 명령줄 도구 sqlite3 정도를 더하면 준비가 끝나고, Supabase와 Vercel 계정은 2부에 가서야 필요하다.
Git이나 Go 같은 도구를 설치해 본 적이 없는 독자를 위해, 필요한 도구의 OS별 설치 방법은 부록 A에 따로 정리해 두었다.
덕분에 독자는 외부 서비스에 가입하지 않고도 바로 만들기 시작할 수 있고, 앞 장들을 따라가는 동안 계정 설정이나 네트워크 문제로 발이 묶일 일이 없다.
1부 안에서도 데이터베이스·Go·HTML과 htmx라는 기초를 먼저 다지고, 그 코드를 만들어 내는 AI 에이전트 도구는 뒤에 두었다.

도구 이야기보다 기초를 앞세운 데는 이유가 있다.
에이전트가 코드의 상당 부분을 써 주더라도 결과를 검증하는 일은 사람에게 남고, 검증은 검증할 지식이 있어야 가능하다.
`api/index.go`가 `internal/` 패키지를 import하면 안 되는 이유(16장)를 모르면, 에이전트가 만든 코드가 로컬에서는 통과하고 배포에서만 실패할 때 원인을 짚을 수 없다.
pgx 연결 설정이 왜 그렇게 되어 있는지(18장) 모르면, 에이전트가 제안한 "더 깔끔한" 수정이 무엇을 깨뜨리는지 판단할 수 없다.
에이전트가 코드를 대신 써 주는 시대에도, 어쩌면 그렇기 때문에 더욱 언어와 프레임워크의 기초는 개발자의 판단 근거로 필요하다.
에이전트 도구 이야기에 앞서 데이터베이스와 Go, 그리고 화면을 이루는 HTML·템플릿·htmx의 기초부터 다지는 이유다.

도입에 이어지는 1장 기술 선택에서는 무료 인프라와 1인 개발이라는 제약에서 출발해 Go와 Gin, 서버에서 HTML을 렌더링하고 htmx로 갱신하는 화면 구성, 그리고 SQLite로 시작해 PostgreSQL로 옮겨 가는 데이터베이스 구성이 어떻게 정해졌는지를 대안과 트레이드오프로 정리한다.
뒤의 장들이 "어떻게 쓰는가"를 다룬다면, 이 장은 "왜 이것인가"에 답한다.

1부 "내 컴퓨터에서 웹 앱 완성하기"는 2장부터 13장까지다.
2장부터 10장까지는 데이터가 저장되는 바닥층에서 출발해 사용자가 만지는 화면 쪽으로 한 층씩 올라가고, 11장부터 13장까지는 그 코드를 만들어 내는 에이전트와 개발 환경을 갖춘다.
13장이 끝나면 앱이 내 컴퓨터에서 완전하게 돌아간다.

2장 데이터베이스 기초에서는 관계형 데이터베이스가 데이터를 담는 방식(테이블과 키), 구조를 만들고 데이터를 다루는 SQL(DDL과 DML), 조회 속도를 좌우하는 인덱스 개념을 실제 테이블과 쿼리로 익힌다.
실습 상대는 서버 설치도 계정도 필요 없는 SQLite다.

3장 데이터베이스 설계에서는 이 장의 요구사항이 어떻게 일곱 개 테이블로 도출되는지 따라가며, 키·제약·인덱스와 SRS 데이터 모델, 그리고 전용 자료형이 적은 SQLite에 설계 의도를 담는 법을 다룬다.
데이터가 어떤 모양으로 저장되는지 알고 나면, 그 위에 얹히는 백엔드 코드와 화면이 왜 그렇게 생겼는지가 훨씬 빨리 눈에 들어온다.

4장 Go 기초에서는 모듈과 패키지, 저장소의 디렉터리 구조, 변수와 함수 같은 기본 문법을 익힌다.

5장 Go 코드 읽기에서는 구조체, 포인터, 에러 처리를 SRS·스마트 덱 코드를 해부하며 배운다.
Go를 처음 접하는 독자가 이 저장소의 백엔드 코드를 읽을 수 있게 되는 것이 4장과 5장의 목표다.

6장 Go 테스트와 품질 도구에서는 테이블 주도 테스트, 그리고 gofmt와 go vet 같은 도구로 코드 품질을 기계적으로 지키는 방법을 다룬다.

7장 Gin에서는 라우터 조립·JSON 바인딩·미들웨어(CORS, JWT 인증)와 handlers→store 계층 분리를 살펴본다.

8장 HTML과 CSS에서는 브라우저가 화면을 그리는 재료인 HTML 문서와 CSS 스타일을 이 앱의 실제 레이아웃과 카드 화면으로 익힌다.
숨긴 체크박스와 CSS만으로 카드 뒤집기 애니메이션이 동작하는 원리도 여기서 본다.

9장 html/template으로 만드는 화면에서는 Go 표준 라이브러리의 템플릿 엔진으로 서버가 HTML을 만들어 내보내는 방법을 다룬다.
공통 레이아웃과 페이지, 조각으로 나뉜 이 저장소의 템플릿 구조와 정적 파일을 바이너리에 내장하는 embed를 함께 살펴본다.

10장 htmx에서는 페이지 전체를 새로 고치지 않고 화면 일부만 갈아 끼우는 동적 화면을 자바스크립트를 직접 쓰지 않고 만드는 방법을 배운다.
학습 세션의 채점 흐름과 사전 자동 채우기가 예제다.

11장 Claude Code에서는 AI 코딩 에이전트가 도구 호출 루프로 동작하는 원리와 CLAUDE.md·권한·세션 개념을 소개하고, 이 프로젝트에서 지시→생성→검증 흐름이 실제로 어떻게 굴러갔는지, 그 안에서 사람의 역할은 무엇인지 나눈다.

12장 서브에이전트와 훅에서는 에이전트가 만든 코드의 품질을 기계적으로 지키는 장치를 다룬다.
gofmt·go vet을 자동 실행하는 훅과, 별도 컨텍스트에서 종합 검증을 수행하고 요약만 반환하는 서브에이전트를 이 저장소의 실제 설정으로 해부한다.

13장 로컬 개발 환경에서는 저장소를 클론한 사람이 앱을 처음 띄우기까지의 길을 정리한다.
환경 변수 하나 없이 명령 하나로 서버를 띄우는 로컬 모드의 동작 원리, SQLite 파일에 쌓인 내 데이터를 직접 열어 보는 실습, `.vscode/` 설정으로 포맷과 검증 명령을 공유하고 함께 디버깅하는 구성을 다룬다.
여기까지 마치면 외부 서비스 없이 내 컴퓨터에서 완결되는 앱이 손에 남는다.

2부 "세상에 공개하고 오래 운영하기"는 14장부터 21장까지다.
1부에서 완성한 앱을 인터넷에 공개하고, 버전 관리와 개발·운영 환경 분리라는 두 축 위에서 오래 운영하는 체계를 세운다.

14장 Git에서는 코드의 변경 이력을 기록하고 언제든 과거로 되돌릴 수 있게 하는 버전 관리의 개념을 익힌다.
브랜치를 어떻게 쓸지는 도구가 정해 주는 것이 아니라 사람이 정하는 정책이라는 관점에서, 개발은 main 브랜치에서 하고 운영 배포는 release 브랜치로 내보내는 이 저장소의 정책도 함께 세운다.

15장 GitHub Actions에서는 저장소에 코드가 도착하는 순간 원격에서 도는 세 번째 품질 게이트를 만든다.
훅과 에디터 태스크가 로컬 게이트라면 CI는 깨끗한 기계에서 저장소만으로 앱이 검증되는지 확인하는 장치이고, 세 게이트가 같은 명령을 공유해야 하는 이유를 짚는다.

16장 Vercel에서는 화면과 API를 겸하는 Go 함수 하나를 배포하는 구성을 다룬다.
Go 서버리스 함수의 제약과 `vercel.json` 설정, 리전 고정의 의미를 상주 서버 대안과 비교하며 살펴보고, 14장의 브랜치 정책을 개발 확인용 배포와 운영 배포의 구분으로 잇는 설정도 여기서 다룬다.

17장 Supabase 인증에서는 1부의 고정 로컬 사용자가 공개된 앱에서 더는 통하지 않는 이유를 짚는 데서 출발한다.
왜 Supabase를 골랐고 그중 무엇을 일부러 쓰지 않았는지 정리한 뒤, Go 서버가 대행하는 OAuth 로그인부터 JWKS 서명 검증과 HttpOnly 쿠키 세션까지 무상태 인증의 전체 흐름을 따라간다.

18장 Supabase 데이터베이스에서는 1부의 SQLite가 환경 변수 하나로 운영 PostgreSQL로 교체되는 구조를 확인하고, 서버리스에서 커넥션이 폭증하는 문제와 트랜잭션 풀러, 그 대가로 pgx에 붙는 설정, 마이그레이션만 직접 연결로 붙어야 하는 이유, 리전 콜로케이션, 그리고 개발 DB와 운영 DB를 나누는 구성을 다룬다.

19장 PWA에서는 배포된 웹 앱을 홈 화면에 설치되는 앱으로 만든다.
매니페스트와 서비스 워커를 읽으며 무엇을 캐시하고 무엇을 캐시하지 않을지의 판단, 그리고 이 앱이 오프라인 학습과 푸시 알림을 포기한 이유를 살펴본다.

20장 무료 티어 운영에서는 Vercel과 Supabase의 한도를 한자리에 놓고 Echo Flip의 규모를 대입해 무엇이 가장 먼저 터지는지 계산한다.
답은 트래픽도 용량도 아니다.

21장 다음 단계에서는 이 책이 일부러 비워 둔 자리를 짚는다.
관측성, 보안, E2E 테스트처럼 앱이 자라면 필요해지는 주제를 "지금 왜 없어도 됐는지"와 "언제 필요해지는지"의 짝으로 정리하고, 무엇부터 손대면 좋을지 안내하며 책을 맺는다.

앞에서부터 순서대로 읽도록 구성했지만, 각 장은 독립적으로도 읽을 수 있다.
Go와 Gin이 익숙한 백엔드 경험자라면 4~7장을 건너뛰고 8장부터, 인프라가 급한 독자라면 2부부터 펼쳐도 된다.

## 이 책이 다루는 것과 다루지 않는 것

범위를 분명히 해 두자.

이 책은 Echo Flip의 모든 화면과 API를 나열하는 매뉴얼이 아니다.
카드 CRUD 핸들러 하나하나, 화면 템플릿 하나하나를 순서대로 해설하지 않는다.
그런 코드는 저장소를 직접 여는 편이 빠르다.

대신 이 책은 두 가지에 집중한다.

첫째, 기술 선택의 이유다.
왜 Go인가, 왜 서버 렌더링과 htmx인가는 1장에서, 왜 Vercel과 Supabase인가는 2부에서 대안과 트레이드오프로 설명한다.
독자의 프로젝트는 Echo Flip과 조건이 다를 것이므로, 결론보다 판단 근거를 전달하는 데 무게를 둔다.

둘째, 다른 앱에도 옮겨 쓸 수 있는 핵심 패턴이다.
로컬 서버와 서버리스 함수가 앱 조립 코드를 공유하는 구조, JWT를 JWKS로 검증하는 무상태 인증, 규칙을 JSON으로 저장했다가 쿼리로 변환하는 스마트 덱, 훅과 서브에이전트로 만드는 품질 게이트 같은 것들이다.
각 장의 코드 예제는 전부 이 저장소에서 실제로 동작하는 코드에서 발췌했으며, 본문에 파일 경로를 함께 적어 두었으니 언제든 원본을 열어 맥락을 확인할 수 있다.

반면 다음 주제는 다루지 않거나 최소한으로만 언급한다.
각 언어의 완결된 입문(공식 튜토리얼을 대체하지 않는다), 시각 디자인(8장의 CSS도 화면이 동작하는 원리에 집중한다), 네이티브 앱 개발, 그리고 대규모 트래픽을 전제한 확장 전략이다.
이 책의 무대는 어디까지나 "개인 개발자가 무료 인프라로 만들고 운영하는 앱"이다.

## 정리

이 장에서는 책 전체의 소재가 될 Echo Flip의 요구사항과 아키텍처를 조망했다.

Echo Flip은 양방향 카드와 덱·태그, 뒤집기와 자기 판정과 재도전 라운드로 이어지는 학습 흐름, SM-2 이진 변형의 간격 반복, 규칙 기반 스마트 덱, 학습 통계, 덱 공유, CSV 입출력, 사전 API 자동 채우기, TTS를 갖춘 암기 카드 앱이다.
개인 개발자가 Claude Code와 함께 만들었고, 완전 무료 인프라·최소 운영 부담·PWA·짧은 콜드스타트라는 비기능 요구사항이 기술 선택을 이끌었다.

그 결과가 세 조각짜리 아키텍처다.
화면 렌더링과 JSON API를 겸하는 Go/Gin 서버리스 함수 하나를 Vercel에 배포하고, PostgreSQL과 OAuth 인증은 Supabase가 맡는다.
화면은 서버가 html/template으로 렌더링하고 htmx가 부분 갱신하며, 직접 작성한 브라우저 자바스크립트는 브라우저 전용 기능을 감싼 파일 하나뿐이다.
데이터 접근은 Go 서버로 일원화하고, RLS는 정책 없이 켜서 우회 경로를 차단했다.
이 모습은 2부에서 완성되는 운영의 모습이고, 1부에서는 같은 코드가 SQLite와 함께 내 컴퓨터 안에서만 돈다.

다음 장에서는 이 아키텍처를 이룬 기술 선택의 이유를 살펴보겠다.
왜 SQLite로 시작해 PostgreSQL로 옮기는가, 왜 Go인가, 왜 서버 렌더링 HTML과 htmx인가를 대안과 트레이드오프로 정리한 뒤, 2장부터 각 기술을 실제 코드로 익힌다.
